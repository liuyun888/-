## 第一部分:核心业务模块

## 一、用户体系模块

用户体系是C端商城的基础设施,提供用户注册、登录、认证、信息管理等核心功能。本模块包含8个主要功能点,是后续所有业务的前提。

### 1.1 注册登录

**开发工时**:前端1天 + 后端1天

**功能概述**:手机号+短信验证码方式完成注册登录,支持微信快捷登录,JWT令牌认证,完善的防刷机制

**核心业务逻辑**:

注册流程:
1. 用户输入11位手机号→正则校验(^1[3-9]\d{9}$)
2. 检查手机号是否已注册(查询tb_user表)
3. 点击"获取验证码"→先弹出图形验证(滑动/点选)
4. 图形验证通过→调用短信服务发送6位数字验证码
5. 验证码存储Redis,key=`sms:code:{mobile}:{scene}`,TTL=300秒
6. 用户输入验证码→后端校验
7. 验证成功→创建用户记录,分配user_id
8. 签发JWT令牌(AccessToken 7天 + RefreshToken 30天)
9. 返回token和用户信息

登录流程:
1. 已注册手机号输入→获取验证短信验证码
2. 验证通过→签发JWT令牌
3. 更新last_login_time、last_login_ip
4. 记录登录日志(tb_login_log表)
5. 异地登录检测→发送安全提醒短信

防刷机制(关键):
1. 手机号限制:同一手机60秒内只能发1次验证码(Redis计数)
2. IP限制:同一IP 1小时内最多发10次(Redis key=`sms:limit:ip:{ip}`)
3. 设备限制:同一设备ID 1小时内最多发5次
4. 错误次数:验证码连续错5次,锁定手机号30分钟
5. 图形验证:使用腾讯云验证码或极验,防机器人
6. 黑名单:异常行为自动拉黑,需人工申诉解除

异常处理:
- 短信发送失败:重试3次,失败提示"系统繁忙"
- 验证码过期:提示"验证码已过期,请重新获取"
- 验证码错误:提示剩余尝试次数
- 账号冻结:提示冻结原因+申诉入口
- 异地登录:短信通知+提供一键冻结功能

**数据库设计**:

tb_user(用户表):
- 主键:user_id BIGINT
- 核心字段:mobile VARCHAR(11) UNIQUE, nickname, avatar, gender, birthday
- 地区字段:province, city, district  
- 会员字段:member_level TINYINT, points INT, balance BIGINT
- 状态字段:status TINYINT (0正常1冻结2注销)
- 时间字段:register_time, last_login_time, last_login_ip
- 索引:UNIQUE(mobile), INDEX(status, deleted)

tb_sms_log(短信日志表):
- 主键:id BIGINT
- 核心字段:mobile, code, type(1注册2登录3找回密码)
- 验证字段:verify_status(0未验证1已验证2已过期), verify_time
- 风控字段:ip_address, device_id, send_time, expire_time
- 索引:INDEX(mobile, type, send_time), INDEX(ip_address, send_time)

**接口设计**:
- POST /api/user/sendSmsCode  发送短信验证码,参数:mobile、scene、captchaToken
- POST /api/user/login  登录/注册,参数:mobile、smsCode  
- POST /api/user/refreshToken  刷新令牌,参数:refreshToken

**技术实现要点**:

1. 短信服务:使用阿里云/腾讯云SDK,配置签名和模板,异步发送,失败重试3次
2. Redis缓存:验证码、限流计数器、错误次数均存Redis,设置合理TTL
3. JWT令牌:Payload包含userId、mobile、timestamp、deviceId,HS256签名
4. 安全加固:验证码不明文日志(仅MD5),登录失败详细记录但对用户模糊提示,IP地理位置识别
5. 性能优化:短信异步发送,验证码校验用Lua脚本保证原子性,用户信息缓存减少DB查询

**注意事项**:
- 短信必须包含退订方式"回复TD退订"
- 手机号脱敏138****8000
- 支持一键登录(运营商网关)
- 注册协议和隐私政策需勾选同意

### 1.2 微信授权

**开发工时**:前端1天 + 后端1天

**功能概述**:微信开放平台OAuth2.0授权登录,自动获取用户信息,首次登录引导绑定手机号

**核心业务逻辑**:

微信授权流程:
1. 点击"微信登录"→判断环境(微信内JSAPI / APP / H5)
2. 构建授权URL→跳转微信授权页(scope=snsapi_userinfo)
3. 用户确认授权→微信回调返回code(5分钟有效)
4. 后端用code换access_token和openid→调用微信API
5. 用access_token获取用户信息(昵称、头像、性别)
6. 查询openid是否已绑定:
   - 已绑定:直接登录签发JWT
   - 未绑定:创建新用户,保存微信信息,签发JWT
7. 返回登录结果

绑定手机号:
1. 微信登录成功检查mobile字段是否为空
2. 为空弹窗:"绑定手机号享受更多服务"
3. 用户选"立即绑定":输入手机号→短信验证→更新mobile字段
4. 选"暂不绑定":可正常使用但部分功能受限

解绑微信:
1. 个人中心"账号安全"→解绑微信
2. 检查:必须已绑定手机号才能解绑
3. 二次确认弹窗→解绑

unionid机制:
- 同一开放平台下所有应用共享unionid
- 优先用unionid关联用户,实现跨应用互通
- 获取unionid需应用通过微信认证

**数据库设计**:

tb_user增加字段:
- wechat_openid VARCHAR(64) INDEX, wechat_unionid VARCHAR(64) INDEX
- wechat_nickname VARCHAR(100), wechat_avatar VARCHAR(255)
- bind_wechat_time DATETIME

tb_wechat_auth_log(微信授权日志):
- id, user_id, openid, unionid, auth_code, access_token
- auth_type TINYINT(1JSAPI 2APP 3H5), auth_result TINYINT(0失败1成功)
- fail_reason VARCHAR(255), auth_time, expire_time

**接口设计**:
- GET /api/wechat/getAuthUrl  生成授权URL,参数:redirectUrl、state
- POST /api/wechat/authCallback  授权回调,参数:code、state
- POST /api/user/bindMobile  绑定手机,参数:mobile、smsCode

**技术实现要点**:

1. access_token管理:Redis缓存,key=`wechat:access_token:{openid}`,TTL=7200,过期前5分钟刷新
2. state防CSRF:用UUID生成,Redis存储10分钟,回调时校验匹配
3. 异常处理:授权失败详细记录,微信API失败重试3次,友好错误提示
4. 头像处理:微信头像URL可能失效,建议下载到OSS永久存储

**注意事项**:
- 回调URL必须HTTPS
- 测试用微信测试号,生产需审核通过
- ruoyi-vue-pro自带社交登录模块可复用

### 1.3 实名认证

**开发工时**:前端1天 + 后端1天

**功能概述**:身份证OCR识别+公安部验证+人脸活体检测,满足保险监管要求

**核心业务逻辑**:

身份证OCR流程:
1. 上传身份证正面照→图片质量检测(清晰度、光线、角度、边缘)
2. 不合格提示重拍→合格调用OCR SDK
3. 识别正面:姓名、身份证号、性别、民族、出生日期、地址
4. 上传反面照→识别签发机关、有效期
5. 自动填充表单,用户可修改
6. 提交认证

实名验证流程:
1. 调用第三方实名认证(公安部/阿里云实人认证)
2. 验证姓名+身份证号是否一致
3. 通过:更新real_name_status=2,保存照片(加密),记录时间,发放奖励
4. 失败:更新status=3,记录原因,允许重试(每日3次)

人脸活体检测(可选):
1. 身份证验证通过后→调用活体检测SDK
2. 检测动作:眨眼、张嘴、摇头
3. 防止照片/视频攻击
4. 通过后完成高级认证

认证等级:
- 初级:仅OCR识别
- 中级:OCR+公安部验证  
- 高级:OCR+公安部+人脸识别

**数据库设计**:

tb_user增加字段:
- real_name VARCHAR(50), id_card_no VARCHAR(18) 加密
- id_card_front_url, id_card_back_url(照片加密存储)
- real_name_status TINYINT(0未认证1认证中2已认证3失败)
- real_name_level TINYINT(1初级2中级3高级)
- real_name_time DATETIME, real_name_fail_reason VARCHAR(255)

tb_real_name_record(认证记录):
- id, user_id, real_name, id_card_no加密
- auth_type TINYINT(1OCR 2公安部 3人脸)
- auth_result TINYINT, fail_reason, ocr_confidence
- apply_time, complete_time

**接口设计**:
- POST /api/user/idCardOcr  身份证OCR,参数:正反面图片
- POST /api/user/submitRealName  提交实名认证
- GET /api/user/getRealNameStatus  查询认证状态

**技术实现要点**:

1. OCR服务:推荐阿里云(准确率>98%),备选腾讯云/百度
2. 图片检测:分辨率≥600x400,边缘完整,非翻拍复印件,未过期
3. 身份证校验:18位格式+校验码算法,提取出生日期性别地区码
4. 加密存储:AES-256-CBC加密身份证号,OSS私有bucket存照片,签名URL访问
5. 活体检测:炫彩光线技术,防照片视频攻击

**注意事项**:
- 身份证信息高度敏感,严格权限控制
- OCR错误率2-5%需人工复核
- 未成年需监护人授权
- 信息变更需重新认证

### 1.4 个人信息管理

**开发工时**:前端1天 + 后端0.5天

**功能概述**:管理基本资料、头像、密码、安全设置、登录设备、注销账号

**核心业务逻辑**:

资料编辑:
- 可编辑:昵称(2-20字,30天限改1次)、性别、生日、地区(省市区)、签名(100字内)
- 不可编辑:手机号(通过"更换手机号"功能)、真实姓名(实名后锁定)、身份证号(实名后锁定)

头像上传:
- 格式:JPG/PNG/GIF,≤2MB,建议200x200-800x800
- 流程:前端裁剪正方形→压缩→上传OSS→生成缩略图(50/100/200px)→更新avatar
- 违规检测:调用内容安全API,拒绝涉黄涉政广告

密码管理:
- 设置密码:8-20位,含字母数字特殊字符,BCrypt加密
- 修改密码:验证旧密码或短信,成功后强制重新登录
- 忘记密码:手机号+验证码重置,清除所有登录设备

安全设置:
- 登录设备:显示设备型号/登录时间/地点,可踢出其他设备
- 操作日志:登录/修改资料/敏感操作,保留3个月
- 注销账号:短信+密码双重验证,检查未完成订单/余额,数据保留180天可恢复

**数据库设计**:

tb_user增加:password(BCrypt), last_pwd_change_time, last_nickname_change_time

tb_user_device(登录设备):
- id, user_id, device_id唯一标识, device_type(1PC 2H5 3iOS 4Android)
- device_model, os_version, app_version, login_ip, login_location
- login_time, last_active_time, token, status(0正常1已踢出)
- INDEX:(user_id, status)

tb_user_operation_log(操作日志):
- id, user_id, operation_type, operation_desc, ip_address, device_info, operation_time

**接口设计**:
- GET /api/user/profile  获取资料
- PUT /api/user/profile  更新资料
- POST /api/user/uploadAvatar  上传头像
- POST /api/user/changePassword  修改密码
- GET /api/user/devices  设备列表
- POST /api/user/kickDevice  踢出设备
- POST /api/user/cancelAccount  注销账号

**技术实现要点**:

1. 头像OSS直传:STS临时凭证,前端直传,回调通知服务端
2. 密码:BCrypt自动加盐,不可逆,连续错5次锁定30分钟
3. 设备管理:deviceId+userId生成token,踢出时token失效,90天未活跃自动清理
4. 注销:逻辑删除deleted=1,敏感数据脱敏,关联数据保留,180天后物理删除

**注意事项**:
- 头像违规及时处理
- 密码强度可配置
- 注销需符合监管
- 日志定期归档

### 1.5 收货地址管理

**开发工时**:前端1天 + 后端0.5天

**功能概述**:管理收货地址用于保单邮寄、理赔材料寄送

**核心业务逻辑**:

添加地址:
- 必填:收货人(2-20字)、手机号(11位)、地区(省市区三级)、详细地址(5-100字)
- 可选:标签(家/公司/学校,可自定义)、是否默认
- 限制:每人最多10个地址

修改地址:全字段可改,保存历史记录审计

删除地址:软删除,检查pending订单是否使用,使用中不可删

默认地址:
- 只能有1个默认
- 设新默认时取消其他(事务保证)
- 下单自动选中默认

智能解析:
- 粘贴"姓名+手机+地址"文本
- 正则识别姓名手机,NLP识别省市区
- 自动填充表单

**数据库设计**:

tb_user_address(收货地址):
- id, user_id, receiver_name, receiver_mobile
- province, city, district, detail_address, address_label
- is_default TINYINT(0否1是), create_time, update_time, deleted
- INDEX:(user_id, deleted), INDEX:(user_id, is_default)

**接口设计**:
- GET /api/user/address/list  地址列表
- POST /api/user/address/add  添加
- PUT /api/user/address/update/{id}  修改
- DELETE /api/user/address/delete/{id}  删除
- POST /api/user/address/setDefault/{id}  设为默认
- POST /api/user/address/parseText  智能解析

**技术实现要点**:

1. 唯一默认:设新默认时先UPDATE旧的is_default=0,再UPDATE新的=1,事务保证
2. 智能解析:正则识别姓名手机,NLP识别地区,匹配地址库
3. 地址校验:省市区匹配标准库,手机号格式验证,详细地址过滤特殊字符
4. 缓存:地址列表缓存10分钟,key=`user:address:{userId}`,增删改刷新

**注意事项**:
- 地址脱敏显示
- 支持导入淘宝京东地址
- 地区数据定期更新

### 1.6 消息通知管理

**开发工时**:前端1天 + 后端1天

**功能概述**:站内信、短信、推送、微信服务号等多渠道消息通知

**核心业务逻辑**:

消息类型:
- 系统通知:平台公告、产品上新、活动
- 订单通知:下单成功、支付成功、出单成功、保单到期
- 理赔通知:进度更新、结果通知
- 营销通知:优惠券、限时活动、新品

通知渠道:
- 站内信:所有通知必发,红点提示,保留3个月
- 短信:重要通知(支付成功、理赔结果),可设置开关
- 推送:APP推送(极光/友盟)、H5浏览器推送,可设置开关
- 微信:服务号模板消息,需关注服务号

消息中心:
- 显示所有消息,时间倒序

2. 防超卖:Redis库存为准,Lua脚本保证原子性,定时任务同步Redis到DB

**注意事项**:

- 库存双写(Redis+DB)定时同步
- 订单超时未支付自动取消释放库存
- 价格折扣合规不能虚标原价

**开发工时**:前端0.5天 + 后端0.5天

**功能概述**:首页展示保险相关资讯,提升用户粘性,提供知识科普和行业动态,增强平台专业性

**核心业务逻辑**:

资讯分类:
- 保险知识:如何选保险、理赔技巧、条款解读
- 行业动态:监管政策、保司新闻、趋势分析
- 案例分享:理赔案例、避坑指南、用户故事
- 产品测评:产品对比、性价比评测、专家点评

内容来源:
- 自营:平台编辑撰写,邀请专家供稿
- 第三方:合作媒体授权转载,聚合优质内容
- UGC:用户投稿,审核后发布

展示策略:
- 首页展示3-5篇精选资讯
- 每篇:封面图(16:9)、标题(限30字)、摘要(限50字)、阅读量、发布时间(相对时间"2小时前")
- 点击跳转详情页

推荐算法:
- 时效性:优先最新发布
- 热度:根据阅读量、点赞量、评论量计算
- 相关性:根据用户浏览历史推荐相关
- 多样性:避免同类型连续出现

**数据库设计**:

tb_article(资讯表):
- id, title标题, summary摘要, cover_image封面
- content正文(富文本HTML), category_id分类
- author作者, source来源, source_url原文链接
- tags标签(逗号分隔), view_count阅读量, like_count点赞, comment_count评论, share_count分享
- is_recommend是否推荐, is_top是否置顶
- status状态(0草稿1已发布2已下架), publish_time发布时间
- create_by, create_time, update_time, deleted
- INDEX:(category_id, status, deleted), INDEX:(is_recommend, is_top, publish_time)

**接口设计**:
- GET /api/home/articleList  首页资讯列表
- GET /api/article/detail/{id}  资讯详情
- POST /api/article/like  点赞,参数:articleId

**技术实现要点**:

1. 内容缓存:列表和详情Redis缓存,热门资讯缓存1小时,普通10分钟
2. 阅读量统计:HyperLogLog统计UV,计数器统计PV,每小时同步DB
3. 图片处理:封面自动裁剪16:9,生成多尺寸缩略图,CDN加速
4. 富文本处理:内容图片懒加载,外部链接rel="nofollow",XSS防护过滤危险标签
5. SEO优化:详情页支持SSR,生成静态HTML,设meta标签和OpenGraph

**注意事项**:
- 转载需授权标注来源
- 内容审核避免虚假宣传
- 图片版权注意
- 支持用户举报不当内容

### 2.6 智能推荐

**开发工时**:前端1天 + 后端2天

**功能概述**:基于用户画像和行为数据,智能推荐个性化保险产品,提升转化率和GMV

**核心业务逻辑**:

用户画像构建:
- 基础属性:年龄、性别、地域、职业、收入、婚姻、子女
- 行为数据:浏览记录(产品、资讯)、搜索关键词、加购记录、购买历史
- 标签体系:人生阶段(单身、新婚、育儿、退休)、风险偏好(保守、平衡、进取)、保障需求(健康、养老、教育、财富)、购买能力(低、中、高)

推荐策略:
- 协同过滤:User-Based CF,找相似用户推荐他们购买的产品,适合冷启动
- 内容推荐:Item-Based CF,推荐与已购产品相似的,适合有购买历史
- 关联规则:购买A经常购买B,如购买重疾→推荐医疗
- 实时推荐:根据当前浏览推荐同类或互补

推荐展示:
- "猜你喜欢"模块:首页底部信息流,滑动加载更多,每次加载6-10个
- 产品卡片:与热销样式一致,添加"推荐理由"标签,如"基于您的浏览记录"
- 多样性控制:避免同质化,保证不同类型产品混合

冷启动处理:
- 新用户无行为:根据注册时基础信息推荐,推荐热门产品,推荐新人专享
- 新产品无销售:基于产品属性匹配,手动设置推荐权重,逐步积累优化

**数据库设计**:

tb_user_behavior(用户行为):
- id, user_id, behavior_type行为类型(1浏览2搜索3加购4购买)
- target_type目标类型(1产品2资讯3活动), target_id目标ID
- behavior_time行为时间, duration停留时长(秒)
- device_type设备类型, ip_address, create_time
- INDEX:(user_id, behavior_type, behavior_time), INDEX:(target_type, target_id)

tb_recommend_cache(推荐结果缓存):
- id, user_id, recommend_type推荐类型(1首页推荐2详情页推荐)
- product_ids推荐产品ID列表(JSON), scores推荐分数列表(JSON)
- algorithm使用的算法, expire_time过期时间, create_time
- UNIQUE INDEX:(user_id, recommend_type)

**接口设计**:
- GET /api/recommend/products  获取推荐产品,参数:page、pageSize、scene(1首页2详情页)
- POST /api/recommend/feedback  推荐反馈,参数:productId、feedbackType(1点击2加购3购买4不感兴趣)

**技术实现要点**:

1. 推荐引擎架构:离线计算(每天凌晨Spark计算),近线计算(每小时更新热门),实时计算(根据当前行为实时推荐)
2. 算法模型:ALS(交替最小二乘法)协同过滤,ItemCF(基于物品),深度学习(Wide&Deep),模型定期训练更新
3. 缓存:推荐结果Redis缓存1小时,key=`recommend:user:{userId}:{type}`,用户行为变化主动刷新
4. 效果评估:点击率CTR、转化率CVR、客单价,A/B测试对比
5. 多样性控制:使用MMR(最大边际相关)算法,保证推荐多样性,避免过于相似

**注意事项**:
- 推荐符合监管不能诱导过度消费
- 用户行为数据脱敏处理
- 推荐结果可解释提供推荐理由
- 支持用户关闭个性化推荐
