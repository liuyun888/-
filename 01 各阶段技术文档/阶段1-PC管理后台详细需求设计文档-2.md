## 1. 部门管理业务逻辑补充

### 1.1 部门新增业务逻辑

部门新增时需要处理的核心逻辑:

**1. 所属机构校验**
- 必须先选择所属机构(org_id),一个部门必须归属于某个具体机构
- 校验org_id对应的机构是否存在且状态为启用
- 如果机构被停用,不允许在其下新增部门

**2. 部门代码唯一性校验**
- 部门代码dept_code在全系统范围内必须唯一(跨机构)
- 查询数据库中是否已存在相同的dept_code(排除已逻辑删除的记录)
- 如果存在,抛出业务异常"部门代码已存在"
- 建议dept_code采用"机构代码+部门编号"的方式生成,如"100001-01"表示100001机构的01部门

**3. 上级部门有效性校验**
- 如果parent_id不为0,需要验证父部门是否存在且状态为启用
- 父部门必须与当前部门属于同一机构(org_id相同)
- 不允许跨机构设置上级部门,如果父部门的org_id与当前部门的org_id不一致,抛出异常"上级部门必须与当前部门属于同一机构"
- 如果父部门被停用,不允许在其下新增子部门

**4. 祖级列表自动构建**
- 根据parent_id自动计算ancestors字段
- 如果是顶级部门(parent_id=0),则ancestors="0"
- 否则ancestors=父部门的ancestors+","+父部门id
- 这个字段用于高效查询部门的所有上级,避免递归查询

**5. 部门层级限制**
- 通过ancestors字段中的逗号数量判断层级深度
- 一般限制部门层级不超过4级,防止组织架构过于复杂
- 计算公式:层级 = ancestors中逗号数量
- 如果超过限制,抛出异常"部门层级不能超过4级"

**6. 负责人冗余存储**
- 虽然有leader_id字段,但同时存储leader_name字段
- 在新增时,如果选择了负责人,需要查询人员表获取姓名并填充leader_name
- 这样做可以避免每次查询部门列表时都要关联人员表,提升查询效率

**7. 负责人有效性校验**
- 如果指定了部门负责人,需要校验该人员是否存在
- 负责人必须是该机构下的人员(查询人员表的org_id字段)
- 负责人的状态必须是启用状态
- 负责人不能是已离职的人员

**8. 默认值设置**
- status默认为1(启用状态)
- sort默认为0
- 创建时间和更新时间自动填充当前时间
- 创建人和更新人从当前登录上下文中获取

### 1.2 部门修改业务逻辑

部门修改时的特殊处理逻辑:

**1. 禁止修改所属机构**
- 部门的org_id一旦创建不允许修改
- 如果需要将部门转移到其他机构,建议新建部门并迁移数据,然后停用或删除原部门
- 在修改接口中,如果发现org_id与数据库中的值不一致,抛出异常"不允许修改部门所属机构"

**2. 禁止修改部门代码**
- 部门代码dept_code一旦创建不允许修改
- 因为部门代码可能被其他业务数据引用(如岗位、人员等)
- 如果确实需要修改,提示用户新建部门并迁移数据

**3. 上级部门变更处理**
- 如果修改了parent_id,需要进行一系列校验和数据更新
- 校验1:不能将部门的上级设置为自己,即parent_id不能等于当前部门id
- 校验2:不能将部门的上级设置为自己的子孙部门,即新的parent_id不能在当前部门的子孙部门列表中
  - 具体做法:查询所有ancestors包含当前部门id的部门,如果新的parent_id在这些部门的id列表中,则不允许修改
- 校验3:新的父部门必须与当前部门属于同一机构
- 校验4:新的父部门必须是启用状态

- 如果校验通过,需要重新计算当前部门及其所有子孙部门的ancestors字段
- 具体步骤:
  1. 计算当前部门的新ancestors值(根据新的parent_id)
  2. 查询出所有子孙部门(ancestors like '%,当前部门id%' 或 ancestors like '%,当前部门id')
  3. 遍历这些子孙部门,将它们ancestors中的旧前缀替换为新前缀
     - 旧前缀 = 当前部门的旧ancestors + "," + 当前部门id
     - 新前缀 = 当前部门的新ancestors + "," + 当前部门id
  4. 批量更新到数据库
  5. 整个操作必须在事务中完成,任何步骤失败都要回滚

**4. 部门状态变更联动**
- 如果将部门状态从启用改为停用,需要进行以下联动操作:
  1. 将该部门下的所有子部门状态改为停用(递归)
  2. 将该部门下的所有岗位状态改为停用
  3. 将该部门下的所有人员的在职状态改为停用(注意不是删除账号,只是停用)
  4. 发送系统通知给部门负责人和相关人员

- 如果将部门状态从停用改为启用,需要进行以下校验:
  1. 检查上级部门是否启用,如果上级被停用,不允许启用当前部门
  2. 检查所属机构是否启用,如果机构被停用,不允许启用部门
  3. 仅启用当前部门,不自动启用子部门(需要手动逐个启用)
  4. 不自动启用岗位和人员(由管理员手动处理)

**5. 负责人变更处理**
- 如果修改了leader_id,需要同步更新leader_name字段
- 查询新负责人的姓名并填充
- 记录变更日志,包括:旧负责人、新负责人、变更时间、操作人

**6. 联系方式变更**
- 如果修改了phone或email,需要进行格式校验
- phone必须是11位手机号
- email必须符合邮箱格式

### 1.3 部门删除业务逻辑

部门删除是一个复杂的业务操作,需要进行多项前置校验:

**1. 子部门检查**
- 查询是否存在以当前部门为parent_id的子部门
- SQL: SELECT COUNT(*) FROM biz_department WHERE parent_id = #{deptId} AND deleted = 0
- 如果count > 0,不允许删除,提示"存在下级部门,请先删除下级"

**2. 岗位关联检查**
- 查询biz_post表中dept_id等于当前部门id的记录数
- SQL: SELECT COUNT(*) FROM biz_post WHERE dept_id = #{deptId} AND deleted = 0
- 如果count > 0,不允许删除,提示"部门下存在{count}个岗位,不允许删除"

**3. 人员关联检查**
- 查询biz_agent表中dept_id等于当前部门id的记录数
- SQL: SELECT COUNT(*) FROM biz_agent WHERE dept_id = #{deptId} AND deleted = 0
- 如果count > 0,不允许删除,提示"部门下存在{count}个业务员,不允许删除"

- 查询biz_staff表中dept_id等于当前部门id的记录数
- SQL: SELECT COUNT(*) FROM biz_staff WHERE dept_id = #{deptId} AND deleted = 0
- 如果count > 0,不允许删除,提示"部门下存在{count}个内勤人员,不允许删除"

**4. 逻辑删除而非物理删除**
- 所有校验通过后,执行逻辑删除(将deleted字段设为1)
- 不是物理删除记录,保留数据用于审计和追溯
- 更新update_time和updater字段

**5. 删除操作记录**
- 在操作日志表中记录删除操作
- 记录内容包括:被删除部门的名称、代码、所属机构、操作人、操作时间

### 1.4 部门树查询业务逻辑

查询部门树时的优化处理:

**1. 按机构查询**
- 部门树的查询必须指定机构org_id,不能跨机构查询
- 一次性查询出该机构下所有启用状态的部门
- SQL: SELECT * FROM biz_department WHERE org_id = #{orgId} AND status = 1 AND deleted = 0 ORDER BY sort ASC

**2. 内存构建树形结构**
- 查询出所有部门后,在内存中构建树形结构,不要使用递归SQL查询
- 构建算法:
  1. 创建一个Map<Long, DepartmentNode>,key是部门id,value是部门节点
  2. 遍历部门列表,将每个部门放入Map中
  3. 再次遍历部门列表,根据parent_id找到父节点,将当前节点添加到父节点的children列表中
  4. 找出所有parent_id=0的节点作为根节点
  5. 返回根节点列表

**3. 状态过滤**
- 默认只返回status=1的部门
- 可以通过参数showDisabled控制是否包含停用的部门
- 如果showDisabled=true,则不过滤status字段

**4. 权限过滤**
- 根据当前登录用户的数据权限,只返回用户有权限查看的部门
- 如果用户是超级管理员,返回所有部门
- 如果用户是某个部门的管理员,只返回该部门及其子部门
- 数据权限的判断逻辑:
  1. 查询用户的角色
  2. 查询角色关联的数据权限规则
  3. 根据数据权限规则过滤部门列表

**5. 懒加载支持**
- 对于大型组织(部门数超过500),支持懒加载模式
- 初次只加载到第二层,点击展开时再加载子节点
- 懒加载接口:/admin-api/biz/department/lazy-tree
- 参数:parentId(父部门ID,0表示查询根节点)
- 返回:该父部门的直接子部门列表

**6. 缓存优化**
- 部门树数据变化不频繁,可以使用Redis缓存
- 缓存key: dept:tree:{orgId}
- 缓存过期时间:30分钟
- 当部门数据发生变更(新增、修改、删除)时,主动清除该机构的缓存
- 清除方法:redisTemplate.delete("dept:tree:" + orgId)

### 1.5 部门合并业务逻辑

部门合并是一个复杂的业务场景,虽然基础功能中未包含,但实际业务中可能需要:

**1. 合并操作流程**
- 选择源部门(被合并方)和目标部门(合并到的部门)
- 校验1:两个部门必须属于同一机构(org_id相同)
- 校验2:两个部门不能有上下级关系
  - 目标部门的id不能在源部门的ancestors中
  - 源部门的id不能在目标部门的ancestors中
- 校验3:源部门和目标部门都必须是启用状态

**2. 数据迁移步骤**
- 步骤1:将源部门下的所有岗位的dept_id改为目标部门id
  - UPDATE biz_post SET dept_id = #{targetDeptId}, updater = #{operator}, update_time = NOW() WHERE dept_id = #{sourceDeptId} AND deleted = 0
  
- 步骤2:将源部门下的所有业务员的dept_id改为目标部门id
  - UPDATE biz_agent SET dept_id = #{targetDeptId}, updater = #{operator}, update_time = NOW() WHERE dept_id = #{sourceDeptId} AND deleted = 0
  
- 步骤3:将源部门下的所有内勤人员的dept_id改为目标部门id
  - UPDATE biz_staff SET dept_id = #{targetDeptId}, updater = #{operator}, update_time = NOW() WHERE dept_id = #{sourceDeptId} AND deleted = 0
  
- 步骤4:将源部门的所有子部门的parent_id改为目标部门id
  - UPDATE biz_department SET parent_id = #{targetDeptId}, updater = #{operator}, update_time = NOW() WHERE parent_id = #{sourceDeptId} AND deleted = 0
  - 同时需要重新计算这些子部门的ancestors字段
  
- 步骤5:将源部门状态改为停用或执行逻辑删除
  - UPDATE biz_department SET deleted = 1, updater = #{operator}, update_time = NOW() WHERE id = #{sourceDeptId}

**3. 事务保证**
- 整个合并操作必须在一个数据库事务中完成
- 任何步骤失败都要回滚所有操作
- 使用@Transactional注解,并设置rollbackFor = Exception.class

**4. 操作日志记录**
- 生成部门变更日志
- 记录内容:源部门名称、目标部门名称、迁移的岗位数、迁移的人员数、操作人、操作时间
- 存入系统操作日志表

**5. 通知相关人员**
- 合并完成后,向涉及的人员发送系统通知
- 通知内容:"您所在的部门【源部门名称】已合并到【目标部门名称】,请知悉"
- 通知方式:站内信、短信(可选)

## 2. 岗位管理业务逻辑补充

### 2.1 岗位新增业务逻辑

岗位新增时需要处理的核心逻辑:

**1. 所属部门校验**
- 必须先选择所属部门(dept_id),一个岗位必须归属于某个具体部门
- 校验dept_id对应的部门是否存在且状态为启用
- 如果部门被停用,不允许在其下新增岗位

**2. 岗位代码唯一性校验**
- 岗位代码post_code在全系统范围内必须唯一(跨部门、跨机构)
- 查询数据库中是否已存在相同的post_code(排除已逻辑删除的记录)
- SQL: SELECT COUNT(*) FROM biz_post WHERE post_code = #{postCode} AND deleted = 0
- 如果count > 0,抛出业务异常"岗位代码已存在"
- 建议post_code采用"部门代码+岗位编号"的方式生成,如"100001-01-P01"表示100001机构01部门的P01岗位

**3. 岗位名称校验**
- 岗位名称post_name在同一部门内必须唯一
- 查询数据库中是否已存在相同部门下的相同岗位名称
- SQL: SELECT COUNT(*) FROM biz_post WHERE dept_id = #{deptId} AND post_name = #{postName} AND deleted = 0
- 如果count > 0,抛出业务异常"该部门下已存在相同名称的岗位"

**4. 岗位类型校验**
- post_type必须是有效的枚举值:1-管理岗 2-业务岗 3-支持岗
- 不同类型的岗位可能有不同的权限和职责
- 管理岗:部门经理、总监等管理人员
- 业务岗:业务员、客户经理等直接产生业绩的岗位
- 支持岗:财务、IT、行政等支持性岗位

**5. 岗位职责字段**
- post_duty字段用于描述岗位职责
- 非必填,但建议填写,便于人员了解岗位要求
- 最大长度1000字符

**6. 岗位等级校验**
- post_level字段表示岗位等级,如1-初级 2-中级 3-高级 4-资深
- 非必填,默认为1(初级)
- 岗位等级可能影响薪资、权限等

**7. 默认值设置**
- status默认为1(启用状态)
- sort默认为0
- 创建时间和更新时间自动填充当前时间
- 创建人和更新人从当前登录上下文中获取

**8. 关联机构信息获取**
- 虽然岗位直接关联部门,但也需要知道所属机构
- 通过dept_id关联查询部门表的org_id字段
- 冗余存储org_id到岗位表中,方便查询和权限控制
- 在新增岗位时,自动填充org_id字段

### 2.2 岗位修改业务逻辑

岗位修改时的特殊处理逻辑:

**1. 禁止修改所属部门**
- 岗位的dept_id一旦创建不允许修改
- 如果需要将岗位转移到其他部门,建议新建岗位并迁移人员,然后停用或删除原岗位
- 在修改接口中,如果发现dept_id与数据库中的值不一致,抛出异常"不允许修改岗位所属部门"

**2. 禁止修改岗位代码**
- 岗位代码post_code一旦创建不允许修改
- 因为岗位代码可能被其他业务数据引用(如人员、权限等)
- 如果确实需要修改,提示用户新建岗位并迁移数据

**3. 岗位名称唯一性校验**
- 如果修改了post_name,需要校验新名称在同一部门下是否唯一
- SQL: SELECT COUNT(*) FROM biz_post WHERE dept_id = #{deptId} AND post_name = #{postName} AND id != #{postId} AND deleted = 0
- 如果count > 0,抛出业务异常"该部门下已存在相同名称的岗位"

**4. 岗位状态变更联动**
- 如果将岗位状态从启用改为停用,需要进行以下联动操作:
  1. 查询该岗位下的所有在职人员
  2. 不强制停用人员账号,但标记人员的岗位状态为停用
  3. 发送系统通知给这些人员:"您的岗位【岗位名称】已停用,请联系管理员处理"

- 如果将岗位状态从停用改为启用,需要进行以下校验:
  1. 检查所属部门是否启用,如果部门被停用,不允许启用岗位
  2. 检查所属机构是否启用,如果机构被停用,不允许启用岗位
  3. 启用岗位后,不自动变更人员的岗位状态,由管理员手动处理

**5. 岗位类型变更影响**
- 如果修改了post_type,可能影响该岗位人员的权限和职责
- 建议在变更前与相关人员沟通确认
- 记录变更日志,包括:旧类型、新类型、变更时间、操作人

**6. 岗位职责更新**
- 如果修改了post_duty,应向该岗位的所有人员发送通知
- 通知内容:"您的岗位职责已更新,请及时查看"

### 2.3 岗位删除业务逻辑

岗位删除前需要进行多项校验:

**1. 人员关联检查**
- 查询biz_agent表中post_id等于当前岗位id的业务员记录数
- SQL: SELECT COUNT(*) FROM biz_agent WHERE post_id = #{postId} AND deleted = 0
- 如果count > 0,不允许删除,提示"岗位下存在{count}个业务员,不允许删除"

- 查询biz_staff表中post_id等于当前岗位id的内勤人员记录数
- SQL: SELECT COUNT(*) FROM biz_staff WHERE post_id = #{postId} AND deleted = 0
- 如果count > 0,不允许删除,提示"岗位下存在{count}个内勤人员,不允许删除"

**2. 权限配置检查**
- 如果岗位与角色、权限有绑定关系,也不允许删除
- 查询系统角色表中是否有角色关联了该岗位
- 如果存在关联,提示"岗位已关联角色,请先解除关联"

**3. 逻辑删除而非物理删除**
- 所有校验通过后,执行逻辑删除(将deleted字段设为1)
- 不是物理删除记录,保留数据用于审计和追溯
- 更新update_time和updater字段

**4. 删除操作记录**
- 在操作日志表中记录删除操作
- 记录内容包括:被删除岗位的名称、代码、所属部门、操作人、操作时间

### 2.4 岗位查询业务逻辑

岗位查询的各种场景:

**1. 按部门查询岗位列表**
- 接口:/admin-api/biz/post/list-by-dept
- 参数:deptId(部门ID)
- 返回:该部门下的所有启用状态岗位
- SQL: SELECT * FROM biz_post WHERE dept_id = #{deptId} AND status = 1 AND deleted = 0 ORDER BY sort ASC

**2. 按机构查询岗位列表**
- 接口:/admin-api/biz/post/list-by-org
- 参数:orgId(机构ID)
- 返回:该机构下的所有启用状态岗位(跨部门)
- SQL: SELECT * FROM biz_post WHERE org_id = #{orgId} AND status = 1 AND deleted = 0 ORDER BY sort ASC

**3. 分页查询岗位**
- 接口:/admin-api/biz/post/page
- 参数:pageNum, pageSize, deptId(可选), postName(可选), status(可选)
- 返回:分页结果
- 支持按岗位名称模糊查询
- 支持按状态过滤
- 支持按部门过滤

**4. 下拉选择用简单列表**
- 接口:/admin-api/biz/post/simple-list
- 参数:deptId(可选)
- 返回:简化的岗位列表,只包含id、postName、postCode
- 用于前端下拉选择框

**5. 数据权限过滤**
- 根据当前登录用户的数据权限,只返回用户有权限查看的岗位
- 如果用户是超级管理员,返回所有岗位
- 如果用户是某个部门的管理员,只返回该部门及其子部门的岗位
- 如果用户是普通员工,只能查看自己所在部门的岗位

**6. 缓存优化**
- 对于简单列表查询,可以使用Redis缓存
- 缓存key: post:simple-list:{deptId}
- 缓存过期时间:1小时
- 当岗位数据发生变更时,主动清除相关缓存

### 2.5 岗位与人员的关联管理

岗位和人员之间的关联关系管理:

**1. 人员分配岗位**
- 在新增或修改人员时,可以指定岗位(post_id)
- 一个人员只能分配一个岗位(主岗)
- 但可以支持兼职岗位(通过扩展表biz_agent_post_relation)

**2. 批量调整岗位**
- 功能:将多个人员批量调整到新岗位
- 接口:/admin-api/biz/post/batch-assign
- 参数:userIds(人员ID列表), newPostId(新岗位ID)
- 业务逻辑:
  1. 校验newPostId对应的岗位是否存在且启用
  2. 遍历userIds,更新每个人员的post_id
  3. 记录岗位变更日志
  4. 发送通知给相关人员

**3. 岗位人员统计**
- 功能:统计每个岗位的在职人员数
- 接口:/admin-api/biz/post/staff-count
- 参数:deptId(可选)
- 返回:岗位列表及每个岗位的人员数
- SQL: 
  ```sql
  SELECT p.id, p.post_name, 
         (SELECT COUNT(*) FROM biz_agent WHERE post_id = p.id AND status = 1 AND deleted = 0) +
         (SELECT COUNT(*) FROM biz_staff WHERE post_id = p.id AND status = 1 AND deleted = 0) AS staff_count
  FROM biz_post p
  WHERE p.dept_id = #{deptId} AND p.deleted = 0
  ```

**4. 岗位空缺提醒**
- 如果某个岗位长期没有人员,系统应发出提醒
- 定时任务:每周一执行
- 查询人员数为0的岗位,向部门负责人发送通知
- 通知内容:"岗位【岗位名称】当前无人员,请及时安排"

## 3. 业务员管理业务逻辑补充

### 3.1 业务员新增业务逻辑

业务员新增时需要处理的核心逻辑:

**1. 工号自动生成**
- agent_code字段表示业务员工号,必须唯一
- 生成规则:前缀 + 序号
  - 前缀从参数配置表获取,默认为"A"
  - 序号为6位数字,从000001开始递增
  - 例如:A000001, A000002
- 生成方法:
  1. 查询数据库中最大的工号序号
  2. SELECT MAX(CAST(SUBSTRING(agent_code, 2) AS UNSIGNED)) AS max_seq FROM biz_agent WHERE agent_code LIKE 'A%' AND deleted = 0
  3. 序号+1,不足6位前面补0
  4. 拼接前缀得到新工号
- 需要考虑并发问题,使用数据库悲观锁或Redis分布式锁保证工号唯一性

**2. 账号信息校验**
- mobile字段作为登录账号,必须唯一
- 校验mobile是否已被其他业务员或内勤人员使用
- SQL: 
  ```sql
  SELECT COUNT(*) FROM (
    SELECT mobile FROM biz_agent WHERE mobile = #{mobile} AND deleted = 0
    UNION ALL
    SELECT mobile FROM biz_staff WHERE mobile = #{mobile} AND deleted = 0
    UNION ALL
    SELECT mobile FROM system_users WHERE mobile = #{mobile} AND deleted = 0
  ) t
  ```
- 如果count > 0,抛出业务异常"手机号已被使用"

**3. 身份证号校验**
- id_card字段必须符合身份证号格式
- 18位身份证号校验规则:
  1. 长度必须是18位
  2. 前17位必须是数字
  3. 第18位可以是数字或X
  4. 第7-14位表示出生日期,必须是有效日期
  5. 最后一位是校验码,根据前17位计算得出,必须匹配
- 使用工具类IdCardUtil进行校验
- 身份证号在系统中必须唯一
- SQL: SELECT COUNT(*) FROM biz_agent WHERE id_card = #{idCard} AND deleted = 0
- 如果count > 0,抛出业务异常"身份证号已存在"

**4. 默认密码设置**
- password字段存储密码的BCrypt加密值
- 新增业务员时,密码使用默认值
- 默认密码从参数配置表获取,参数键:agent.default.password,默认值:123456
- 使用BCrypt加密后存入数据库
- 代码示例:String encodedPassword = BCrypt.hashpw(defaultPassword, BCrypt.gensalt());

**5. 机构和部门关联校验**
- org_id和dept_id必须关联正确
- 校验部门是否属于指定机构:
  - 查询biz_department表,WHERE id = #{deptId} AND org_id = #{orgId}
  - 如果查询结果为空,抛出异常"部门不属于该机构"
- 校验机构和部门是否都是启用状态
- 如果机构或部门被停用,不允许新增业务员

**6. 岗位关联校验**
- post_id字段表示业务员的岗位
- 如果指定了岗位,需要校验:
  1. 岗位是否存在
  2. 岗位是否属于指定部门
  3. 岗位状态是否启用
  4. 岗位类型是否为业务岗(post_type=2)
- 如果未指定岗位,post_id可以为NULL

**7. 执业证书信息校验**
- cert_type字段表示证书类型:1-代理人证 2-经纪人证 3-公估人证
- cert_no字段表示证书编号,必填
- cert_level字段表示证书等级,如1级、2级等
- cert_issue_date字段表示发证日期
- cert_expire_date字段表示证书到期日期,必须大于发证日期
- 证书编号在系统中必须唯一:
  - SQL: SELECT COUNT(*) FROM biz_agent WHERE cert_no = #{certNo} AND deleted = 0
  - 如果count > 0,抛出业务异常"证书编号已存在"

**8. 银行卡信息校验**
- bank_id字段表示开户银行,关联biz_bank表
- bank_card_no字段表示银行卡号
- 银行卡号格式校验:
  1. 长度必须是16-19位
  2. 必须全部是数字
  3. 使用Luhn算法校验银行卡号是否有效
- 银行卡号在系统中不要求唯一(同一张卡可能被多人使用,如家属代发)

**9. 佣金比例校验**
- commission_rate字段表示业务员的佣金比例,单位为百分比
- 取值范围:0-100
- 如果不填,默认为0
- 不同险种可以设置不同的佣金比例(通过扩展表biz_agent_commission_rate)

**10. 实名认证状态初始化**
- verify_status字段表示实名认证状态:0-未认证 1-待审核 2-已认证 3-已拒绝
- 新增时默认为0(未认证)
- 业务员需要上传身份证正反面照片完成实名认证

**11. 账号状态初始化**
- status字段表示账号状态:0-停用 1-启用
- 新增时默认为1(启用)
- account_status字段表示在职状态:1-在职 2-离职
- 新增时默认为1(在职)

**12. 默认值自动填充**
- join_date字段表示入职日期,默认为当前日期
- creator和create_time从当前登录上下文中获取
- tenant_id从当前登录上下文中获取,用于SaaS多租户隔离

**13. 同步创建系统账号**
- 业务员新增后,需要在系统用户表(system_users)中同步创建账号
- 用户名:业务员手机号
- 密码:业务员密码
- 昵称:业务员姓名
- 关联角色:默认的"业务员"角色
- 数据权限:只能查看自己的数据
- 这样业务员可以使用手机号和密码登录系统

### 3.2 业务员修改业务逻辑

业务员修改时的特殊处理逻辑:

**1. 禁止修改工号**
- agent_code一旦生成不允许修改
- 工号是业务员的唯一标识,可能被订单、佣金等业务数据引用
- 如果确实需要修改,只能新建账号并迁移数据

**2. 手机号唯一性校验**
- 如果修改了mobile,需要校验新手机号是否已被使用
- SQL: 
  ```sql
  SELECT COUNT(*) FROM (
    SELECT mobile FROM biz_agent WHERE mobile = #{mobile} AND id != #{agentId} AND deleted = 0
    UNION ALL
    SELECT mobile FROM biz_staff WHERE mobile = #{mobile} AND deleted = 0
    UNION ALL
    SELECT mobile FROM system_users WHERE mobile = #{mobile} AND deleted = 0
  ) t
  ```
- 如果count > 0,抛出业务异常"手机号已被使用"
- 修改手机号后,需要同步更新system_users表中的username字段

**3. 身份证号唯一性校验**
- 如果修改了id_card,需要校验新身份证号是否已存在
- SQL: SELECT COUNT(*) FROM biz_agent WHERE id_card = #{idCard} AND id != #{agentId} AND deleted = 0
- 如果count > 0,抛出业务异常"身份证号已存在"

**4. 机构和部门变更校验**
- 如果修改了org_id或dept_id,需要校验新的机构和部门关联关系
- 部门必须属于新机构
- 新机构和新部门都必须是启用状态
- 机构或部门变更会影响业务员的数据权限和业绩归属

**5. 机构/部门变更的数据影响**
- 如果变更了org_id或dept_id,需要考虑历史数据的处理:
  - 方案1:保留历史订单、佣金等数据的原机构/部门归属,不做变更
  - 方案2:将历史数据也迁移到新机构/部门(需谨慎,可能影响绩效统计)
  - 推荐方案1,保持历史数据不变,只影响后续新产生的数据
- 生成人员调动记录,记录:原机构、新机构、原部门、新部门、调动时间、操作人

**6. 岗位变更校验**
- 如果修改了post_id,需要校验新岗位是否属于当前部门
- 新岗位必须是启用状态
- 记录岗位变更日志

**7. 执业证书变更**
- 如果修改了cert_no,需要校验新证书编号的唯一性
- SQL: SELECT COUNT(*) FROM biz_agent WHERE cert_no = #{certNo} AND id != #{agentId} AND deleted = 0
- 如果count > 0,抛出业务异常"证书编号已存在"
- 证书到期日期必须大于发证日期

**8. 银行卡信息变更**
- 如果修改了银行卡号,需要重新校验银行卡号格式
- 银行卡号变更会影响佣金发放,需要记录变更日志

**9. 佣金比例变更**
- 如果修改了commission_rate,需要记录变更日志
- 佣金比例变更只影响后续新订单,不影响历史订单的佣金计算
- 向业务员发送通知:"您的佣金比例已调整为{newRate}%,自{effectiveDate}起生效"

**10. 账号状态变更联动**
- 如果将status从启用改为停用:
  1. 同步停用system_users表中的对应账号
  2. 中止该业务员的所有进行中的订单
  3. 冻结该业务员的待结算佣金
  4. 发送通知给业务员:"您的账号已被停用,如有疑问请联系管理员"

- 如果将status从停用改为启用:
  1. 检查所属机构和部门是否启用
  2. 同步启用system_users表中的对应账号
  3. 不自动解冻佣金,由财务手动处理

**11. 在职状态变更联动**
- 如果将account_status从在职改为离职:
  1. 记录离职日期leave_date为当前日期
  2. 停用账号(status改为0)
  3. 中止所有进行中的订单
  4. 结算所有待结算佣金
  5. 生成离职人员档案
  6. 通知业务员的上级领导

- 如果将account_status从离职改为在职(返聘):
  1. 清空离职日期leave_date
  2. 启用账号(status改为1)
  3. 发送欢迎回归通知

**12. 实名认证状态变更**
- 如果verify_status从未认证或待审核改为已认证:
  1. 更新verify_time为当前时间
  2. 发送通知给业务员:"您的实名认证已通过"
  3. 如果是首次认证,可以发放新人奖励

- 如果verify_status从待审核改为已拒绝:
  1. 记录拒绝原因verify_remark
  2. 发送通知给业务员:"您的实名认证未通过,原因:{reason},请重新提交"

**13. 同步更新系统账号**
- 业务员信息修改后,需要同步更新system_users表
- 更新字段:昵称(姓名)、头像、邮箱、状态等
- 保持业务员表和系统用户表的数据一致性

### 3.3 业务员删除业务逻辑

业务员删除前需要进行严格校验:

**1. 订单数据检查**
- 查询biz_order表中agent_id等于当前业务员id的订单数
- SQL: SELECT COUNT(*) FROM biz_order WHERE agent_id = #{agentId} AND deleted = 0
- 如果count > 0,不允许删除,提示"业务员存在{count}条订单记录,不允许删除,请使用离职功能"

**2. 佣金数据检查**
- 查询biz_commission表中agent_id等于当前业务员id的佣金记录数
- SQL: SELECT COUNT(*) FROM biz_commission WHERE agent_id = #{agentId} AND deleted = 0
- 如果count > 0,不允许删除,提示"业务员存在{count}条佣金记录,不允许删除"

**3. 客户数据检查**
- 查询biz_customer表中agent_id等于当前业务员id的客户数
- SQL: SELECT COUNT(*) FROM biz_customer WHERE agent_id = #{agentId} AND deleted = 0
- 如果count > 0,不允许删除,提示"业务员存在{count}个客户,不允许删除"

**4. 业务数据转移建议**
- 如果业务员有业务数据,不应删除,而应:
  1. 将业务员的account_status改为离职
  2. 将业务员的status改为停用
  3. 将业务员的客户转移给其他业务员
  4. 结算完所有佣金后,可以考虑数据归档

**5. 无业务数据时可删除**
- 只有当业务员完全没有业务数据时,才允许删除
- 执行逻辑删除(将deleted字段设为1)
- 同步删除system_users表中的对应账号(逻辑删除)
- 记录删除操作日志

### 3.4 业务员实名认证业务逻辑

业务员实名认证是重要的合规要求:

**1. 提交认证申请**
- 接口:/admin-api/biz/agent/submit-verify
- 业务员上传身份证正面照片(id_card_front_img)
- 业务员上传身份证反面照片(id_card_back_img)
- 自动OCR识别身份证信息:
  1. 调用阿里云OCR服务识别身份证正面
  2. 提取姓名、身份证号、性别、民族、出生日期、住址
  3. 调用阿里云OCR服务识别身份证反面
  4. 提取签发机关、有效期限
  5. 将识别结果填充到对应字段
- 校验OCR识别的身份证号与业务员填写的id_card是否一致
- 如果不一致,提示"身份证号与上传的证件不一致,请核实"
- 将verify_status改为1(待审核)
- 记录提交时间

**2. 审核认证申请**
- 接口:/admin-api/biz/agent/audit-verify
- 管理员审核业务员的实名认证申请
- 审核通过:
  1. 将verify_status改为2(已认证)
  2. 记录审核时间verify_time
  3. 发送通知给业务员
- 审核拒绝:
  1. 将verify_status改为3(已拒绝)
  2. 填写拒绝原因verify_remark
  3. 发送通知给业务员,说明拒绝原因

**3. 认证状态查询**
- 业务员可以查询自己的认证状态
- 管理员可以查询所有业务员的认证状态
- 支持按认证状态筛选:未认证、待审核、已认证、已拒绝

**4. 认证过期提醒**
- 身份证有有效期,需要定期检查
- 定时任务:每天凌晨执行
- 查询身份证距离到期日期少于90天的业务员
- 向业务员发送提醒:"您的身份证将于{expireDate}到期,请及时更新"
- 向管理员发送汇总报告

**5. 认证信息更新**
- 如果业务员更换了身份证(如到期换证),需要重新提交认证
- 将verify_status改回1(待审核)
- 重新上传新的身份证照片
- 重新走审核流程

### 3.5 业务员密码管理业务逻辑

业务员密码相关的业务逻辑:

**1. 密码重置(管理员操作)**
- 接口:/admin-api/biz/agent/reset-password
- 管理员可以重置业务员密码
- 新密码使用默认密码(从参数配置获取)
- 使用BCrypt加密后更新到数据库
- 同步更新system_users表中的密码
- 发送通知给业务员:"您的密码已被管理员重置为默认密码,请登录后及时修改"
- 记录密码重置操作日志

**2. 业务员修改密码**
- 接口:/admin-api/biz/agent/change-password
- 业务员登录后可以修改自己的密码
- 参数:oldPassword(旧密码), newPassword(新密码)
- 校验逻辑:
  1. 验证旧密码是否正确
  2. 新密码长度必须6-20位
  3. 新密码必须包含数字和字母
  4. 新密码不能与旧密码相同
- 使用BCrypt加密后更新到数据库
- 同步更新system_users表中的密码
- 记录密码修改操作日志

**3. 忘记密码(短信验证码重置)**
- 接口:/admin-api/biz/agent/forget-password
- 业务员忘记密码时,可以通过短信验证码重置
- 流程:
  1. 输入手机号,发送验证码
  2. 输入验证码,验证成功后允许设置新密码
  3. 使用BCrypt加密后更新到数据库
  4. 同步更新system_users表中的密码
  5. 发送通知:"您的密码已重置成功"

**4. 首次登录强制修改密码**
- 新增业务员时,密码为默认密码
- 业务员首次登录时,强制要求修改密码
- 在system_users表中增加字段:need_change_password(是否需要修改密码)
- 新增业务员时,need_change_password设为1
- 业务员修改密码后,need_change_password设为0
- 登录时检查,如果need_change_password=1,跳转到修改密码页面

**5. 密码安全策略**
- 密码必须定期更换,如90天一次
- 在system_users表中增加字段:password_update_time(密码更新时间)
- 每次修改密码时,更新password_update_time为当前时间
- 定时任务:每天凌晨执行,查询距离上次修改密码超过90天的账号
- 向这些业务员发送提醒:"您的密码已超过90天未修改,为了账号安全,请及时修改密码"
- 可以配置参数:超过90天未修改密码的账号强制下线,要求修改密码后才能登录

### 3.6 业务员业绩统计业务逻辑

业务员的业绩数据统计:

**1. 个人业绩统计**
- 接口:/admin-api/biz/agent/performance-stats
- 参数:agentId(业务员ID), startDate(开始日期), endDate(结束日期)
- 统计指标:
  1. 保费总额:该业务员在指定时间范围内的总保费
  2. 保单数量:该业务员在指定时间范围内的保单总数
  3. 佣金总额:该业务员在指定时间范围内的佣金总额
  4. 客户数量:该业务员的客户总数
  5. 新增客户:该业务员在指定时间范围内新增的客户数
  6. 退保率:退保保单数 / 总保单数
  7. 续保率:续保保单数 / 到期保单数
- SQL查询示例:
  ```sql
  SELECT 
    COUNT(DISTINCT o.id) AS policy_count,
    SUM(o.premium_amount) AS total_premium,
    SUM(c.amount) AS total_commission
  FROM biz_order o
  LEFT JOIN biz_commission c ON o.id = c.order_id
  WHERE o.agent_id = #{agentId}
    AND o.status IN (2, 3)  -- 已支付、已完成
    AND o.create_time >= #{startDate}
    AND o.create_time <= #{endDate}
    AND o.deleted = 0
  ```

**2. 业绩排行榜**
- 接口:/admin-api/biz/agent/performance-rank
- 参数:orgId(机构ID,可选), deptId(部门ID,可选), startDate, endDate, rankType(排名类型:1-保费 2-保单数 3-佣金)
- 返回:业务员列表,按指定指标降序排列,包含排名、姓名、机构、部门、业绩数据
- 支持按机构或部门筛选
- 可以分页展示,每页显示Top 50

**3. 业绩目标设置与完成率**
- 为每个业务员设置月度/季度/年度业绩目标
- 扩展表:biz_agent_target
  - agent_id:业务员ID
  - target_type:目标类型,1-月度 2-季度 3-年度
  - target_year:目标年份
  - target_period:目标期间,如1-12月,1-4季度
  - target_premium:保费目标
  - target_policy_count:保单数目标
  - actual_premium:实际保费
  - actual_policy_count:实际保单数
  - completion_rate:完成率
- 定时任务:每天凌晨执行,更新所有业务员的实际业绩和完成率
- 完成率计算:actual_premium / target_premium * 100%

**4. 业绩数据可视化**
- 在业务员个人中心展示业绩数据图表
- 折线图:展示近12个月的保费趋势
- 柱状图:展示各险种的保费占比
- 饼图:展示客户来源渠道占比
- 使用ECharts前端组件进行数据可视化

### 3.7 业务员客户管理业务逻辑

业务员的客户管理功能:

**1. 客户归属规则**
- 客户首次购买保险时,由哪个业务员服务,客户就归属于该业务员
- 在biz_customer表中,agent_id字段记录客户归属的业务员
- 客户可以通过多种方式产生:
  1. 业务员主动开发:业务员手动添加客户信息
  2. 客户自助下单:客户在C端商城下单,系统自动分配业务员
  3. 客户转介绍:老客户推荐新客户,新客户归属于老客户的业务员

**2. 客户自动分配逻辑**
- 当客户在C端商城自助下单时,需要自动分配业务员
- 分配规则(按优先级):
  1. 如果客户之前有归属业务员,直接分配给该业务员
  2. 如果客户通过推广链接进入,分配给推广链接对应的业务员
  3. 如果客户选择了服务网点,分配给该网点的值班业务员
  4. 如果以上都不满足,采用轮询算法分配给在线业务员
- 轮询算法:
  1. 查询当前在线的业务员列表(status=1, account_status=1)
  2. 按上次分配时间升序排列
  3. 取第一个业务员进行分配
  4. 更新该业务员的last_assign_time字段

**3. 客户转移**
- 接口:/admin-api/biz/agent/transfer-customer
- 管理员可以将客户从一个业务员转移到另一个业务员
- 参数:customerIds(客户ID列表), fromAgentId(原业务员), toAgentId(新业务员)
- 业务逻辑:
  1. 校验fromAgentId和toAgentId都是有效的业务员
  2. 校验所有customerIds确实属于fromAgentId
  3. 更新biz_customer表的agent_id字段
  4. 生成客户转移记录表biz_customer_transfer_log
  5. 发送通知给原业务员和新业务员

**4. 客户保护期**
- 客户归属业务员后,有一定的保护期,其他业务员不能抢单
- 保护期默认为365天,可以通过参数配置
- 保护期内,如果客户再次下单,仍然由原业务员服务
- 保护期外,如果客户通过其他渠道下单,可以分配给新业务员

**5. 公海客户池**
- 对于长期未跟进的客户,自动回收到公海客户池
- 回收规则:
  1. 业务员离职,客户自动进入公海
  2. 客户最后跟进时间超过90天,自动进入公海
  3. 管理员手动回收
- 公海客户可以由其他业务员领取
- 领取规则:
  1. 业务员可以主动领取公海客户(有数量限制,如每天最多领取10个)
  2. 管理员可以手动分配公海客户给业务员

**6. 客户标签管理**
- 业务员可以为客户打标签,如"高净值客户"、"价格敏感"、"有车一族"等
- 标签用于客户分类和精准营销
- 扩展表:biz_customer_tag_relation
  - customer_id:客户ID
  - tag_id:标签ID
  - agent_id:打标签的业务员ID
  - create_time:打标签时间

### 3.8 业务员数据权限控制

业务员的数据权限规则:

**1. 数据权限基本原则**
- 业务员只能查看和操作自己的数据
- 数据包括:客户、订单、佣金、业绩等
- 不能查看其他业务员的数据

**2. 数据权限实现方式**
- 在查询SQL中,增加agent_id的过滤条件
- 从当前登录上下文中获取业务员ID
- 示例:
  ```sql
  SELECT * FROM biz_customer 
  WHERE agent_id = #{currentAgentId} 
    AND deleted = 0
  ```

**3. 管理员权限**
- 部门经理可以查看本部门所有业务员的数据
- 机构负责人可以查看本机构所有业务员的数据
- 总公司管理员可以查看全公司所有数据

**4. 权限判断逻辑**
- 查询当前登录用户的角色
- 如果是超级管理员,不限制数据权限
- 如果是机构负责人,只能查看本机构数据
  - SQL增加条件:org_id = #{currentUserOrgId}
- 如果是部门经理,只能查看本部门数据
  - SQL增加条件:dept_id = #{currentUserDeptId}
- 如果是普通业务员,只能查看自己的数据
  - SQL增加条件:agent_id = #{currentUserId}

**5. 数据权限注解**
- 使用自定义注解@DataScope标注需要数据权限控制的方法
- AOP切面拦截,自动注入数据权限SQL条件
- 示例:
  ```java
  @DataScope(deptAlias = "d", userAlias = "a")
  public List<Customer> selectCustomerList(CustomerQuery query) {
      // 查询逻辑
  }
  ```

## 4. 内勤人员管理业务逻辑补充

### 4.1 内勤人员与业务员的区别

**1. 职能定位不同**
- 业务员:直接面向客户,从事保险销售,有佣金收入
- 内勤人员:从事公司内部管理、支持工作,无佣金收入,有固定工资

**2. 数据表分离**
- 业务员存储在biz_agent表
- 内勤人员存储在biz_staff表
- 两个表结构类似,但字段有差异

**3. 权限和功能不同**
- 业务员:主要使用B端业务系统,如报价、出单、客户管理等
- 内勤人员:主要使用PC管理后台,如审核、统计、配置等

### 4.2 内勤人员新增业务逻辑

内勤人员新增时的核心逻辑:

**1. 工号自动生成**
- staff_code字段表示内勤工号,必须唯一
- 生成规则:前缀 + 序号
  - 前缀为"S"(Staff的首字母)
  - 序号为6位数字,从000001开始递增
  - 例如:S000001, S000002
- 生成方法与业务员类似,查询最大序号+1

**2. 账号信息校验**
- mobile字段作为登录账号,必须唯一
- 校验mobile是否已被其他内勤或业务员使用
- 校验逻辑同业务员

**3. 身份证号校验**
- id_card字段必须符合身份证号格式
- 身份证号在系统中必须唯一
- 校验逻辑同业务员

**4. 默认密码设置**
- password字段存储密码的BCrypt加密值
- 新增内勤时,密码使用默认值
- 默认密码从参数配置表获取,参数键:staff.default.password,默认值:123456

**5. 机构和部门关联校验**
- org_id和dept_id必须关联正确
- 校验逻辑同业务员

**6. 岗位关联校验**
- post_id字段表示内勤的岗位
- 如果指定了岗位,需要校验:
  1. 岗位是否存在
  2. 岗位是否属于指定部门
  3. 岗位状态是否启用
  4. 岗位类型是否为管理岗(post_type=1)或支持岗(post_type=3)

**7. 银行卡信息校验**
- 内勤人员也需要填写银行卡信息,用于发放工资
- 校验逻辑同业务员

**8. 同步创建系统账号**
- 内勤人员新增后,需要在系统用户表(system_users)中同步创建账号
- 关联角色:根据岗位自动分配角色
  - 如果是部门经理岗,分配"部门经理"角色
  - 如果是财务岗,分配"财务"角色
  - 如果是客服岗,分配"客服"角色
  - 等等

### 4.3 内勤人员修改业务逻辑

内勤人员修改时的特殊处理逻辑:

**1. 禁止修改工号**
- staff_code一旦生成不允许修改

**2. 手机号唯一性校验**
- 如果修改了mobile,需要校验新手机号是否已被使用
- 校验逻辑同业务员

**3. 机构和部门变更校验**
- 如果修改了org_id或dept_id,需要校验新的机构和部门关联关系
- 生成人员调动记录

**4. 岗位变更联动**
- 如果修改了post_id,可能需要调整角色和权限
- 例如:从普通客服岗调整为客服主管岗,需要增加主管权限
- 提示管理员手动调整角色

**5. 账号状态变更联动**
- 逻辑同业务员

**6. 在职状态变更联动**
- 逻辑同业务员

### 4.4 内勤人员删除业务逻辑

内勤人员删除前需要进行严格校验:

**1. 业务数据检查**
- 查询该内勤是否有审核记录、操作记录等
- 如果有业务数据,不允许删除,建议使用离职功能

**2. 无业务数据时可删除**
- 只有当内勤完全没有业务数据时,才允许删除
- 执行逻辑删除
- 同步删除system_users表中的对应账号

### 4.5 内勤人员角色和权限管理

内勤人员的角色和权限配置:

**1. 角色分配**
- 内勤人员创建时,根据岗位自动分配初始角色
- 管理员可以手动调整内勤的角色
- 一个内勤可以拥有多个角色

**2. 权限控制**
- 内勤的权限由角色决定
- 角色关联菜单权限、按钮权限、数据权限
- 使用ruoyi-vue-pro框架的RBAC权限模型

**3. 数据权限**
- 内勤的数据权限根据角色和岗位确定
- 例如:
  - 部门经理:可以查看本部门所有数据
  - 机构负责人:可以查看本机构所有数据
  - 财务:可以查看所有佣金和财务数据
  - 客服:只能查看分配给自己的工单

**4. 权限审批流程**
- 内勤申请新权限时,需要走审批流程
- 提交申请 -> 部门经理审批 -> IT管理员审批 -> 生效
- 记录权限变更日志

## 5. 产品管理业务逻辑补充

### 5.1 保险产品新增业务逻辑

保险产品新增时需要处理的核心逻辑:

**1. 产品代码唯一性校验**
- product_code字段表示产品代码,必须唯一
- 通常由保险公司提供,格式如:PICC-CAR-001
- 查询数据库中是否已存在相同的product_code
- SQL: SELECT COUNT(*) FROM biz_product WHERE product_code = #{productCode} AND deleted = 0
- 如果count > 0,抛出业务异常"产品代码已存在"

**2. 保险公司关联校验**
- company_id字段表示产品所属的保险公司
- 校验company_id对应的保险公司是否存在且启用
- SQL: SELECT status FROM biz_insurance_company WHERE id = #{companyId} AND deleted = 0
- 如果查询结果为空或status=0,抛出异常"保险公司不存在或已停用"

**3. 产品类型校验**
- product_type字段表示产品类型:1-车险 2-非车险
- 必填且只能是这两个值之一

**4. 险种类型校验**
- insurance_type字段表示险种类型:1-车险 2-健康险 3-意外险 4-寿险 5-财产险等
- 必填且必须是系统定义的有效枚举值
- 可以从数据字典表sys_dict_data中查询有效值

**5. 产品名称校验**
- product_name字段表示产品名称
- 长度必须在2-100字符之间
- 建议同一保险公司下的产品名称不重复,但不强制

**6. 保费范围校验**
- min_premium和max_premium字段表示保费范围
- min_premium必须大于0
- max_premium必须大于等于min_premium
- 如果不填,表示不限制保费范围

**7. 保额范围校验**
- min_coverage和max_coverage字段表示保额范围
- min_coverage必须大于0
- max_coverage必须大于等于min_coverage
- 如果不填,表示不限制保额范围

**8. 保障期限校验**
- coverage_period字段表示保障期限,单位:月
- 取值范围:1-360(1个月到30年)
- 常见值:12(一年)、60(五年)、240(二十年)

**9. 佣金比例校验**
- commission_rate字段表示默认佣金比例,单位:百分比
- 取值范围:0-100
- 如果不填,默认为0

**10. 产品图片上传**
- product_image字段存储产品图片URL
- 图片上传到阿里云OSS
- 图片格式:JPG/PNG
- 图片大小:不超过2MB
- 图片尺寸:建议800x600

**11. 产品详情填写**
- product_detail字段存储产品详情,富文本内容
- 可以包含HTML标签
- 最大长度:65535字符
- 建议使用富文本编辑器(如TinyMCE)编辑

**12. 产品状态初始化**
- status字段表示产品状态:0-下架 1-上架
- 新增时默认为0(下架),需要手动上架

**13. 库存初始化**
- stock字段表示产品库存,默认为99999
- 如果是限量产品,需要填写实际库存
- 库存为0时,前端显示"已售罄",不允许下单

**14. 产品标签**
- 可以为产品打标签,如"热销"、"新品"、"爆款"等
- 扩展表:biz_product_tag_relation
- 标签用于前端展示和筛选

### 5.2 保险产品修改业务逻辑

保险产品修改时的特殊处理逻辑:

**1. 禁止修改产品代码**
- product_code一旦创建不允许修改
- 因为产品代码可能被订单、佣金等业务数据引用

**2. 禁止修改保险公司**
- company_id一旦创建不允许修改
- 如果需要修改,建议新建产品

**3. 禁止修改产品类型**
- product_type一旦创建不允许修改
- 车险和非车险的业务逻辑差异很大,不允许相互转换

**4. 产品名称唯一性校验**
- 如果修改了product_name,建议同一保险公司下不重复
- 但不强制,只给出提示

**5. 保费、保额范围调整**
- 如果修改了min_premium、max_premium、min_coverage、max_coverage
- 需要校验新的范围是否合理
- 记录变更日志

**6. 佣金比例调整**
- 如果修改了commission_rate,需要记录变更日志
- 佣金比例变更只影响后续新订单,不影响历史订单

**7. 产品状态变更**
- 如果将status从上架改为下架:
  1. 产品在前端不再展示
  2. 不允许新下单
  3. 已有订单继续服务
  4. 向已授权该产品的机构发送通知

- 如果将status从下架改为上架:
  1. 检查保险公司是否启用
  2. 检查产品信息是否完整(名称、图片、详情等)
  3. 产品在前端展示
  4. 允许下单

**8. 库存调整**
- 如果修改了stock,需要记录变更日志
- 如果库存从大于0改为0,产品自动下架
- 如果库存从0改为大于0,产品可以上架

**9. 产品详情更新**
- 如果修改了product_detail,建议向已购买该产品的客户发送通知
- 通知内容:"您购买的产品【产品名称】的详情已更新,请及时查看"

### 5.3 保险产品删除业务逻辑

保险产品删除前需要进行严格校验:

**1. 订单数据检查**
- 查询biz_order表中product_id等于当前产品id的订单数
- SQL: SELECT COUNT(*) FROM biz_order WHERE product_id = #{productId} AND deleted = 0
- 如果count > 0,不允许删除,提示"产品存在{count}条订单记录,不允许删除,请使用下架功能"

**2. 机构授权检查**
- 查询biz_product_org表中product_id等于当前产品id的授权记录数
- SQL: SELECT COUNT(*) FROM biz_product_org WHERE product_id = #{productId} AND deleted = 0
- 如果count > 0,建议先取消授权再删除,或者直接不允许删除

**3. 无业务数据时可删除**
- 只有当产品完全没有订单和授权数据时,才允许删除
- 执行逻辑删除(将deleted字段设为1)
- 记录删除操作日志

**4. 建议使用下架而非删除**
- 对于已有业务数据的产品,建议使用下架功能,而非删除
- 下架后产品不再展示,但历史数据保留

### 5.4 保险产品上下架业务逻辑

产品上下架是高频操作,需要详细定义:

**1. 产品上架**
- 接口:/admin-api/biz/product/on-shelf
- 参数:productId
- 前置校验:
  1. 产品是否存在
  2. 所属保险公司是否启用
  3. 产品名称、图片、详情是否完整
  4. 保费、保额范围是否合理
  5. 库存是否大于0
- 校验通过后,将status改为1(上架)
- 记录上架时间on_shelf_time
- 向已授权该产品的机构发送通知:"产品【产品名称】已上架,可以开始销售"

**2. 产品下架**
- 接口:/admin-api/biz/product/off-shelf
- 参数:productId
- 前置校验:
  1. 产品是否存在
  2. 产品当前是否是上架状态
- 校验通过后,将status改为0(下架)
- 记录下架时间off_shelf_time
- 向已授权该产品的机构发送通知:"产品【产品名称】已下架,不能再销售"

**3. 批量上下架**
- 接口:/admin-api/biz/product/batch-shelf
- 参数:productIds(产品ID列表), shelfAction(上架/下架)
- 遍历productIds,逐个执行上架或下架操作
- 如果某个产品操作失败,继续处理下一个,最后汇总结果

**4. 定时上下架**
- 支持设置产品的自动上架时间和下架时间
- 扩展字段:auto_on_shelf_time(自动上架时间), auto_off_shelf_time(自动下架时间)
- 定时任务:每小时执行一次
- 查询auto_on_shelf_time小于等于当前时间且status=0的产品,执行上架
- 查询auto_off_shelf_time小于等于当前时间且status=1的产品,执行下架

### 5.5 保险产品库存管理业务逻辑

产品库存的管理逻辑:

**1. 库存扣减**
- 当客户下单时,需要扣减产品库存
- 时机:订单支付成功后
- 方法:UPDATE biz_product SET stock = stock - #{quantity} WHERE id = #{productId} AND stock >= #{quantity}
- 如果更新影响行数为0,说明库存不足,订单支付失败

**2. 库存回退**
- 当订单取消或退款时,需要回退产品库存
- 方法:UPDATE biz_product SET stock = stock + #{quantity} WHERE id = #{productId}

**3. 库存预警**
- 当产品库存低于预警值时,发送预警通知
- 预警值可以在产品信息中配置,默认为10
- 定时任务:每小时执行一次
- 查询stock小于等于预警值的产品
- 向产品管理员发送通知:"产品【产品名称】库存不足,当前库存:{stock},请及时补货"

**4. 库存同步**
- 如果产品库存由保险公司系统管理,需要定时同步
- 定时任务:每天凌晨执行
- 调用保险公司API,查询产品库存
- 更新biz_product表的stock字段

**5. 无限库存产品**
- 某些产品可能是无限库存的,如电子保单
- stock字段设为-1表示无限库存
- 下单时不扣减库存

### 5.6 保险产品佣金方案管理

产品的佣金方案配置:

**1. 默认佣金比例**
- biz_product表的commission_rate字段是产品的默认佣金比例
- 适用于所有业务员,除非有特殊配置

**2. 分级佣金方案**
- 支持根据业务员等级设置不同的佣金比例
- 扩展表:biz_product_commission_plan
  - product_id:产品ID
  - agent_level:业务员等级,1-初级 2-中级 3-高级 4-资深
  - commission_rate:佣金比例
- 查询时,先查该业务员等级的佣金比例,如果没有配置,使用产品默认佣金比例

**3. 阶梯佣金方案**
- 支持根据销售数量或保费金额设置阶梯佣金
- 扩展表:biz_product_commission_ladder
  - product_id:产品ID
  - tier_start:阶梯起始值
  - tier_end:阶梯结束值
  - commission_rate:佣金比例
- 示例:
  - 0-10单:5%
  - 11-50单:8%
  - 51单以上:10%

**4. 特殊佣金方案**
- 支持为特定业务员或机构设置特殊佣金比例
- 扩展表:biz_product_special_commission
  - product_id:产品ID
  - agent_id:业务员ID(可为NULL)
  - org_id:机构ID(可为NULL)
  - commission_rate:佣金比例
- 查询优先级:特殊佣金 > 阶梯佣金 > 分级佣金 > 默认佣金

**5. 佣金方案生效时间**
- 佣金方案可以设置生效时间和失效时间
- 字段:effective_time(生效时间), expire_time(失效时间)
- 查询时,只取当前时间在生效时间和失效时间之间的方案

## 6. 产品机构授权业务逻辑补充

### 6.1 产品授权业务概述

产品授权是保险中介业务的核心概念:

**1. 授权的意义**
- 保险公司的产品不是所有机构都能销售
- 需要保险公司授权给中介机构后,中介机构才能销售
- 授权信息存储在biz_product_org表中

**2. 授权的维度**
- 产品维度:哪些产品被授权
- 机构维度:哪些机构获得授权
- 时间维度:授权的有效期

**3. 授权的影响**
- 未授权的产品,机构无法查看和销售
- 已授权的产品,机构可以查看产品信息、报价、出单
- 授权到期后,自动失效,不能再销售

### 6.2 产品授权新增业务逻辑

新增产品授权时需要处理的核心逻辑:

**1. 产品有效性校验**
- product_id字段表示被授权的产品
- 校验product_id对应的产品是否存在且上架
- SQL: SELECT status FROM biz_product WHERE id = #{productId} AND deleted = 0
- 如果查询结果为空或status=0,抛出异常"产品不存在或已下架"

**2. 机构有效性校验**
- org_id字段表示被授权的机构
- 校验org_id对应的机构是否存在且启用
- SQL: SELECT status FROM biz_organization WHERE id = #{orgId} AND deleted = 0
- 如果查询结果为空或status=0,抛出异常"机构不存在或已停用"

**3. 重复授权检查**
- 同一产品不能重复授权给同一机构(在有效期内)
- 查询是否已存在授权记录
- SQL: 
  ```sql
  SELECT COUNT(*) FROM biz_product_org 
  WHERE product_id = #{productId} 
    AND org_id = #{orgId} 
    AND status = 1 
    AND (expire_date IS NULL OR expire_date >= CURDATE())
    AND deleted = 0
  ```
- 如果count > 0,抛出业务异常"该产品已授权给该机构,不能重复授权"

**4. 授权有效期校验**
- start_date字段表示授权开始日期
- expire_date字段表示授权到期日期
- expire_date必须大于start_date
- 如果不填expire_date,表示永久授权

**5. 授权佣金比例设置**
- commission_rate字段表示该机构销售该产品的佣金比例
- 如果不填,使用产品的默认佣金比例
- 如果填写,必须在0-100之间
- 可以为不同机构设置不同的佣金比例,体现差异化合作

**6. 授权状态初始化**
- status字段表示授权状态:0-停用 1-启用
- 新增时默认为1(启用)

**7. 授权后通知**
- 授权成功后,向机构负责人发送通知
- 通知内容:"产品【产品名称】已授权给您的机构,授权有效期至{expireDate},可以开始销售"
- 通知方式:站内信、短信、邮件

### 6.3 产品授权修改业务逻辑

产品授权修改时的特殊处理逻辑:

**1. 禁止修改产品和机构**
- product_id和org_id一旦创建不允许修改
- 如果需要修改,建议删除旧授权,新增新授权

**2. 授权有效期调整**
- 如果修改了start_date或expire_date,需要校验新的日期范围是否合理
- expire_date必须大于start_date
- 记录变更日志

**3. 授权佣金比例调整**
- 如果修改了commission_rate,需要记录变更日志
- 佣金比例变更只影响后续新订单,不影响历史订单
- 向机构负责人发送通知:"产品【产品名称】的佣金比例已调整为{newRate}%"

**4. 授权状态变更**
- 如果将status从启用改为停用:
  1. 该机构不能再销售该产品
  2. 已有订单继续服务
  3. 向机构负责人发送通知:"产品【产品名称】的授权已停用"

- 如果将status从停用改为启用:
  1. 检查产品是否上架
  2. 检查机构是否启用
  3. 检查授权是否在有效期内
  4. 该机构可以继续销售该产品
  5. 向机构负责人发送通知:"产品【产品名称】的授权已恢复"

### 6.4 产品授权删除业务逻辑

产品授权删除前需要进行校验:

**1. 订单数据检查**
- 查询biz_order表中product_id和org_id都匹配的订单数
- SQL: 
  ```sql
  SELECT COUNT(*) FROM biz_order 
  WHERE product_id = #{productId} 
    AND org_id = #{orgId} 
    AND deleted = 0
  ```
- 如果count > 0,不允许删除,提示"该授权存在{count}条订单记录,不允许删除,请使用停用功能"

**2. 无业务数据时可删除**
- 只有当授权完全没有订单数据时,才允许删除
- 执行逻辑删除(将deleted字段设为1)
- 向机构负责人发送通知:"产品【产品名称】的授权已取消"
- 记录删除操作日志

**3. 建议使用停用而非删除**
- 对于已有订单数据的授权,建议使用停用功能,而非删除
- 停用后机构不能再销售,但历史数据保留

### 6.5 产品授权查询业务逻辑

产品授权查询的各种场景:

**1. 按产品查询授权机构列表**
- 接口:/admin-api/biz/product-org/list-by-product
- 参数:productId
- 返回:该产品已授权的所有机构列表
- SQL: 
  ```sql
  SELECT po.*, o.org_name, o.org_code
  FROM biz_product_org po
  LEFT JOIN biz_organization o ON po.org_id = o.id
  WHERE po.product_id = #{productId}
    AND po.deleted = 0
  ORDER BY po.create_time DESC
  ```

**2. 按机构查询授权产品列表**
- 接口:/admin-api/biz/product-org/list-by-org
- 参数:orgId
- 返回:该机构已获得授权的所有产品列表
- SQL: 
  ```sql
  SELECT po.*, p.product_name, p.product_code
  FROM biz_product_org po
  LEFT JOIN biz_product p ON po.product_id = p.id
  WHERE po.org_id = #{orgId}
    AND po.status = 1
    AND (po.expire_date IS NULL OR po.expire_date >= CURDATE())
    AND po.deleted = 0
  ORDER BY po.create_time DESC
  ```

**3. 分页查询授权记录**
- 接口:/admin-api/biz/product-org/page
- 参数:pageNum, pageSize, productId(可选), orgId(可选), status(可选)
- 返回:分页结果
- 支持按产品、机构、状态筛选

**4. 检查机构是否有产品授权**
- 接口:/admin-api/biz/product-org/check-auth
- 参数:productId, orgId
- 返回:true/false
- 用于业务员报价前的权限检查
- SQL: 
  ```sql
  SELECT COUNT(*) FROM biz_product_org 
  WHERE product_id = #{productId} 
    AND org_id = #{orgId} 
    AND status = 1 
    AND (expire_date IS NULL OR expire_date >= CURDATE())
    AND deleted = 0
  ```
- 如果count > 0,返回true,否则返回false

### 6.6 产品授权批量操作业务逻辑

批量授权和批量取消授权:

**1. 批量授权产品给机构**
- 接口:/admin-api/biz/product-org/batch-grant
- 参数:productIds(产品ID列表), orgId(机构ID), startDate, expireDate, commissionRate
- 业务逻辑:
  1. 校验机构是否存在且启用
  2. 遍历productIds,逐个进行授权
  3. 对于每个产品,检查是否已授权,如果已授权,跳过
  4. 如果未授权,新增授权记录
  5. 汇总操作结果:成功授权{successCount}个产品,跳过{skipCount}个产品

**2. 批量取消机构的产品授权**
- 接口:/admin-api/biz/product-org/batch-revoke
- 参数:productIds(产品ID列表), orgId(机构ID)
- 业务逻辑:
  1. 遍历productIds,逐个取消授权
  2. 对于每个产品,检查是否有订单数据
  3. 如果有订单数据,将授权状态改为停用,而不是删除
  4. 如果没有订单数据,执行逻辑删除
  5. 汇总操作结果

**3. 批量授权产品给多个机构**
- 接口:/admin-api/biz/product-org/batch-grant-orgs
- 参数:productId(产品ID), orgIds(机构ID列表), startDate, expireDate, commissionRate
- 业务逻辑:
  1. 校验产品是否存在且上架
  2. 遍历orgIds,逐个进行授权
  3. 对于每个机构,检查是否已授权,如果已授权,跳过
  4. 如果未授权,新增授权记录
  5. 汇总操作结果

### 6.7 产品授权到期处理业务逻辑

授权到期的自动处理:

**1. 授权到期检查**
- 定时任务:每天凌晨执行
- 查询expire_date小于等于当前日期且status=1的授权记录
- SQL: 
  ```sql
  SELECT * FROM biz_product_org 
  WHERE expire_date <= CURDATE() 
    AND status = 1 
    AND deleted = 0
  ```

**2. 到期自动停用**
- 遍历查询结果,将这些授权记录的status改为0(停用)
- 记录停用时间和原因

**3. 到期提醒**
- 在授权到期前30天、15天、7天、3天、1天发送提醒
- 向机构负责人发送通知:"产品【产品名称】的授权将于{expireDate}到期,请及时续期"
- 向平台管理员发送汇总报告

**4. 授权续期**
- 接口:/admin-api/biz/product-org/renew
- 参数:authId(授权ID), newExpireDate(新到期日期)
- 业务逻辑:
  1. 校验authId对应的授权记录是否存在
  2. 校验newExpireDate必须大于当前到期日期
  3. 更新expire_date字段
  4. 如果授权已停用,恢复为启用状态
  5. 发送通知给机构负责人:"产品【产品名称】的授权已续期至{newExpireDate}"
