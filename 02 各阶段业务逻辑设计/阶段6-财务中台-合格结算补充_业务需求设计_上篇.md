## FN-01 批量修改上游手续费并结算

### 1.1 功能入口

菜单路径：**财务 → 上游结算 → 对账**  
页面上点击按钮：**【批量导入对账】**

### 1.2 整体流程

```
Step1: 用户在"上游结算-对账"页面 → 点击【批量导入对账】→ 弹出对话框
Step2: 对话框内点击【下载对账模板】→ 下载标准Excel模板
Step3: 用户填写Excel模板 → 点击【选择文件】上传 → 选择处理方式 → 点击【确定】
Step4: 系统解析Excel → 更新系统内手续费金额（对账完成）
Step5: 菜单切换至 财务 → 上游结算 → 结算 → 点击【批量导入结算】
Step6: 下载结算模板 → 填写结算信息 → 上传 → 选择"以导入应结金额为准，修改系统数据并结算"
Step7: 系统生成结算单 → 在"结算记录"中找到该结算单 → 点击【审批】→ 完成上游结算
```

### 1.3 Step1-4：批量导入对账（修改手续费）

#### 1.3.1 弹窗页面字段

点击【批量导入对账】后弹出对话框，包含以下内容：

| 元素 | 类型 | 说明 |
|------|------|------|
| 【下载对账表格】按钮 | Button | 下载含当前筛选条件保单的Excel模板 |
| 文件上传区 | Upload | 仅支持.xlsx/.xls，文件大小≤10MB |
| 处理方式 | Radio 单选 | 选项1：以导入金额为准，完成对账；选项2：以导入应结金额为准，修改系统数据并结算 |
| 【确定】按钮 | Button | 提交执行 |
| 【取消】按钮 | Button | 关闭弹窗 |

#### 1.3.2 下载对账模板

- 模板文件名：`上游对账导入模板_${yyyyMMdd}.xlsx`
- 模板Sheet结构（必填列标红）：

| 列名 | 字段 | 是否必填 | 说明 |
|------|------|---------|------|
| 保单号 | policy_no | ✅ 必填 | 用于匹配系统保单 |
| 保险公司 | insurer_name | ✅ 必填 | 与系统保单保险公司一致 |
| 应结手续费金额（元） | should_settle_amount | ✅ 必填 | 保司应付手续费金额 |
| 实结手续费金额（元） | actual_settle_amount | ✅ 必填 | 本次实际收到金额；修改手续费场景填0 |
| 上游手续费比例（%） | upstream_rate | 选填 | 若填写则同步更新费率 |
| 备注 | remark | 选填 | — |

#### 1.3.3 上传校验（后端）

后端收到文件后执行以下校验，任一校验失败则整批拒绝导入，返回错误明细：

1. **格式校验**：文件是否为xlsx/xls，行数 ≤ 500条，否则提示"单次最多500条"。
2. **保单存在性校验**：`policy_no` 在 `insurance_policy` 中存在且状态不为已撤保/已作废，否则标记该行"保单不存在或状态异常"。
3. **保险公司匹配校验**：导入行的 `insurer_name` 与系统保单的保险公司名称一致（允许模糊匹配，差异超过阈值时警告）。
4. **金额格式校验**：`should_settle_amount`、`actual_settle_amount` 必须为数字，≥0，精度≤2位小数。
5. **重复性校验**：同一保单号在本次导入中不允许重复。

校验通过后进入二次确认弹窗（见1.3.4）。

#### 1.3.4 二次确认弹窗

弹窗显示以下汇总信息：

```
本次导入保单共 N 条
涉及保险公司：XX保险、XX财险（列举）
修改前系统手续费合计：¥ XXX.XX
修改后（导入值）手续费合计：¥ XXX.XX
差异金额：¥ XXX.XX（红色显示）

[以导入金额为准，完成对账]  [取消]
```

用户点击**【以导入金额为准，完成对账】**后执行写库。

#### 1.3.5 后端处理逻辑（对账写库）

遍历每条导入记录，在**同一数据库事务**内执行：

```sql
-- 1. 若 upstream_rate 有值，更新保单表手续费比例
UPDATE insurance_policy
SET upstream_rate = #{upstream_rate},
    upstream_amount = #{should_settle_amount},
    update_time = NOW(), updater = #{operatorId}
WHERE policy_no = #{policy_no};

-- 2. 在结算明细表中插入/更新对账记录
INSERT INTO ins_settlement_detail (
    policy_no, should_settle_amount, actual_settle_amount,
    settle_type, settle_status, operate_type, created_time, creator
) VALUES (
    #{policy_no}, #{should_settle_amount}, #{actual_settle_amount},
    'UPSTREAM', 'RECONCILED', 'BATCH_IMPORT', NOW(), #{operatorId}
) ON DUPLICATE KEY UPDATE
    should_settle_amount = VALUES(should_settle_amount),
    actual_settle_amount = VALUES(actual_settle_amount),
    update_time = NOW();

-- 3. 写操作日志
INSERT INTO sys_operate_log (...) VALUES (...);
```

对账完成后，保单的上游对账状态更新为 `RECONCILED`（已对账）。

### 1.4 Step5-7：批量导入结算

#### 1.4.1 入口

菜单：**财务 → 上游结算 → 结算 → 【批量导入结算】**

弹出对话框，与对账弹窗结构相似，但处理方式选项变为：
- **以导入应结金额为准，修改系统数据并结算**（关键选项）
- 以系统已有数据结算

#### 1.4.2 结算模板字段

| 列名 | 字段 | 是否必填 |
|------|------|---------|
| 保单号 | policy_no | ✅ |
| 保险公司 | insurer_name | ✅ |
| 应结手续费（元） | should_settle_amount | ✅ |
| 实结手续费（元） | actual_settle_amount | ✅ |
| 结算周期开始日期 | period_start | ✅ |
| 结算周期结束日期 | period_end | ✅ |

#### 1.4.3 选择"以导入应结金额为准，修改系统数据并结算"时的处理逻辑

此选项的核心是：**先修改手续费，再执行结算，一步到位**。

后端处理步骤：
1. 执行1.3.5的对账写库逻辑（更新手续费）。
2. 校验 `actual_settle_amount > 0`，若为0则跳过结算步骤，仅更新手续费。
3. 按保司聚合所有保单，生成上游结算单 `ins_settlement_upstream`：
   - `settle_no`：系统自动生成（规则：`US${yyyyMMdd}${6位序号}`）
   - `insurer_code`：保险公司编码
   - `period_start` / `period_end`：结算周期
   - `total_amount`：本批次应结合计
   - `actual_amount`：本批次实结合计
   - `settle_status`：`PENDING_APPROVE`（待审批）
4. 生成结算明细记录（一保单一条）关联至结算单。
5. 写 `sys_operate_log`，操作类型为 `BATCH_SETTLE_WITH_RATE_MODIFY`。

#### 1.4.4 审批流程

结算单生成后状态为 `PENDING_APPROVE`，需财务主管在**结算记录**页找到该结算单，点击【审批】按钮：

- 弹出审批对话框，展示结算单汇总信息：结算单号、保司、期间、应结/实结金额、涉及保单数。
- 审批人可填写**审批意见**（选填，最长200字）。
- 点击【通过】：结算单状态变更为 `SETTLED`（已结算），写审批记录。
- 点击【拒绝】：结算单状态变更为 `REJECTED`，写审批记录，保单上游结算状态回滚为 `RECONCILED`。

#### 1.4.5 已发放佣金的差额处理

若同批次中存在保单**已完成下游佣金发放**（`ins_commission_record.status = 'PAID'`），且本次修改导致手续费金额发生变化，系统需：

1. 计算差额：`delta = 新手续费金额 - 原手续费金额`。
2. 若 `delta ≠ 0`，自动生成**差额佣金记录** `ins_commission_record`：
   - `type = 'ADJUSTMENT'`（差额调整）
   - `amount = delta`（正值为补发，负值为扣回）
   - `ref_policy_no`：原保单号
   - `status = 'PENDING_APPROVE'`（进入正常审批发放流程）
3. 前端在二次确认弹窗中**高亮提示**"以下 X 条保单已发放佣金，本次修改将产生差额调整单，需财务审批后发放"。

## FN-02 未填手续费跟单结算

### 2.1 业务背景

保单导入时保司尚未确认费率，`upstream_rate` 字段为空（NULL 或 0），该类保单需进入"跟单队列"暂缓结算，等待保司确认费率后再处理。

### 2.2 功能入口

菜单路径：**财务 → 上游结算 → 跟单队列**（新增菜单项）

### 2.3 跟单队列页面

#### 2.3.1 页面筛选条件

| 字段 | 控件类型 | 说明 |
|------|---------|------|
| 保险公司 | 下拉多选 | 关联保险公司字典 |
| 险种大类 | 下拉多选 | 车险/非车险/寿险 |
| 签单日期 | 日期范围 | 起止日期 |
| 跟单状态 | 下拉单选 | 待确认/已跳过/已结算 |
| 是否超期 | 下拉单选 | 是/否（超过配置天数未处理） |
| 保单号 | 文本框 | 模糊查询 |

点击【查询】后刷新列表。

#### 2.3.2 列表字段

| 列名 | 字段 | 说明 |
|------|------|------|
| 保单号 | policy_no | — |
| 保险公司 | insurer_name | — |
| 险种 | insurance_type | — |
| 投保人 | insured_name | — |
| 签单日期 | sign_date | — |
| 保费 | premium_amount | — |
| 当前上游手续费 | upstream_amount | 为空时显示"—" |
| 跟单状态 | pending_status | PENDING_RATE/SKIP_CURRENT_PERIOD/SETTLED |
| 跟单天数 | pending_days | 距导入日期天数，超期标红 |
| 操作 | — | 【填入手续费】【标记跳过】 |

列表支持**分组展示**：默认按保险公司分组折叠，可展开查看该保司下所有跟单保单。

#### 2.3.3 状态标识

超期判断规则：`pending_days > sys_config['pending_rate_expire_days']`（系统配置，默认45天）

超期保单在列表中以**红色文字**高亮显示 pending_days 列，并在行尾显示红色标签"⚠超期"。

### 2.4 跟单保单的产生（保单导入时）

在保单导入（车险批量录入、非车保单导入）的处理逻辑中，增加以下判断：

```
保单导入后处理：
IF upstream_rate IS NULL OR upstream_rate = 0
    → 将该保单插入 ins_pending_rate_queue：
        policy_no = 保单号
        insurer_code = 保险公司编码
        import_time = 当前时间
        pending_status = 'PENDING_RATE'
        expire_days_config = 系统配置值
        creator = 操作人
```

同时在 `insurance_policy` 表中设置 `upstream_settle_status = 'PENDING_RATE'`。

### 2.5 操作A：手动填入手续费

#### 2.5.1 单条操作

点击列表行的**【填入手续费】**按钮，弹出输入框：

| 字段 | 类型 | 校验 |
|------|------|------|
| 上游手续费比例（%） | 数字输入 | 选填；0~100，最多2位小数 |
| 上游手续费金额（元） | 数字输入 | 必填；>0，最多2位小数 |
| 备注 | 文本域 | 选填，最长200字 |

点击【确认】后：
1. 更新 `insurance_policy.upstream_rate`、`upstream_amount`。
2. 更新 `ins_pending_rate_queue.pending_status = 'SETTLED'`，记录处理时间和处理人。
3. 更新 `insurance_policy.upstream_settle_status = 'RECONCILED'`。
4. **自动触发佣金计算**：调用佣金计算引擎，以新手续费为基础重新计算 `ins_commission_record`，若已有记录则追加差额记录（type=ADJUSTMENT）。
5. 写操作日志，操作类型为 `PENDING_RATE_FILLED`。

#### 2.5.2 批量填入手续费（Excel导入方式）

跟单队列页面顶部提供**【批量导入手续费】**按钮：
- 点击后弹出与FN-01相同的导入对话框
- 导入模板列：保单号、上游手续费金额、上游手续费比例（选填）、备注
- 处理逻辑与单条操作一致，但批量执行，最多500条/次
- 批量操作加**分布式锁**（Redis Redisson），防止并发操作同一批保单

### 2.6 操作B：标记跳过（等下期补算）

点击列表行的**【标记跳过】**按钮，弹出确认框：

```
确认将该保单标记为"跳过本期"？
标记后本期不再结算此保单手续费，待下期补录费率后一并结算。
保单号：XXXXXXXXX
[确认跳过]  [取消]
```

点击【确认跳过】后：
1. 更新 `ins_pending_rate_queue.pending_status = 'SKIP_CURRENT_PERIOD'`。
2. 在 `ins_pending_rate_queue` 中记录跳过时间、跳过人员、跳过原因（可选）。
3. 该保单不出现在当期结算汇总中，但下期进行结算时重新扫描该保单（状态恢复为PENDING_RATE，重新进入跟单队列）。
4. 写操作日志。

### 2.7 跟单完整结算流程（确认费率后）

用户在跟单队列填入手续费后，后续结算流程遵循标准上游结算流程：

```
跟单队列填入手续费（PENDING_RATE → SETTLED）
    ↓
在「上游结算-结算」中可查询到该保单（因已有手续费数据）
    ↓
通过「批量导入结算」导入结算，填写：
    - 应结手续费 = 已确认的手续费金额
    - 实结手续费 = 本次实际收到金额（与应结相同，代表已收到）
    ↓
生成结算单 → 财务主管审批 → 完成结算
    ↓
进行开票（非本期关注，遵循标准开票流程）、收款
```

注：跟单结算完成后，监管报表才会包含该保单数据（`has_tracking_settled = true`）。

### 2.8 超期告警

通过 **XXL-Job** 定时任务每日凌晨1:00执行：

```
扫描 ins_pending_rate_queue where pending_status = 'PENDING_RATE'
计算 DATEDIFF(NOW(), import_time) > expire_days_config（取系统配置）
对超期保单：
    1. 更新 is_expired = 1
    2. 推送站内信给财务主管（角色：FINANCE_MANAGER）：
       标题："跟单队列超期提醒"
       内容："共有 X 条保单上游手续费超 N 天未处理，请及时处理。[查看详情]"
    3. 写告警日志 sys_alarm_log（防重复发送：同一批次同一天只发一次）
```

系统配置项（sys_config）：`pending_rate_expire_days`，默认值45，可在财务配置页面修改。

### 2.9 完整审计轨迹

`ins_pending_rate_queue` 表需记录以下时间戳和操作人字段：

| 字段 | 说明 |
|------|------|
| `import_time` | 保单导入时间 |
| `filled_time` | 手续费填入时间 |
| `filled_by` | 填入操作人ID |
| `skip_time` | 跳过时间 |
| `skip_by` | 跳过操作人ID |
| `skip_reason` | 跳过原因 |
| `expire_time` | 超期时间（计算字段或定时更新） |
| `settle_time` | 最终结算时间 |
| `update_time` | 最后更新时间 |
| `updater` | 最后更新人 |

## 数据库设计（FN-01 & FN-02）

### 跟单队列表

```sql
CREATE TABLE `ins_pending_rate_queue` (
  `id`                  BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键',
  `policy_no`           VARCHAR(64)  NOT NULL COMMENT '保单号',
  `insurer_code`        VARCHAR(32)  NOT NULL COMMENT '保险公司编码',
  `insurer_name`        VARCHAR(64)  NOT NULL COMMENT '保险公司名称',
  `insurance_type`      VARCHAR(32)  NOT NULL COMMENT '险种大类（CAR/NON_CAR/LIFE）',
  `premium_amount`      DECIMAL(12,2) DEFAULT NULL COMMENT '保费',
  `upstream_rate`       DECIMAL(8,4)  DEFAULT NULL COMMENT '上游手续费比例（%）',
  `upstream_amount`     DECIMAL(12,2) DEFAULT NULL COMMENT '上游手续费金额',
  `pending_status`      VARCHAR(32)  NOT NULL DEFAULT 'PENDING_RATE' COMMENT '跟单状态（PENDING_RATE/SKIP_CURRENT_PERIOD/SETTLED）',
  `is_expired`          TINYINT(1)   NOT NULL DEFAULT 0 COMMENT '是否超期（0否 1是）',
  `expire_days_config`  INT          NOT NULL DEFAULT 45 COMMENT '超期天数配置值（快照）',
  `import_time`         DATETIME     NOT NULL COMMENT '导入时间',
  `filled_time`         DATETIME     DEFAULT NULL COMMENT '手续费填入时间',
  `filled_by`           BIGINT       DEFAULT NULL COMMENT '填入人ID',
  `skip_time`           DATETIME     DEFAULT NULL COMMENT '跳过时间',
  `skip_by`             BIGINT       DEFAULT NULL COMMENT '跳过人ID',
  `skip_reason`         VARCHAR(200) DEFAULT NULL COMMENT '跳过原因',
  `settle_time`         DATETIME     DEFAULT NULL COMMENT '结算完成时间',
  `remark`              VARCHAR(500) DEFAULT NULL COMMENT '备注',
  `creator`             BIGINT       NOT NULL COMMENT '创建人',
  `create_time`         DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updater`             BIGINT       DEFAULT NULL,
  `update_time`         DATETIME     DEFAULT NULL ON UPDATE CURRENT_TIMESTAMP,
  `deleted`             TINYINT(1)   NOT NULL DEFAULT 0,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_policy_no` (`policy_no`),
  KEY `idx_insurer_status` (`insurer_code`, `pending_status`),
  KEY `idx_import_time` (`import_time`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='上游手续费跟单队列';
```
