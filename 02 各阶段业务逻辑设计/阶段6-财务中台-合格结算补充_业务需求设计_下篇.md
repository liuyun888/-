## FN-05 合格-上游结算完整流程

### 5.0 功能概述

本功能是针对**"合格"品类保单**的上游结算专项流程，整合合格认定、汇总统计、对账单生成、结算确认，并与上篇的"跟单结算（FN-02）"和"批量修改手续费（FN-01）"场景形成完整闭环。

**合格品类**：指已通过保司核实、满足可结算条件的保单，区别于普通保单需要额外的合格认定步骤。

---

### 5.1 Step1：合格认定

#### 5.1.1 功能入口

菜单路径：**财务 → 上游结算 → 合格认定**

#### 5.1.2 合格认定方式

系统提供两种方式让保单进入"合格"状态：

**方式A：系统自动认定（规则引擎）**

系统按可配置规则自动扫描保单，满足以下**全部条件**的保单自动标记为合格：

| 规则项 | 说明 | 可配置 |
|--------|------|-------|
| 保单状态 = 有效 | `insurance_policy.status = 'VALID'` | ❌ 固定 |
| 已回款 | `insurance_policy.payment_status = 'PAID'` | ✅ 可关闭 |
| 满足保司核实条件 | `insurance_policy.insurer_verified = 1` | ✅ 可关闭 |
| 保单录入时间超过N天 | `DATEDIFF(NOW(), sign_date) >= config_days`（默认0天，即不限制）| ✅ |
| 上游手续费不为空 | `upstream_rate IS NOT NULL AND upstream_rate > 0` | ✅ 可关闭 |

规则配置入口：**财务 → 财务配置 → 合格认定规则**，每条规则可单独启用/禁用，修改后立即生效。

自动认定通过**XXL-Job**定时任务每天凌晨2:00执行扫描，也可在页面点击【立即执行认定】手动触发。

**方式B：人工批量导入合格名单**

当保司提供合格保单清单（Excel）时，财务人员直接导入，系统标记这批保单为合格。

操作步骤：
1. 点击【批量导入合格名单】按钮
2. 下载合格名单导入模板（列：保单号、保险公司、备注）
3. 填写后上传
4. 后端校验（见5.1.3），通过后二次确认
5. 确认后批量更新保单合格状态

#### 5.1.3 批量导入合格名单校验

| 校验项 | 规则 | 不通过处理 |
|--------|------|----------|
| 保单号存在 | 在系统中能匹配到对应保单 | 标记该行错误，整批不导入或跳过该行（用户选择） |
| 保单未合格过 | 防止重复导入同一保单 | 警告提示，用户确认是否覆盖 |
| 文件格式/行数 | xlsx/xls，≤1000行 | 整体拒绝 |
| 保险公司一致 | 导入行保司与系统保单保司一致 | 标记警告 |

二次确认弹窗显示：
- 本次导入总行数 N
- 其中新合格保单 X 条
- 已有合格记录（覆盖）Y 条
- 错误行 Z 条（可下载错误明细）

确认后入库（以下写库在事务中执行）：

```sql
-- 更新保单合格状态
UPDATE insurance_policy
SET qualify_status = 'QUALIFIED',
    qualify_time   = NOW(),
    qualify_by     = #{operatorId},
    qualify_source = 'MANUAL_IMPORT'  -- 区分自动/手动
WHERE policy_no = #{policyNo};

-- 写合格认定记录（可审计）
INSERT INTO ins_qualify_record (
    policy_no, insurer_code, qualify_status,
    qualify_source, qualify_time, operator, remark
) VALUES (...);
```

#### 5.1.4 合格认定页面列表

| 列名 | 字段 |
|------|------|
| 保单号 | policy_no |
| 保险公司 | insurer_name |
| 险种 | insurance_type |
| 签单日期 | sign_date |
| 合格状态 | qualify_status（未合格/合格/已结算） |
| 合格来源 | qualify_source（AUTO/MANUAL_IMPORT）|
| 合格时间 | qualify_time |
| 操作 | 【撤销合格】（仅对"合格"状态且未进入结算的保单可用） |

支持按保司、险种、合格状态、日期范围筛选查询。

### 5.3 Step3：一键生成对账单

#### 5.3.1 触发方式

在合格汇总表中，勾选一行或多行保司数据，点击【生成对账单】按钮（或点击单行的【生成对账单】）。

#### 5.3.2 生成对账单逻辑（后端）

1. 查询勾选保司下所有**合格且未结算**的保单及手续费明细。
2. 按保司聚合，计算应得手续费合计、已收手续费合计、差异。
3. 生成对账单记录 `ins_reconcile_bill`：
   - `bill_no`：系统生成（`RB${yyyyMMdd}${6位序号}`）
   - `insurer_code`：保险公司
   - `bill_period`：对账月份
   - `total_policy_count`：涉及保单数
   - `should_amount`：应收手续费合计
   - `actual_amount`：已收手续费合计（初始为0）
   - `diff_amount`：差异金额
   - `bill_status`：`GENERATED`（已生成）
   - `generate_time`：当前时间
   - `generator`：操作人
4. 生成对账单明细 `ins_reconcile_bill_detail`（一保单一条）。

#### 5.3.3 对账单预览页面

对账单生成后弹出预览对话框（或跳转至预览页），支持**HTML预览**和**Excel下载**两种输出：

**HTML预览内容结构：**

### 5.4 Step4：结算确认（财务到账确认）

#### 5.4.1 功能入口

菜单路径：**财务 → 上游结算 → 结算记录**（在现有结算记录页扩展）

#### 5.4.2 到账确认操作

财务人员收到保司打款后，在结算记录页找到对应结算单，点击【确认到账】：

弹出确认到账对话框：

| 字段 | 类型 | 是否必填 | 说明 |
|------|------|---------|------|
| 到账金额（元） | 数字输入 | ✅ | 系统预填应收金额，可修改为实际到账金额 |
| 到账日期 | 日期选择 | ✅ | 默认今日 |
| 到账银行账户 | 下拉选择 | ✅ | 公司收款账户，从 `finance_company_account` 读取 |
| 流水号/参考号 | 文本输入 | 选填 | 银行流水号，便于核对 |
| 附件 | 文件上传 | 选填 | 上传银行回单截图/PDF，≤5个文件，每个≤10MB |
| 备注 | 文本域 | 选填 | — |

点击【确认】后：

```sql
-- 1. 更新结算单状态
UPDATE ins_settlement_upstream
SET settle_status  = 'SETTLED',
    actual_amount  = #{actualAmount},
    arrive_date    = #{arriveDate},
    bank_ref_no    = #{bankRefNo},
    confirm_time   = NOW(),
    confirm_by     = #{operatorId}
WHERE settle_no = #{settleNo};

-- 2. 批量更新明细保单的上游结算状态
UPDATE insurance_policy
SET upstream_settle_status = 'SETTLED',
    settle_time = NOW()
WHERE policy_no IN (
    SELECT policy_no FROM ins_settlement_detail WHERE settle_no = #{settleNo}
);

-- 3. 写操作日志
INSERT INTO sys_operate_log (...);
```

#### 5.4.3 差额处理（实结 ≠ 应结时）

若用户输入的"到账金额"不等于"应收金额"：
- 系统自动计算差额：`diff = actualAmount - shouldAmount`
- 显示差额说明，要求用户填写**差异原因**（必填）：
  - 选项：保司扣税、汇率差异、本期调减、其他（手动输入）
- 差额记录写入 `ins_reconcile_diff` 表，后续可进行差额跟踪

若多次部分到账（保司分批打款），支持**多次确认到账**，每次填写本次到账金额，系统累计已收金额，直到累计实收 ≥ 应收后自动将结算单状态更新为 `SETTLED`。

### 5.5 各场景与主流程的衔接说明

#### 5.5.1 跟单结算（FN-02）与本流程的衔接

```
跟单队列中的保单（PENDING_RATE）
    ↓ 财务人员填入手续费
跟单保单状态变为 SETTLED
    ↓ 后台自动更新 qualify_status
若保单满足合格认定规则 → qualify_status = QUALIFIED
    ↓ 进入 Step2 合格汇总统计
    ↓ 生成对账单 → 结算确认
```

注意：跟单手续费填入后，监管报表才会包含该保单的手续费数据（`has_tracking_settled = 1`）。

#### 5.5.2 批量修改手续费（FN-01）与本流程的衔接

```
批量修改手续费并结算 → 生成结算单（PENDING_APPROVE）
    ↓ 审批通过
结算单状态变为 SETTLED
    ↓ 系统自动将关联保单的 qualify_status 标记为 SETTLED（已结算，锁定）
    ↓ 对账单中显示该批保单已结算
```

若保单在批量修改手续费前已生成对账单（bill_status = GENERATED），则修改手续费后：
1. 对账单的差异金额自动重新计算（重新汇总）。
2. 对账单状态变更为 `UPDATED`（已更新，需财务人员重新确认）。
3. 系统推送站内信给生成对账单的操作人："对账单 RBxxxxxx 包含的保单手续费已被修改，请重新确认。"

#### 5.5.3 整体流程状态机

```
保单状态机（合格结算维度）：

NEW（录入）
    ↓ 触发合格认定规则
QUALIFIED（已合格）
    ↓ 被加入结算单/对账单
IN_SETTLE（结算中）
    ↓ 财务确认到账
SETTLED（已结算，终态）

分支：
QUALIFIED → [批量修改手续费] → IN_SETTLE → SETTLED
PENDING_RATE（跟单待费率）→ [填入手续费] → QUALIFIED → ...
QUALIFIED → [撤销合格] → NEW（仅结算前可撤销）
```

### 5.8 关键业务规则补充

#### 5.8.1 合格认定幂等性

自动认定任务每日执行，对已合格保单不重复处理（`qualify_status = 'QUALIFIED'` 的保单跳过）。

手动导入合格名单时，若保单已经是 `QUALIFIED` 或 `SETTLED` 状态：
- `QUALIFIED` 状态：警告提示"该保单已合格，是否重新导入覆盖合格时间和来源"，用户确认后覆盖。
- `SETTLED` 状态：不允许覆盖，该行标记错误。

#### 5.8.2 对账单锁定规则

对账单一旦财务确认到账（进入 `CONFIRMED` 状态），其关联的保单手续费不允许再修改。若用户尝试在FN-01批量修改这些保单的手续费，系统提示：
```
以下 X 条保单已关联已确认的对账单（单号：RBxxxxxxxxx），无法修改手续费：
保单号1、保单号2...
请先联系财务主管解锁对账单后再操作。
```

财务主管可在对账单详情页点击【解锁】，解锁后对账单状态回到 `GENERATED`，关联保单可重新修改手续费。

#### 5.8.3 监管报表数据完整性

以下保单类型**必须先完成跟单结算**才会出现在监管报表的手续费统计中：
- `upstream_rate` 为空的保单（通过 FN-02 跟单填入手续费后结算）

系统在监管报表查询逻辑中过滤条件：
```sql
-- 监管报表只统计已有手续费且已结算的保单
WHERE upstream_rate IS NOT NULL
  AND upstream_rate > 0
  AND upstream_settle_status = 'SETTLED'
```
