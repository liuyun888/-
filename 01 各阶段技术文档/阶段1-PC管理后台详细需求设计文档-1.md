## 1. 组织架构管理

### 1.1 机构管理

#### 1.1.1 功能概述

**业务目标**:
- 管理保险中介公司的总分支机构树形结构
- 支持总公司、分公司、营业部等多级机构
- 实现机构的增删改查和层级调整

**使用场景**:
- 集团型中介公司需要管理遍布全国的分支机构
- 业务数据需要按机构维度进行统计和权限隔离
- 支持机构合并、拆分等组织变更

#### 1.1.6 业务逻辑详细说明

**1. 机构新增业务逻辑**

新增机构时需要处理以下核心逻辑:

- **机构代码唯一性校验**:查询数据库中是否已存在相同的org_code(排除已逻辑删除的记录)。如果存在,抛出业务异常"机构代码已存在"
- **上级机构有效性校验**:如果parent_id不为0,需要验证父机构是否存在且状态为启用。如果父机构被停用,不允许在其下新增子机构
- **祖级列表自动构建**:根据parent_id自动计算ancestors字段。如果是顶级机构(parent_id=0),则ancestors="0";否则ancestors=父机构的ancestors+","+父机构id。这个字段用于高效查询机构的所有上级
- **负责人冗余存储**:虽然有leader_id,但同时存储leader_name字段,避免每次查询都要关联人员表。当负责人变更时需要同步更新leader_name
- **许可证有效期校验**:permit_end_date必须大于permit_start_date,并且不能选择已过期的日期
- **机构层级限制**:通过ancestors字段中的逗号数量判断层级深度,一般限制不超过5级,防止组织架构过于复杂
- **创建人和租户ID自动填充**:从当前登录上下文中获取creator和tenant_id,确保数据隔离

**2. 机构修改业务逻辑**

修改机构时的特殊处理:

- **禁止修改机构代码**:机构代码一旦创建不允许修改,因为可能被其他业务数据引用。如果需要修改,提示用户新建机构并迁移数据
- **上级机构变更处理**:如果修改了parent_id,需要重新计算当前机构及其所有子机构的ancestors字段。这是一个递归更新操作,需要使用事务保证数据一致性
- **状态变更联动**:如果将机构状态从启用改为停用,需要同时停用该机构下的所有子机构、该机构的所有业务员账号,并中止该机构的所有进行中的业务流程
- **负责人变更同步**:如果修改了leader_id,需要同步更新leader_name,并记录变更日志

**3. 机构删除业务逻辑**

机构删除是一个复杂的业务操作,需要进行多项前置校验:

- **子机构检查**:查询是否存在以当前机构为parent_id的子机构。如果存在,不允许删除,提示"存在下级机构,请先删除下级"
- **业务员关联检查**:查询biz_agent表中org_id等于当前机构id的记录数。如果大于0,不允许删除,提示"机构下存在{count}个业务员,不允许删除"
- **订单数据检查**:查询biz_order表中org_id等于当前机构id的记录数。如果大于0,不允许删除,提示"机构存在{count}条历史订单数据,不允许删除"
- **产品关联检查**:查询biz_product_org表中是否有该机构的产品授权记录。如果存在,不允许删除
- **佣金数据检查**:查询biz_commission表中是否有该机构的佣金记录。如果存在,不允许删除
- **逻辑删除而非物理删除**:所有校验通过后,执行逻辑删除(将deleted字段设为1),而不是物理删除记录,保留数据用于审计和追溯

**4. 机构树查询业务逻辑**

查询机构树时的优化处理:

- **一次性查询所有机构**:不要使用递归查询,而是一次性查询出所有启用状态的机构,然后在内存中构建树形结构。这样可以减少数据库查询次数
- **权限过滤**:根据当前登录用户的数据权限,只返回用户有权限查看的机构。如果用户是某个分公司的管理员,只返回该分公司及其子机构
- **状态过滤**:默认只返回status=1的机构,可以通过参数控制是否包含停用的机构
- **懒加载支持**:对于大型组织(机构数超过1000),支持懒加载模式,初次只加载到第二层,点击展开时再加载子节点
- **缓存优化**:机构树数据变化不频繁,可以使用Redis缓存,设置过期时间为1小时。当机构数据发生变更时,主动清除缓存

**5. 祖级列表(ancestors)维护逻辑**

ancestors字段是机构树的关键优化字段,需要严格维护:

- **新增时自动计算**:如前所述,根据parent_id自动拼接
- **修改上级时批量更新**:当某个机构的parent_id变更时,需要更新该机构及其所有子孙机构的ancestors。具体步骤:
  1. 计算当前机构的新ancestors值
  2. 查询出所有子孙机构(ancestors like '%,当前id%')
  3. 遍历这些机构,将它们ancestors中的旧前缀替换为新前缀
  4. 批量更新到数据库
- **定时校验任务**:由于ancestors是冗余字段,可能因为异常情况导致不一致。建议添加定时任务(每天凌晨执行),遍历所有机构,根据parent_id重新计算ancestors,发现不一致则修复并记录日志

**6. 机构启用/停用联动逻辑**

当机构状态变更时的联动处理:

- **停用机构时**:
  1. 将该机构下的所有子机构状态改为停用(递归)
  2. 将该机构下的所有业务员账号status改为停用
  3. 将该机构相关的产品授权记录状态改为停用
  4. 中止该机构的所有进行中的订单(标记为已取消)
  5. 发送系统通知给机构负责人和相关业务员
  
- **启用机构时**:
  1. 检查上级机构是否启用,如果上级被停用,不允许启用当前机构
  2. 仅启用当前机构,不自动启用子机构(需要手动逐个启用)
  3. 不自动启用业务员账号(由人事手动处理)

**7. 机构合并业务逻辑**

虽然基础功能中未包含合并功能,但这是常见需求:

- **合并操作流程**:
  1. 选择源机构(被合并方)和目标机构(合并到的机构)
  2. 校验:两个机构不能有上下级关系
  3. 将源机构下的所有业务员的org_id改为目标机构id
  4. 将源机构下的所有订单的org_id改为目标机构id
  5. 将源机构下的所有佣金记录的org_id改为目标机构id
  6. 将源机构的所有子机构的parent_id改为目标机构id,并重新计算ancestors
  7. 将源机构状态改为停用或删除
  8. 生成机构变更日志和业务员调动记录
  
- **数据迁移事务**:整个合并操作必须在一个事务中完成,任何步骤失败都要回滚

**8. 许可证到期预警逻辑**

保险中介机构的经营许可证有有效期,需要提前预警:

- **预警规则**:当许可证距离到期日期少于90天时,系统自动发出预警
- **预警方式**:
  1. 在管理后台首页显示预警提示
  2. 向机构负责人发送站内信和短信
  3. 向平台管理员发送邮件通知
- **定时检查**:每天凌晨执行定时任务,扫描所有机构的permit_end_date,计算距离今天的天数,生成预警记录
- **续期处理**:提供续期功能,上传新的许可证图片和有效期,更新记录并清除预警状态

| 新增部门 | POST | / | 创建部门 |
| 修改部门 | PUT | / | 更新部门信息 |
| 删除部门 | DELETE | /{id} | 删除部门 |

#### 1.2.5 业务逻辑详细说明

**1. 部门与机构的关系**

每个部门必须归属于某个机构,部门不能跨机构:

- **所属机构限制**:部门的org_id确定后不能修改。如果需要调整,应该在新机构下创建部门,然后迁移人员
- **部门树范围**:构建部门树时,必须先指定org_id,然后查询该机构下的所有部门。不同机构的部门是独立的树
- **权限控制**:用户只能查看和管理自己所属机构的部门,不能跨机构操作
- **机构停用联动**:当机构被停用时,该机构下的所有部门自动停用

**2. 部门新增业务逻辑**

- **部门代码规则**:dept_code建议采用"机构代码+部门序号"的格式,如"100001-001"表示机构100001的第1个部门
- **父部门校验**:如果指定了parent_id,需要验证父部门是否属于同一个机构。不允许将A机构的部门挂到B机构的部门下
- **ancestors自动计算**:逻辑同机构表,保证查询效率
- **负责人校验**:如果指定了leader_id,需要验证该人员是否属于当前部门所在的机构

**3. 部门删除业务逻辑**

- **子部门检查**:如果存在下级部门,不允许删除
- **人员关联检查**:查询biz_agent表中dept_id等于当前部门的记录数,如果大于0不允许删除
- **岗位关联检查**:查询biz_post表中dept_id等于当前部门的记录数,如果大于0不允许删除
- **逻辑删除**:同机构删除逻辑

**4. 部门调整业务逻辑**

当需要调整部门架构时:

- **部门合并**:将A部门合并到B部门,需要将A部门下的所有人员、岗位、子部门转移到B部门,然后删除A部门
- **部门拆分**:从A部门拆分出B部门,需要先创建B部门,然后将部分人员和岗位转移到B部门
- **部门调动**:将A部门从机构X调整到机构Y,需要修改A部门及其所有子部门的org_id,同时调整相关人员的org_id

#### 1.3.5 业务逻辑详细说明

**1. 岗位设计原则**

岗位是组织管理的核心概念,设计时需要注意:

- **岗位vs部门**:岗位是职责的抽象,部门是组织的划分。一个部门可以有多个岗位,一个岗位可以存在于多个部门
- **岗位vs角色**:岗位偏业务属性,角色偏系统权限。岗位决定你做什么工作,角色决定你在系统中能做什么操作。一个岗位通常对应一个或多个角色
- **岗位层级**:虽然表结构未设计parent_id,但实际业务中岗位是有层级的(如销售专员<销售主管<销售总监)。可以通过post_code编码规则体现,也可以扩展表结构增加层级字段

**2. 岗位与人员的关系**

人员与岗位是多对多关系,需要中间表:

```sql
CREATE TABLE `biz_agent_post` (
  `id` BIGINT NOT NULL AUTO_INCREMENT,
  `agent_id` BIGINT NOT NULL COMMENT '业务员ID',
  `post_id` BIGINT NOT NULL COMMENT '岗位ID',
  `is_primary` TINYINT NOT NULL DEFAULT 0 COMMENT '是否主岗位:0-否 1-是',
  `creator` VARCHAR(64) DEFAULT '',
  `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_agent_post` (`agent_id`, `post_id`),
  KEY `idx_post` (`post_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='业务员岗位关系表';
```

- **主岗位标识**:一个人可以兼任多个岗位,但必须有且仅有一个主岗位(is_primary=1)
- **岗位变更**:当人员调岗时,需要删除旧的agent_post记录,新增新的记录,并生成岗位变更日志
- **兼岗管理**:如果允许兼岗,需要在业务规则中明确兼岗的数量限制(如最多兼任3个岗位)

**3. 岗位删除业务逻辑**

- **人员关联检查**:查询biz_agent_post表中是否有该岗位的关联记录,如果存在不允许删除,提示"该岗位下有{count}个人员,请先调整人员岗位"
- **角色关联检查**:如果系统中有"岗位-角色"关联表,需要检查是否有角色绑定,存在则不允许删除
- **建议停用而非删除**:对于已经使用过的岗位,建议停用(status=0)而不是删除,保留历史数据

**4. 岗位与权限的关联**

岗位可以直接关联系统角色,实现基于岗位的权限管理(RBAC):

- **岗位角色关联表**:
```sql
CREATE TABLE `biz_post_role` (
  `id` BIGINT NOT NULL AUTO_INCREMENT,
  `post_id` BIGINT NOT NULL COMMENT '岗位ID',
  `role_id` BIGINT NOT NULL COMMENT '角色ID',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_post_role` (`post_id`, `role_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='岗位角色关系表';
```

- **权限计算逻辑**:用户的最终权限=个人角色权限 ∪ 岗位角色权限。即取并集
- **岗位权限继承**:如果岗位有层级关系,下级岗位自动继承上级岗位的权限

**5. 岗位类别管理**

post_category字段将岗位分为5类,不同类别的岗位有不同的管理规则:

- **管理岗(1)**:如总经理、部门经理等,通常具有较高的数据权限和审批权限
- **销售岗(2)**:如业务员、销售主管等,核心是销售业绩管理和佣金计算
- **职能岗(3)**:如人事、财务、客服等,有专门的职能系统权限
- **技术岗(4)**:如IT运维、开发等,有系统管理权限
- **其他(5)**:临时工、实习生等

根据类别可以设置不同的默认角色模板,新增岗位时自动关联对应角色。

#### 2.1.5 业务逻辑详细说明

**1. 业务员新增业务逻辑**

新增业务员是一个涉及多表操作的复杂流程:

- **唯一性校验**:
  1. 检查agent_code是否已存在(排除逻辑删除的记录)
  2. 检查id_card是否已存在,防止同一人重复入职
  3. 检查mobile是否已存在,一个手机号只能绑定一个业务员

- **创建系统用户账号**:
  1. 在system_users表中创建一条用户记录,username使用agent_code或mobile
  2. 密码使用系统配置的默认密码(如123456),加密后存储
  3. 设置user_type为业务员类型(如2)
  4. 记录创建的user_id,用于关联

- **业务员信息入库**:
  1. 将业务员基本信息插入biz_agent表
  2. user_id字段关联到刚创建的系统用户
  3. org_id从当前登录用户的机构权限中获取或手动选择
  4. status默认为0(待激活),需要业务员首次登录修改密码后才激活

- **岗位关联**:
  1. 如果指定了岗位,在biz_agent_post表中插入关联记录
  2. is_primary字段标记为1,表示这是主岗位

- **发送激活通知**:
  1. 向业务员手机号发送短信,告知工号和初始密码
  2. 发送站内信,引导首次登录修改密码
  3. 记录通知发送日志

- **生成入职档案**:
  1. 在人员档案表中创建一条档案记录
  2. 包含入职日期、推荐人、合同信息等

**2. 业务员修改业务逻辑**

修改业务员信息时需要注意:

- **敏感字段限制**:
  1. agent_code、id_card、mobile等关键字段一旦创建不允许修改
  2. 如果确需修改,需要特殊审批流程,并记录变更日志

- **机构部门变更**:
  1. 如果修改了org_id或dept_id,视为调动
  2. 需要在biz_agent_transfer表中记录调动历史
  3. 同时更新业务员的数据权限范围

- **银行卡信息变更**:
  1. 银行卡号关系到佣金发放,变更需要审批
  2. 记录变更前后的值,生成变更日志
  3. 变更后,未发放的佣金按新卡号发放

- **联动更新**:
  1. 如果修改了real_name,同步更新system_users表的nickname
  2. 如果修改了mobile,同步更新system_users表的mobile
  3. 如果修改了avatar,同步更新system_users表的avatar

**3. 业务员状态流转逻辑**

业务员有5种状态,状态之间的流转规则如下:

```
待激活(0) -> 正常(1):业务员首次登录并修改密码
正常(1) -> 停用(2):管理员手动停用
停用(2) -> 正常(1):管理员手动启用
正常(1) -> 离职(3):办理离职手续
正常(1)/停用(2) -> 黑名单(4):违规行为,加入黑名单
黑名单(4) -> 正常(1):申诉成功,解除黑名单
```

每种状态的业务规则:

- **待激活(0)**:
  - 账号已创建,但业务员未登录过
  - 不能进行任何业务操作
  - 不计入在职人数统计
  - 超过30天未激活自动失效

- **正常(1)**:
  - 可以正常登录系统
  - 可以进行业务操作(报价、出单等)
  - 可以参与佣金分配
  - 计入在职人数和绩效考核

- **停用(2)**:
  - 不能登录系统
  - 不能进行新的业务操作
  - 已有业务继续跟进(如理赔)
  - 暂停佣金发放,恢复后补发
  - 不计入在职人数

- **离职(3)**:
  - 不能登录系统
  - 所有业务移交给其他业务员
  - 结算应发未发的佣金
  - 从在职人数中移除
  - 保留历史数据用于审计

- **黑名单(4)**:
  - 永久禁止登录系统
  - 冻结所有未发佣金
  - 列入行业黑名单,其他机构可查询
  - 标记原因和证据

**4. 业务员离职业务逻辑**

离职是一个复杂的业务流程,涉及多个环节:

- **前置校验**:
  1. 检查是否有未完成的订单(状态为进行中)
  2. 检查是否有未结算的佣金
  3. 检查是否有正在处理的理赔案件
  4. 如果有以上情况,提示"请先处理完XXX再办理离职"

- **业务移交**:
  1. 弹出客户移交界面,选择接手人
  2. 将离职业务员的客户关系转移给接手人(更新biz_customer表的agent_id)
  3. 将未完成订单的跟进人改为接手人
  4. 将正在处理的理赔案件移交
  5. 生成移交清单,双方签字确认

- **佣金结算**:
  1. 计算离职业务员的应发未发佣金总额
  2. 生成离职佣金结算单
  3. 财务审核通过后发放
  4. 如果有佣金扣款(如退保扣佣),在结算时扣除

- **账号处理**:
  1. 将status改为3(离职)
  2. 填写leave_date为当前日期
  3. 将关联的system_users记录status改为停用
  4. 清空角色权限

- **档案归档**:
  1. 生成离职证明
  2. 归档劳动合同、佣金记录、业绩报表等
  3. 在人员档案表中记录离职原因、离职类型(主动/被动)

- **后续处理**:
  1. 离职后,如果业务员名下的客户发生退保,需要追回已发佣金
  2. 离职后6个月内,如果名下保单出险理赔,业务员有义务协助处理
  3. 离职后1年内,禁止到竞争对手公司入职(根据竞业协议)

**5. 业务员黑名单逻辑**

将业务员加入黑名单是最严厉的处罚:

- **加入黑名单的场景**:
  1. 严重违规:如私自收取客户保费、伪造保单等
  2. 欺诈行为:虚假承保、骗取佣金等
  3. 泄露客户隐私
  4. 恶意诋毁公司
  5. 其他严重违反公司规定的行为

- **加入流程**:
  1. 合规部门发起黑名单申请,提交证据材料
  2. 法务部门审核
  3. 高管审批
  4. 生效后,status改为4,记录黑名单原因和证据
  5. 冻结所有未发佣金

- **黑名单数据同步**:
  1. 上报到中国保险行业协会的代理人黑名单数据库
  2. 同步到集团内部其他分公司,防止其他分公司误招
  3. 同步到第三方背景调查平台

- **解除黑名单**:
  1. 业务员可提起申诉,提交证据
  2. 复核小组重新审查
  3. 如果认定处罚过重,可以解除黑名单
  4. 解除后status恢复为1或2,解冻佣金

**6. 业务员数据权限逻辑**

业务员的数据权限决定了他能看到哪些数据:

- **数据权限级别**:
  1. 仅本人:只能看到自己的客户和订单
  2. 本部门:可以看到所在部门所有业务员的数据
  3. 本机构:可以看到所在机构所有业务员的数据
  4. 全部:可以看到所有机构的数据(仅限超级管理员)

- **数据权限配置**:
  1. 普通业务员默认"仅本人"
  2. 部门主管配置"本部门"
  3. 机构经理配置"本机构"
  4. 集团总部配置"全部"

- **权限实现**:
  1. 在查询SQL中动态拼接WHERE条件
  2. 根据当前登录用户的数据权限级别,过滤org_id、dept_id、agent_id
  3. 使用MyBatis拦截器实现自动注入

**7. 业务员调动逻辑**

业务员在机构间或部门间调动:

- **调动类型**:
  1. 跨机构调动:从机构A调到机构B
  2. 机构内调动:在同一机构内,从部门X调到部门Y
  3. 晋升调动:从普通岗位晋升到管理岗位

- **调动流程**:
  1. 调出方主管发起调动申请
  2. 调入方主管确认接收
  3. HR审核
  4. 高管审批(跨机构调动需要)
  5. 审批通过后,修改org_id或dept_id
  6. 在biz_agent_transfer表中记录调动历史

- **调动影响**:
  1. 数据权限范围变更
  2. 客户归属变更(调动后,老客户是否跟随?)
  3. 佣金归属变更(调动后成交的订单,佣金归调入方机构)
  4. 业绩统计:调动前的业绩计入调出方,调动后计入调入方

**8. 业务员导入逻辑**

批量导入业务员,加快入职效率:

- **Excel模板**:
  - 提供标准Excel模板下载
  - 包含必填字段:姓名、身份证、手机号、性别、入职日期、机构代码
  - 可选字段:部门代码、岗位代码、邮箱、银行卡号等

- **导入校验**:
  1. 文件格式校验:仅支持.xlsx格式,大小不超过5MB
  2. 数据格式校验:身份证15/18位,手机号11位等
  3. 业务规则校验:
     - 身份证、手机号不能重复(包括已有数据)
     - 机构代码、部门代码必须存在
     - 入职日期不能早于机构成立日期
  4. 汇总校验结果,区分"校验通过"和"校验失败"的记录

- **导入执行**:
  1. 仅导入校验通过的记录
  2. 使用批量插入,提高性能
  3. 每条记录都创建关联的system_users账号
  4. 生成导入日志,记录成功/失败条数
  5. 将失败的记录导出为Excel,告知失败原因

- **导入后处理**:
  1. 批量发送激活短信
  2. 生成导入报告,邮件通知HR

#### 2.2.5 业务逻辑详细说明

**1. 证书提交认证流程**

业务员提交证书认证的完整流程:

- **图片上传**:
  1. 前端使用文件上传组件,选择证书图片
  2. 调用文件上传接口(框架自带),上传到OSS
  3. 返回图片URL,填入cert_image字段

- **OCR识别**:
  1. 调用OCR接口(如百度OCR、腾讯OCR),识别证书图片中的文字
  2. 提取关键信息:证书编号、姓名、发证日期、有效期等
  3. 自动填充表单,业务员确认或修改

- **信息校验**:
  1. 校验证书编号格式(如代理人证为18位)
  2. 校验有效期不能早于今天
  3. 检查该证书编号是否已被其他业务员使用

- **提交审核**:
  1. 将认证记录插入数据库,status=1(待审核)
  2. 向合规部门发送待审核通知
  3. 业务员收到"提交成功,等待审核"的提示

**2. 证书审核逻辑**

合规部门审核证书的流程:

- **审核入口**:
  1. 合规人员登录管理后台
  2. 进入"业务员认证审核"页面
  3. 看到所有status=1(待审核)的认证申请

- **审核操作**:
  1. 查看证书图片,核对信息真实性
  2. 到保险行业协会网站查询证书编号是否真实有效
  3. 检查证书姓名是否与业务员真实姓名一致
  4. 检查证书有效期是否在有效期内

- **审核通过**:
  1. status改为2(已认证)
  2. 填写审核人、审核时间
  3. 在biz_agent表中标记该业务员已认证(增加字段:is_certified=1)
  4. 向业务员发送通过通知

- **审核拒绝**:
  1. status改为3(已拒绝)
  2. 填写拒绝原因(如"证书图片模糊,请重新上传")
  3. 向业务员发送拒绝通知,告知原因
  4. 业务员可以修改信息后重新提交

**3. 证书到期预警逻辑**

证书有有效期,到期前需要提醒续证:

- **预警规则**:
  1. 证书距离到期日期少于30天时,系统自动预警
  2. 证书已过期,业务员不能进行业务操作

- **预警方式**:
  1. 在业务员个人中心首页显示红色预警提示
  2. 每天早上8点,向到期业务员发送短信提醒
  3. 向业务员的直属主管发送邮件,告知团队中有哪些证书即将到期
  4. 在管理后台显示到期证书列表,HR可导出跟进

- **定时任务**:
  1. 每天凌晨1点执行定时任务
  2. 查询expire_date在30天内的认证记录
  3. 生成预警记录到biz_cert_warning表
  4. 触发短信和邮件发送

- **过期处理**:
  1. 证书过期后(expire_date < 今天),业务员不能报价和出单
  2. 前端在业务操作入口进行拦截,提示"证书已过期,请续证"
  3. 后端在业务接口中增加证书有效期校验
  4. 过期后已成交但未完成的订单,可以继续跟进售后

**4. 证书续期逻辑**

证书到期后需要续期:

- **续期方式**:
  1. 业务员参加继续教育培训,考试合格后获得新证
  2. 上传新证书图片,提交认证
  3. 审核通过后,旧证书记录status不变,增加新的认证记录
  4. 系统始终使用最新的有效认证记录

- **历史记录保留**:
  1. 旧证书记录不删除,保留历史
  2. 在查询时,优先使用expire_date最晚的认证记录
  3. 业务员可以在个人中心查看自己的所有认证历史

**5. 多证书管理逻辑**

一个业务员可能持有多种证书:

- **证书类型**:
  1. 代理人证:允许销售保险产品
  2. 经纪人证:可以代表客户选择保险产品
  3. 公估人证:可以从事保险理赔勘察和评估

- **业务权限**:
  1. 只有代理人证:只能销售本公司代理的保险产品
  2. 同时持有代理人证和经纪人证:可以销售多家保险公司的产品
  3. 公估人证:主要用于理赔业务,不能销售

- **认证状态判断**:
  1. 如果业务员有至少一个已认证且未过期的证书,视为已认证
  2. 在biz_agent表中增加字段:cert_status(0-未认证 1-已认证 2-已过期)
  3. 定时任务每天更新cert_status字段

**6. 证书真伪验证逻辑**

为了防止伪造证书,需要对接行业数据库验证:

- **验证接口**:
  1. 对接中国保险行业协会的代理人信息查询接口
  2. 输入证书编号和姓名,查询该证书是否真实存在
  3. 返回证书状态:有效、注销、吊销等

- **自动验证**:
  1. 在审核环节,点击"验证真伪"按钮
  2. 调用行业协会接口进行查询
  3. 显示查询结果,辅助审核人员判断

- **异常处理**:
  1. 如果查询结果为"证书不存在",自动拒绝认证
  2. 如果查询结果为"证书已注销",提示业务员重新考证
  3. 如果查询接口调用失败,记录日志,人工审核

**7. 证书数据统计**

为了管理需要,提供证书数据统计:

- **统计维度**:
  1. 按机构统计:各机构的已认证/未认证业务员数量
  2. 按证书类型统计:代理人证、经纪人证、公估人证的数量
  3. 按有效期统计:30天内到期、60天内到期、90天内到期的证书数
  4. 按认证状态统计:待审核、已认证、已拒绝的数量

- **数据可视化**:
  1. 在管理后台首页显示证书统计看板
  2. 使用饼图、柱状图展示各维度数据
  3. 点击图表可以下钻查看明细

  2. 包含课程ID、培训日期、考试成绩等
  3. 业务员可以在个人中心查看自己的培训历史

**2. 继续教育培训**

在职业务员每年需要参加一定学时的继续教育:

- **学时要求**:

  3. 通知业务员参加

- **学时统计**:
  1. 根据培训记录,统计每个业务员的年度培训学时
  2. 未完成学时要求的业务员,年底考核扣分
  3. 提供培训学时报表,供监管检查

**3. 产品培训**

新产品上线前,需要对业务员进行产品培训:

- **培训目标**:
  1. 了解产品特点和保障范围

- **培训方式**:
  1. 线上视频课程

- **培训后授权**:
  1. 产品培训考试合格后,才能销售该产品

#### 3.1.5 业务逻辑详细说明

**1. 产品新增业务逻辑**

新增保险产品时的处理流程:

- **产品代码规则**:
  1. product_code采用"保险公司代码+险种代码+序号"的格式
  2. 如"PICC-CAR-001"表示人保财险车险产品001
  3. 系统自动校验代码唯一性

- **保险公司关联**:
  1. insurance_company_id必须是biz_insurance_company表中已存在的记录
  2. 外键约束保证数据完整性

- **投保规则校验**:
  1. 年龄上限必须大于年龄下限
  2. 最高保额必须大于最低保额
  3. 最高保费必须大于最低保费
  4. 保障期限格式校验(如"1年"、"至70岁"、"终身"等)

- **条款文件上传**:
  1. 上传PDF格式的保险条款
  2. 存储到OSS,保存文件URL
  3. 提供条款文件下载功能

- **产品默认状态**:
  1. 新增产品默认status=0(已下架),需要审核通过后才能上架
  2. 产品经理审核产品信息无误后,手动上架

**2. 产品上下架业务逻辑**

产品上下架是重要的业务操作:

- **上架前置条件**:
  1. 产品信息完整,必填字段都已填写
  2. 条款文件已上传
  3. 费率表已配置(如果是费率型产品)
  4. 产品已通过审核

- **上架操作**:
  1. 修改status=1(在售)
  2. 记录上架时间和操作人
  3. 向业务员推送新品上线通知
  4. 在C端商城展示该产品

- **下架操作**:
  1. 修改status=0(已下架)
  2. 已出单但未生效的保单继续生效
  3. 业务员不能再报价该产品
  4. C端商城隐藏该产品
  5. 记录下架原因(如"产品停售"、"费率调整中"等)

- **下架影响**:
  1. 进行中的报价单自动作废
  2. 购物车中的该产品自动移除
  3. 业务员收到产品下架通知

**3. 产品信息修改业务逻辑**

产品上架后,某些信息可能需要调整:

- **允许修改的字段**:
  1. product_intro、product_detail等描述性字段可以随时修改
  2. 修改后立即生效,无需审核

- **限制修改的字段**:
  1. product_code、insurance_type等关键字段不允许修改
  2. 如果确需修改,建议下架旧产品,创建新产品

- **影响定价的字段**:
  1. 保额范围、年龄范围等影响定价的字段修改后,需要重新计算费率
  2. 修改后需要产品经理审核
  3. 审核通过前,产品自动下架

**4. 产品与机构的授权逻辑**

并非所有机构都可以销售所有产品,需要授权管理:

- **授权表设计**:
```sql
CREATE TABLE `biz_product_org` (
  `id` BIGINT NOT NULL AUTO_INCREMENT,
  `product_id` BIGINT NOT NULL COMMENT '产品ID',
  `org_id` BIGINT NOT NULL COMMENT '机构ID',
  `commission_rate` DECIMAL(5,4) NOT NULL COMMENT '佣金比例(如0.1500表示15%)',
  `status` TINYINT NOT NULL DEFAULT 1 COMMENT '状态:0-停用 1-启用',
  `start_date` DATE NOT NULL COMMENT '授权开始日期',
  `end_date` DATE DEFAULT NULL COMMENT '授权结束日期',
  `creator` VARCHAR(64) DEFAULT '',
  `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_product_org` (`product_id`, `org_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='产品机构授权表';
```

- **授权逻辑**:
  1. 保险公司将产品授权给某些机构销售
  2. 未授权的机构,业务员看不到该产品
  3. 授权时可以设置该机构的佣金比例
  4. 授权有有效期,过期后自动失效

- **授权管理**:
  1. 产品经理在产品详情页点击"机构授权"
  2. 勾选要授权的机构,设置佣金比例和有效期
  3. 批量授权或单个授权
  4. 可以随时取消授权或调整佣金比例

**5. 产品费率管理逻辑**

保险产品的费率是核心,需要单独管理:

- **费率表设计**:
  - 对于简单产品(如意外险),费率可以直接配置在产品表中
  - 对于复杂产品(如车险),需要单独的费率表
  - 费率表根据多个因子计算(如年龄、性别、保额、保障期限等)

- **费率表结构**(以健康险为例):
```sql
CREATE TABLE `biz_product_rate` (
  `id` BIGINT NOT NULL AUTO_INCREMENT,
  `product_id` BIGINT NOT NULL COMMENT '产品ID',
  `age_min` INT NOT NULL COMMENT '年龄下限',
  `age_max` INT NOT NULL COMMENT '年龄上限',
  `gender` TINYINT DEFAULT NULL COMMENT '性别:1-男 2-女 NULL-不限',
  `coverage_amount` DECIMAL(15,2) NOT NULL COMMENT '保额',
  `coverage_period` VARCHAR(20) NOT NULL COMMENT '保障期限',
  `premium` DECIMAL(10,2) NOT NULL COMMENT '保费',
  PRIMARY KEY (`id`),
  KEY `idx_product` (`product_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='产品费率表';
```

- **费率计算**:
  1. 根据投保人的年龄、性别、所选保额等条件
  2. 查询费率表,匹配对应的费率记录
  3. 返回应缴保费

- **费率调整**:
  1. 保险公司调整费率后,需要更新费率表
  2. 费率调整生效日期之后的新单使用新费率
  3. 已出单的保单不受影响,续保时使用新费率

**6. 产品推荐逻辑**

为了提高销售转化率,需要产品推荐功能:

- **推荐规则**:
  1. 根据客户画像(年龄、职业、收入等)推荐合适的产品
  2. 热销产品优先推荐
  3. 佣金高的产品优先推荐(但不能误导客户)

- **个性化推荐**:
  1. 记录客户的浏览历史、咨询历史
  2. 使用协同过滤算法,推荐相似客户购买的产品
  3. 在C端商城首页展示推荐产品

**7. 产品库存管理逻辑**(针对限量产品)

某些保险产品有销售限额:

- **限额场景**:
  1. 保险公司对某些高风险产品设置销售限额
  2. 促销活动限量销售

- **库存表设计**:
```sql
CREATE TABLE `biz_product_stock` (
  `id` BIGINT NOT NULL AUTO_INCREMENT,
  `product_id` BIGINT NOT NULL COMMENT '产品ID',
  `total_stock` INT NOT NULL COMMENT '总库存',
  `sold_stock` INT NOT NULL DEFAULT 0 COMMENT '已售库存',
  `lock_stock` INT NOT NULL DEFAULT 0 COMMENT '锁定库存(待支付订单)',
  `available_stock` INT GENERATED ALWAYS AS (total_stock - sold_stock - lock_stock) STORED COMMENT '可用库存',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_product` (`product_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='产品库存表';
```

- **库存扣减**:
  1. 客户提交订单时,锁定库存(lock_stock+1)
  2. 客户支付成功,锁定库存转为已售库存(lock_stock-1,sold_stock+1)
  3. 订单超时未支付,释放锁定库存(lock_stock-1)
  4. 库存不足时,不允许下单

  4. 佣金金额:累计佣金、待结算佣金
  5. 出单成功率:报价数 vs 成交数

## 6. 附录

### 6.1 通用技术规范

#### 6.1.1 接口规范

- 使用RESTful风格
- 统一返回格式:CommonResult<T>
- 统一异常处理:GlobalExceptionHandler
- 参数校验:使用@Valid + BindingResult

#### 6.1.2 命名规范

- **数据库表名**:小写 + 下划线,前缀biz_
- **Java类名**:大驼峰
- **变量/方法名**:小驼峰
- **常量名**:大写 + 下划线

#### 6.1.3 代码分层

```
controller/   // 控制器层
├── dto/      // 数据传输对象
├── vo/       // 视图对象
service/      // 业务逻辑层
├── impl/     // 实现类
mapper/       // 数据访问层(MyBatis)
entity/       // 实体类(对应数据库表)
enums/        // 枚举类
convert/      // 对象转换类
```

---

### 6.2 数据库设计规范

1. **字段必备**:creator, create_time, updater, update_time, deleted, tenant_id
2. **索引规范**:
   - 主键:id (BIGINT AUTO_INCREMENT)
   - 唯一索引:uk_前缀
   - 普通索引:idx_前缀
3. **字符集**:utf8mb4
4. **引擎**:InnoDB
5. **注释**:每个表和字段必须添加COMMENT

---

### 6.3 安全规范

1. **敏感数据加密存储**:身份证、银行卡号使用AES加密
2. **密码加密**:使用BCrypt
3. **数据脱敏**:日志、导出文件中的敏感数据必须脱敏
4. **权限校验**:所有接口必须进行权限校验
5. **SQL注入防护**:使用MyBatis参数绑定,禁止字符串拼接
