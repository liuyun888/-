## 一、自动对账模块

### 1.1 功能概述

自动对账模块负责将保险公司下发的保单数据（通过Excel导入）与系统内已有的业务员投保订单进行自动匹配比对，识别差异并推送人工处理，最终生成对账报表。核心流程：**导入 → 匹配 → 差异标记 → 人工处理 → 出报表**。

### 1.2 保单导入

#### 1.2.1 功能入口

PC管理后台左侧菜单 → 财务中台 → 自动对账 → 保单导入，进入保单导入列表页。

#### 1.2.2 导入列表页

| 导入文件名 | 上传的Excel文件名 |
| 保单总数 | 本批次导入保单数量 |
| 匹配成功数 | 智能匹配后成功匹配的保单数 |
| 差异数量 | 存在差异的保单数 |

| 导入时间 | 操作时间 |
| 操作 | 查看详情 / 下载原文件 / 重新匹配 |

列表顶部右侧提供 **"导入保单"** 按钮。

#### 1.2.3 点击"导入保单"按钮

弹出导入对话框，包含以下字段：

| 字段 | 类型 | 是否必填 | 说明 |
|------|------|---------|------|
| 保险公司 | 下拉选择 | 必填 | 选择本次导入数据对应的保险公司 |
| 对账月份 | 月份选择器 | 必填 | 本次对账对应的自然月，格式YYYY-MM |
| 导入模板 | 下载链接 | - | 点击可下载对应保险公司的Excel模板 |
| 上传文件 | 文件上传 | 必填 | 支持.xlsx/.xls，单次最大20MB |
| 备注 | 文本框 | 非必填 | 最多200字 |

点击"确认导入"按钮后：
1. 前端校验必填项，未填则在字段下方提示红字"请填写XX"。
2. 校验文件格式，非.xlsx/.xls弹出提示"请上传Excel格式文件"。
3. 校验通过后，上传文件并提交，弹出"导入任务已提交，系统将在后台处理，可在列表中查看进度"提示框，点击"确定"关闭对话框，列表刷新。

#### 1.2.4 后端导入处理逻辑

**接收文件（同步部分）：**
1. 校验文件大小（≤20MB）、格式（xlsx/xls）。
2. 生成导入批次号（IMP + yyyyMMdd + 4位序号，Redis INCR保证序号唯一）。
3. 将文件存储至OSS（路径：finance/reconciliation/import/年/月/批次号.xlsx）。
4. 在数据库中插入导入记录，状态为"待处理"，立即返回批次号给前端。

**异步处理部分（MQ消费者）：**
1. 更新状态为"处理中"。
2. 使用EasyExcel读取文件，逐行解析，每1000行提交一次事务写库（finance_policy_import_detail表）。
3. 解析规则：
   - 必须列：保单号、投保人姓名、投保人身份证/手机号、险种类型、保费金额、起保日期、保险公司保单流水号。
   - 空行自动跳过，表头行自动识别（第一行为表头）。
   - 金额列强制转为BigDecimal，非数字格式记录行错误信息。
   - 日期列解析失败记录行错误信息，不中断整体导入。
4. 解析完成后，更新导入记录：总数、成功数、失败数、状态改为"处理完成"（有失败行则"部分失败"）。
5. 自动触发智能匹配流程（见1.3节）。

### 1.3 智能匹配

#### 1.3.1 触发方式

有两种触发方式：
1. **自动触发**：保单导入处理完成后，系统自动发起匹配任务（通过MQ消息）。
2. **手动触发**：在导入批次列表中，点击"重新匹配"按钮，重新发起匹配（覆盖上次匹配结果）。

#### 1.3.2 匹配算法

匹配引擎从高到低按优先级依次执行以下规则，一旦某规则匹配成功则停止：

**规则一（精确匹配，优先级最高）：**
- 条件：导入数据的`insurer_policy_no`（保险公司保单号）= 系统订单表中的`insurer_policy_no`字段，且`insurer_id`相同。
- 若找到唯一结果 → 标记为"已匹配"（match_status=1）。
- 若找到多条（数据异常）→ 标记为"差异-重复匹配"。

**规则二（投保人+保费匹配）：**
- 条件：`policy_holder_name` = 订单投保人姓名，`premium` = 订单保费（精确相等），`start_date` = 订单起保日期，`insurer_id`相同。
- 若找到唯一结果 → 标记为"已匹配"。
- 若找到多条 → 标记为"差异-多条匹配"。

**规则三（手机号+险种+保费模糊匹配）：**
- 条件：`mobile` = 订单投保人手机号，`product_type` 大类相同，保费差异≤5%（`ABS(imported_premium - order_premium) / order_premium <= 0.05`）。
- 若找到唯一结果 → 标记为"已匹配（模糊）"，差异类型记录为"金额差异"。

**规则四（身份证+险种匹配）：**
- 条件：`id_card_no` = 订单身份证号，`product_type` 大类相同，`start_date`相差≤3天。
- 若找到唯一结果 → 标记为"已匹配（模糊）"。

**规则五（未匹配）：**
- 以上规则均不能匹配 → 标记为"无法匹配"（match_status=3）。

#### 1.3.3 差异类型定义

匹配到订单但数据存在差异时，记录差异类型（可多选，逗号分隔）：

| 差异类型代码 | 差异描述 |
|-------------|---------|
| PREMIUM_DIFF | 保费金额差异（导入值≠系统值） |
| COMMISSION_DIFF | 佣金金额差异 |
| DATE_DIFF | 起保/止保日期差异 |
| POLICY_NO_DIFF | 保单号差异（模糊匹配但号码不同） |
| NO_MATCH | 保险公司有该保单但系统无对应订单 |
| DUPLICATE | 重复保单（同一保单号导入多次） |

#### 1.3.4 后端实现要点

1. 匹配在独立Spring Batch Job或CompletableFuture中运行，不阻塞主线程。
2. 每1000条导入记录作为一个分片并行处理，使用ThreadPoolTaskExecutor。
3. 匹配结果写入`finance_import_detail`的match_status、system_order_id字段，差异信息写入`finance_reconcile_diff`表。
4. 处理完成后更新batch的match_count、diff_count、状态。

```sql
-- 对账差异记录表
CREATE TABLE finance_reconcile_diff (
  id                  BIGINT PRIMARY KEY AUTO_INCREMENT,
  batch_id            BIGINT NOT NULL COMMENT '批次ID',
  import_detail_id    BIGINT NOT NULL COMMENT '导入明细ID',
  system_order_id     BIGINT COMMENT '系统订单ID（可为空，表示无法匹配）',
  diff_type           VARCHAR(200) COMMENT '差异类型，逗号分隔',
  import_premium      DECIMAL(12,2) COMMENT '导入保费',
  system_premium      DECIMAL(12,2) COMMENT '系统保费',
  import_commission   DECIMAL(12,2) COMMENT '导入佣金',
  system_commission   DECIMAL(12,2) COMMENT '系统佣金',
  import_start_date   DATE COMMENT '导入起保日期',
  system_start_date   DATE COMMENT '系统起保日期',
  process_status      TINYINT DEFAULT 0 COMMENT '0待处理 1已处理 2已忽略',
  process_user_id     BIGINT COMMENT '处理人ID',
  process_time        DATETIME COMMENT '处理时间',
  process_remark      VARCHAR(500) COMMENT '处理备注',
  creator             BIGINT,
  create_time         DATETIME DEFAULT CURRENT_TIMESTAMP,
  updater             BIGINT,
  update_time         DATETIME,
  deleted             TINYINT DEFAULT 0,
  tenant_id           BIGINT
) ENGINE=InnoDB COMMENT='财务对账-差异记录';
```

### 1.5 人工处理差异

#### 1.5.1 操作入口

在差异列表中，点击"处理"按钮，右侧弹出侧滑面板（宽度800px），展示差异详情。

#### 1.5.2 侧滑面板内容

**上半部分：对比视图（左右分栏）**

| 字段 | 保险公司数据（左） | 系统数据（右） |
|------|----------|---------|
| 保单号 | 显示，若不同则红色高亮 | 显示，若不同则红色高亮 |
| 投保人姓名 | 显示 | 显示 |
| 保费金额 | 显示，若不同则红色高亮 | 显示，若不同则红色高亮 |
| 佣金金额 | 显示，若不同则红色高亮 | 显示，若不同则红色高亮 |
| 起保日期 | 显示 | 显示 |
| 业务员 | - | 显示 |

若为"无法匹配"类型，右侧显示"系统中无对应订单"，并提供**手动关联**按钮（弹出搜索订单弹窗）。

**下半部分：处理区**

| 字段 | 类型 | 是否必填 | 说明 |
|------|------|---------|------|
| 处理方式 | 单选 | 必填 | 以系统数据为准 / 以保险公司数据为准 / 手动输入修正值 / 忽略此差异 |
| 修正保费 | 数字输入 | 条件必填 | 选择"手动输入修正值"时必填 |
| 修正佣金 | 数字输入 | 条件必填 | 选择"手动输入修正值"时必填 |
| 处理备注 | 文本域 | 非必填 | 最多500字 |

底部操作按钮：**确认处理** / **取消**。

#### 1.5.3 后端处理逻辑

点击"确认处理"后：

1. 后端校验：process_status必须为0（待处理），否则返回"该差异已被处理"错误。
2. 根据处理方式执行不同逻辑：
   - **以系统数据为准**：保留系统订单数据不变，标记差异为已忽略的修正类型，更新diff的process_status=1。
   - **以保险公司数据为准**：更新系统订单的`premium`、`commission_amount`等字段（保留原值备份到订单变更记录），更新diff的process_status=1。
   - **手动输入修正值**：更新系统订单的相关字段为修正值，更新diff。
   - **忽略此差异**：直接标记process_status=2（已忽略），不修改订单数据。
3. 记录处理人（process_user_id）、处理时间、备注。
4. 检查该批次是否所有差异已处理完毕，若是则更新batch状态为"对账完成"。

#### 1.5.4 批量操作

差异列表支持勾选多条记录，点击顶部"批量忽略"按钮，将选中的差异全部标记为已忽略（仅适用于处理状态为"待处理"的记录）。最多一次批量操作200条。
