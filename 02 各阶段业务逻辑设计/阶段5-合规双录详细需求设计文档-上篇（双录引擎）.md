## 二、功能模块：双录触发判断

### 2.1 触发规则

投保流程在提交保单前，后端自动判断是否需要双录。满足以下**任意一条**即触发：

| 规则 | 条件 | record_scene值 |
|------|------|---------------|
| 年龄规则 | 投保人或被保险人年龄 ≥ 60岁 | 1 |
| 趸交保费 | 趸交保费金额 ≥ 50000元 | 2 |
| 期交保费 | 期交年保费 ≥ 20000元 | 3 |
| 新型产品 | 产品类型为分红险/万能险/投连险 | 4 |
| 保司要求 | 产品配置中标记 `need_double_record=true` | 5 |

**后端校验逻辑（订单提交时）**：
1. 从订单中取投保人生日，计算年龄，判断规则1
2. 取缴费方式和保费金额，判断规则2/3
3. 查产品表 `product_type` 字段，判断规则4
4. 查产品扩展配置 `need_double_record` 字段，判断规则5
5. 若触发，在订单表更新 `need_double_record=true`，`record_scene`=对应场景值
6. 返回前端触发标记，前端引导业务员进入双录流程

**接口**：

```
POST /admin-api/insurance/double-record/check-trigger

Request:
{
  "orderId": 123456
}

Response:
{
  "code": 0,
  "data": {
    "needDoubleRecord": true,
    "triggerReasons": ["投保人年龄65岁", "趸交保费6万元"],
    "recordScene": 1,
    "recommendTemplateId": 10
  }
}
```

## 三、功能模块：话术模板配置（PC管理后台）

### 3.1 模板列表页

**页面路径**：合规管理 → 话术模板管理

**列表展示字段**：模板编号、模板名称、适用产品类型、节点总数、版本号、是否启用、创建时间、操作（编辑/删除/查看节点）

**搜索条件**：模板名称（模糊）、产品类型（下拉选）、是否启用（下拉选）

**接口**：
```
GET /admin-api/insurance/double-record/template/page
Query: pageNo=1&pageSize=20&templateName=&productType=&isActive=
```

### 3.2 新增/编辑模板

**点击「新增话术模板」或「编辑」按钮** → 打开全页表单（非弹窗，内容较多）

**基本信息区（必填）**：

| 字段 | 类型 | 校验 |
|------|------|------|
| 模板名称 | 文本 | 必填，最长100字 |
| 适用产品类型 | 下拉单选 | 必填 |
| 版本号 | 文本 | 必填，如 1.0、2.1 |
| 最短双录时长（秒） | 数字 | 选填，最小60 |
| 最大重录次数 | 数字 | 选填，默认3，最大10 |
| 全局禁用词 | 标签输入框 | 选填，按回车添加标签 |
| 备注 | 文本域 | 选填 |

**话术节点配置区**：

- 点击「添加节点」按钮，在列表末尾插入一行节点编辑区
- 每个节点包含如下子字段：

| 字段 | 类型 | 校验 |
|------|------|------|
| 节点名称 | 文本 | 必填，如「身份确认」 |
| 节点类型 | 单选 | 必填：必读/选读 |
| 话术内容 | 文本域 | 必填 |
| 必读关键词 | 标签输入框 | 选填 |
| 禁用词 | 标签输入框 | 选填 |
| 最短时长（秒） | 数字 | 选填，最小0 |

- 节点支持上移/下移调整顺序，支持删除（至少保留1个节点）
- 点击「预览话术」，以只读方式展示完整话术内容

**后端校验（保存时）**：
1. 同一租户内 `template_code`/`template_name` 不允许重复
2. `script_nodes` 数组不能为空，至少含1个 `MUST` 类型节点
3. 每个节点的 `scriptContent` 不能为空
4. 保存时自动计算 `total_nodes`、`must_node_indexes` 并写入

**接口**：
```
POST /admin-api/insurance/double-record/template/create
PUT  /admin-api/insurance/double-record/template/update
DELETE /admin-api/insurance/double-record/template/{id}  // 逻辑删除，已被双录记录引用的不允许删除
GET  /admin-api/insurance/double-record/template/{id}
```

## 四、功能模块：发起双录

### 4.1 业务流程

1. 投保订单触发双录后，业务员在App/PC端点击「发起双录」按钮
2. 系统匹配话术模板（按 `product_type` + 客户年龄区间匹配，若有多个取最新启用版本）
3. 后端创建 `ins_double_record` 记录，状态为 `0-待开始`
4. 若为**远程双录**：后端调用 RTC SDK 创建房间，返回 `rtcRoomId` 和客户端邀请链接
5. 业务员将邀请链接通过短信/微信发给客户
6. 双方进入 RTC 房间，业务员端展示话术节点列表（左侧），视频画面（右侧）
7. 若为**现场双录**：双方面对面，使用同一设备，客户摄像头录制客户面部

### 4.2 发起双录接口

```
POST /admin-api/insurance/double-record/start

Request:
{
  "orderId": 123456,
  "recordType": 2,            // 1-现场 2-远程
  "templateId": 10,           // 若不传，后端自动匹配
  "customerMobile": "138xxxx" // 远程时必填，用于发送邀请短信
}

后端处理：
1. 校验订单存在且状态合法
2. 校验 need_double_record=true
3. 确认 templateId 有效且启用
4. 若 templateId 未传，自动按产品类型+年龄匹配最新模板，匹配不到则报错
5. 生成唯一 record_code：DR + yyyyMMdd + Redis自增6位序号
6. 若为远程双录，调用 RTC SDK 创建房间获取 rtcRoomId
7. 插入 ins_double_record 记录，status=0
8. 若为远程，发送邀请短信给客户（短信模板：[平台名]您的业务员邀请您参与双录确认，请点击链接：{url}，有效期30分钟）

Response:
{
  "code": 0,
  "data": {
    "recordId": 789,
    "recordCode": "DR202602140001",
    "rtcRoomId": "room_abc123",     // 仅远程双录返回
    "inviteUrl": "https://h5.xx.com/double-record?token=xxx&expire=1800",
    "templateId": 10,
    "totalNodes": 6
  }
}
```

### 4.3 双录进行中：节点推进

双录界面左侧显示话术节点列表，业务员逐节点朗读，读完后点击「完成本节点」按钮推进。

**前端界面逻辑**：
- 进入双录后调用接口将状态更新为 `1-进行中`
- 展示当前节点的话术内容（大字显示便于朗读）
- 展示关键词提示（红色标注必须说到的词）
- 展示禁用词提示（灰色标注不能说的词）
- 每个节点有倒计时，未达到 `minDuration` 秒时「完成本节点」按钮置灰
- 必读节点不可跳过（节点列表中，未完成必读节点时，其后所有节点禁止点击）

**完成节点接口**：
```
POST /admin-api/insurance/double-record/node/complete

Request:
{
  "recordId": 789,
  "nodeIndex": 2,
  "actualDuration": 35  // 该节点实际耗时（秒）
}

后端处理：
1. 校验 recordId 存在且 status=1
2. 校验 nodeIndex 合法（≤ totalNodes）
3. 将 nodeIndex 追加到 node_completed 数组（JSON追加）
4. 更新 current_node = nodeIndex + 1
5. 返回成功

Response:
{
  "code": 0,
  "data": {
    "completedNodes": [1, 2],
    "currentNode": 3,
    "allCompleted": false
  }
}
```

### 4.4 结束双录

所有节点完成后，业务员点击「结束双录」按钮。

**前端动作**：
1. 停止 RTC 录制
2. 视频文件上传至 OSS（分片上传，最大支持 2GB）
3. 上传完成后调用结束接口

**结束接口**：
```
POST /admin-api/insurance/double-record/finish

Request:
{
  "recordId": 789,
  "videoUrl": "oss://bucket/tenant/2026/02/DR202602140001_video.mp4",
  "videoSize": 52428800,
  "audioUrl": "oss://bucket/tenant/2026/02/DR202602140001_audio.mp3",
  "totalDuration": 360
}

后端处理：
1. 校验 recordId 存在且 status=1
2. 校验 node_completed 数组是否包含所有 must_node_indexes（若未全部完成，返回错误 "必读节点未全部完成"）
3. 校验 totalDuration ≥ template 的 min_duration（若不满足，返回警告但不阻断，记录到日志）
4. 更新 ins_double_record：status=2, end_time=now, video_url, audio_url, duration
5. 发送 MQ 消息（topic: double_record_completed, body: recordId）触发异步 AI 质检
6. 更新订单表 double_record_status=2（已完成双录）

Response: { "code": 0, "msg": "双录完成，AI质检任务已提交" }
```

## 五、功能模块：断点重录

### 5.1 触发场景

- 场景1：RTC 网络中断，自动触发重录提示
- 场景2：业务员主动点击「重录当前节点」
- 场景3：AI 质检不通过，质检员要求重录

### 5.2 重录逻辑

```
POST /admin-api/insurance/double-record/retry

Request:
{
  "recordId": 789,
  "retryReason": "网络中断",     // 重录原因
  "fromNodeIndex": 3             // 从第几个节点开始重录
}

后端处理：
1. 查询 ins_double_record，取 retry_count
2. 判断 retry_count 是否已达 template 的 max_retry：
   - 未超限：直接允许重录
   - 超限（如 retry_count >= 3）：检查是否有主管审批记录，无则返回错误"重录次数超限，请联系主管审批"
3. retry_count + 1，last_retry_time = now，status = 1（进行中）
4. 清空 fromNodeIndex 及之后的节点完成记录（node_completed 中移除 >= fromNodeIndex 的元素）
5. current_node = fromNodeIndex
6. 插入重录日志（记录到 ins_double_record_retry_log 表，含 reason、operator）

Response:
{
  "code": 0,
  "data": {
    "retryCount": 2,
    "maxRetry": 3,
    "fromNodeIndex": 3
  }
}
```

### 5.3 重录审批（超限时）

超过 `max_retry` 次数后，需主管审批才能继续重录：

```
POST /admin-api/insurance/double-record/retry-approve

Request:
{
  "recordId": 789,
  "approveResult": 1,    // 1-同意 2-拒绝
  "approveRemark": "特殊情况，客户网络条件差"
}

后端处理：
1. 验证当前操作人为该业务员的直属主管（或合规角色）
2. 审批通过：将 max_retry 临时扩大1次（更新 retry_count - 1）
3. 审批拒绝：双录状态改为 6-已作废，订单标记需要重新发起
```

## 六、功能模块：录制与云存储

### 6.1 服务端录制方案

推荐使用**声网 Agora 服务端合流录制**（Cloud Recording），避免前端录制上传的稳定性问题：

1. 后端调用声网 `start` 接口开始录制，传入 `channelName`（即 `rtcRoomId`）
2. 声网自动合并音视频流，录制完成后推送到指定 OSS Bucket
3. 后端通过声网回调（Webhook）接收录制完成通知，获取 `video_url`
4. 前端无需上传视频，`finish` 接口中 `videoUrl` 可为空，后端等待录制回调填充

**录制参数配置**：
- 分辨率：1280×720（720P）
- 帧率：25fps，码率：1500kbps
- 格式：MP4（H.264）
- 文件命名：`{tenant_id}/{year}/{month}/{record_code}_video.mp4`

### 6.2 OSS 访问控制

- Bucket 权限：**私有**，禁止公开访问
- 访问方式：后端生成**临时签名 URL**，有效期 2 小时
- 视频播放接口（质检查看时）：后端生成临时 URL 返回给前端，不暴露真实 OSS 路径

```
GET /admin-api/insurance/double-record/video-url/{recordId}

后端处理：
```

## 八、重要业务规则汇总

1. **双录状态流转**：`0-待开始` → `1-进行中` → `2-已完成` → `3-质检中` → `4-质检通过`/`5-质检不通过`；随时可流转为 `6-已作废`
2. **模板匹配优先级**：产品类型精确匹配 > 通用模板（product_type=0）；同类型取版本号最大且 is_active=true 的
3. **质检顺序**：AI 质检先行（异步，双录完成后自动触发）→ AI 通过后仍需按比例抽取人工质检（抽检比例由系统配置，默认 20%）
4. **数据加密**：`customer_id_card` 字段使用 AES-256 加密，密钥由 KMS 管理，查询时自动解密展示脱敏值（前6后4）
5. **双录视频保留 5 年**（监管要求），通过 OSS 生命周期规则实现：0~3月标准存储 → 3~12月低频存储 → 12月+归档存储，满5年后需合规审批才可删除
