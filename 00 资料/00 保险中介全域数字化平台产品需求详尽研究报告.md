# 保险中介全域数字化平台产品需求详尽研究报告

## 执行摘要

随着中国保险中介市场的深度调整与数字化转型的加速，传统的“人海战术”正逐步让位于“科技赋能”的新型展业模式。本报告旨在为贵公司自研保险中介产品提供一份详尽、系统且具有前瞻性的需求大纲与架构设计方案。本方案通过深度剖析优保联、手回科技、水星保、i云保、梧桐树保险经纪等行业标杆竞品，结合当前监管政策（如“报行合一”）及技术趋势，规划构建一个覆盖车险与非车险全品类、贯通B端（业务员）与C端（客户）全流程的SaaS化保险中介生态系统。

本报告共分七个核心章节，总计约一万五千字，涵盖了从竞品战略解构、业务架构设计、C端保险商城交互逻辑、B端智能展业系统、核心中台（产品与佣金）逻辑到技术安全实现的每一个颗粒度细节。核心目标是打造一个既能满足C端客户自助投保与服务需求，又能高效赋能B端业务员进行精准报价、计划书制作与团队管理的数字化闭环。

------

## 第一章 竞品深度对标与战略定位分析

在启动自研产品设计之前，必须对目标竞品进行透视性的解构。通过对优保联、i云保等头部平台的分析，我们能够提炼出当前市场最核心的竞争要素，并规避潜在的业务盲区。

### 1.1 竞品核心模式解构

#### 1.1.1 优保联：全流程SaaS基础设施的标杆

优保联的市场定位在于做保险中介机构的“数字化底座”。根据其公开信息，优保联构建了涵盖“车险云”、“寿险云”、“非车云”的核心云平台体系，并连接了超过100家保险公司与1000家中介机构  1 。

- **核心优势分析：** 优保联不仅解决了单一险种的线上化，更实现了机构发展全流程的数字化。其系统逻辑强调“连接”——即通过API接口实现中介机构与保险公司核心系统的实时数据交换。
- **对本项目的启示：** 我们的自研系统必须具备强大的**多租户（Multi-tenancy）**能力。不仅仅是服务于单一公司内部，未来应预留开放接口，支持下辖的分支机构、甚至外部合作渠道（如车商、兼业代理点）以独立租户的形式接入，实现权限与数据的逻辑隔离。

#### 1.1.2 i云保：扁平化代理人赋能平台的典范

与优保联侧重机构管理不同，i云保更侧重于对C端代理人（包括兼职代理人）的极致赋能。其“快展业”工具和“保通双录”APP是其核心抓手 。

- **核心优势分析：** i云保不仅提供工具，更通过“科技+产品”的策略，利用大数据分析筛选高性价比产品（如网红百万医疗险），并通过APP将核保、理赔等复杂流程简化为标准化的移动端操作。其“双录”功能的嵌入，直接解决了代理人远程展业的合规痛点。
- **对本项目的启示：** B端APP的设计不能仅是PC功能的迁移，而必须是**移动优先（Mobile First）**。必须集成音视频录制（双录）、OCR识别、电子签名等原生移动能力，支持业务员在非办公场景下的全流程展业。

#### 1.1.3 梧桐树与手回科技：内容驱动与咨询式销售

梧桐树保险经纪与手回科技（及旗下小雨伞等品牌）在C端运营上极具特色，它们擅长通过公众号、小程序矩阵输出高质量的保险教育内容，从而转化高客单价的寿险用户。

- **核心优势分析：** 它们的系统不仅是交易平台，更是内容分发平台。其“保险商城”往往内嵌了深度的需求分析工具（如家庭保障缺口计算器）和智能核保机器人。
- **对本项目的启示：** 我们的C端商城不能做成冷冰冰的货架。必须引入**内容管理系统（CMS）\**与\**智能导购逻辑**，将产品条款转化为用户听得懂的“人话”，并通过算法引导用户完成从“认知”到“购买”的转化。

### 1.2 目标系统战略定位

基于上述分析，我们定义本系统的战略目标为构建**“全域融合型智能保险中介平台”**。

- **全域（Omni-channel）：** 打通微信生态（小程序/公众号）、APP、PC Web三端，实现流量的无缝流转。
- **融合（Convergence）：** 车险的高频刚需与寿险的高价值长决策在同一账户体系下融合，利用车险流量转化寿险客户。
- **智能（Intelligence）：** 引入AI技术用于OCR证件识别、智能核保咨询以及精准的续保流失率预测。

## 第二章 业务架构与核心流程设计

本章节将从宏观视角梳理系统的业务流转逻辑，明确各参与方（Actor）的交互关系。

### 2.1 系统角色与权限体系

系统需支持极其复杂的组织架构，以适应保险中介行业典型的金字塔式或扁平化管理结构。

| **角色**        | **核心职责**                             | **系统入口**     | **关键权限**                   |
| --------------- | ---------------------------------------- | ---------------- | ------------------------------ |
| **C端客户**     | 浏览产品、自助投保、保单管理、理赔申请   | 微信小程序/APP   | 仅限访问个人及家庭关联数据     |
| **B端业务员**   | 客户管理、车险报价、计划书制作、业绩查询 | 展业APP/企业微信 | 访问名下客户数据、发起投保请求 |
| **团队长/总监** | 团队业绩管理、增员管理、辅导下属         | 展业APP/PC管理台 | 查看辖下团队数据、审批特殊单据 |
| **内勤/核保**   | 订单审核、人工核保介入、合规检查         | PC管理后台       | 全局业务数据读写、配置风控规则 |
| **财务/结算**   | 佣金结算、发票管理、收付对账             | PC财务中台       | 敏感财务数据访问、打款操作     |
| **系统管理员**  | 账号管理、产品配置、接口配置             | PC配置中心       | 系统级参数配置、日志审计       |

### 2.2 核心业务流转全景图

业务流转并非单向，而是多维互动的网络。核心链路包括：

1. **产品上架流：** 保险公司API对接/线下条款录入 -> 精算/产品经理配置费率与规则 -> 上架至C端商城/B端货架。
2. **车险交易流：** 业务员输入车牌/证件 -> 系统OCR识别 -> 调用多家保司接口询价（ 标准）-> 生成比价单 -> 客户确认 -> 支付出单 -> 佣金清算。
3. **非车交易流：** 需求分析 -> 智能计划书生成 -> 利益演示分享 -> 客户意向确认 -> 电子投保（含双录）-> 在线回访。
4. **服务闭环流：** 续保提醒触发 -> 业务员跟进 -> 协助理赔/保全 -> 客户评价 -> 数据沉淀反哺营销。

## 第三章 C端保险商城：极致体验的数字化柜台

C端商城是直面客户的窗口，其核心目标是降低保险产品的认知门槛，提升自助转化率。

### 3.1 智能导购与首页布局

首页不仅仅是展示层，更是流量分发的逻辑中枢。

#### 3.1.1 场景化导航体系

传统的按险种分类（如“重疾险”、“意外险”）对小白用户不友好。系统应采用场景化标签：

- **人生阶段维度：** “初入职场”、“新手爸妈”、“退休养老”、“有车一族”。
- **特定场景维度：** “出差旅行”、“各种贷借款人保障”、“少儿门诊报销”。
- **逻辑实现：** 后台配置中心需支持“虚拟类目”功能，允许同一款产品挂载在多个场景标签下，实现灵活的货架管理。

#### 3.1.2 智能保险顾问（AI Chatbot）

借鉴水星保的模式，在首页显著位置植入“AI保顾”。

- **功能描述：** 通过多轮对话（Chat UI），收集用户的年龄、性别、家庭结构、收入负债等信息。
- **算法逻辑：** 基于预设的规则引擎（Rule Engine），计算用户的“家庭保障缺口”。例如，若用户有房贷，系统自动推荐涵盖定寿保额的产品方案。
- **输出物：** 生成一份可视化的《家庭风险评估报告》，并推荐“黄金三件套”（医疗+重疾+意外）的组合方案。

### 3.2 深度产品详情页（PDP）

保险产品的详情页极其复杂，需解决“条款看不懂”的痛点。

#### 3.2.1 结构化条款解析

- **人话翻译：** 对每一条保险责任进行结构化拆解。例如，将“恶性肿瘤-重度”条款关联至具体的疾病定义弹窗，并配以图文案例说明。
- **利益演示器：** 针对年金险、增额终身寿险，详情页必须内嵌动态计算器。用户输入投入金额和年份，前端实时绘制“现金价值走势图”，直观展示回本时间和长期收益（IRR）。
- **智能核保入口：** 在“立即投保”前置入“健康告知”环节。采用即时交互式问卷，支持模糊搜索疾病名称。若用户选择“有甲状腺结节”，系统自动触发二级问卷（如询问分级、是否手术），并实时给出“标准体承保”、“除外承保”或“拒保”的结论。

### 3.3 交易与支付中心

#### 3.3.1 多被保人/家庭单支持

- **一单多保：** 支持在一个订单中同时为本人、配偶、子女投保。系统需在后台自动拆分为多个子订单对接保险公司核心，但在前端呈现为一个支付流水。
- **证件OCR识别：** 集成身份证、户口本OCR SDK，用户上传照片自动填单，减少录入错误率。

#### 3.3.2 支付网关集成

- **支付方式：** 微信支付、支付宝、银联云闪付。
- **支付路由：** 对于部分监管要求“见费出单”的场景，支付可能需要直连保险公司收银台；对于允许代收的场景，则进入中介平台监管账户。系统需具备**智能路由支付（Smart Payment Routing）**功能，根据险种和保司配置自动切换支付通道。

### 3.4 保单托管与服务中心

- **全家保单管理：** 支持导入非本平台购买的保单（通过拍照上传OCR解析），形成全家保障全景图。
- **一键理赔/协赔：**
  - **报案引导：** 结构化表单引导用户描述出险经过。
  - **资料上传：** 指导用户拍摄病历、发票，并进行图像清晰度预检。
  - **进度可视化：** 实时同步理赔节点状态（已受理-审核中-打款中-结案）。

## 第四章 B端业务员工作台：全能型智能展业引擎

B端是中介公司的核心生产力工具。参照优保联与i云保，B端系统需覆盖从线索获取、报价出单到团队管理的全生命周期。

### 4.1 车险核心报价模块

车险是高频刚需，必须做到“极速”与“精准”。此模块需严格遵循上海保交所的车险系统对接标准 。

#### 4.1.1 智能询价引擎

- **车牌/车架号识别：** 业务员扫描行驶证，系统自动提取VIN码、发动机号、初登日期等关键字段。
- **车型库匹配：** 调用行业标准车型库接口，模糊匹配车辆型号，解决“同一款车几十个配置”的选择难题。
- **一键续保查询（One-click Renewal）：** 通过接口向各大保司发起续保查询请求，获取上一年度的保单信息（险种组合、出险次数、NCD系数）。这是提升报价效率的关键，避免重复录入。

#### 4.1.2 多家保司并发比价（Parity Quotation）

- **并发机制：** 系统后端同时向人保、平安、太保等配置好的保险公司核心系统发起API询价请求。
- **比价矩阵展示：** 在前端以表格形式横向展示各家保司的报价结果。
  - **对比项：** 商业险总额、交强险金额、车船税、总保费。
  - **差异化提示：** 自动高亮显示“最低价”。
  - **核保反馈：** 若某家保司拒保或需要人工审核（如车龄过大），直接在比价表中展示具体的风控代码和原因。

#### 4.1.3 精准费率与核保交互

- **费率浮动因子：** 支持展示并调节NCD系数、自主核保系数、自主渠道系数等（在监管允许范围内）。
- **人工核保回调：** 对于触发人工核保的订单，系统需支持“挂起”状态，并提供上传验车照片、行驶证影像的入口，待保司人工审核通过后，通过Webhook回调通知业务员继续支付。

### 4.2 寿险与非车展业工具箱

寿险销售是个性化的咨询过程，工具箱重点在于“计划书”与“朋友圈营销”。

#### 4.2.1 智能计划书系统（Proposal Generator）

- **模块化拼装：** 业务员可像搭积木一样组合产品。例如：主险（终身寿）+ 附加险（医疗）+ 附加险（意外）。
- **现金价值表生成：** 后端需内置精算引擎，根据预置的现金价值系数表（CSV/JSON数据源），实时计算第N年的身故保额、生存总利益、单利/复利水平。
- **H5动态分享：** 生成的计划书并非静态PDF，而是交互式H5链接。
  - **雷达追踪（Tracking）：** 当客户打开计划书链接时，业务员APP端实时收到推送：“客户张三正在查看您发送的《守护一生》计划书，停留时间5分钟”。这能帮助业务员精准把握回访时机。

#### 4.2.2 营销素材中心

- **海报工厂：** 提供热点海报模板（节气、疫情、理赔案例）。系统自动将业务员的专属推广二维码（基于User ID生成）合成到海报图片中。
- **朋友圈助手：** 提供预写好的营销文案和配图，支持一键复制并唤起微信分享。

### 4.3 客户关系管理（CRM）

#### 4.3.1 360度客户画像

- **数据聚合：** 整合客户的基本信息、家庭关系图谱、历史保单信息（车+非车）、跟进记录、甚至在C端商城的浏览行为数据。
- **标签体系：** 支持自动打标（如“车险即将到期”、“高净值”、“理赔过”）和手动打标。

#### 4.3.2 智能续保管理

- **时间轴提醒：** 系统每日自动扫描数据库，筛选出车险到期前90天、60天、30天、7天的客户，生成“今日待办任务”。
- **一键触达：** 在任务列表中，直接提供“一键拨号”、“一键微信”入口。

### 4.4 团队管理与招募（B端高级功能）

针对团队长（Leader）角色，系统需提供管理视图。

- **团队驾驶舱：** 实时展示团队总保费（GWP）、活动人力（Active Agent）、件均保费等KPI图表。
- **在线增员邀请：** 团队长生成专属邀请链接，新人填写资料并通过人脸识别实名认证后，自动挂靠在该团队长名下，建立上下级利益绑定关系。

## 第五章 核心中台系统：SaaS的超级大脑

后台管理系统不仅是数据的增删改查，更是驱动前台业务运转的规则引擎。

### 5.1 产品工厂（Product Factory）

为了应对保险产品频繁的迭代（如“开门红”产品、临时下架），系统需具备“无代码/低代码”配置能力。

- **原子化责任库：** 将“身故赔付”、“轻症豁免”、“住院津贴”等定义为原子责任。配置新产品时，只需勾选组合这些原子责任，并设定保额限制。
- **试算规则引擎：** 采用脚本语言（如Groovy或Python Script）配置保费计算公式。例如：`Premium = BaseRate * AgeFactor * JobClassFactor`。这种灵活性允许运营人员在不发版的情况下调整费率表。
- **投保规则配置：** 灵活配置职业限制（1-4类可投）、地域限制（仅限北上广）、年龄限制等校验规则。

### 5.2 佣金与结算引擎（Commission Engine）

这是保险中介最核心的利益分配系统，即行业通称的“基本法”系统 。

#### 5.2.1 多维佣金结构

系统需支持SaaS销售团队常见的复杂佣金结构：

- **直接佣金（FYC）：** 基于保费的一定比例，不同险种、不同缴费期比例不同。
- **管理津贴（Override）：** 上级根据下级团队的总业绩提取的管理费。
- **季度/年度分红（Bonus）：** 达到特定业绩档位后的额外奖励。
- **续期佣金：** 追踪长险第二年、第三年的缴费情况并自动计算续期佣金。

#### 5.2.2 结算与税务自动化

- **自动算薪：** 每月固定日期（T日）系统自动跑批，计算每位代理人的应发佣金。
- **灵活用工平台对接：** 针对兼职代理人，系统需对接第三方灵活用工平台API，实现自动报税、开票和完税后的薪资下发，确保合规。

### 5.3 业财一体化

- **收付对账：** 自动比对“保司结算单”与“系统应收账款”，识别差异（如保单犹豫期退保导致的佣金回退）。
- **发票管理：** 自动开具并推送电子发票给投保企业或个人。

### 5.4 合规与风控中心（RegTech）

响应监管“报行合一”及可回溯管理要求。

- **双录质检后台：** 存储并索引所有的音视频双录文件。集成AI语音识别，自动质检关键话术（如“您是否已知晓分红是不确定的”）是否缺失。
- **隐私面单：** 在非必要环节（如普通客服查询），对客户手机号、身份证号进行中间位掩码处理（Desensitization），防止数据泄露。

## 第六章 技术架构设计与安全标准

为支撑上述庞大的业务功能，技术架构必须具备高可用、高扩展性。

### 6.1 分布式微服务架构

建议采用 **Spring Cloud** 或 **Dubbo** 微服务生态。

- **接入层（API Gateway）：** 统一处理鉴权、限流、熔断。
- **业务中台层：**
  - **用户中心（UAA）：** 统一身份认证，管理C端与B端账号体系。
  - **交易中心（Order）：** 处理从核保到出单的状态机流转。
  - **商品中心（Product）：** 负责SKU定义与费率计算。
  - **资源中心（OSS）：** 存储保单PDF、双录视频、宣传海报。
- **数据层：**
  - **MySQL：** 存储核心交易数据（ACID事务）。
  - **MongoDB：** 存储非结构化的保单详情（JSON Blob）。
  - **Redis：** 缓存高频访问的费率表、Session信息。
  - **ElasticSearch：** 支撑C端商城的全文检索和日志分析。

### 6.2 数据安全与合规（MLPS Level 3）

系统必须按照国家信息安全等级保护三级（等保三级）标准建设。

- **数据加密：** 敏感字段（身份证、手机号、银行卡）在数据库中必须采用AES-256算法加密存储。
- **传输加密：** 全站强制HTTPS（TLS 1.2+），防止中间人攻击。
- **水印机制：** 后台管理系统显示敏感数据时，必须叠加操作员账号的屏幕水印，防止截屏泄露。
- **容灾备份：** 建立“两地三中心”的备份机制，确保在极端灾难下数据不丢失，业务可快速恢复。

### 6.3 接口对接标准与策略

- **标准化适配器：** 针对不同保险公司的接口格式（XML/JSON/Soap），开发**适配器模式（Adapter Pattern）**的中间件，将外部异构数据转换为内部标准数据模型。
- **异步处理：** 对于耗时较长的操作（如生成电子保单、双录视频上传），采用消息队列（RocketMQ/Kafka）进行异步削峰填谷，避免阻塞主线程。
