## 一、功能点详细业务逻辑

### 1.1 我的寿险保单（C端）

> **参考 PDF**：`#109`（个险保单目录）、`#111`（保单查询）、`#166`（系统配置-保单状态配置）

#### 1.1.1 页面入口

小程序底部 Tab「我的」页面，在已有的「非车险保单」Tab 旁新增「寿险」Tab（两个 Tab 并列展示，可左右滑动切换）。默认展示第一条有效保单。

#### 1.1.2 保单列表加载

前端调用 `GET /app-api/ins/life/policy/my-list`，接口返回当前登录用户的所有寿险保单列表，按 `create_time` 降序排列。

**筛选 Tab**（列表顶部切换，默认"全部"）：

| Tab | 过滤状态 |
|---|---|
| 全部 | 不过滤 |
| 生效中 | `status = ACTIVE` |
| 核保中 | `status = UNDERWRITING` |
| 已失效 | `status IN (CANCELLED, LAPSED, REJECTED, MATURE)` |

> 对应 PDF `#111`：保单查询列表按状态分类展示，已承保未回执/已承保未回访/中止/终止/退保等均有对应展示逻辑；PC端不同状态的操作按钮不同，C端同理根据保单状态动态显示可用操作。

#### 1.1.3 保单卡片展示

每张保单卡片展示：

- 险种图标（从 `ins_life_product.category_icon` 取，对应 PDF `#170` 寿险产品管理中产品分类）
- 产品名称
- 保险公司名称（来自 `#169` 合作保司管理中维护的保司信息）
- 被保人姓名（脱敏，仅显示姓 + *，如"张*"）
- 保障额度（如"保额 100 万"）
- 缴费状态标签（颜色区分）：

| 缴费状态标签 | 触发条件 |
|---|---|
| 🟢 保障中 | `status = ACTIVE` 且 `next_payment_date > 今日` |
| 🟡 待缴费 | `status = ACTIVE` 且 `next_payment_date` 在7天内（含今日） |
| 🔴 已逾期 | `status = ACTIVE` 且 `next_payment_date < 今日` |
| ⚪ 核保中 | `status = UNDERWRITING` |
| ⚫ 已失效 | `status IN (CANCELLED, LAPSED)` |

- 下次缴费日（格式：2025-06-01，趸缴/已到期产品不展示）

> **特别说明**（参考 `#111`）：PC端保单查询中，险种后方蓝色数字可点击展开该保单的缴费次数、缴费方式、关系人信息等；C端保单卡片同样需要支持点击展开更多摘要信息（可滑动展开卡片），再点击进入详情页。

**逾期缴费提醒弹窗**：若列表中存在 `status = ACTIVE` 且 `next_payment_date < 今日` 的保单，页面加载完成后自动弹出弹窗（每次进入此页面最多弹出一次，使用 Session 变量控制）：

- 弹窗内容："您有 X 份保单已逾期，请尽快补缴，以免保单失效"
- 「立即补缴」按钮：关闭弹窗，自动滚动到第一条逾期保单卡片
- 「稍后再说」按钮：关闭弹窗

#### 1.1.4 后端接口逻辑

`GET /app-api/ins/life/policy/my-list`：

1. 查询 `ins_life_policy`，`member_id = 当前登录用户ID`，`deleted = 0`。
2. 按 `status_filter`（Tab 筛选参数）过滤。
3. 关联查询 `ins_life_policy_insured`（取 `record_type = 1` 的第一条被保人姓名）。
4. 关联查询 `ins_life_product` 取产品名称和险种图标。
5. 关联查询 `ins_insurer` 取保险公司名称。
6. 计算缴费状态标签（根据 `next_payment_date` 与当前日期比较）。
7. 返回列表，每条数据包含 `policy_id`、`policy_no`、`product_name`、`category_icon`、`insurer_name`、`insured_name_masked`、`coverage_amount`、`payment_status_label`、`next_payment_date`、`status`。

> 注意：`#166`（系统配置-保单状态配置）中可配置"允许参与续期操作的保单状态"（有效、退保、终止、中止），后端需读取此配置以决定是否允许用户发起续期缴费，C端"立即补缴"按钮的可用性也需同步此逻辑。

### 1.2 寿险保单详情（C端）

> **参考 PDF**：`#110`（保单录入字段结构）、`#111`（保单查询-保单日志）、`#116`（回执管理）、`#118`（新增回访）、`#125~#127`（保全维护）、`#138`（理赔管理）

#### 1.2.1 页面入口

点击保单列表中任意卡片，进入保单详情页。调用 `GET /app-api/ins/life/policy/{policy_id}/detail`。

#### 1.2.2 页面区域说明

**基本信息区域**（始终展示）：

- 产品名称、保险公司 Logo、保单号（完整展示，可点击复制）
- 保单状态大标签（核保中 / 生效中 / 已失效等）
- 保障期限（起止日期）

**分 Tab 展示内容**（默认"保障详情"）：

**Tab 1：保障详情**

- 保障额度
- 险种类型（对应 `#165` H5后台配置中的产品分类，如重疾险/寿险/医疗险/意外险/储蓄险）
- 缴费方式与期限（如"年缴，20年期"；趸缴件标注"趸缴"）
- 保障责任列表（从 `ins_life_product.product_detail_json.coverage_items` 读取，展示各项保障责任名称与保额）
- 代扣状态（开启中 / 未开启 / 已到期），如已到期显示"代扣协议已到期，请重新签约"
- 回执状态：已回执（显示回执日期）/ 待回执（参考 `#116` 回执管理流程，若保单已承保但未回执，C端展示"待回执"提示并允许用户确认回执）
- 回访状态：已回访 / 待回访（参考 `#118` 回访管理，已承保未回访的保单在C端可展示"待回访"状态，并由业务员在App端操作，C端只读展示）

**Tab 2：缴费历史**

调用 `GET /app-api/ins/life/policy/{policy_id}/payment-records`，返回历史缴费记录列表，每条显示：缴费日期、缴费金额、缴费方式、缴费状态（成功 / 失败 / 退款）、对应保单年度（第1年/第2年...，参考 `#166` 续期保单配置中"以实收日期为准"的结算逻辑）。分页，每页 10 条，下拉加载更多。

**Tab 3：受益人**

展示受益人列表（来自 `ins_life_policy_insured` 表 `record_type = 2` 的记录），每条显示：受益人姓名（脱敏，显示姓+*）、与被保人关系、受益比例、受益顺序。法定受益人则显示"法定受益人"文字。

> 对应 `#110` 保单录入中的关系人信息结构：投保人、被保人、受益人均存储在关系人信息表中，类型字段区分角色。C端展示时仅展示受益人相关信息。

**Tab 4：保全记录**

调用 `GET /app-api/ins/life/policy/{policy_id}/conservation-records`，返回该保单的保全变更历史（变更类型、变更时间、变更内容摘要、申请状态）。若无保全记录则显示"暂无保全记录"。

> 参考 `#126`（保全新增/修改）：保全记录包含保全类型、申请时间、处理状态；`#126` 中注明"有【撤销】按钮是可以自行撤销，没有按钮想撤销可联系客服人员进行处理"，C端同样需要对 `status = PENDING` 的保全申请提供"撤销申请"功能，点击后调用 `DELETE /app-api/ins/life/policy/{policy_id}/conservation/{conservation_id}`，将 `status` 更新为 `CANCELLED`，同时撤销 Flowable 流程实例。

**Tab 5：保单日志**

> 新增 Tab，参考 `#111`（保单日志）：

调用 `GET /app-api/ins/life/policy/{policy_id}/logs`，返回该保单的所有操作日志，包含：操作类型（承保/回执/回访/保全/续期缴费/状态变更等）、操作时间、操作人（业务员名称或"用户自助"）、操作摘要。按时间倒序排列，前端以时间轴形式展示。

#### 1.2.3 操作按钮区域（底部固定栏）

根据保单状态动态显示可用操作：

| 操作按钮 | 显示条件 | 点击逻辑 |
|---|---|---|
| 联系顾问 | 始终显示 | 若归因业务员存在，展示业务员联系信息弹窗（姓名+微信二维码+手机号）；否则展示平台客服联系方式 |
| 申请保全 | `status = ACTIVE` 且已回执已回访 | 跳转保全申请页（见 1.2.4） |
| 下载保单 | `e_policy_url` 不为空 | 调用微信小程序 `wx.downloadFile` + `wx.openDocument` 打开 PDF；若 `e_policy_url` 为空则显示"保单文件生成中，请稍后再试" |
| 理赔报案 | `status = ACTIVE` | 跳转理赔报案页（参考 `#138` 理赔管理：填写理赔信息后提交，写入 `ins_life_claim` 表） |
| 立即补缴 | `status = ACTIVE` 且逾期 | 跳转续期缴费页（见 1.3.5） |

> **理赔报案说明**（参考 `#138`）：C端用户新增理赔报案，需填写保险公司、事故时间、事故描述、联系方式等信息，保存后可在保单详情的保单日志中查看理赔记录状态；PC管理端（`#138`）支持筛选条件查询、修改和查看明细，C端只做提交和状态查询。

**逾期弹窗**：若 `status = ACTIVE` 且 `next_payment_date < 今日`，进入详情页时弹出醒目提示弹窗："您的保单已逾期 X 天，如逾期超过宽限期则保单将失效，请立即补缴"，「立即补缴」按钮直接发起补缴支付流程。

#### 1.2.4 申请保全

> **参考 PDF**：`#125`（保全维护目录）、`#126`（保全新增/修改）、`#127`（批量保全）

点击「申请保全」后，进入保全申请选择页，展示可申请的保全类型（来自产品配置的允许保全项）：

| 保全类型 | 说明 | 对应 PDF 字段 |
|---|---|---|
| 变更受益人 | 修改受益人信息 | `#126` 保全信息必填项包含受益人相关字段 |
| 变更缴费账户 | 修改代扣银行卡/支付宝 | `#126` 缴费信息变更 |
| 修改联系方式 | 修改联系电话/地址 | `#126` 联系方式字段 |
| 申请退保 | 提前终止保单，退还现金价值 | `#121` 保单修改-状态变更（退保） |
| 申请减保 | 降低保障额度 | `#122` 保单修改-保单修改 |

用户选择保全类型后，进入对应表单页。表单字段参考 `#126` 中保全新增时的必填项（标星字段为必填）。填写完成后调用 `POST /app-api/ins/life/policy/{policy_id}/conservation`，后端处理：

1. 写入 `ins_life_conservation`（保全申请表）记录，`status = PENDING`。
2. 触发 BPM 工作流（Flowable）发起审批流程，经内部客服审核后，审核通过则调用保司 API 执行变更，更新保单相关数据；审核拒绝则通知用户并说明原因。
3. 推送站内信："您的[保全类型]申请已提交，审核周期1-3个工作日"。

**撤销保全说明**（参考 `#126`）：

PC端保全详情页有"撤销"按钮，说明用户可以在特定条件下自行撤销保全申请（无撤销按钮的需联系客服）。C端同样支持对 `PENDING` 状态的保全申请进行撤销，撤销后 `status = CANCELLED`，Flowable 流程实例同步终止。若保全已进入 `APPROVED/EXECUTED` 状态则不可撤销，前端隐藏撤销按钮。

### 1.3 寿险续期缴费提醒（C端）

> **参考 PDF**：`#128`（续期管理目录）、`#129`（续期跟踪目录）、`#130`（续期查询）、`#131`（续期批量导入）、`#166`（系统配置-续期保单配置及保单状态配置）

#### 1.3.1 定时提醒触发逻辑

XXL-Job 定时任务 `LifePolicyRenewalReminderJob`，每天 **09:00** 执行一次：

1. 查询所有 `ins_life_policy` 中 `status = ACTIVE`、`deleted = 0`、`auto_deduct_status != ACTIVE`（未开启代扣的保单才需要提醒）的保单。
2. 计算距离 `next_payment_date` 的天数差 `days_until_payment`。
3. 满足以下任一条件则触发提醒：

| 提醒节点 | 触发条件 | 提醒内容 |
|---|---|---|
| 提前30天 | `days_until_payment = 30` | "您的[产品名]保单将于30天后到期缴费，缴费金额¥xxx，请提前准备" |
| 提前15天 | `days_until_payment = 15` | "您的[产品名]保单将于15天后到期缴费，缴费金额¥xxx，请及时缴纳" |
| 提前7天 | `days_until_payment = 7` | "您的[产品名]保单还有7天到期缴费，缴费金额¥xxx，请尽快操作" |
| 到期当天 | `days_until_payment = 1` | "今日是您[产品名]保单的缴费日，缴费金额¥xxx，请立即缴纳" |
| 逾期提醒 | `next_payment_date < 今日` 且 `days_overdue IN (1, 3, 7)` | "您的[产品名]保单已逾期X天，如超过宽限期保单将失效，请立即补缴" |

4. 对满足条件的每条保单，分别执行：
   - 发送**极光推送**（App 推送，若用户未安装则忽略）。
   - 发送**短信**（调用短信服务，模板：`SMS_LIFE_RENEWAL_REMIND`，变量：产品名、缴费日期、缴费金额、补缴入口链接）。
   - 写入用户**消息中心**记录（`ins_member_message` 表，`message_type = RENEWAL_REMIND`）。
5. 防重复发送：Redis 中记录当日已发送记录，Key：`ins:remind:sent:{policy_id}:{remind_type}:{date}`，TTL 25 小时，存在则跳过。

> **系统配置关联**（参考 `#166`）：系统配置中「续期保单配置」支持设置"续期保单佣金默认结算标准以实收日期为准"；「保单状态配置」中可配置参与续期操作的保单状态（有效、退保、终止、中止），C端续期缴费逻辑需读取此配置，非允许状态的保单不展示"立即补缴"按钮。

#### 1.3.2 消息中心展示（缴费待办）

小程序「我的」→「消息中心」页面，新增「缴费待办」分类 Tab。

调用 `GET /app-api/member/message/list?type=RENEWAL_REMIND`，返回未读的缴费提醒消息列表，每条显示：

- 消息标题（如"[平安福]保单缴费提醒"）
- 消息摘要（缴费日期、金额）
- 消息时间
- 是否已读（未读消息左侧显示红点）

点击消息卡片，跳转至对应保单详情页，同时将该消息标记为已读（调用 `PUT /app-api/member/message/{id}/read`）。

#### 1.3.3 代扣签约到期提醒

单独定时任务 `AutoDeductExpiryReminderJob`，每天 09:30 执行：

1. 查询 `ins_life_policy.auto_deduct_status = ACTIVE` 且代扣签约到期日（从代扣签约记录表中取）距今 ≤ 30 天的保单。
2. 向用户发送提醒："您的[产品名]保单代扣协议将于[到期日]到期，到期后将无法自动续费，请及时重新签约"。
3. 推送方式：极光推送 + 短信。
4. 前端：用户进入对应保单详情页时，顶部展示橙色 Banner："代扣协议即将到期，点击重新签约"，点击 Banner 跳转代扣重新签约流程（调起微信支付/支付宝签约 API）。

#### 1.3.4 年金险 / 两全险到期给付提醒

单独定时任务 `LifePolicyMaturityReminderJob`，每天 10:00 执行：

1. 查询 `ins_life_policy.status = ACTIVE` 且 `expiry_date` 在 30 天内到期（含当日）的保单（险种为年金险或两全险，通过 `ins_life_product.category_code IN ('ANNUITY', 'ENDOWMENT')` 判断）。
2. 发送提醒消息："您的[产品名]保单将于[到期日]到期，届时将按约定给付满期金，如有疑问请联系顾问"。
3. 同时触发业务员侧通知，在业务员 App 的"续期管理"模块中新增一条"即将到期给付"待办记录。

#### 1.3.5 续期缴费入口与流程

> **参考 PDF**：`#128`（续期管理）、`#130`（续期查询-续期撤回/续期修改）、`#166`（系统配置-续期保单配置）

用户通过以下任意入口发起续期缴费：

- 点击消息中心的缴费提醒 → 保单详情页 → 「立即补缴」按钮
- 直接进入保单详情页，底部「立即补缴」按钮
- 保单列表页卡片右下角的「立即缴费」快捷按钮（仅待缴费/已逾期状态显示）

点击「立即补缴」后，前端调用 `POST /app-api/ins/life/policy/{policy_id}/renewal-pay`，后端处理：

1. 查询保单当前应缴保费（年缴 / 月缴，根据缴费期限确定），计算逾期罚息（若产品有罚息配置则计入，否则不计）。
2. 创建续期支付订单（`ins_order`，`order_type = LIFE_RENEWAL`）。同时写入续期跟踪记录（`ins_life_renewal_track`），与 PC 端续期管理（`#130`）的续期跟踪数据共用，支持 PC 端管理员查询、修改和撤回。
3. 调用支付模块 `PayOrderApi.createOrder()` 发起支付，返回支付参数。

前端唤起支付 SDK，支付成功后：

4. 支付回调更新 `ins_order.status = COMPLETED`。
5. 更新 `ins_life_policy.next_payment_date`（在现有日期基础上加一个缴费周期，如年缴则 +365 天，月缴则 +30 天）。以实收日期作为续期结算基准（参考 `#166` 续期保单配置）。
6. 若保单处于逾期状态（`next_payment_date < 今日`），缴费成功后恢复为正常生效状态，更新 `status = ACTIVE`（若之前已因逾期设为 `LAPSED` 且在宽限期内，则恢复；超出宽限期的失效保单不可补救，不允许发起续费）。
7. 推送"续费成功"通知。
8. 续期缴费成功后，如果业务员在 PC 端对该续期记录执行了"撤回"操作（参考 `#130` 续期撤回），则 C 端对应保单需同步回退 `next_payment_date` 并退款，退款处理逻辑参考常规退款流程。

## 五、全模块状态机总结

### 5.1 寿险保单（`ins_life_policy.status`）状态流转

> 对应 PDF `#121`（保单修改-状态变更）、`#166`（系统配置-保单状态配置）

```
PAYING（支付中）
    ↓ 支付成功
UNDERWRITING（核保中）
    ↓ 核保通过（STANDARD）
ACTIVE（生效中）
    ↓ 附加条件承保
WAITING_CONDITION_CONFIRM（等待用户确认条件）
    ↓ 用户接受
ACTIVE（生效中）
    ↓ 用户拒绝
CANCELLED（已取消退款）

ACTIVE
    ↓ 逾期超过宽限期
LAPSED（已失效）
    ↓ 宽限期内补缴（#130续期缴费）
ACTIVE（恢复）

ACTIVE
    ↓ 保障期届满
MATURE（已满期）

UNDERWRITING / WAITING_CONDITION_CONFIRM / ACTIVE
    ↓ 核保拒保 / 用户拒绝条件 / 退保申请通过（#121状态变更）
REJECTED / CANCELLED

ACTIVE
    ↓ 投保人/保司申请中止（#121状态变更）
SUSPENDED（中止）
    ↓ 恢复缴费
ACTIVE（恢复）

ACTIVE / SUSPENDED
    ↓ 终止（#121状态变更）
TERMINATED（终止）
```

> 说明：`#166`（保单状态配置）中可设置参与续期操作的保单状态，默认支持"有效、退保、终止、中止"，C端"立即补缴"按钮需与此配置联动。

### 5.2 投保草稿（`ins_life_order_draft.status`）

```
0 进行中 → 1 已提交（支付发起后）
0 进行中 → 2 已过期（超24小时未完成，定时任务更新）
0 进行中 → 3 已放弃（用户主动放弃）
```

### 5.3 保全申请（`ins_life_conservation.status`）

> 对应 PDF `#126`（保全详情-撤销按钮逻辑）

```
PENDING（待审核）
    ↓ 客服审核通过
APPROVED（审核通过）
    ↓ 调用保司API执行变更
EXECUTED（已执行）

PENDING
    ↓ 客服审核拒绝
REJECTED（已拒绝）

PENDING（仅此状态可撤销）
    ↓ 用户自行撤销（#126：有撤销按钮则可自行撤销）
CANCELLED（已撤销）
```

## 六、重要注意事项与边界处理

**1. 数据脱敏**：所有被保人/受益人/投保人的身份证号在存入数据库前需加密存储（AES-256），前端展示时服务端解密后脱敏返回（如：`110***1990***001*`），不允许接口直接返回明文完整身份证号。手机号同理（如：`138****5678`）。

**2. 并发防重**：用户同一时间对同一产品只能存在一条 `status = 进行中` 的草稿。调用 `POST /app-api/ins/life/order/start` 时，先查询是否存在未过期的同产品草稿，若存在则返回已有 `draft_id` 并询问用户"您有未完成的投保申请，是否继续？"，确认后返回已有草稿，拒绝则将旧草稿设为"已放弃"，新建草稿。

**3. 支付超时处理**：支付订单创建后，若用户 30 分钟内未完成支付，XXL-Job 定时任务将订单状态设为 `EXPIRED`，同时将草稿状态回退到"缴费确认"步骤（`step = 3`），允许用户重新发起支付。

**4. 退款到账时间**：前端所有涉及退款的场景，展示文案统一为"退款将在 1-7 个工作日退回原支付账户，具体到账时间以支付平台为准"，不承诺具体天数。

**5. 保司API调用失败兜底**：核保 API 调用失败时，不直接拒保，而是将保单状态置为 `UNDERWRITING`，触发告警通知客服，由客服人工处理后手动更新核保结果。

**6. 分布式事务**：支付成功后写入正式保单数据（`ins_life_policy`、`ins_life_policy_insured`、`ins_order` 状态更新）必须在同一 Seata 分布式事务中完成，任一步骤失败则全部回滚，并通过 RocketMQ 延迟消息在 5 分钟后重试（最多重试 3 次）。

**7. 续期撤回联动**（参考 `#130`）：PC端管理员对续期跟踪记录执行"撤回"操作后，系统需触发 MQ 消息通知保单中台，若对应支付已完成则发起退款，并回退 `ins_life_policy.next_payment_date`，同时向用户推送"续期记录已被撤回，款项将退回原支付账户"通知。

**8. 回执/回访状态影响C端展示**（参考 `#111`、`#116`、`#118`）：已承保但未完成回执/回访的保单，C端保单详情页顶部展示提示条（黄色），内容为"您的保单正在处理中，如有疑问请联系顾问"；此状态下保单的"申请保全"功能禁用（灰色），防止在回执/回访未完成时发起保全变更。

**9. 自保件识别**（参考 `#166` 系统配置）：若系统开启了"自动识别自保件"配置，C端投保时后端需检测投保人/被保人/受益人是否为公司业务员，如是则在保单中标记 `is_self_insurance = 1`，保单详情页展示"自保件"标签，并在PC端相关报表中体现。
