## 一、产品库模块 - 业务逻辑补充

### 1.1 产品上下架业务逻辑

#### 1.1.1 产品下架触发条件

**自动下架场景:**

```
场景1: 保险公司通知产品停售
  触发方式: 管理员手动下架
  处理流程:
    1. 修改产品status=0(下架)
    2. 检查是否有正在使用该产品的草稿计划书
    3. 如有草稿计划书,发送通知给业务员
    4. 同步更新ElasticSearch索引(设置可搜索=false)
    5. 清除Redis缓存
    6. 记录下架日志(操作人、操作时间、原因)
  
场景2: 产品达到停售日期
  触发方式: 定时任务每日检查
  处理流程:
    1. 扫描产品表的stop_sale_date字段
    2. 如果stop_sale_date <= 当前日期,自动下架
    3. 批量更新status=2(停售)
    4. 发送停售通知给所有业务员
    5. 同步更新ES和Redis
```

**下架影响范围:**

```
影响1: 产品列表不再展示
影响2: 产品详情页显示"已停售"标识
影响3: 无法创建新的计划书包含该产品
影响4: 已生成的计划书不受影响(保留历史数据)
影响5: 搜索结果不再包含该产品
```

#### 1.1.2 产品信息变更同步逻辑

**变更类型与同步策略:**

```
类型1: 基础信息变更(产品名称、亮点、FAQ等)
  同步时机: 实时同步
  同步范围: MySQL → Redis → ElasticSearch
  同步方式: 监听UPDATE事件,异步更新

类型2: 费率调整(premium_formula、费率表)
  同步时机: 实时同步
  特殊处理:
    - 保留历史费率表(新增版本号)
    - 历史订单使用历史费率
    - 新订单使用新费率
    - 发送费率变更通知

类型3: 保障责任变更(coverage_detail)
  同步时机: 延迟同步(审核通过后)
  特殊处理:
    - 需要管理员审核
    - 审核通过后生效
    - 记录变更历史
    - 通知相关业务员
```

**数据一致性保证:**

```
机制1: 分布式锁
  - 使用Redis分布式锁防止并发修改
  - 锁Key: product:lock:{productId}
  - 锁过期时间: 30秒

机制2: 版本号机制
  - 每次更新检查version字段
  - UPDATE product SET ... WHERE id=? AND version=?
  - 如果version不匹配,返回更新失败

机制3: 事务保证
  - MySQL更新、Redis删除、ES更新在一个分布式事务中
  - 如果ES更新失败,记录失败日志,定时重试
  - 使用消息队列(RocketMQ)保证最终一致性
```

### 1.2 产品对比深度逻辑

#### 1.2.1 对比产品数量限制

**限制规则:**

```
最少对比: 2个产品
最多对比: 4个产品
原因: 超过4个产品,对比表格过宽,用户体验差
```

**异常处理:**

```
场景1: 用户选择少于2个产品
  处理: 前端禁用"对比"按钮
  提示: "请至少选择2个产品进行对比"

场景2: 用户选择超过4个产品
  处理: 前端最多允许选择4个
  提示: "最多对比4个产品,请取消勾选部分产品"

场景3: 选择的产品已下架
  处理: 后端过滤已下架产品
  提示: "部分产品已停售,已自动移除"
```

#### 1.2.2 对比维度动态生成

**动态维度提取算法:**

```
输入: 产品A、产品B、产品C的coverage_detail JSON

步骤1: 提取所有产品的责任项
  产品A责任: [重疾赔付, 轻症赔付, 中症赔付, 身故保障]
  产品B责任: [重疾赔付, 轻症赔付, 身故保障, 特定疾病额外赔]
  产品C责任: [重疾赔付, 轻症赔付, 医疗费用报销]

步骤2: 合并去重,得到全量责任项
  全量责任: [重疾赔付, 轻症赔付, 中症赔付, 身故保障, 特定疾病额外赔, 医疗费用报销]

步骤3: 按重要性排序
  排序规则:
    - 核心责任优先(重疾、轻症、中症、身故)
    - 附加责任其次(特定疾病、医疗费用)
    - 增值服务最后(绿通、特药)

步骤4: 标记差异项
  - 只有部分产品有的责任 → 标记为"差异项"
  - 所有产品都有的责任 → 标记为"共同项"

步骤5: 构建对比矩阵
  [
    {责任名称: "重疾赔付", 产品A: "100%保额", 产品B: "100%保额", 产品C: "100%保额", 类型: "共同"},
    {责任名称: "中症赔付", 产品A: "50%保额", 产品B: "-", 产品C: "-", 类型: "差异"},
    ...
  ]
```

**差异高亮逻辑:**

```
数值型责任:
  - 赔付比例: 最高值绿色高亮
  - 赔付次数: 最多次数绿色高亮
  - 等待期: 最短等待期绿色高亮

布尔型责任:
  - 有该责任: 显示 "√" + 绿色
  - 无该责任: 显示 "-" + 灰色

独有责任:
  - 只有该产品有: 红色星号标记 + "独有"标签
```

#### 1.3.1 搜索关键词处理

**关键词预处理:**

```
步骤1: 去除特殊字符
  输入: "平安的@重疾险!"
  输出: "平安的重疾险"

步骤2: 分词处理

步骤3: 去除停用词
  停用词表: ["的", "吗", "呢", "啊", ...]
  结果: ["平安", "重疾险", "重疾", "险"]

步骤4: 提取核心词
  核心词: ["平安", "重疾险"]
```

**搜索意图识别:**

```
意图1: 搜索保险公司
  关键词: "平安"、"人寿"、"泰康"、"太平洋"
  处理: 优先搜索companyName字段

意图2: 搜索产品分类
  关键词: "重疾"、"医疗"、"意外"、"年金"

意图3: 搜索保障内容
  关键词: "癌症"、"心脏病"、"住院"、"身故"

意图4: 搜索产品名称
  关键词: 完整产品名或产品代码
  处理: 精确匹配productName和productCode
```

#### 1.3.2 搜索结果排序优化

**多维度排序算法:**

```
排序因子:
  F1 = 相关性得分(ES内置算分)
  F2 = 销量得分 = log10(salesCount + 1)
  F3 = 佣金得分 = commissionRate * 100
  F4 = 新品加权 = created_time > 30天前 ? 1.2 : 1.0

最终得分 = F1 * 0.5 + F2 * 0.3 + F3 * 0.1 + F4 * 0.1

排序规则:
  1. 按最终得分降序
  2. 得分相同时,按销量降序
  3. 销量相同时,按上架时间降序
```

**搜索结果过滤:**

```
过滤1: 状态过滤
  - 只返回status=1(在售)的产品
  - 已下架和停售产品不返回

过滤2: 年龄过滤(如果用户提供年龄)
  - 年龄 >= minAge AND 年龄 <= maxAge
  - 不符合年龄范围的产品不返回

过滤3: 职业过滤(如果用户提供职业类别)
  - occupation_class字段匹配
  - 高危职业过滤部分产品

过滤4: 保额过滤(如果用户提供保额预算)
  - 保额 >= minAmount AND 保额 <= maxAmount
```

**搜索日志记录:**

```
记录内容:
```

```
策略1: 搜索词联想

  - 用户输入时实时提示联想词
```

## 二、保费试算模块 - 业务逻辑补充

### 2.1 试算参数校验逻辑

#### 2.1.1 年龄校验规则

**校验维度:**

```
校验1: 投保年龄范围
  规则: minAge <= 年龄 <= maxAge
  示例: 某产品投保年龄0-60岁
    - 年龄 = -1 → 错误: "年龄不能为负数"
    - 年龄 = 70 → 错误: "超过最大投保年龄60岁"
    - 年龄 = 30 → 通过

校验2: 保障期限与年龄匹配
  规则: 年龄 + 保障年限 <= 合理上限
  示例: 保障至70岁
    - 年龄50岁, 保障至70岁 → 实际保障20年 → 通过
    - 年龄65岁, 保障至70岁 → 实际保障5年 → 通过但提示
    - 年龄75岁, 保障至70岁 → 错误: "年龄已超过保障期限"

校验3: 缴费期限与年龄匹配
  规则: 年龄 + 缴费年限 <= 退休年龄(通常60或65岁)
  示例: 缴费20年
    - 年龄30岁 → 50岁缴完 → 通过
    - 年龄50岁 → 70岁缴完 → 警告: "超过退休年龄,建议缩短缴费期"
```

**特殊年龄处理:**

```
场景1: 儿童投保(0-17岁)
  - 自动关联父母为投保人
  - 保费可能有优惠
  - 部分产品有儿童专属保障

场景2: 老年人投保(60岁以上)
  - 费率大幅提高
  - 健康告知更严格
  - 保额可能受限
  - 提示: "建议优先配置医疗险和防癌险"

场景3: 退休年龄临界点
  - 如果缴费期结束在退休后,提示调整
  - 建议: "可选择较短缴费期,降低后期压力"
```

#### 2.1.2 保额校验规则

**基础校验:**

```
校验1: 保额范围
  规则: minAmount <= 保额 <= maxAmount
  示例: 某重疾险保额范围10万-100万
    - 保额 = 5万 → 错误: "低于最低保额10万"
    - 保额 = 50万 → 通过
    - 保额 = 150万 → 错误: "超过最高保额100万"

校验2: 保额递增单位
  规则: 保额必须是increment的整数倍
  示例: 保额递增单位为5万
    - 保额 = 50万 → 通过
    - 保额 = 53万 → 错误: "保额请按5万的整数倍选择"
```

**收入倍数校验:**

```
规则: 保额不得超过年收入的N倍(通常5-10倍)
场景1: 重疾险
  - 年收入20万,建议保额50-200万
  - 如果保额>200万,警告: "保额过高,可能影响核保"

场景2: 寿险
  - 年收入50万,建议保额250-500万
  - 适用于家庭经济支柱

场景3: 医疗险
  - 保额通常固定(100万、200万、300万等)
  - 不做收入倍数校验
```

**保额匹配校验:**

```
规则1: 主附险保额匹配
  - 附加险保额 <= 主险保额
  - 示例: 主险50万,附加住院医疗险最多50万

规则2: 多产品保额协调
  - 重疾险 + 医疗险: 医疗险保额可大于重疾险
  - 重疾险 + 定期寿险: 寿险保额应≥重疾险保额

规则3: 职业保额限制
  - 高危职业(4-6类): 意外险保额受限
  - 示例: 6类职业意外险最高50万
```

### 2.2 保费计算引擎详细逻辑

#### 2.2.1 Groovy脚本执行机制

**脚本结构规范:**

```groovy
// 脚本模板
import java.math.BigDecimal

class PremiumCalculator {
    
    // 主计算方法
    static BigDecimal calculate(Map params) {
        // 1. 提取参数
        int age = params.age
        int gender = params.gender  // 1-男 2-女
        BigDecimal amount = params.amount
        int paymentYears = params.paymentYears
        int coverageYears = params.coverageYears
        int occupationClass = params.occupationClass
        
        // 2. 查询费率表
        Map rateTable = params.rateTable
        BigDecimal baseRate = getRateFromTable(rateTable, age, gender, paymentYears)
        
        // 3. 计算基础保费
        BigDecimal basePremium = amount.multiply(baseRate)
        
        // 4. 应用系数调整
        basePremium = applyOccupationFactor(basePremium, occupationClass)
        basePremium = applyCoverageFactor(basePremium, coverageYears)
        
        // 5. 应用优惠折扣
        basePremium = applyDiscount(basePremium, params)
        
        // 6. 四舍五入到元
        return basePremium.setScale(0, BigDecimal.ROUND_HALF_UP)
    }
    
    // 费率表查询
    static BigDecimal getRateFromTable(Map table, int age, int gender, int paymentYears) {
        String genderKey = gender == 1 ? "male" : "female"
        return table[genderKey][age.toString()][paymentYears.toString()]
    }
    
    // 职业系数
    static BigDecimal applyOccupationFactor(BigDecimal premium, int occupationClass) {
        Map factors = [
            1: 1.0, 2: 1.1, 3: 1.3, 
            4: 1.5, 5: 1.8, 6: 2.0
        ]
        return premium.multiply(new BigDecimal(factors[occupationClass]))
    }
    
    // 保障期限系数
    static BigDecimal applyCoverageFactor(BigDecimal premium, int coverageYears) {
        if (coverageYears == -1) {  // -1表示终身
            return premium.multiply(new BigDecimal("1.2"))
        }
        return premium
    }
    
    // 优惠折扣
    static BigDecimal applyDiscount(BigDecimal premium, Map params) {
        BigDecimal discount = new BigDecimal("1.0")
        
        // 非吸烟优惠
        if (params.nonSmoker) {
            discount = discount.multiply(new BigDecimal("0.95"))
        }
        
        // 线上投保优惠
        if (params.online) {
            discount = discount.multiply(new BigDecimal("0.98"))
        }
        
        return premium.multiply(discount)
    }
}

// 执行计算
PremiumCalculator.calculate(params)
```

```
  - 禁止文件读写
  - 禁止网络访问
  - 禁止反射调用
  - 禁止线程操作

限制2: 资源限制
  - 单次执行时间 <= 5秒

限制3: 沙箱隔离

```

#### 2.2.2 费率表版本管理

```
数据结构:

版本选择规则:
```

#### 2.2.3 保费试算缓存策略

**缓存Key设计:**

```
Key结构: 
  quotation:{productId}:{age}:{gender}:{amount}:{paymentYears}:{coverageYears}:{occupationClass}

示例:
  quotation:123:30:1:500000:20:70:1

Value: 
  {
    "annualPremium": 6580,
    "totalPremium": 131600,
    "calculateTime": "2025-02-15 10:30:00",
    "version": "v2.0"
  }

过期时间: 24小时
```

**缓存使用流程:**

```
流程1: 试算请求到达
  → 计算缓存Key
  → 查询Redis
  → 如果命中缓存,直接返回
  → 如果未命中,执行Groovy脚本计算
  → 计算结果写入Redis
  → 返回结果

流程2: 费率版本更新
  → 清除该产品的所有试算缓存
  → 使用Redis SCAN命令扫描: quotation:{productId}:*
  → 批量删除

流程3: 缓存预热
  → 新产品上线时,预先计算热门组合的保费
  → 典型年龄: 25, 30, 35, 40岁
  → 典型保额: 30万, 50万, 100万
  → 典型缴费期: 10年, 20年, 30年
```

### 2.3 利益演示深度逻辑

#### 2.3.1 现金价值计算

**现金价值定义:**

```
现金价值 = 退保时可以拿回的金额
影响因素:
  - 已缴保费
  - 保单年度
  - 产品类型(消费型vs储蓄型)
  - 投资收益(年金险)
```

**计算方法:**

```
方法1: 固定现金价值表(寿险、重疾险)
  - 产品配置cash_value_table字段
  - 格式: {年份: 现金价值}
  - 示例: {"1": 500, "2": 2000, "3": 4500, ...}
  - 直接查表返回

方法2: 动态计算(年金险、万能险)
  公式:
    现金价值 = Σ(已缴保费 - 已扣费用) × (1 + 投资收益率)^年份
  
  示例: 年金险
    年缴保费: 10000元
    缴费期: 10年
    保单年度: 第15年
    投资收益率: 4.5%
    
    已缴总保费 = 10000 × 10 = 100000
    已扣费用 = 初始费用 + 管理费用 ≈ 5000
    净值 = 100000 - 5000 = 95000
    累计收益 = 95000 × (1.045^15 - 1) ≈ 92000
    现金价值 = 95000 + 92000 = 187000元
```

**现金价值表生成:**

```
输入参数:
  - 年缴保费
  - 缴费期限
  - 保障期限(或年金领取期)
  - 产品类型

生成逻辑:
  for (年度 from 1 to 保障期限) {
    if (年度 <= 缴费期限) {
      累计保费 += 年缴保费
    }
    
    if (产品类型 == "消费型") {
      现金价值 = 累计保费 × 现金价值系数[年度]
    } else if (产品类型 == "储蓄型") {
      现金价值 = 累计保费 × (1 + 投资收益率)^年度 - 扣费
    }
    
    现金价值表[年度] = 现金价值
  }

输出: 
  [
    {year: 1, cashValue: 500, paidPremium: 10000},
    {year: 2, cashValue: 2000, paidPremium: 20000},
    ...
    {year: 20, cashValue: 150000, paidPremium: 200000}
  ]
```

#### 2.3.2 IRR(内部收益率)计算

**IRR含义:**

```
IRR = 使得净现值NPV=0的折现率
衡量指标: 投资回报率
应用场景: 年金险、增额终身寿等储蓄型产品
```

**NPV公式:**

```
NPV = Σ(现金流 / (1 + IRR)^年份) = 0

现金流定义:
  - 缴费期内: 现金流为负(支出保费)
  - 领取期内: 现金流为正(领取年金或现金价值)
```

**IRR计算算法:**

```
方法: 牛顿迭代法

步骤1: 初始化IRR猜测值 r = 0.05(5%)

步骤2: 计算NPV
  NPV = Σ(CF[t] / (1 + r)^t)
  其中CF[t]为第t年的现金流

步骤3: 计算NPV的导数
  NPV' = Σ(-t × CF[t] / (1 + r)^(t+1))

步骤4: 更新r
  r_new = r - NPV / NPV'

步骤5: 判断收敛
  if (|NPV| < 0.01) {
    收敛,返回r
  } else {
    r = r_new
    继续迭代
  }

步骤6: 迭代上限
  最多迭代100次,避免死循环
  如果不收敛,返回错误
```

**IRR计算示例:**

```
年金险案例:
  年缴保费: 10万
  缴费期: 5年
  60岁开始领取
  每年领取: 8万
  领取至80岁

现金流:
  第1年: -10万
  第2年: -10万
  ...
  第5年: -10万
  第6-25年: 0
  第26年: +8万
  ...
  第45年: +8万

计算IRR:
  初始r = 0.05
  迭代后r ≈ 0.038 (3.8%)
  
结论: 该年金险的内部收益率约为3.8%
```

**IRR结果解读:**

```
IRR > 5%: 收益较高,值得购买
IRR 3%-5%: 收益一般,适合稳健型客户
IRR < 3%: 收益较低,不如其他理财产品
IRR < 0%: 亏损,不建议购买
```

### 3.1 计划书状态流转逻辑

#### 3.1.1 状态定义与流转规则

**状态枚举:**

```

  - 点击"分享"按钮后的状态
  - 记录分享时间
  - 计入分享统计
  - 可追踪客户查看行为
```

**状态流转图:**

```
草稿(1) → 已生成(2) → 已分享(3)
  ↓          ↑
重新编辑 ←┘
```

**状态流转触发:**

```
触发1: 创建计划书
  状态: null → 草稿(1)
  条件: 填写标题和客户信息
  操作: 保存到数据库

触发2: 生成PDF
  状态: 草稿(1) → 已生成(2)

    - 至少配置1个产品

  操作:
    - 渲染模板
    - 生成PDF
    - 上传OSS
    - 生成二维码
    - 更新状态和URL字段

触发3: 分享计划书
  状态: 已生成(2) → 已分享(3)
  条件: PDF和H5链接已生成
  操作:
    - 更新状态
    - 记录分享时间
    - 生成分享海报(可选)

触发4: 重新编辑
  状态: 已生成(2) → 草稿(1)
  条件: 用户点击"重新编辑"
  操作:
    - 状态回退
    - 清空PDF和H5链接
```

```
限制2: 草稿状态不可直接分享
  原因: 未生成PDF和H5链接

限制3: 删除已分享的计划书需二次确认
```

### 3.2 产品方案配置深度逻辑

#### 3.2.1 产品组合合法性校验

**主附险关联校验:**

```
规则1: 附加险必须依附主险
  示例:
    主险: 平安福终身寿险(主险)
    附加险: 平安附加长期意外险
  校验逻辑:
    - 选择附加险时,必须先选择对应主险
    - 删除主险时,自动删除依附的附加险
    - 前端提示: "该附加险需要先选择主险XXX"

规则2: 附加险数量限制
  限制: 一个主险最多关联5个附加险
  原因: 过多附加险会导致方案复杂,不易理解
  处理: 超过5个时禁止继续添加
```

**保额匹配校验:**

```
规则1: 附加险保额 <= 主险保额
  示例:
    主险保额: 50万
    附加医疗险保额: 100万 → 错误
  处理: 自动调整附加险保额为50万

规则2: 意外险保额根据职业限制
  职业类别与保额上限:
    1-2类: 无限制
    3类: 最高200万
    4类: 最高100万
    5-6类: 最高50万
  校验: 超过上限时警告并自动调整

规则3: 多产品总保额协调
  场景: 重疾险 + 定期寿险
  建议: 定期寿险保额 >= 重疾险保额
  原因: 身故保障应覆盖重疾保障
  提示: "建议定期寿险保额不低于重疾险"
```

**缴费期限协调:**

```
规则: 尽量统一缴费期限
  原因: 
    - 便于客户记忆
    - 减少缴费混乱
    - 降低脱保风险
  
  示例:
    产品A: 缴费20年
    产品B: 缴费10年
    产品C: 缴费20年
  提示: "产品B的缴费期与其他产品不一致,是否调整为20年?"
```

#### 3.2.2 保障责任汇总算法

**汇总维度:**

```
维度1: 按责任类型
  - 重疾保障
  - 轻症保障
  - 中症保障
  - 身故保障
  - 全残保障
  - 医疗报销
  - 意外伤害
  - 意外医疗

维度2: 按赔付方式
  - 一次性给付
  - 多次给付
  - 费用报销
  - 津贴给付

维度3: 按产品
  - 产品A提供的保障
  - 产品B提供的保障
  - 产品C提供的保障
```

**汇总算法:**

```
输入: 
  产品列表 = [产品A, 产品B, 产品C]

步骤1: 提取每个产品的保障责任
  产品A.coverageDetail = {
    "重疾": {amount: 500000, type: "一次给付", limit: 1},
    "轻症": {amount: 150000, type: "多次给付", limit: 3}
  }
  
  产品B.coverageDetail = {
    "重疾": {amount: 300000, type: "一次给付", limit: 1},
    "身故": {amount: 300000, type: "一次给付", limit: 1}
  }

步骤2: 合并相同责任的保额
  重疾保障 = 产品A.重疾 + 产品B.重疾
           = 50万 + 30万
           = 80万

步骤3: 处理赔付次数
  轻症保障:
    - 产品A提供3次
    - 总计: 3次,每次15万

步骤4: 标记特色保障
  如果某个责任只有一个产品提供:
    标记为"特色保障"
    示例: 身故保障(仅产品B)

步骤5: 生成汇总表
  [
    {
      type: "重疾保障",
      totalAmount: 800000,
      source: "产品A(50万) + 产品B(30万)",
      paymentTimes: 1,
      special: false
    },
    {
      type: "轻症保障",
      totalAmount: 150000,
      source: "产品A",
      paymentTimes: 3,
      special: false
    },
    {
      type: "身故保障",
      totalAmount: 300000,
      source: "产品B",
      paymentTimes: 1,
      special: true
    }
  ]
```

**汇总展示优化:**

```
优化1: 按重要性排序
  排序规则:
    1. 重疾保障(最重要)
    2. 轻症/中症保障
    3. 身故/全残保障
    4. 医疗报销
    5. 意外伤害/医疗
    6. 其他增值服务

优化2: 可视化展示
  使用雷达图展示保障全面性:
    - 重疾保障维度
    - 医疗保障维度
    - 意外保障维度
    - 身故保障维度
    - 理财规划维度

优化3: 保障缺口提示
  对比标准保障需求:
    - 重疾保障缺口 = 标准保额 - 实际保额
    - 如果缺口 > 0,提示补充保障
```

#### 4.1.1 敏感字段加密

**需要加密的字段:**

```
字段3: 银行卡号(bank_account)

  特殊处理: 保留后4位明文
  存储格式: 前12-16位加密 + 后4位明文
```

**加密实现逻辑:**

```java
// 加密工具类
public class CryptoUtil {
    
    private static final String ALGORITHM = "AES/CBC/PKCS5Padding";
    private static final String KEY = getKeyFromConfigCenter();  // 从配置中心获取
    
    // 手机号加密
    public static String encryptMobile(String mobile) {
        if (StringUtils.isBlank(mobile)) {
            return null;
        }
        // 完全加密
        byte[] encrypted = aesEncrypt(mobile.getBytes(), KEY);
        return Base64.getEncoder().encodeToString(encrypted);
    }
    
    // 身份证号加密(保留后4位)
    public static String encryptIdCard(String idCard) {
        if (StringUtils.isBlank(idCard) || idCard.length() < 18) {
            return null;
        }
        // 前14位加密
        String prefix = idCard.substring(0, 14);
        byte[] encrypted = aesEncrypt(prefix.getBytes(), KEY);
        String encryptedPrefix = Base64.getEncoder().encodeToString(encrypted);
        
        // 后4位明文
        String suffix = idCard.substring(14);
        
        return encryptedPrefix + "|" + suffix;
    }
    
    // 解密
    public static String decrypt(String encrypted) {
        byte[] decrypted = aesDecrypt(Base64.getDecoder().decode(encrypted), KEY);
        return new String(decrypted);
    }
}
```

**查询处理:**

```

  操作: INCR order:seq:20250215
  特点:
    - 线程安全
    - 高性能
    - 每天自动重置

  代码逻辑:
    String today = DateUtil.format(new Date(), "yyyyMMdd");
    String key = "order:seq:" + today;
    Long seq = redisTemplate.opsForValue().increment(key, 1);
    
    // 设置过期时间(第二天凌晨)
    if (seq == 1) {
      long expireSeconds = getSecondsUntilMidnight();
      redisTemplate.expire(key, expireSeconds, TimeUnit.SECONDS);
    }
    
    // 格式化流水号(补足8位)
    String seqStr = String.format("%08d", seq);
    String orderNo = "ORD" + today + seqStr;
    
    return orderNo;

方案2: 使用数据库自增ID
  缺点: 性能较低,存在并发问题
  不推荐

方案3: 雪花算法(Snowflake)
  特点: 分布式唯一ID,但不够人性化
  不适合订单号(订单号需要可读性)
```

**唯一性保证:**

```
保证机制:
  1. Redis INCR保证单机唯一性
  2. 订单表设置UNIQUE索引
  3. 插入前检查订单号是否存在
  4. 如果存在,重新生成(理论上不会发生)

异常处理:
  if (orderMapper.existsByOrderNo(orderNo)) {
    // 重新生成
    orderNo = generateOrderNo();
  }
```

#### 5.1.2 订单创建前置校验

**校验项:**

```
1. 产品状态校验
   规则: 产品必须在售(status=1)

   规则: 必填字段(姓名、身份证、手机号)已填写
```

### 5.2 订单状态流转深度逻辑

#### 5.2.1 状态机设计

**状态定义:**

```
状态1: 待支付(PENDING_PAYMENT = 1)
  含义: 订单已创建,等待客户支付
  允许操作: 支付、取消

状态2: 核保中(UNDERWRITING = 2)
  含义: 客户已支付,保险公司核保中
  允许操作: 查看进度、撤单(扣手续费)

状态3: 已承保(INSURED = 3)
  含义: 核保通过,保单已生效
  允许操作: 查看保单、申请理赔

状态4: 已取消(CANCELLED = 4)
  含义: 订单已取消
  子状态:
    - 用户主动取消
    - 支付超时取消
    - 核保拒保
  允许操作: 查看取消原因
```

**状态流转矩阵:**

```
         │ 待支付 │ 核保中 │ 已承保 │ 已取消
─────────┼────────┼────────┼────────┼────────
待支付   │   ✓    │   支付  │   ✗    │  取消
核保中   │   ✗    │   ✓    │  通过  │  拒保
已承保   │   ✗    │   ✗    │   ✓    │   ✗
已取消   │   ✗    │   ✗    │   ✗    │   ✓

说明:
  ✓ - 允许处于该状态
  ✗ - 不允许流转
  文字 - 流转条件
```

**流转触发机制:**

```
触发1: 支付回调
  from: 待支付(1)
  to: 核保中(2)
  条件: 支付成功
  操作:
    - 更新order_status=2
    - 更新payment_status=1
    - 更新payment_time=当前时间
    - 调用保险公司核保接口
    - 发送核保中通知给客户

触发2: 核保结果回调
  from: 核保中(2)
  to: 已承保(3) 或 已取消(4)
  条件: 核保通过/拒保
  操作:
    - 更新order_status=3或4
    - 更新issue_time=承保时间
    - 更新policy_no=保单号
    - 计算并记录佣金
    - 发送承保/拒保通知

触发3: 用户撤单
  from: 待支付(1) 或 核保中(2)
  to: 已取消(4)
  条件: 用户主动取消
  操作:
    - 更新order_status=4
    - 如已支付,发起退款
    - 如核保中,通知保险公司撤单
    - 记录取消原因

触发4: 支付超时
  from: 待支付(1)
  to: 已取消(4)
  条件: 创建24小时后仍未支付
  操作:
    - 定时任务扫描
    - 批量更新order_status=4
    - 记录取消原因="支付超时"
```

### 6.1 业绩指标计算逻辑

#### 6.1.1 核心指标定义

**指标1: 件数(保单数量)**

```
定义: 已承保订单数量

  SELECT COUNT(*) FROM ins_order
```

**指标2: 保费(总保费)**

```
定义: 已承保订单的保费总和
SQL:
  SELECT SUM(premium) FROM ins_order

  AND order_status = 3
  AND issue_time BETWEEN ? AND ?

注意: 
  - 只统计首年保费
  - 续年保费单独统计
```

**指标3: FYC(首年佣金)**

```
定义: 首年佣金总额
SQL:
  SELECT SUM(commission_amount) FROM ins_commission
  WHERE agent_id = ?
  AND commission_type = 1  -- FYC
  AND created_time BETWEEN ? AND ?
```

**指标4: RYC(续年佣金)**

```
定义: 续年佣金总额
SQL:
  SELECT SUM(commission_amount) FROM ins_commission
  WHERE agent_id = ?
  AND commission_type = 2  -- RYC
  AND created_time BETWEEN ? AND ?
```

**指标5: 件均保费**

```
定义: 总保费 / 件数
公式: 件均保费 = Σ保费 / 件数

示例:

  件数: 10件
  件均保费: 5万/件
```

**指标6: 成交率**

```
定义: 已承保订单数 / 总订单数
公式: 成交率 = 已承保订单数 / (已承保 + 已取消 + 待支付 + 核保中)

示例:
  已承保: 20单
  总订单: 50单
  成交率: 40%
```

#### 6.1.2 统计周期处理

**周期类型:**

```
日: 今日、昨日、本周、本月
周: 本周、上周、近4周、近12周
月: 本月、上月、近3月、近6月、近12月
季: 本季、上季、近4季
年: 本年、去年、近3年
自定义: 指定开始和结束日期
```

**周期计算逻辑:**

```
1. 今日
   startDate = 今天00:00:00
   endDate = 今天23:59:59

2. 本周(周一到周日)
   startDate = 本周周一00:00:00
   endDate = 本周周日23:59:59

3. 本月
   startDate = 本月1日00:00:00
   endDate = 本月最后一天23:59:59

4. 本季
   Q1: 1-3月
   Q2: 4-6月
   Q3: 7-9月
   Q4: 10-12月
   
   if (当前月份 <= 3) {
     startDate = 1月1日
     endDate = 3月31日
   } else if (当前月份 <= 6) {
     startDate = 4月1日
     endDate = 6月30日
   } ...

5. 近N个月/周
   startDate = 当前日期 - N个月/周
   endDate = 当前日期
```

#### 6.2.2 同比环比计算

**同比增长率:**

```
定义: 与去年同期相比的增长率

公式:
  同比增长率 = (本期值 - 去年同期值) / 去年同期值 × 100%

示例:
  本月保费: 30万
  去年同月保费: 25万
  同比增长率 = (30 - 25) / 25 × 100% = 20%

计算逻辑:
  currentMonth = "2025-02"
  lastYearMonth = "2024-02"
  
  currentValue = queryPremium(agentId, currentMonth)
  lastYearValue = queryPremium(agentId, lastYearMonth)
  
  if (lastYearValue == 0) {
    yoyGrowth = null  // 无法计算
  } else {
    yoyGrowth = (currentValue - lastYearValue) / lastYearValue * 100
  }
```

**环比增长率:**

```
定义: 与上一期相比的增长率

公式:
  环比增长率 = (本期值 - 上期值) / 上期值 × 100%

示例:
  本月保费: 30万
  上月保费: 28万
  环比增长率 = (30 - 28) / 28 × 100% = 7.14%

计算逻辑:
  currentMonth = "2025-02"
  lastMonth = "2025-01"
  
  currentValue = queryPremium(agentId, currentMonth)
  lastValue = queryPremium(agentId, lastMonth)
  
  if (lastValue == 0) {
    momGrowth = null
  } else {
    momGrowth = (currentValue - lastValue) / lastValue * 100
  }
```

## 七、佣金管理模块 - 业务逻辑补充

### 7.1 佣金计算深度逻辑

#### 7.1.1 多级佣金体系

**佣金层级:**

```
层级1: 直接佣金(业务员)
  - 自己出单,获得全额佣金
  - 比例: 产品佣金率(如60%)

层级2: 管理津贴(团队长)
  - 团队成员出单,获得管理津贴
  - 比例: 10-15%

层级3: 机构佣金(分公司经理)
  - 分公司所有订单,获得机构佣金
  - 比例: 3-5%

层级4: 总部佣金(公司)
  - 所有订单的剩余佣金归总部
```

**佣金计算示例:**

```
假设:
  订单保费: 10000元
  产品佣金率: 60%
  总佣金: 10000 × 0.6 = 6000元

分配:
  业务员A(直接佣金): 6000 × 100% = 6000元
  
  如果A有上级团队长B:
    业务员A: 6000 × 85% = 5100元
    团队长B: 6000 × 15% = 900元
  
  如果B有上级经理C:
    业务员A: 6000 × 85% = 5100元
    团队长B: 6000 × 10% = 600元
    经理C: 6000 × 5% = 300元
```

**佣金计算触发:**

```
触发时机: 订单状态变为"已承保"

计算流程:
  1. 获取订单信息(保费、产品佣金率)
  2. 计算总佣金 = 保费 × 佣金率
  3. 查询业务员的上级关系链
  4. 按层级分配佣金
  5. 生成佣金记录
  6. 更新佣金账户余额
```

### 7.2 提现流程深度逻辑

#### 7.2.1 提现条件校验

**校验项:**

```
1. 实名认证校验
   规则: 必须完成实名认证
   校验: 查询agent表的id_card_verified字段
   错误: "请先完成实名认证"

2. 银行卡绑定校验
   规则: 必须绑定银行卡
   校验: 查询bank_account字段是否为空
   错误: "请先绑定提现银行卡"

3. 可提现金额校验
   规则: 提现金额 <= 已结算佣金余额
   计算: 
     已结算佣金 = SUM(commission WHERE settlement_status=1)
     已提现金额 = SUM(withdrawal WHERE status IN (2,3))
     可提现金额 = 已结算佣金 - 已提现金额
   错误: "可提现金额不足"

4. 最低提现金额校验
   规则: 提现金额 >= 100元
   错误: "提现金额不得低于100元"

5. 单日提现次数限制
   规则: 每天最多提现3次
   校验: 查询今日提现记录数量
   错误: "您今日提现次数已达上限"

6. 提现手续费计算
   规则: 
     - 提现金额 < 1000元: 免手续费
     - 提现金额 >= 1000元: 手续费2元/笔
   计算:
     if (金额 >= 1000) {
       手续费 = 2元
       实际到账 = 提现金额 - 2
     }
```

## 八、性能优化 - 业务逻辑补充

### 8.1 缓存策略深化

#### 8.1.1 多级缓存架构

**缓存层次:**

```
L1缓存: 本地缓存(Caffeine)
  - 容量: 10000条
  - 过期时间: 5分钟
  - 适用: 热点数据(产品详情、字典数据)

L2缓存: Redis缓存
  - 过期时间: 1小时 ~ 1天
  - 适用: 产品列表、客户信息、试算结果

L3缓存: 数据库
  - 持久化存储
  - 适用: 所有数据
```

**缓存读取流程:**

```
查询产品详情:
  1. 查L1缓存(Caffeine)
  2. 如果命中,直接返回
  3. 如果未命中,查L2缓存(Redis)
  4. 如果命中,写入L1,返回
  5. 如果未命中,查数据库
  6. 写入L2和L1,返回
```

**缓存更新策略:**

```
策略1: 主动失效(推荐)
  - 数据更新时,主动删除缓存
  - 下次查询时自动加载最新数据

策略2: 定时刷新
  - 定时任务每隔N分钟刷新缓存
  - 适用于准实时数据

策略3: 延迟双删
  - 更新数据库
  - 删除缓存
  - 等待1秒
  - 再次删除缓存(防止脏数据)
```

#### 8.1.2 缓存穿透、击穿、雪崩防护

**缓存穿透:**

```
问题: 查询不存在的数据,缓存和DB都没有,每次都查DB

防护方案1: 布隆过滤器
  - 在Redis中维护布隆过滤器
  - 记录所有存在的productId
  - 查询前先判断,如果不存在直接返回

防护方案2: 缓存空值
  - 查询不存在的数据时,缓存null值
  - 设置较短过期时间(5分钟)
  - 下次查询直接返回null,不查DB
```

**缓存击穿:**

```
问题: 热点数据缓存过期,大量请求同时查DB

防护方案: 分布式锁
  public ProductDO getProduct(Long id) {
    String cacheKey = "product:" + id;
    
    // 1. 查缓存
    ProductDO cached = redis.get(cacheKey);
    if (cached != null) {
      return cached;
    }
    
    // 2. 缓存未命中,加锁
    String lockKey = "lock:product:" + id;
    boolean locked = redis.setNX(lockKey, "1", 10);  // 10秒锁
    
    if (locked) {
      try {
        // 3. 查DB
        ProductDO product = db.selectById(id);
        
        // 4. 写缓存
        redis.set(cacheKey, product, 3600);
        
        return product;
      } finally {
        // 5. 释放锁
        redis.del(lockKey);
      }
    } else {
      // 6. 未获取锁,等待后重试
      Thread.sleep(100);
      return getProduct(id);  // 递归重试
    }
  }
```

**缓存雪崩:**

```
问题: 大量缓存同时过期,DB压力骤增

防护方案1: 过期时间随机化
  int expireTime = baseTime + random(0, 300);  // 基础时间 + 0-5分钟随机
  redis.set(key, value, expireTime);

防护方案2: 永不过期 + 后台更新
  - 缓存设置为永不过期
  - 定时任务后台刷新缓存
  - 适用于核心数据

防护方案3: 限流降级
  - 如果DB压力过大,触发降级
  - 返回默认值或提示"系统繁忙"
```

```
    CREATE INDEX idx_mobile ON ins_customer(mobile);

    SELECT * FROM ins_order
    WHERE agent_id = 123

    WHERE created_time >= '2025-02-15 00:00:00'
```
