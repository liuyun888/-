## 模块一：用户体系

### 1.1 注册登录

**工时：前端1天 + 后端1天**

#### 页面与交互

- 首次进入 App / H5，弹出登录页（全屏遮罩）。
- 页面字段：手机号输入框（必填）、图形验证区（滑动/点选）、短信验证码输入框（必填）、"获取验证码"按钮、勾选"已阅读并同意《用户协议》《隐私政策》"（必须勾选，否则按钮置灰）。
- 点击"获取验证码"：先校验手机号格式（`^1[3-9]\d{9}$`），未通过提示"请输入正确的手机号"；通过后弹出图形验证（腾讯云 CAPTCHA 或极验），图形验证通过后调接口发短信。
- 验证码发送成功：按钮变为倒计时 60 秒，不可重复点击。
- 验证码输入框：6 位数字，输入完毕自动提交（无需点击按钮）。
- 验证通过：自动判断是否首次注册，首次则创建用户，否则直接登录，两种情况均跳转首页。
- 登录失败（错误/过期/锁定）：原地提示具体原因，不跳页。

#### 后端业务逻辑

**发送短信（POST /api/c/user/sms/send）**
1. 接收参数：`mobile`、`scene`（1注册登录）、`captchaToken`（图形验证 token）。
2. 向腾讯云/极验校验 `captchaToken`，失败返回"图形验证失败"。
3. 查询 Redis key=`sms:limit:mobile:{mobile}`：若值 ≥ 1 且距上次发送 < 60 秒，返回"发送过于频繁，请 XX 秒后重试"。
4. 查询 Redis key=`sms:limit:ip:{ip}`：值 ≥ 10 返回"该 IP 请求超限"。
5. 生成 6 位随机数字验证码。
6. 调用短信服务（异步，失败重试 3 次），发送短信内容"【xx保险】您的验证码为 XXXXXX，5 分钟内有效。"
7. 将验证码存入 Redis，key=`sms:code:{mobile}:{scene}`，TTL=300 秒，同时存入 `tb_sms_log`。
8. Redis 计数器 `sms:limit:mobile:{mobile}` 加 1，TTL=60 秒；`sms:limit:ip:{ip}` 加 1，TTL=3600 秒。

**登录（POST /api/c/user/login）**
1. 接收参数：`mobile`、`smsCode`。
2. 从 Redis 取 `sms:code:{mobile}:1`，不存在返回"验证码已过期"。
3. 错误次数检查：Redis key=`sms:error:{mobile}` ≥ 5，返回"验证码错误次数过多，已锁定 30 分钟"，TTL=1800 秒。
4. 验证码不匹配：错误次数 +1，返回"验证码错误，还可尝试 X 次"。
5. 验证通过：删除 Redis 验证码和错误计数。
6. 查询 `tb_user` 是否存在该手机号：不存在则插入新用户记录（status=0，member_level=0，points=0）。
7. 检查用户 status：1=冻结（返回冻结原因+申诉入口），2=注销（提示已注销）。
8. 更新 `last_login_time`、`last_login_ip`，插入 `tb_login_log`。
9. 检测是否异地登录（IP 归属地与上次登录省份不同）：是则异步发短信提醒。
10. 签发 JWT：AccessToken（有效期 7 天）+ RefreshToken（有效期 30 天），payload 含 `userId`、`mobile`。
11. 返回：`accessToken`、`refreshToken`、`userInfo`（userId、nickname、avatar、memberLevel）。

#### 数据库

```sql
-- 用户表
tb_user
  user_id       BIGINT PK AUTO_INCREMENT
  mobile        VARCHAR(11) UNIQUE NOT NULL
  nickname      VARCHAR(50)                   -- 默认"用户+手机号后4位"
  avatar        VARCHAR(255)                  -- 头像URL
  gender        TINYINT DEFAULT 0             -- 0未知1男2女
  birthday      DATE
  province      VARCHAR(20)
  city          VARCHAR(20)
  member_level  TINYINT DEFAULT 0             -- 0普通1银卡2金卡3钻石
  points        INT DEFAULT 0
  balance       BIGINT DEFAULT 0              -- 余额（分）
  status        TINYINT DEFAULT 0             -- 0正常1冻结2注销
  freeze_reason VARCHAR(255)
  register_time DATETIME NOT NULL
  last_login_time DATETIME
  last_login_ip VARCHAR(50)
  creator/updater/create_time/update_time/deleted/tenant_id  -- 框架标准字段
  INDEX(mobile), INDEX(status, deleted)

-- 短信日志表
tb_sms_log
  id            BIGINT PK AUTO_INCREMENT
  mobile        VARCHAR(11) NOT NULL
  code          VARCHAR(10) NOT NULL           -- 存加密值（MD5）
  scene         TINYINT NOT NULL               -- 1注册登录2找回密码
  verify_status TINYINT DEFAULT 0              -- 0未验证1已验证2过期
  verify_time   DATETIME
  ip_address    VARCHAR(50)
  send_time     DATETIME NOT NULL
  expire_time   DATETIME NOT NULL
  INDEX(mobile, scene, send_time), INDEX(ip_address, send_time)

-- 登录日志表
tb_login_log
  id, user_id, mobile, login_ip, login_region, device_type, login_time, login_result
  INDEX(user_id, login_time)
```

### 1.2 微信授权

**工时：前端1天 + 后端1天**

#### 页面与交互

- 登录页展示"微信一键登录"按钮。
- 点击后：在微信内调用 JSSDK 授权；在 APP 内调起微信 APP 授权；在 H5 外部浏览器跳转授权 URL。
- 授权成功后：若该微信已绑定手机号，直接登录跳首页；若未绑定，弹出"绑定手机号"弹窗（字段：手机号、验证码），可选择"暂不绑定"跳过。
- 解绑微信：个人中心 → 账号安全 → 解绑微信 → 确认弹窗（提示"解绑后需使用手机号登录"）→ 确认解绑。

#### 后端业务逻辑

**微信回调（POST /api/c/wechat/callback）**
1. 接收 `code`、`state`；校验 `state` 是否在 Redis 存在（防 CSRF），不存在返回"授权已过期"。
2. 用 `code` 换取 `access_token` 和 `openid`（调微信 API），失败返回"微信授权失败"。
3. 用 `access_token` + `openid` 拉取用户信息（昵称、头像）。
4. 尽量获取 `unionid`（需应用已接入开放平台）。
5. 查 `tb_user` where `wechat_openid = ?`：已存在则直接登录（同 1.1 签发 JWT 步骤）；不存在则创建用户（mobile 为空），保存 openid/unionid/昵称/头像，签发 JWT。
6. 返回 JWT + `needBindMobile`（true/false）。

**绑定手机（POST /api/c/user/bindMobile）**
1. 校验当前登录用户 mobile 字段是否为空，非空返回"已绑定手机号"。
2. 校验短信验证码（同登录逻辑）。
3. 检查手机号是否已被其他用户注册：已注册则提示"该手机号已被其他账号使用，是否合并？"（引导账号合并流程）。
4. 更新 `tb_user.mobile`，记录 `bind_mobile_time`。

#### 数据库（在 tb_user 追加字段）

```
wechat_openid     VARCHAR(64)   INDEX
wechat_unionid    VARCHAR(64)   INDEX
wechat_nickname   VARCHAR(100)
wechat_avatar     VARCHAR(255)
bind_wechat_time  DATETIME
bind_mobile_time  DATETIME
```

### 1.3 实名认证

**工时：前端1天 + 后端1天**

#### 页面与交互

- 入口：个人中心 → 实名认证（未认证显示"去认证"红点提示）。
- 投保流程中若未实名，自动跳转至认证页。
- 页面：拍照上传身份证正面、身份证反面（各有"拍照"和"从相册选取"入口）；上传后自动 OCR 识别填充姓名、身份证号、有效期；用户确认或手动修改；点击"提交认证"。
- 认证状态展示：未认证 / 认证中 / 已认证（显示姓名和证件号后4位）/ 认证失败（显示失败原因）。

#### 后端业务逻辑

**提交认证（POST /api/c/user/realname/submit）**
1. 校验用户是否已实名认证（status=1 已认证则直接返回，禁止重复提交）。
2. 接收参数：`realName`（必填，2-20个汉字）、`idCardNo`（必填，18位，最后一位大写 X）、`idCardFront`（OSS URL）、`idCardBack`（OSS URL）。
3. 校验身份证号格式（正则+校验位验证）；校验姓名与身份证是否匹配（可选：调二要素认证 API）。
4. 调用人脸核身 SDK 前置OCR识别：将 `idCardFront` 发给 OCR 服务，返回姓名、证件号、有效期，与用户填写内容比对，偏差大则提示"证件信息与照片不符"。
5. 创建 `tb_user_realname` 记录，status=0（审核中）；更新 `tb_user.realname_status=1`。
6. 异步发起第三方实名认证（阿里云实人认证 / 腾讯云慧眼），结果回调更新 status。
7. 认证通过：status=2，更新 `tb_user.real_name`、`id_card_no`（脱敏存储：`3101**********1234`）、`realname_status=2`。
8. 认证失败：status=3，记录 `fail_reason`，更新 `tb_user.realname_status=3`，推送消息通知用户。

#### 数据库

```sql
tb_user_realname
  id, user_id BIGINT NOT NULL
  real_name     VARCHAR(50) NOT NULL
  id_card_no    VARCHAR(50) NOT NULL        -- 脱敏存储
  id_card_no_encrypt VARCHAR(255)           -- AES加密的完整号
  id_card_front VARCHAR(255)               -- OSS URL
  id_card_back  VARCHAR(255)
  status        TINYINT DEFAULT 0          -- 0审核中1通过2失败
  fail_reason   VARCHAR(255)
  submit_time   DATETIME NOT NULL
  verify_time   DATETIME
  verify_channel VARCHAR(50)               -- 认证渠道
  UNIQUE(user_id)
```

### 1.4 人脸识别

**工时：前端1天 + 后端1.5天**

#### 页面与交互

- 场景：高保额投保（保额≥50万）、账户提现时触发。
- 进入人脸识别页：展示提示文字"请正对摄像头，保持光线充足"→ 开始录制（3 秒活体动作检测：点头/摇头/张嘴）→ 提交识别。
- 识别中：展示 loading；识别完成：成功则继续流程，失败显示原因和"重试"按钮（最多3次）。

#### 后端业务逻辑

**发起人脸识别（POST /api/c/user/face/start）**
1. 校验用户已完成实名认证，否则引导先实名。
2. 调用腾讯云 / 阿里云慧眼 SDK 创建核验任务，返回 `bizToken`（有效期 5 分钟）。
3. 存 Redis key=`face:token:{userId}`，TTL=300。

**验证结果（POST /api/c/user/face/verify）**
1. 接收 `bizToken`，从 Redis 校验有效性。
2. 调 SDK 查询核验结果：相似度 ≥ 60 认证通过；< 60 失败。
3. 连续失败 3 次，锁定24小时（Redis key=`face:lock:{userId}`，TTL=86400）。
4. 通过：更新 `tb_user.face_verify_status=1`，记录 `tb_user_face_log`。
5. 入库：`tb_user_face_log`（user_id, biz_token, similarity, verify_result, fail_reason, verify_time）。

### 1.5 个人中心

**工时：前端1.5天 + 后端1天**

#### 页面与交互

- 个人中心页展示：头像（可点击修改）、昵称（可点击修改）、手机号（脱敏）、会员等级角标。
- 功能卡片区：我的保单、我的订单、实名认证、账号安全、消息通知设置、帮助与反馈、退出登录。
- 编辑昵称：点击昵称 → 弹出输入框，限2-20字符，不含特殊符号，点击保存。

- 退出登录：弹确认弹窗，确认后清除本地 token，跳登录页。

#### 后端业务逻辑

**获取个人信息（GET /api/c/user/info）**：返回用户完整资料（头像、昵称、手机号脱敏、会员等级、积分、实名状态）。

**更新个人信息（PUT /api/c/user/info）**
1. 可更新字段：nickname、avatar、gender、birthday、province、city。
2. 昵称：过敏词过滤（调内容安全 API），违规返回"昵称含有不当内容"。
3. 头像：校验 OSS URL 合法性（域名白名单）；异步做图片安全审核（暴力/色情检测）。
4. 更新 `tb_user` 对应字段。

### 1.6 家人管理

**工时：前端1天 + 后端0.5天**

#### 页面与交互

- 入口：个人中心 → 家人管理。
- 列表页：展示已添加家庭成员（头像、姓名、关系、年龄），最多添加10人，超出提示"最多添加10位家庭成员"。
- 添加成员：点击"+"弹出表单：

  - 与本人关系（必填，下拉选：配偶、子女、父母、兄弟姐妹、其他）

  - 证件号（必填，按证件类型做格式校验）
  - 出生日期（必填，日期选择器，不能晚于今天）
  - 手机号（选填）
- 点击保存：后端校验并入库，成功刷新列表。

**保存家庭成员（POST /api/c/user/family）**
1. 校验同一用户下同姓名+证件号不能重复。

3. 出生日期与身份证号出生日期段（7-14位）需匹配（仅身份证）。
4. 成员数量校验：该用户已有成员数 ≥ 10 则拒绝。
5. 入库 `tb_user_family`。

## 模块二：商城首页

### 2.1 Banner 轮播

**工时：前端0.5天 + 后端0.5天**

#### 页面与交互

- 首页顶部展示 Banner 轮播，图片宽高比 16:9（建议 750×422px）。
- 自动播放（间隔3秒），底部显示圆点指示器，支持手动左右滑动。
- 每张 Banner 可配置跳转链接（站内页面或外部 URL）；点击触发跳转。
- 最多展示8张，按配置排序。

#### 后端逻辑

**GET /api/c/home/banners**
1. 查 `tb_banner` where `status=1 AND type=1（首页Banner）AND now() BETWEEN start_time AND end_time`，按 `sort` 升序。
2. 结果缓存 Redis，key=`cache:home:banners`，TTL=300 秒。
3. 运营后台修改 Banner 后清除该缓存。

#### 数据库

```sql
tb_banner
  id, title VARCHAR(100), image_url VARCHAR(255)
```

### 3.2 筛选条件

**工时：前端1.5天 + 后端1天**

#### 页面与交互

- 产品列表页右上角"筛选"按钮 → 从右侧滑出筛选面板（半屏抽屉）。
- 筛选项：
  - 保险公司（多选，Checkbox，展示公司 logo + 名称，最多选5个）
  - 价格区间（范围滑块或快捷标签：100以下/100-300/300-500/500以上）
  - 保障期限（单选：1年/多年/终身）
  - 被保年龄（范围输入：最小年龄-最大年龄）
  - 缴费方式（多选：一次性/年缴/月缴）
- 底部两个按钮：重置（清空所有筛选）、确定（关闭面板，刷新列表）。
- 已选条件在列表顶部以 Tag 形式展示，单个 Tag 右侧有"×"可单独删除。

#### 后端逻辑

**GET /api/c/product/list**（筛选 + 排序 + 分页合一接口）
参数：`categoryId`、`insurerIds`（多值，逗号分隔）、`priceMin`、`priceMax`、`coverPeriod`（1/multi/lifelong）、`ageMin`、`ageMax`、`payMethod`（多值）、`sort`（comprehensive/sales/priceAsc/priceDesc）、`page`、`pageSize`。

SQL 构建逻辑：
1. 基础条件：`tb_product.status = 1`（上架）、`deleted = 0`。
2. 分类筛选：若传 `categoryId`，关联 `tb_product_category` 做分类过滤（含子分类，用 `parent_id` 递归或提前存完整路径）。
3. 保险公司：若传 `insurerIds`，`WHERE insurer_id IN (...)`。
4. 价格区间：`WHERE min_price BETWEEN priceMin AND priceMax`（保费使用最低保费字段）。
5. 保障期限：`WHERE cover_period_type = ?`。
6. 年龄：`WHERE min_insure_age <= ageMin AND max_insure_age >= ageMax`（查询用户年龄范围内可买的产品）。
7. 缴费：`WHERE pay_method LIKE '%monthly%'`（JSON 字段或逗号分隔字段）。
8. 分页：使用 MyBatis Plus 分页插件。
9. 结果不缓存（动态筛选），但产品基础信息可二级缓存。
