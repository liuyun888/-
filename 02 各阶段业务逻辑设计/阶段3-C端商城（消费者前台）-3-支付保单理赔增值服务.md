## 模块七：支付模块

### 7.1 支付方式选择

**工时：前端1.5天 + 后端1.5天**

#### 页面与交互

- 支付页面（跳转自投保确认页，携带 `orderId`）。
- 展示订单信息：产品名、保障期限、应付金额（醒目红色大字）。
- 支付方式列表（Radio 单选）：
  - 微信支付（图标+文字）
  - 支付宝支付
  - 银行卡支付（显示已绑卡列表，选择具体卡片）
  - 余额支付（显示当前余额，余额不足时置灰+提示"余额不足"）
- 优惠券区域（展示可用优惠券数量标识）：点击进入优惠券选择弹窗（见 7.1.1）。
- 积分抵扣区域：显示当前可用积分，勾选后显示可抵扣金额（如"1000积分 = 1元"）。
- 底部展示"实付金额：XX元"（动态计算 = 应付 - 优惠券 - 积分抵扣）。
- 点击"立即支付"按钮（若选择微信/支付宝则调起支付，若选银行卡/余额则弹输入支付密码弹窗）。

#### 7.1.1 优惠券选择弹窗（工时：前端1天 + 后端1天）

- 弹窗展示用户已有且可用的优惠券列表（`status=1 AND expire_time > now() AND min_amount <= 订单金额`）。
- 每张券卡片：面额（大号红色）、使用条件（如"满200可用"）、有效期、适用险种。
- 不可用优惠券（置灰，悬浮展示不可用原因）在列表底部展示。
- 选择一张后关闭弹窗，支付页实时更新应付金额。

#### 后端逻辑

**发起支付（POST /api/c/pay/create）**
1. 校验 `orderId` 属于当前用户且 status=0（待支付）且未超过 expire_time。
2. 校验优惠券：`couponId` 非空时，查 `tb_user_coupon`：券状态=1（未使用）、未过期、`min_amount ≤ 订单金额`、适用险种包含当前险种；不满足则返回错误。
3. 校验积分抵扣：`usePoints > 0` 时，查用户当前积分 ≥ `usePoints`；积分上限=订单金额的10%（业务规则）。
4. 计算最终支付金额：`actualAmount = 订单金额 - 优惠券面额 - 积分换算金额`，不能 < 0（若 < 0 按 0 处理，并退还差额为积分）。
5. 若 `actualAmount = 0`（全额优惠）：直接完成支付，更新订单状态为已支付，无需调支付接口。
6. 锁定优惠券（status=2 使用中，防止重复使用），扣减用户积分（临时冻结，支付成功后确认扣减）。
7. 创建 `tb_payment_record`（status=0 待支付）。
8. 根据支付方式调不同支付 SDK：
   - 微信/支付宝：返回支付参数（prepayId / payInfo）给前端调起支付组件。
   - 银行卡/余额：需用户输入支付密码后验证（见下方）。
9. 返回 `paymentId` + 支付参数。

**验证支付密码并扣款（POST /api/c/pay/verify-password）**
1. 接收 `paymentId`、`password`（MD5 加密）。
2. 查用户 `tb_user.pay_password`（MD5 存储），不匹配则记录错误次数（≥5次锁定24小时）。
3. 密码正确：调余额扣减或银行卡扣款；成功则更新支付记录和订单状态。

**微信/支付宝回调（POST /api/c/pay/callback/{channel}）**
1. 验签（微信：RSA2048 验签；支付宝：RSA2 验签）。
2. 查 `tb_payment_record` by `tradeNo`（第三方交易号），防重复通知（幂等处理）。
3. 支付成功：更新 `tb_payment_record.status=1`，更新 `tb_order.status=1`（已支付）。
4. 确认扣减优惠券（status=3 已使用）、积分正式扣减。
5. 发布"订单支付成功"MQ 消息 → 触发保单生成流程。
6. 立即返回 HTTP 200（避免重复通知）。

#### 数据库

```sql
tb_payment_record
  id, order_id BIGINT NOT NULL, user_id BIGINT NOT NULL
  payment_no   VARCHAR(64) NOT NULL UNIQUE     -- 系统支付流水号
  channel      VARCHAR(20) NOT NULL            -- wechat/alipay/bankcard/balance
  amount       BIGINT NOT NULL                 -- 实付金额（分）
  trade_no     VARCHAR(100)                    -- 第三方交易号
  status       TINYINT DEFAULT 0               -- 0待支付1成功2失败3已退款
  pay_time     DATETIME
  expire_time  DATETIME NOT NULL               -- 支付过期时间
  notify_content TEXT                          -- 回调原始报文
  INDEX(order_id), INDEX(payment_no), INDEX(trade_no)

tb_user_coupon
  id, user_id BIGINT, coupon_id BIGINT
  status TINYINT DEFAULT 1                -- 1未使用2使用中3已使用4已过期
  coupon_type TINYINT                     -- 1满减2折扣3直减
  face_value  BIGINT                      -- 面额（分）
  min_amount  BIGINT DEFAULT 0            -- 最低使用金额
  product_category_id BIGINT DEFAULT 0   -- 0不限险种，否则指定险种
  expire_time DATETIME NOT NULL
  use_time DATETIME, use_order_id BIGINT
  INDEX(user_id, status, expire_time)
```

### 7.2 支付密码

**工时（含在 7.1 内）**

#### 页面与交互

- 首次使用余额/银行卡支付：引导设置支付密码（6位数字）。
- 设置页面：输入6位 → 确认6位（再次输入）→ 保存。
- 输入支付密码弹窗：6个密码框（每输入一位自动跳下一位）→ 输满6位自动提交（无确认按钮）。
- 忘记密码入口：跳验证手机号流程（短信验证后可重设）。

#### 后端逻辑

**设置支付密码（POST /api/c/user/pay-password/set）**
1. 校验用户已实名认证。
2. 已设置过则返回"请通过修改密码流程操作"。
3. 密码 MD5 加盐（`MD5(password + userId)`）存 `tb_user.pay_password`。
4. 短信通知用户"支付密码设置成功"。

### 7.3 支付结果

**工时：前端1天 + 后端1天**

#### 页面与交互

- 支付完成（或从第三方支付页面返回）后进入支付结果页。
- 结果页：
  - 成功：绿色对勾 + "支付成功" + 订单金额 + 产品名 + 操作按钮（查看保单、返回首页）。
  - 处理中（轮询超时）：黄色 loading 图标 + "支付处理中，请稍后查看保单" + "查看订单"按钮。
  - 失败：红色叉 + "支付失败" + 失败原因 + "重新支付"按钮。
- 支付成功后，系统自动发送成功通知（短信+App Push）。

#### 后端逻辑

**查询支付结果（GET /api/c/pay/result/{orderId}）**
1. 查 `tb_order.status`：1=成功，0=待支付。
2. 若 status=0 且 `tb_payment_record.expire_time > now()`：返回"处理中"（前端轮询，每3秒一次，最多10次）。
3. 若轮询结束仍为待支付：主动调三方查询接口确认状态，同步更新。
4. 支付成功后，异步触发：短信通知（保费已扣除）、App Push 推送、保单生成 MQ。

### 8.1 保单列表

**工时：前端1天 + 后端0.5天**

#### 页面与交互

- 入口：个人中心 → 我的保单。
- 顶部 Tab：全部 / 生效中 / 已过期 / 待支付 / 待核保。
- 保单卡片信息：产品名称、保险公司 logo、保险期限（起止日期）、被保人姓名、保单状态标签（彩色）、到期提醒天数（生效中且距到期≤30天显示"还有 X 天到期"红色提示）。

- 待支付保单展示倒计时（`expire_time`），过期自动变为"已取消"状态。

#### 后端逻辑

**GET /api/c/policy/list**：参数 `status`（可选）、`page=1`、`pageSize=10`。
1. 查 `tb_policy` where `user_id=?`（+ status 筛选），`ORDER BY create_time DESC`，分页。
2. 计算到期提醒：`expiry_date - today <= 30天` 时设置提醒标志。
3. 返回列表含 `canRenew`（是否可续保）、`canClaim`（是否可报案）字段。

### 8.3 电子保单

**工时：前端0.5天 + 后端0.5天**

#### 页面与交互

- 保单详情页"下载电子保单"按钮。
- 点击后：展示 loading → 生成/获取 PDF → 唤起浏览器下载或在 App 内打开预览。
- 同时支持"分享到微信"（分享 PDF 文件）。

#### 后端逻辑

**GET /api/c/policy/{policyId}/download**
1. 校验 `policyId` 属于当前用户。
2. 查 `tb_policy.pdf_url`：若已有则直接返回 OSS 签名 URL（有效期 5 分钟）。
3. 若无 PDF（首次请求）：
   a. 获取保险公司出具的电子保单 URL（调对接接口或读取回传文件）。
   b. 若保险公司无电子保单接口：使用 JasperReport / iText 按模板生成 PDF，上传 OSS，存 `tb_policy.pdf_url`。
4. 返回签名 URL，前端直接下载。

### 8.5 保单变更

**工时：前端1天 + 后端1天**

#### 页面与交互

- 保单详情 → "申请变更"。
- 弹出可变更项选择（根据保险公司支持情况显示）：
  - 变更联系方式（手机号/邮箱）
  - 变更受益人（仅寿险/重疾险）
  - 变更地址（邮寄地址）
- 选择变更项后填写新信息 → 提交。
- 提交后状态变为"变更审核中"，运营后台审核。

#### 后端逻辑

**POST /api/c/policy/change/apply**
1. 校验保单 status=1（生效中）。
2. 校验变更类型在该产品/保险公司支持范围内（`tb_insurer_support_change`）。
3. 创建 `tb_policy_change_record`（status=0 审核中）。
4. 同时通知运营后台待处理。
5. 审核通过后（后台操作）更新原保单对应字段并推送通知。

#### 数据库

```sql
tb_policy_change_record
  id, policy_id BIGINT NOT NULL, user_id BIGINT NOT NULL
  change_type  VARCHAR(50)                -- mobile/beneficiary/address
  old_value    JSON, new_value JSON
  status       TINYINT DEFAULT 0          -- 0审核中1已通过2已拒绝
  apply_time   DATETIME, audit_time DATETIME
  audit_remark VARCHAR(500)
  INDEX(policy_id, status)
```

## 模块九：理赔服务

### 9.1 报案

**工时：前端1.5天 + 后端1天**

#### 页面与交互

- 入口：保单详情 → "申请理赔"（保单状态为生效中才可申请）/ 首页 → 理赔服务。
- 报案第一步：选择出险保单（列表选择，仅展示生效中保单）。
- 报案信息填写页（必填）：
  - 出险时间（日期时间选择器，不能晚于当前时间）
  - 出险地点（文本输入 + "定位"按钮，自动获取当前位置）
  - 出险原因（文本输入，最多500字，必填）
  - 伤亡情况（下拉：无伤亡/轻伤/重伤/死亡）
  - 预估损失金额（数字输入，选填）
  - 联系人姓名（必填，默认投保人姓名）
  - 联系人电话（必填，默认投保人手机号）
  - 上传现场照片（最多9张，必传至少1张）
- 提交报案 → 展示报案成功页（报案号、下一步指引"上传理赔材料"、预计处理时间）。

#### 后端逻辑

**提交报案（POST /api/c/claim/report）**
1. 校验 `policyId` 属于当前用户且 `status=1`（生效中）。
2. 出险时间不能晚于当前时间。
3. 出险时间需在保障期内（`policy_start_date ≤ accident_time ≤ policy_end_date`）。
4. 校验图片 OSS URL 合法性（至少1张）。
5. 生成报案号（`CLME` + 年月日 + 8位流水号）。
6. 创建 `tb_claim_record`（status=0 已报案）。
7. 异步通知：发送报案受理短信给联系人；发 MQ 通知保险公司（若有对接接口）；运营后台提醒待处理。
8. 返回 `claimId` + `claimNo`。

#### 数据库

```sql
tb_claim_record
  id, claim_no VARCHAR(30) NOT NULL UNIQUE
  user_id BIGINT, policy_id BIGINT NOT NULL
  accident_time    DATETIME NOT NULL
  accident_location VARCHAR(500)
  accident_lng DECIMAL(10,7), accident_lat DECIMAL(10,7)   -- 经纬度
  accident_reason  TEXT NOT NULL
  injury_level     TINYINT     -- 0无伤亡1轻伤2重伤3死亡
  estimated_amount BIGINT      -- 预估损失（分）
  contact_name     VARCHAR(50) NOT NULL
  contact_mobile   VARCHAR(11) NOT NULL
  scene_images     JSON        -- 现场照片URL列表
  status           TINYINT DEFAULT 0   -- 0已报案1材料审核中2理赔审核中3已赔付4已拒赔5撤案
  INDEX(user_id, status), INDEX(policy_id), UNIQUE(claim_no)
```

### 9.2 材料上传

**工时：前端1天 + 后端1天**

#### 页面与交互

- 报案成功后引导进入"上传理赔材料"页。
- 页面顶部展示"理赔所需材料清单"（根据险种和出险类型从后台配置中加载）。
- 材料分类列表（如：身份证明、医疗证明、损失证明）：
  - 每类材料下方显示已上传文件缩略图 + "继续添加"按钮。
  - 每类材料可上传多个文件（图片或 PDF）。
  - 必传材料有红色"*"标注。
  - 已上传文件可删除（删除需二次确认）。
- 底部"提交材料"按钮（所有必传材料上传完成后高亮）。
- 支持多次补传（提交后仍可追加）。

#### 后端逻辑

**获取材料清单（GET /api/c/claim/{claimId}/material-list）**
1. 查 `tb_claim_material_template`（根据险种类型和出险原因匹配）。
2. 同时查该理赔已上传的材料 `tb_claim_material`，合并返回（每个模板项含已上传文件列表）。

**上传材料（POST /api/c/claim/material/upload）**
1. 接收 `claimId`、`materialTypeId`、`fileUrl`（OSS URL）。
2. 校验 `claimId` 属于当前用户且 status ≠ 4（已赔付）。
3. 文件格式校验：仅支持 jpg/png/pdf，大小≤10MB。
4. 入库 `tb_claim_material`。

**提交所有材料（POST /api/c/claim/material/submit）**
1. 校验所有必传材料类型均已有文件。
2. 更新 `tb_claim_record.status=1`（材料审核中）。
3. 发送短信"您的理赔材料已收到"，发 MQ 通知审核人员。

#### 数据库

```sql
tb_claim_material_template
  id, product_category_id BIGINT  -- 险种类型
  material_name VARCHAR(100) NOT NULL
  material_desc TEXT               -- 说明（如"需清晰可见，大小不超过10M"）
  is_required TINYINT DEFAULT 1
  sort INT DEFAULT 0, status TINYINT DEFAULT 1

tb_claim_material
  id, claim_id BIGINT NOT NULL
  template_id  BIGINT NOT NULL
  file_url     VARCHAR(255) NOT NULL
  file_type    VARCHAR(20)          -- image/pdf
  upload_time  DATETIME NOT NULL
  status       TINYINT DEFAULT 1    -- 1有效0已删除
  INDEX(claim_id, template_id)
```

### 9.3 进度查询

**工时：前端1天 + 后端0.5天**

#### 页面与交互

- 入口：我的保单 → 保单详情 → "查看理赔进度" / 理赔记录列表。
- 进度详情页：
  - 顶部：报案号 + 当前状态标签。
  - 时间轴（从上到下最新在前）：每个节点 = 状态变更时间 + 状态描述 + 备注（如审核人意见）。
  - 进度节点：已报案 → 材料审核中 → 理赔审核中 → 已赔付/已拒赔。
  - 若状态为"已赔付"：展示赔付金额 + 赔付时间 + 赔付方式（银行卡尾号）。
  - 若状态为"已拒赔"：展示拒赔原因 + "申请复核"按钮。
  - 补传材料提示：若状态为"材料审核中"且有补材料通知，展示"需补充材料"红色提示 + 补传入口。

#### 后端逻辑

**GET /api/c/claim/{claimId}/progress**
1. 查 `tb_claim_record` 基础信息 + `tb_claim_progress_log` 进度日志列表（按 create_time DESC）。
2. 查赔付记录 `tb_claim_payment`（若已赔付）。

#### 数据库

```sql
tb_claim_progress_log
  id, claim_id BIGINT NOT NULL
  status       TINYINT NOT NULL
  status_desc  VARCHAR(200)     -- 状态说明文字
  remark       VARCHAR(500)     -- 备注
  operator     VARCHAR(50)      -- 操作人（系统/理赔员/用户）
  create_time  DATETIME NOT NULL
  INDEX(claim_id, create_time)

tb_claim_payment
  id, claim_id BIGINT NOT NULL
  amount       BIGINT NOT NULL        -- 赔付金额（分）
  pay_method   VARCHAR(20)            -- bank/wechat/alipay
  pay_account  VARCHAR(100)           -- 脱敏账号
  pay_time     DATETIME NOT NULL
```

## 模块十：增值服务

### 10.1 保额计算器

**工时：前端1.5天 + 后端1天**

#### 页面与交互

- 入口：首页工具区 / 个人中心 → 保额计算器。
- 问卷形式逐步填写基本信息：
  - 性别（单选）
  - 年龄（滑块，0-80岁）
  - 已婚（是/否）+ 子女数量
  - 赡养老人（是/否）+ 人数
  - 月收入（区间选择：3000以下/3-5千/5-1万/1-2万/2万以上）
  - 现有负债（房贷/车贷/其他，多选 + 金额输入）
  - 现有保障（医社保/商业险，多选）
- 点击"开始计算"：展示推荐保额和分险种建议。
- 结果页：总推荐保额（大号字体）+ 各险种（寿险/重疾险/意外险/医疗险）建议保额及原因说明。
- 底部"根据建议投保"按钮 → 跳转对应产品列表。

#### 后端逻辑

**POST /api/c/tool/insurance-calculator**
1. 接收问卷数据 JSON。
2. 按规则计算：
   - 寿险需求 = 年收入 × 10 + 负债总额 - 现有保障
   - 重疾险需求 = 年收入 × 3~5（治疗费用+收入补偿）
   - 意外险：100万（基础推荐）
   - 医疗险：百万医疗（标配）
3. 返回各险种推荐保额 + 产品推荐（关联查询 `tb_product`）。
4. 将计算结果保存 `tb_calculator_record`（可选，用于用户历史查看）。

### 10.5 在线客服

**工时：前端2天 + 后端2天**

#### 页面与交互

- 入口：产品详情 / 保单详情 / 个人中心 / 理赔页面均有入口。
- 客服页面分两层：
  - **智能客服**（默认进入）：输入框 + 对话气泡列表；底部快捷问题按钮（常见问题快速触达，如"如何报案""如何续保"）。
  - **转人工客服**：智能客服回复"抱歉我无法解答"或用户点击"转人工"按钮 → 排队中（展示排队号）→ 接入人工。
- 人工客服：
  - 支持发送文字、图片、文件。
  - 客服坐席接入后，头像变为客服真实头像+名字。
  - 会话超时（客服10分钟未回复）：展示"客服暂时离开，可留言"提示。
  - 结束会话后弹出满意度评价（1-5星 + 文字）。
- 离线留言：工作时间外（非9:00-21:00）进入提示"当前非服务时间"，可提交留言（问题描述 + 手机号），工作日回复。

#### 后端逻辑

**智能客服（POST /api/c/cs/chat/robot）**
1. 接收用户消息 `content`。
2. 调内置知识库（`tb_faq_knowledge`）做关键词匹配（TF-IDF 相似度）或直接对接第三方智能客服 SDK（如云问/腾讯云企点）。
3. 匹配到答案：返回答案内容（文本/富文本）+ 相关推荐问题列表。
4. 未匹配：返回"抱歉，我没有找到相关答案，为您转接人工客服"。
5. 记录对话 `tb_cs_conversation`。

**创建人工客服会话（POST /api/c/cs/chat/human/create）**
1. 查当前用户是否已有进行中的人工会话，有则返回已有 `sessionId`。
2. 创建 `tb_cs_session`（status=0 排队中），推送到客服系统坐席队列。
3. 返回 `sessionId`，前端通过 WebSocket 连接 `/ws/cs/{sessionId}` 实时通讯。

**WebSocket 消息处理**
1. 用户消息：推送到 Redis Pub/Sub `cs:msg:{sessionId}`，客服端订阅接收。
2. 客服消息：同样通过 Pub/Sub 反向推送到用户端 WebSocket。
3. 消息存入 `tb_cs_message`（用于历史记录）。

#### 数据库

```sql
tb_cs_session
  id, user_id BIGINT NOT NULL
  session_type TINYINT         -- 1智能2人工
  agent_id BIGINT              -- 客服ID（人工时）
  status TINYINT DEFAULT 0     -- 0排队1进行中2已结束
  start_time DATETIME, end_time DATETIME
  satisfy_score TINYINT        -- 1-5满意度评分
  satisfy_comment VARCHAR(500)
  source_page VARCHAR(100)     -- 来源页面（产品详情/保单/理赔）
  INDEX(user_id, status)

tb_cs_message
  id, session_id BIGINT NOT NULL
  sender_type TINYINT          -- 1用户2机器人3客服
  sender_id BIGINT
  content_type TINYINT         -- 1文字2图片3文件
  content TEXT NOT NULL
  send_time DATETIME NOT NULL
  INDEX(session_id, send_time)
```
