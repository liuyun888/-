## 一、功能模块：区块链存证

### 1.1 存证触发时机

以下三个场景触发区块链存证（异步，通过 MQ 消息驱动）：

| 场景 | 触发条件 | evidence_type |
|------|---------|---------------|
| 双录视频存证 | 双录状态变为「已完成」（status=2）后，**24小时内**自动存证 | 1 |
| 行为轨迹存证 | 保单承保成功后，对该订单行为轨迹汇总存证 | 2 |
| 质检报告存证 | 人工质检完成后，对生成的质检报告PDF存证 | 3 |

**MQ 消息**：
- Topic: `double_record_blockchain_evidence`
- Body: `{ "evidenceType": 1, "relatedId": 789 }`

### 1.2 存证处理流程

```
BlockchainEvidenceConsumer 消费 MQ 消息后：

1. 根据 evidenceType + relatedId 查询文件地址（video_url / report_url）
2. 下载文件到临时内存（或流式），计算 SHA256 哈希值
3. 调用区块链 SDK 上链（建议使用蚂蚁链 BaaS 平台）：
   - 入参：fileHash, fileSize, relatedCode, timestamp
   - 出参：txId（交易ID）、blockHeight、blockchainTimestamp
4. 插入 ins_blockchain_evidence 记录，写入所有字段
5. 更新 ins_double_record.blockchain_hash = txId，blockchain_time = now
6. 生成在线验证 URL（区块链平台提供的公开验证页面）
7. 若步骤3调用失败：写入失败日志，MQ 消息重试（最多3次），超过3次告警通知合规负责人
```

**存证接口（供内部调用 + 管理员手动触发）**：
```
POST /admin-api/insurance/blockchain/evidence/create

Request:
{
  "evidenceType": 1,
  "relatedId": 789,
  "forceRecreate": false   // false=若已存在则跳过；true=强制重新存证（管理员手动触发用）
}

后端处理（若 forceRecreate=false）：
- 先查 ins_blockchain_evidence WHERE related_id=789 AND evidence_type=1
- 若已存在且区块链成功：直接返回已有存证信息
- 若不存在或上次失败：执行存证流程

Response:
{
  "code": 0,
  "data": {
    "evidenceCode": "EV202602190001",
    "blockchainTxId": "0x1234abcd...",
    "blockHeight": 12345678,
    "blockchainTimestamp": 1740000000000,
    "verifyUrl": "https://antchain.com/verify?txId=0x1234abcd"
  }
}
```

**验证存证接口（公开可访问）**：
```
GET /api/insurance/blockchain/evidence/verify/{evidenceCode}

后端处理：
1. 查 ins_blockchain_evidence WHERE evidence_code=evidenceCode
2. 调用区块链 SDK 查询链上数据（传入 txId）
3. 对比链上 fileHash 与数据库 fileHash 是否一致
4. 更新 is_verified=true，verify_time=now
5. 返回验证结果

Response:
{
  "code": 0,
  "data": {
    "isValid": true,
    "evidenceCode": "EV202602190001",
    "fileHash": "a1b2c3...",
    "relatedCode": "DR202602190001",
    "createTime": "2026-02-19 10:30:00",
    "blockchainTxId": "0x1234abcd",
    "conclusion": "存证完整，文件未被篡改"
  }
}
```

### 1.3 存证管理列表（PC 管理后台）

| 播放视频 | ✅ 本人 | ✅ | ✅ | ✅ |

| 手动触发存证 | ❌ | ❌ | ❌ | ✅ |

### 3.2 数据权限规则

**业务员数据隔离**：查询双录列表时，若当前用户为「业务员」角色，后端自动追加 `WHERE agent_id = currentUserId`，禁止越权查看他人双录。

**团队长权限**：可查看本团队所有成员的双录（通过 `org_id` 范围限定），但不可质检或修改。

**敏感字段脱敏**（接口返回时，非合规主管/管理员）：
- `customer_id_card`：返回脱敏值，如 `110101****1234`（前6后4）
- `customer_name`：返回脱敏值，如 `李*`（名字只保留姓）
- `customer_mobile`：不在双录接口中返回

### 3.3 操作角色编码（与 yudao 权限体系集成）

在 `system_role` 表中新增以下角色（或通过系统管理配置）：

| 角色编码 | 角色名称 |
|----------|----------|
| `COMPLIANCE_INSPECTOR` | 合规质检员 |
| `COMPLIANCE_MANAGER` | 合规主管 |

对应权限点（`system_menu` 表，type=3 按钮权限）：
- `insurance:double-record:check`：执行人工质检
- `insurance:double-record:template:manage`：管理话术模板
- `insurance:blockchain:evidence:manage`：区块链存证管理
- `insurance:double-record:retry-approve`：审批重录申请

## 四、时效性要求 & 超时处理

| 环节 | 时限 | 超时处理 |
|------|------|----------|
| 双录完成 → 区块链存证 | 24小时内 | 定时任务每小时扫描，超时触发站内消息通知合规负责人 |
| 双录完成 → AI质检 | 72小时内 | 定时任务兜底，若72小时未完成AI质检则自动重新提交MQ |
| AI质检完成 → 人工质检 | 7个工作日内 | 超时自动生成催办工单，分配给质检员组长 |
| 双录视频存储期限 | ≥5年 | OSS生命周期规则：5年后变更为「锁定」状态，需管理员手动申请删除并留存审批记录 |

## 五、定时任务设计

| 任务名称 | 执行频率 | 功能说明 |
|----------|----------|----------|
| ASR结果轮询 | 每30秒 | 扫描 `asr_task_id` 不为空且 `check_status=1` 的记录，调用ASR服务轮询结果 |
| 双录超时告警 | 每小时 | 扫描 `status=2` 且超24小时未存证的记录，推送告警 |
| 质检催办 | 每天9:00 | 扫描超过7个工作日未完成人工质检的记录，生成催办通知 |
| 存证失败重试 | 每小时 | 扫描 `blockchain_hash` 为空且超1小时的完成双录记录，重新触发存证 |

## 六、异常处理规范

### 6.1 双录中断处理

| 异常情况 | 前端处理 | 后端处理 |
|----------|----------|----------|
| RTC网络中断 | 展示「网络中断，3秒后尝试重连」倒计时；重连3次失败后弹窗「网络不稳定，建议重录」 | 监听 RTC 断线回调，更新 `ins_double_record` 的 `remark` 记录断线时间 |
| 视频上传失败 | 使用 OSS 分片上传，失败时自动重试3次；仍失败则提示「上传失败，请点击重新上传」 | 接口幂等，相同 `recordId` 可多次调用 `finish` 接口，以最后一次为准 |
| 声网录制回调超时 | — | 定时任务30分钟后查询录制状态；若录制失败，将 `record_status` 改回 `1-进行中`，并通知业务员重新结束双录 |

### 6.2 AI 质检异常处理

- ASR 服务调用失败（超时/5xx）：重试3次，间隔指数退避（1s / 2s / 4s）
- 重试均失败：将质检状态置为「待人工处理」，通知质检员手动处理
- 人脸识别/证件识别失败：单项标记为「检测失败（-）」，不影响其他项得分，但在质检报告中注明「部分检测失败，需人工复核」

## 八、开发注意事项

### 8.1 多租户隔离

所有表均含 `tenant_id` 字段，使用 yudao-cloud 框架的 `@TenantIgnore` 和 `TenantContextHolder` 机制，确保跨租户数据严格隔离。区块链存证验证接口（`/api/insurance/blockchain/evidence/verify`）为公开接口，需绕过租户过滤。

### 8.2 身份证加密

`customer_id_card` 字段存储加密值，写入前调用 `SensitiveFieldEncryptor.encrypt(idCard)`（AES-256/CBC/PKCS5Padding），密钥从 Nacos 配置中心读取，不硬编码在代码中。查询时框架自动解密，业务层使用明文比对，接口返回时再次脱敏。

### 8.3 OSS 上传规范

- 文件路径规范：`{tenant_id}/double-record/{yyyy}/{MM}/{record_code}_{type}.{ext}`
  - type: video / audio / idcard / screenshot / report
- 上传前在后端生成预签名上传 URL（有效期30分钟），前端直传 OSS，不经过后端
- 上传完成后，前端通知后端 URL，后端校验文件是否存在（OSS head 请求）

### 8.4 RTC 房间安全

- 每次发起双录时生成 RTC Token（有效期8小时）
- Token 中绑定 uid（业务员ID），防止他人加入房间
- 客户端邀请 URL 使用 JWT 签名（`inviteToken`），包含 recordId + 有效期30分钟
- 客户通过邀请 URL 进入后，后端校验 token 并分配只读 RTC Token

### 8.5 幂等设计

以下接口需保证幂等（防止网络重试导致重复操作）：
- `finish`（结束双录）：以 `record_code` 为幂等键，重复调用时若状态已为 2 则直接返回成功
- `manual-check`（提交质检）：以 `recordId` 为幂等键，使用 Redis 分布式锁
- `evidence/create`（区块链存证）：同 `related_id + evidence_type` 已存在则跳过
