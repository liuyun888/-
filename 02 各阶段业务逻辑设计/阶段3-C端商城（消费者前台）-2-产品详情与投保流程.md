## 模块四：产品详情

### 4.1 产品介绍页

**工时：前端1.5天 + 后端0.5天**

#### 页面与交互

- 进入路径：产品列表点击卡片 / 分享链接。
- 顶部悬浮栏（随页面滚动固定）：产品名称 + "立即投保"按钮。
- Tab 切换：产品介绍 / 保障详情 / 投保案例 / 常见问题。

**【产品介绍 Tab】**
- 产品图片轮播（Swiper，最多5张）。
- 产品名称、保险公司 logo + 名称 + 官方备案号。
- 核心标签（最多5个，横向滚动）：如"住院不限社保用药""保证续保20年"。
- 核心卖点区：3-5条，图标 + 文字（后台配置）。
- 销售数据：已售 XX 份（实时，每10分钟更新）。
- 评分：5星 + 综合评分（如4.8分，X人评价）。
- 产品有效期：若有效期结束后7天内，展示"即将下架"提示。
- 投保须知（富文本，折叠，默认收起）。
- 底部固定：收藏图标 + "立即投保"按钮。

**【保障详情 Tab】**
- 保障责任列表（折叠面板 Accordion）：保障项名称 → 展开后显示详细说明。
- 免责条款（重要，红色底部提示）：醒目展示。
- 条款文件下载入口：PDF 链接。

**【投保案例 Tab】**
- 真实理赔案例列表（后台录入）：每条案例包含案例描述、理赔结果（金额）、案例日期。

**【常见问题 Tab】**
- FAQ 手风琴列表：问题 → 点击展开回答；支持关键词搜索（前端过滤）。

#### 后端逻辑

**GET /api/c/product/{id}**
1. 查 `tb_product` 基础信息。
2. 关联查 `tb_product_coverage`（保障明细列表）。
3. 关联查 `tb_product_exclusion`（免责条款）。
4. 关联查 `tb_product_case`（投保案例，最多5条）。
5. 关联查 `tb_product_faq`（FAQ 列表）。
6. 查询该用户是否已收藏（需传 token）。
7. 结果缓存 `cache:product:{id}`，TTL=300 秒，产品信息更新后主动清除。
8. 异步：浏览量 +1（Redis INCR）。

#### 数据库

```sql
-- 产品保障明细
tb_product_coverage
  id, product_id BIGINT NOT NULL
  coverage_name  VARCHAR(200)          -- 保障名称
  coverage_amount VARCHAR(100)          -- 保额，如"100万"
  coverage_desc  TEXT                  -- 详细说明
  sort INT DEFAULT 0
  INDEX(product_id, sort)

-- 产品免责条款
tb_product_exclusion
  id, product_id, exclusion_text TEXT, sort INT
  INDEX(product_id)

-- 投保案例
tb_product_case
  id, product_id, case_title VARCHAR(200), case_content TEXT
  claim_amount BIGINT                  -- 理赔金额（分）
  case_date DATE, sort INT
  INDEX(product_id)

-- FAQ
tb_product_faq
  id, product_id, question VARCHAR(500), answer TEXT, sort INT
  INDEX(product_id)
```

### 4.5 用户评价

**工时：前端1天 + 后端1天**

#### 页面与交互

- 入口：产品详情页顶部评分区"查看评价"→ 跳转评价列表页。
- 评价列表页：顶部综合评分（5星展示 + 数字）、标签筛选（全部/好评/中评/差评/有图）、评价列表（头像+昵称+星级+内容+图片+时间+官方回复）。
- 发起评价入口：我的保单 → 保单详情 → "评价"按钮（仅保单生效后可评价，每个保单只能评价1次）。
- 评价表单：星级（1-5星，必选）、文字评价（20-500字，必填）、图片上传（最多9张，OSS）、是否匿名（勾选框）。

#### 后端逻辑

**提交评价（POST /api/c/product/review）**
1. 校验 `orderId` 对应保单属于当前用户且保单状态为已生效。
2. 校验该订单未评价过（查 `tb_product_review` where `order_id = ?`）。
3. 内容安全审核：调阿里云内容安全 API 检测评价文字，违规返回"评价内容包含违规信息"。
4. 图片数量校验：≤9张。
5. 入库 `tb_product_review`，status=1（直接显示，人工复核制）。
6. 异步更新 `tb_product` 的 `avg_score`（AVG 计算）和 `review_count`（COUNT）。
7. 更新 `tb_order.is_reviewed = 1`。

**获取评价列表（GET /api/c/product/{id}/reviews）**
- 参数：`type`（0全部1好评2中评3差评4有图）、`page`、`pageSize=10`。
- 查 `tb_product_review` where `product_id=? AND status=1`，关联用户信息（匿名则脱敏昵称）。

#### 数据库

```sql
tb_product_review
  id, product_id BIGINT NOT NULL
  user_id BIGINT NOT NULL, order_id BIGINT NOT NULL UNIQUE
  score TINYINT NOT NULL              -- 1-5
  content TEXT NOT NULL
  images JSON                         -- ["url1","url2",...]
  is_anonymous TINYINT DEFAULT 0
  status TINYINT DEFAULT 1            -- 0待审核1正常2违规删除
  reply_content TEXT                  -- 官方回复
  reply_time DATETIME
  review_time DATETIME NOT NULL
  INDEX(product_id, status, score), INDEX(user_id)
```

## 模块五：车险投保

### 5.1 车牌识别

**工时：前端1天 + 后端1天**

#### 页面与交互

- 车险投保第一步页面：展示输入框"请输入车牌号"和"拍照识别"按钮。
- 手动输入：省份缩写选择键盘（左）+ 字母数字键盘（右），最多8位。
- 拍照识别：调用摄像头 → OCR 识别车牌 → 自动填入输入框（用户确认）。
- 输入车牌后点击"查询车辆信息"：后端查历史投保记录，自动回填车辆信息（如已有）。

#### 后端逻辑

**OCR 识别车牌（POST /api/c/vehicle/ocr/plate）**
1. 接收图片 base64 或 OSS URL。
2. 调腾讯云 / 阿里云 OCR 识别车牌，返回 `licensePlate`、`plateType`（蓝/黄/绿）。
3. 结果直接返回前端，不入库（入库在提交投保时）。

**查询车辆历史记录（GET /api/c/vehicle/history?plate=粤A12345）**
1. 查 `tb_user_vehicle` where `user_id=? AND license_plate=?`，返回上次录入的车辆信息（VIN、品牌、车型等）用于回填。
2. 若无记录返回空，前端引导用户手动填写。

### 5.2 车型选择

**工时：前端1.5天 + 后端1天**

#### 页面与交互

- 车辆信息录入页（支持 OCR 上传行驶证自动识别）。
- 页面字段（全部必填）：
  - 车牌号（已填，可修改）
  - 车辆类型：家用/营运（单选）
  - 初次登记日期（日期选择器，不能晚于今日）
  - 品牌（文本搜索 + 列表选择，首字母分组）
  - 车系（依赖品牌，联动刷新）
  - 车型（依赖车系，联动刷新，选择后自动填入购置价、座位数）
  - 年款（依赖车型，联动刷新）
  - 发动机号（文本输入，格式：字母+数字，5-20位）
  - 车架号 VIN（17位，不含 I/O/Q；可扫码/拍照 OCR 识别）
  - 新车购置价（数字输入，单位：万元，精确到0.1）
  - 座位数（数字输入，2-9位，含司机）
- 上传行驶证（拍照或相册）→ OCR 识别后自动填充上述字段 → 用户确认。
- 点击"下一步"：前端校验所有必填项 → 调接口保存并跳险种选择页。

#### 后端逻辑

**保存车辆信息（POST /api/c/vehicle/save）**
1. 必填校验：所有字段非空。
2. 格式校验：VIN 17位（正则 `[A-HJ-NPR-Z0-9]{17}`）；初登日期不晚于今日；车牌格式（`[京津沪渝冀豫云辽黑湘皖鲁新苏浙赣鄂桂甘晋蒙陕吉闽贵粤川青藏琼宁夏][A-Z][A-Z0-9]{4,5}[A-Z0-9挂学警港澳]`）。
3. 购置价需 ≥ 1万 ≤ 500万。
4. 更新或插入 `tb_user_vehicle`（同一用户同车牌覆盖更新）。
5. 返回 `vehicleId` 用于后续报价。

**品牌车系车型接口**
- `GET /api/c/vehicle/brands`：查 `tb_vehicle_brand`，按首字母分组，缓存 TTL=3600。
- `GET /api/c/vehicle/series?brandId=`：查 `tb_vehicle_series`，缓存同上。
- `GET /api/c/vehicle/models?seriesId=`：查 `tb_vehicle_model`，返回含购置参考价、座位数。

#### 数据库

```sql
tb_user_vehicle
  id, user_id BIGINT NOT NULL
  license_plate  VARCHAR(20) NOT NULL
  plate_type     TINYINT                  -- 1蓝2黄3绿（新能源）
  vehicle_type   TINYINT NOT NULL         -- 1家用2营运
  register_date  DATE NOT NULL            -- 初登日期
  brand_id BIGINT, series_id BIGINT, model_id BIGINT
  vehicle_brand VARCHAR(50), vehicle_series VARCHAR(50), vehicle_model VARCHAR(100)
  model_year     VARCHAR(10)
  vin_code       VARCHAR(17)
  engine_no      VARCHAR(50)
  purchase_price DECIMAL(10,2)            -- 购置价（万元）
  seat_count     TINYINT
  owner_name     VARCHAR(50)
  owner_id_no    VARCHAR(50)              -- 脱敏
  owner_mobile   VARCHAR(11)
  INDEX(user_id, license_plate)

tb_vehicle_brand / tb_vehicle_series / tb_vehicle_model  -- 车辆品牌数据库（定期同步）
```

### 5.3 方案选择（险种选择）

**工时：前端1.5天 + 后端1天**

#### 页面与交互

- 险种选择页，分为交强险区和商业险区。
- **交强险**：固定保障，仅展示投保说明，无需选择。
- **商业险**（多选 Checkbox）：
  - 车辆损失险（车损险）：必选，默认勾选，保额选择（按购置价比例：实际价值/70万/100万）
  - 第三者责任险：必选，默认勾选，保额选择（20万/50万/100万/200万）
  - 座位险：选填，投保人数（1-9座位），每座保额选择
  - 自燃险：选填
  - 涉水险：选填
  - 不计免赔险：选填（勾选商业险中任一后推荐勾选）
  - 玻璃险：选填，选单独玻璃险还是含框
- 险种选择后，底部实时显示"预计保费：XXXX元"（前端估算公式，供参考）。
- 点击"立即报价"：提交所选险种，跳报价结果页。

#### 后端逻辑

**提交险种方案（POST /api/c/car-insurance/scheme）**
1. 校验 `vehicleId` 存在且属于当前用户。
2. 校验商业险：车损险和三者险必须至少选一。
3. 三者险保额须在允许档位内（20/50/100/200 万）。
4. 保存险种方案到 `tb_car_quote_scheme`，返回 `schemeId`。

### 5.4 保费试算（在线报价）

**工时：前端1天 + 后端1天**

#### 页面与交互

- 报价结果页：顶部 loading"正在向多家保险公司询价..."（动画）。
- 逐步展示各公司报价结果（先到先展示）：保险公司 logo + 名称 + 总保费 + 明细折叠展示。
- 每家报价卡片：商业险保费 + 交强险保费 = 合计保费，"选择"按钮。
- 若某公司报价失败：显示"暂无报价"，不影响其他公司展示。
- 点击"选择"：跳转该公司投保信息填写页，携带 `quotationId`。

#### 后端逻辑

**发起报价（POST /api/c/car-insurance/quote）**
1. 校验 `schemeId` 有效且属于当前用户。
2. 创建 `tb_car_quotation`记录（status=0 报价中），返回 `quotationId`。
3. 异步并发调用平台配置的3-5家保险公司报价接口（线程池，每家超时5秒跳过）。
4. 每家报价结果实时更新 `tb_car_quote_result`（status=1成功/2失败）。
5. 全部完成或超时后，更新 `tb_car_quotation.status=1`（已完成）。

**轮询报价状态（GET /api/c/car-insurance/quote/{quotationId}/status）**
- 前端每2秒轮询，返回当前已出结果的公司列表；`finished=true` 时停止轮询。

#### 数据库

```sql
tb_car_quotation
  id, user_id, vehicle_id, scheme_id
  status TINYINT             -- 0报价中1完成2失败
  quote_time DATETIME
  INDEX(user_id, status)

tb_car_quote_result
  id, quotation_id BIGINT NOT NULL
  insurer_id BIGINT, insurer_name VARCHAR(100)
  traffic_premium   BIGINT    -- 交强险保费（分）
  commercial_premium BIGINT   -- 商业险保费（分）
  total_premium     BIGINT
  discount_rate     DECIMAL(4,2)   -- 折扣率
  quote_detail      JSON           -- 各险种明细
  status    TINYINT            -- 0报价中1成功2失败
  fail_reason VARCHAR(255)
  INDEX(quotation_id, status)
```

### 5.5 投保信息填写

**工时：前端1.5天 + 后端1天**

#### 页面与交互

- 投保信息填写页，分3个区块：投保人信息、被保人信息（被保车辆车主）、车辆附加信息。
- **投保人信息**（必填）：
  - 姓名（必填，可从实名认证自动带入）
  - 证件类型（默认身份证）
  - 证件号（必填，格式校验）
  - 手机号（必填，默认登录手机号）
  - 邮箱（选填，用于发送保单）
- **被保车主信息**：
  - 与投保人是否同一人（勾选框，勾选后自动同步）
  - 姓名、证件类型、证件号（必填）
- **车险特有字段**：
  - 上年保险公司（下拉选择，选项来自 `tb_insurer`）
  - 上年出险次数（0/1/2/3/3次以上，影响折扣）
  - 是否新车（是/否）
- 保险起止日期：起保日期（日历选择，最早明天，最晚30天后），自动计算终止日期（交强险1年）。
- 勾选"我已阅读《投保须知》《产品条款》"（必须勾选）。
- 点击"确认投保信息"：提交校验 → 跳至核保流程或直接跳支付页。

#### 后端逻辑

**提交投保信息（POST /api/c/car-insurance/apply-info）**
1. 校验所有必填项，证件号格式校验。
2. 起保日期：不允许今天（需明天起），不允许超过30天后。
3. 保存 `tb_insurance_apply`（status=0 待核保）。
4. 触发核保：调保险公司核保接口（同步等待，超时30秒）。
5. 核保结果：通过（status=1）→ 跳支付；手动（status=2）→ 提示"需人工审核"；拒保（status=3）→ 展示拒保原因。

## 模块六：非车险投保

### 6.1 健康告知

**工时：前端2天 + 后端1.5天**

#### 页面与交互

- 非车险投保第一步（健康类险种必须）：展示"健康告知书"问卷页面。
- 问卷题目列表，每道题目格式：题干 + 是/否单选按钮（或多选 Checkbox）。
- **跳题逻辑**：某题选"是"时，展开子问题（如"是否治愈？治愈时间？"）；选"否"则收起子问题。
- 所有题目展示完毕后，顶部进度条显示完成进度。
- 若任何关键题目（`is_critical=1`）选"是"，提交后后端可能触发"不可承保"或"加费承保"。
- 底部"下一步"按钮：所有必答题均回答后才可点击。

#### 后端逻辑

**获取健康告知题目（GET /api/c/health-disclosure/questions?productId=）**
1. 查 `tb_health_question` where `product_id=? AND status=1`，按 `sort` 排序。
2. 返回树形结构（父题 + 子题数组）。
3. 缓存 key=`cache:health:questions:{productId}`，TTL=3600。

**提交健康告知（POST /api/c/health-disclosure/submit）**
1. 接收参数：`productId`、`answers`（数组：questionId + answer + subAnswers）。
2. 校验所有必答题（`required=1`）均有回答。
3. 逐条检查关键题目（`is_critical=1`）回答为"是"的情况：
   - 全部为"否"：标记 `disclosure_result=0`（通过），继续流程。
   - 存在"是"但非拒保题：调智能核保引擎，判定结果（通过/加费/拒保）。
   - 存在拒保关键题：直接判定 `disclosure_result=2`（不可承保），展示"很遗憾，您当前健康状况不符合本产品投保条件"。
4. 将回答存入 `tb_health_disclosure_record`。
5. 返回 `disclosureId` 和 `disclosureResult`（0通过1待评估2不可承保）。

#### 数据库

```sql
tb_health_question
  id, product_id BIGINT
  parent_id BIGINT DEFAULT 0        -- 子题的父ID
  question_text TEXT NOT NULL
  answer_type TINYINT               -- 1单选是否2多选
  is_critical TINYINT DEFAULT 0     -- 1关键题（选是可能触发拒保）
  is_required TINYINT DEFAULT 1
  sort INT DEFAULT 0, status TINYINT DEFAULT 1
  INDEX(product_id, status, sort)

tb_health_disclosure_record
  id, user_id, product_id, apply_id BIGINT
  answers JSON                       -- [{questionId, answer, subAnswers:[...]}]
  disclosure_result TINYINT          -- 0通过1待评估2不可承保
  submit_time DATETIME
  INDEX(user_id, product_id)
```

### 6.2 智能核保

**工时：前端1.5天 + 后端1.5天**

#### 页面与交互

- 健康告知提交后，若需智能核保，展示"正在智能核保中..."（loading，最多10秒）。
- 核保结果展示：
  - 标准承保：绿色提示"您符合标准承保条件"，继续填写投保信息。
  - 加费承保：橙色提示"您可以投保，但将加收 XX% 保费，如继续请点击确认"。
  - 除外承保：提示"已除外 [某疾病]，其他保障正常生效"。
  - 拒保：红色提示"很遗憾，您不符合本产品投保条件"，展示拒保原因，提供"查看其他产品"按钮。

#### 后端逻辑

**智能核保引擎（内部调用）**
1. 接收健康告知回答 + 被保人基本信息（年龄、性别）。
2. 匹配 `tb_underwrite_rule` 规则表：
   - 规则条件（JSON：questionId + answer + 被保人属性）
   - 规则结果（0标准1加费2除外3拒保）+ 加费比例 + 除外责任描述
3. 所有规则顺序执行，取最严结果（多条规则命中时取最高等级：拒保 > 除外 > 加费 > 标准）。
4. 将核保结果存 `tb_underwrite_result`，关联 `apply_id`。

#### 数据库

```sql
tb_underwrite_rule
  id, product_id BIGINT
  rule_name VARCHAR(100)
  conditions JSON          -- [{questionId, operator, value}, {ageMin, ageMax}]
  result_type TINYINT      -- 0标准1加费2除外3拒保
  extra_rate DECIMAL(5,2)  -- 加费比例
  exclusion_desc VARCHAR(500)
  priority INT DEFAULT 0   -- 规则优先级
  status TINYINT DEFAULT 1
  INDEX(product_id, status, priority)

tb_underwrite_result
  id, user_id, product_id, apply_id BIGINT
  result_type TINYINT, extra_rate DECIMAL(5,2)
  exclusion_desc VARCHAR(500)
  underwrite_time DATETIME
```

### 6.3 职业选择

**工时：前端1天 + 后端0.5天**

#### 页面与交互

- 弹窗结构：搜索框 + 职业分类列表（二级，点击一级展开二级，点击二级选中）。

### 6.4 受益人信息

**工时：前端1天 + 后端0.5天**

#### 页面与交互

- 非车险（寿险、重疾险）投保信息页中受益人模块（可选填，不填默认法定继承）。
- 默认展示"法定受益人"已勾选，用户可切换为"指定受益人"。
- 选"指定受益人"后展示添加表单（最多5人）：
  - 姓名（必填）
  - 与被保人关系（必填，下拉：配偶/子女/父母/其他）
  - 证件类型+证件号（必填）
  - 受益顺序（1/2，拖拽排序或下拉）
  - 受益比例（必填，同顺序受益人合计必须 = 100%，精确到1%）
- 添加时，实时校验同顺序受益比例合计。

#### 后端逻辑

**保存受益人（POST /api/c/insurance/beneficiary）**
1. 若类型=1（法定）：直接记录，无需表单字段。
2. 若类型=2（指定）：
   - 人数校验：1≤人数≤5。
   - 同顺序受益比例合计必须 = 100%（可有多个顺序，如顺序1多人合计100，顺序2多人合计100）。
   - 证件号格式校验。
3. 存 `tb_insurance_beneficiary`，关联 `apply_id`。

#### 数据库

```sql
tb_insurance_beneficiary
  id, apply_id BIGINT NOT NULL
  beneficiary_type TINYINT    -- 1法定2指定
  name VARCHAR(50)
  relation VARCHAR(20)
  id_type TINYINT, id_no VARCHAR(50)
  benefit_order TINYINT       -- 受益顺序 1/2
  benefit_ratio TINYINT       -- 受益比例 1-100
  INDEX(apply_id)
```

### 6.5 投保确认

**工时：前端1天 + 后端0.5天**

#### 页面与交互

- 投保信息填写完成后，最后展示"投保确认页"（信息汇总核对页）。
- 页面内容：
  - 产品名称 + 保险公司
  - 保险期限（起止日期）
  - 保费金额（红色大号字体）
  - 投保人信息摘要（姓名+手机号）
  - 被保人信息摘要（可多人）
  - 受益人信息摘要
  - 优惠券使用情况（已选则展示）
  - 应付金额 = 保费 - 优惠券抵扣
- 底部两个按钮：修改信息（返回上一步）、确认投保（跳支付）。
- 点击"确认投保"前需勾选《投保须知》《产品条款》《个人信息授权协议》（三个链接，均可点击查看）。

#### 后端逻辑

**GET /api/c/insurance/apply/confirm/{applyId}**：返回汇总信息供确认页展示。

**确认提交（POST /api/c/insurance/apply/confirm）**
1. 校验 `applyId` 属于当前用户，status=0（待确认）。
2. 校验健康告知、核保、受益人等前置步骤均已完成。
3. 更新 `tb_insurance_apply.status=1`（待支付）。
4. 创建 `tb_order`（status=0 待支付），返回 `orderId`。
5. 返回 `orderId`，前端跳支付页。

#### 数据库

```sql
tb_insurance_apply
  id, user_id BIGINT NOT NULL, product_id BIGINT NOT NULL
  insurer_id BIGINT, policy_start_date DATE, policy_end_date DATE
  premium BIGINT NOT NULL              -- 保费（分）
  status TINYINT DEFAULT 0             -- 0待确认1待支付2已支付3核保中4拒保5撤单
  insured_info JSON                    -- 被保人信息列表
  holder_info JSON                     -- 投保人信息
  apply_time DATETIME NOT NULL
  INDEX(user_id, status), INDEX(product_id)

tb_order
  id (订单号，雪花算法), user_id, apply_id BIGINT
  order_type TINYINT             -- 1车险2非车险
  product_name VARCHAR(200)
  amount BIGINT NOT NULL         -- 应付金额（分）
  discount_amount BIGINT DEFAULT 0  -- 优惠金额
  coupon_id BIGINT               -- 使用的优惠券
  status TINYINT DEFAULT 0       -- 0待支付1已支付2已取消3退款中4已退款
  pay_time DATETIME, pay_method VARCHAR(20)
  expire_time DATETIME NOT NULL  -- 订单过期时间（创建后30分钟）
  INDEX(user_id, status), INDEX(status, expire_time)
```

## 附录：非车险投保总体流程

```
选产品 → 立即投保
  → 填写被保人信息（可多人，来自家人管理）
  → 职业选择
  → 健康告知（问卷）
  → 智能核保（若有问题）
  → 填写受益人（寿险/重疾险）
  → 投保确认（汇总核对 + 勾选协议）
  → 支付
  → 保单生成
```

每步均保存中间状态到 `tb_insurance_apply`，用户中途退出后可从断点继续。
