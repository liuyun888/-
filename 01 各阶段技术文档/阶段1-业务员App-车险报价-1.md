## 第一部分:项目概述与目标

### 1.1 业务背景与痛点分析

#### 1.1.1 传统车险展业痛点

当前保险中介行业车险展业面临以下核心问题:

**效率低下**
- 传统询价需要业务员逐个保司打电话或登录系统,单次报价耗时20-30分钟
- 信息录入重复,每家保司都要重新填写车辆和车主信息
- 比价困难,无法直观对比多家保司方案

**数据孤岛**
- 客户信息分散在纸质单据、Excel表格、微信聊天记录中
- 历史报价无沉淀,续保时需要重新询价
- 缺乏客户画像,无法精准营销

**合规风险**
- 纸质单据易丢失,无法追溯
- 口头承诺与实际条款不符导致投诉
- 缺乏完整的操作日志审计

**客户体验差**
- 等待时间长,当场无法给出准确报价
- 报价单格式不统一,客户难以理解
- 续保提醒靠人工记忆,易遗漏

#### 1.1.2 数字化转型目标

本系统旨在通过技术手段解决上述痛点,实现以下核心目标:

**效率目标**: 车险报价流程从30分钟压缩至3分钟以内
**准确目标**: OCR识别准确率>95%,多保司报价成功率>90%
**体验目标**: 移动端全流程操作,所见即所得的报价单生成
**数据目标**: 沉淀客户档案,为续保和交叉销售提供数据支撑

### 1.2 系统核心价值

#### 1.2.1 对业务员的价值

- **提升展业效率**: 拍照识别车辆信息,一键获取多家保司报价
- **增强专业能力**: 系统自动生成专业的PDF报价单和方案对比
- **客户管理工具**: 自动提醒续保,记录跟进历史
- **业绩透明化**: 实时查看个人业绩和佣金

#### 1.2.2 对管理层的价值

- **数据驾驶舱**: 实时掌握团队业绩、报价转化率等核心指标
- **渠道管控**: 所有报价和出单行为可追溯
- **风控能力**: 异常交易预警,合规操作监督
- **决策支持**: 基于数据分析优化产品策略和渠道策略

#### 1.2.3 对客户的价值

- **透明化**: 多家保司方案一目了然
- **便捷性**: 续保一键触达,理赔协助在线化
- **专业性**: 获得定制化的保障方案建议

### 1.3 系统边界与范围

#### 1.3.1 本期实现范围

**核心功能**:
- 业务员管理(注册、认证、权限)

#### 1.3.2 后期扩展方向

- 电子投保与支付
- 保单管理与理赔协助

- 智能推荐引擎
- 营销工具箱(海报、朋友圈素材)

### 2.2 核心设计原则

#### 2.2.1 高可用设计

**服务降级**:
- 当某一保司接口超时或异常时,不影响其他保司报价
- 报价引擎采用熔断器模式,自动隔离故障保司

**数据容错**:
- 报价结果存储双写(MySQL+Redis),防止数据丢失
- 重要操作日志落库,支持业务回溯

**限流保护**:
- API层面限流: 单业务员每分钟最多发起10次报价请求
- 保司接口限流: 防止频繁调用被封禁

#### 2.2.2 高性能设计

**并发报价**:
- 使用CompletableFuture实现多保司并发询价
- 线程池隔离,避免某一保司慢响应影响整体

**缓存策略**:
- 车型库数据缓存1小时,减少重复查询
- 车辆基础信息缓存7天,加速续保查询
- 报价结果缓存30分钟,支持快速回显

**异步处理**:
- PDF生成异步化,避免阻塞用户请求
- 续保提醒通过MQ异步发送,削峰填谷

#### 2.2.3 安全设计

**数据脱敏**:
- 车主身份证号中间位数脱敏存储和展示
- 车主手机号中间4位脱敏
- 敏感日志不记录完整信息

**权限控制**:
- 业务员只能查看本人录入的客户数据
- 团队长可查看下属团队数据
- 管理员拥有全局查看权限但需审计

**操作审计**:
- 所有修改操作记录操作人、操作时间、修改内容
- 敏感操作(删除客户、修改报价)需二次确认

### 3.1 车险报价完整流程

#### 3.1.1 业务流程图

```
[业务员] 发起报价
    ↓
[步骤1] 获取车辆信息
    ├─ 方式A: 拍摄行驶证 → OCR识别 → 自动填充
    ├─ 方式B: 输入车牌号 → 查询历史档案 → 回显信息
    └─ 方式C: 手动输入 → 完整录入所有字段
    ↓
[步骤2] 车辆信息确认与补充
    ├─ 核对车牌、VIN、发动机号
    ├─ 选择或修正车型(品牌/车系/车款)
    ├─ 确认初登日期、座位数
    └─ 填写车主姓名、手机号
    ↓
[步骤3] 选择投保方案
    ├─ 交强险(必选)
    ├─ 商业险
    │   ├─ 三者险(保额选择: 50万/100万/150万/200万/300万)

    │   ├─ 盗抢险(是/否)
    │   └─ 附加险(玻璃/划痕/涉水/自燃等)
    └─ 增值服务
        ├─ 道路救援(基础版/豪华版)
        └─ 代驾服务(是/否)
    ↓
[步骤4] 发起多保司询价
    ├─ 系统校验: 必填项完整性检查
    ├─ 并发调用已配置的保司API
    │   ├─ 人保财险API
    │   ├─ 平安车险API
    │   ├─ 太保车险API
    │   ├─ 太平洋车险API
    │   └─ 国寿财险API
    ├─ 超时控制: 单保司最长等待10秒
    └─ 结果聚合: 成功+失败结果汇总
    ↓
[步骤5] 报价结果展示
    ├─ 成功报价
    │   ├─ 按保费升序排列
    │   ├─ 高亮最低价方案
    │   ├─ 展示明细
    │   │   ├─ 交强险: ¥950
    │   │   ├─ 商业险: ¥2,310(含各险种明细)
    │   │   ├─ 车船税: ¥420
    │   │   └─ 总保费: ¥3,680
    │   └─ 费率系数说明
    │       ├─ NCD系数(无赔款优待)
    │       ├─ 自主核保系数
    │       └─ 自主渠道系数
    ├─ 失败报价
    │   ├─ 显示失败原因
    │   │   ├─ 超时无响应
    │   │   ├─ 接口异常
    │   │   ├─ 车辆拒保(高风险)
    │   │   └─ 需人工核保
    │   └─ 提供人工核保引导
    └─ 支持重新询价
    ↓
[步骤6] 生成报价单
    ├─ 选择目标保司方案
    ├─ 生成PDF报价单
    │   ├─ 包含车辆信息
    │   ├─ 方案明细
    │   ├─ 费率说明
    │   ├─ 业务员信息与二维码
    │   └─ 免责条款

        ├─ 关联业务员

    ↓

    ├─ 微信发送PDF
    ├─ 短信发送H5链接

    ├─ 客户查看行为通知业务员
    ├─ 业务员发起回访
    ├─ 客户确认投保 → 转至出单流程(后期开发)
    └─ 未成交 → 记录跟进记录,加入续保池
```

#### 3.1.2 关键决策点说明

**决策点1: OCR识别失败如何处理?**
- 系统提示识别失败原因(光线不足、角度倾斜、图片模糊)
- 提供"重新拍摄"和"手动输入"两个选项
- 部分识别成功的字段保留,仅需补充未识别字段

**决策点2: 车型库匹配不到如何处理?**
- 提供模糊搜索功能,允许按品牌/车系筛选
- 支持手动填写车款名称
- 记录未匹配车型,后台人工补录到车型库

**决策点3: 所有保司都报价失败如何处理?**
- 记录失败原因(网络异常、保司系统维护、车辆拒保)
- 提供"稍后重试"按钮
- 若多次失败,引导转人工线下报价

**决策点4: 客户要求修改方案如何处理?**
- 支持在报价结果页面直接调整方案(如修改三者险保额)
- 无需返回上一步重新填写车辆信息
- 调整后一键重新询价,保持车辆信息不变

### 3.2 续保管理流程

#### 3.2.1 续保提醒触发机制

```
[系统定时任务] 每日凌晨2:00执行
    ↓
[扫描策略]
    ├─ 查询条件
    │   ├─ 保单状态=有效
    │   ├─ 到期日期在 [今天, 今天+90天] 范围内
    │   └─ 未标记为"已流失"
    ├─ 提醒节点
    │   ├─ 90天前: 首次提醒
    │   ├─ 60天前: 二次提醒
    │   ├─ 30天前: 加强提醒
    │   ├─ 15天前: 紧急提醒
    │   └─ 7天前: 最后提醒
    └─ 生成续保任务
        ├─ 任务状态: 待跟进
        ├─ 关联客户: 车主信息
        ├─ 关联业务员: 原报价业务员
        └─ 优先级: 根据到期天数动态计算
    ↓
[推送通知]
    ├─ 业务员APP推送
    │   └─ "您有3个客户车险即将到期,请及时跟进"
    ├─ 短信通知(可配置)
    │   └─ 发送给车主手机
    └─ 邮件通知(可配置)
        └─ 发送给业务员邮箱
    ↓
[业务员跟进]
    ├─ 查看续保任务列表
    │   ├─ 按到期日期排序
    │   └─ 高亮临期客户(<=15天)
    ├─ 一键拨打客户电话
    ├─ 发起续保报价
    │   └─ 自动带入上年度车辆信息和方案
    ├─ 记录跟进结果
    │   ├─ 已联系-有意向
    │   ├─ 已联系-拒绝
    │   ├─ 未接通
    │   └─ 已在他处购买
    └─ 更新任务状态
        ├─ 已续保 → 任务完成
        ├─ 已流失 → 任务关闭
        └─ 待跟进 → 设置下次跟进时间
```

#### 3.2.2 续保转化率统计逻辑

**统计维度**:
- 个人维度: 业务员A的续保转化率 = (已续保数 / 到期提醒总数) × 100%
- 团队维度: 团队B的续保转化率 = (团队已续保数 / 团队到期提醒总数) × 100%
- 时间维度: 本月续保转化率、季度续保转化率

**数据钻取**:
- 支持按车型、保费区间、出险次数等维度分析流失原因
- 流失客户标签分析: 价格敏感型、服务不满型、已迁移型

### 4.1 OCR识别核心逻辑

#### 4.1.1 行驶证识别业务规则

**识别字段映射关系**:

| OCR返回字段 | 业务字段 | 校验规则 | 必填 |
|------------|---------|---------|------|
| plate_num | 车牌号 | 正则: `^[京津沪渝冀豫云辽黑湘皖鲁新苏浙赣鄂桂甘晋蒙陕吉闽贵粤青藏川宁琼使领][A-Z][A-HJ-NP-Z0-9]{5}$` | 是 |
| vin | 车架号 | 长度17位,字母数字组合 | 是 |
| engine_num | 发动机号 | 长度6-20位 | 是 |
| register_date | 初登日期 | 格式: YYYY-MM-DD, 不能晚于今天 | 是 |
| owner | 所有人 | 中文姓名或公司名称 | 是 |
| vehicle_type | 车辆类型 | 小型轿车/小型普通客车/载货汽车等 | 是 |
| use_character | 使用性质 | 非营运/营运/租赁 | 是 |
| model | 品牌型号 | 自由文本,用于车型库匹配 | 否 |

**异常处理策略**:

1. **识别置信度过低** (confidence < 0.7)
   - 字段标红提示人工确认
   - 保留原始识别值供参考
   - 提供"重新拍摄"入口

2. **关键字段缺失**
   - 若车牌号、VIN、发动机号任一缺失,直接提示识别失败
   - 引导用户手动输入缺失字段

3. **日期格式异常**
   - 识别到非标准日期格式(如"二〇二〇年"、"2020.1.1")
   - 系统尝试智能转换为YYYY-MM-DD
   - 转换失败则提示手动选择日期

4. **车牌省份与当前业务员归属地不符**
   - 仅提示,不阻断流程
   - 记录跨地域业务标记,用于后续风控分析

#### 4.1.2 车型库匹配逻辑

**匹配步骤**:

1. **第一步: 品牌识别**
   - 从行驶证"品牌型号"字段提取品牌关键词(如"本田"、"丰田")
   - 与车型库品牌表进行模糊匹配
   - 匹配成功率统计: 目标>85%

2. **第二步: 车系定位**
   - 提取车系关键词(如"雅阁"、"凯美瑞")
   - 在匹配到的品牌下搜索车系
   - 支持同音字纠错(如"雅歌"→"雅阁")

3. **第三步: 车款精确匹配**
   - 提取排量、年款、配置(如"2.0L 自动豪华版 2020款")
   - 计算与车型库中各车款的相似度评分
   - 返回Top3候选车款供用户选择

4. **第四步: 人工兜底**
   - 自动匹配失败时,提供手动搜索功能
   - 支持按品牌→车系→车款逐级下钻
   - 支持输入关键词全文检索

**新车购置价获取**:
- 匹配到车款后,自动带出该车款的指导价
- 支持业务员手动调整(考虑优惠幅度)
- 新车购置价用于车损险保额计算

### 4.2 报价引擎核心逻辑

#### 4.2.1 报价请求参数校验

**必填字段校验清单**:

```
车辆信息:
├─ 车牌号 (plate_no)
├─ 车架号 (vin)
├─ 发动机号 (engine_no)
├─ 品牌名称 (brand_name)
├─ 车款名称 (model_name)
├─ 初登日期 (register_date)
├─ 车辆类型 (vehicle_type)
├─ 使用性质 (use_character)
├─ 座位数 (seat_count)
└─ 新车购置价 (purchase_price)

车主信息:
├─ 车主姓名 (owner_name)
├─ 车主证件类型 (owner_id_type): 身份证/组织机构代码证
├─ 车主证件号 (owner_id_card)
└─ 车主手机号 (owner_mobile)

投保方案:
├─ 交强险 (jqx): 必须为true
├─ 商业险总开关 (commercial_insurance): true/false
└─ 若commercial_insurance=true,需包含:
    ├─ 三者险保额 (third_party_liability): 单位万元
    ├─ 车损险 (vehicle_damage): true/false
    ├─ 司乘险 (driver_liability): {座位数×保额}
    └─ 其他附加险...

保险期间:
├─ 起保日期 (effective_date): 不早于明天,不晚于90天后
└─ 保险期限 (period): 默认1年
```

**业务规则校验**:

1. **时间规则**
   - 初登日期不能晚于今天
   - 起保日期不能早于明天(防止逆选择)
   - 车龄 = 当前年份 - 初登年份
   - 若车龄>15年,部分险种限制投保

2. **保额规则**
   - 三者险保额可选值: 5/10/15/20/30/50/100/150/200/300万
   - 车损险保额 = 新车购置价 × 折旧系数(根据车龄计算)
   - 司乘险单座保额可选: 1/2/3/5/10万

3. **逻辑组合规则**
   - 若选择"车损险",系统自动推荐"玻璃险"、"划痕险"
   - 若使用性质=营运,不能投保"非营业用汽车损失保险"
   - 若车辆类型=新能源,自动去除"涉水险"

#### 4.2.2 多保司并发询价实现

**并发控制策略**:

```
执行流程:
1. 获取已启用的保司列表
   └─ 从配置表 carins_insurer_config 中查询 status=1 的保司

2. 创建异步任务池
   ├─ 核心线程数: 10
   ├─ 最大线程数: 20
   ├─ 队列容量: 100
   └─ 拒绝策略: CallerRunsPolicy(调用者线程执行)

3. 并发发起询价
   ├─ 为每个保司创建CompletableFuture任务
   ├─ 设置超时时间: 单保司最长10秒
   └─ 异常隔离: 单保司失败不影响其他保司

4. 等待聚合结果
   ├─ 使用 CompletableFuture.allOf() 等待所有任务完成
   ├─ 或者 anyOf() 只要有一个成功就返回(快速响应模式)
   └─ 收集成功与失败结果

5. 结果排序与返回
   ├─ 按总保费升序排列
   ├─ 标记最低价方案
   └─ 返回统一数据结构
```

**保司接口适配器模式**:

每个保司实现统一的接口规范:

```
InsurerQuoteService接口定义:
├─ String getInsurerCode()  // 返回保司代码(如"PICC"、"PAIC")

├─ QuoteResultDTO quote(QuoteRequestDTO request)  // 核心询价方法

```

- `ChinalifePCQuoteServiceImpl`: 对接国寿财险API

**异常处理细化**:

| 异常类型 | 处理策略 | 返回给用户的提示 |
|---------|---------|----------------|
| 网络超时 | 记录日志,返回失败结果 | "XX保险公司询价超时,请稍后重试" |

#### 4.2.3 报价结果计算逻辑

**费率计算公式**(以三者险为例):

```
三者险保费 = 基准保费 × NCD系数 × 自主核保系数 × 自主渠道系数

其中:
- 基准保费: 根据车型、保额查费率表
- NCD系数(无赔款优待): 
  └─ 上年度0次出险: 0.85
  └─ 上年度1次出险: 1.0
  └─ 上年度2次出险: 1.25
  └─ 上年度3次及以上: 1.5
- 自主核保系数: [0.85, 1.15], 保司根据风险评估决定
- 自主渠道系数: [0.85, 1.15], 保司给渠道的优惠系数
```

**商业险总保费**:
```
商业险总保费 = Σ(各险种保费)
         = 三者险保费 + 车损险保费 + 司乘险保费 + ...
```

**总保费**:
```
总保费 = 交强险 + 商业险总保费 + 车船税
```

**费率系数展示要求**:
- 在报价明细中展示各系数值
- 用户可点击"?"查看系数说明
- NCD系数高亮显示,提示客户安全驾驶的优惠

### 4.4 续保管理核心逻辑

#### 4.4.1 续保任务生成规则

**数据来源**:
- 方式1: 从历史报价记录提取即将到期保单
- 方式2: 业务员手动创建续保任务
- 方式3: 对接保司API获取到期保单清单

**生成条件**:
```
SELECT 
  v.id AS vehicle_id,
  v.plate_no,
  v.owner_name,
  v.owner_mobile,
  qr.agent_id,
  qr.create_time AS last_quote_time,
  DATE_ADD(qr.create_time, INTERVAL 1 YEAR) AS expire_date
FROM carins_vehicle v
INNER JOIN carins_quote_request qr ON v.id = qr.vehicle_id
WHERE 
  qr.status = 1  -- 报价成功
  AND qr.deleted = 0
  AND DATEDIFF(DATE_ADD(qr.create_time, INTERVAL 1 YEAR), CURDATE()) IN (90, 60, 30, 15, 7)  -- 到期前指定天数
  AND NOT EXISTS (  -- 排除已有续保任务
    SELECT 1 FROM carins_renewal_task rt
    WHERE rt.vehicle_id = v.id AND rt.status != 2  -- 状态!=已流失
  )
```

**任务优先级计算**:
```
优先级 = 100 - 距到期天数

示例:
- 距到期7天: 优先级 = 100 - 7 = 93 (高)
- 距到期30天: 优先级 = 100 - 30 = 70 (中)
- 距到期90天: 优先级 = 100 - 90 = 10 (低)

加权因子:
- 上年保费>5000元: 优先级 +20
- 上年出险次数=0: 优先级 +10
- 客户为VIP标签: 优先级 +15
```

#### 4.4.2 续保跟进记录

**跟进记录字段**:

```
carins_renewal_follow_up (续保跟进记录表)
├─ id: 主键
├─ task_id: 关联续保任务ID
├─ agent_id: 跟进业务员
├─ follow_type: 跟进方式
│   ├─ 1-电话联系
│   ├─ 2-微信沟通
│   ├─ 3-上门拜访
│   └─ 4-短信触达
├─ follow_result: 跟进结果
│   ├─ 1-有意向
│   ├─ 2-考虑中
│   ├─ 3-拒绝(价格)
│   ├─ 4-拒绝(服务)
│   ├─ 5-已在他处购买
│   └─ 6-未接通
├─ follow_content: 跟进内容(文本记录)
├─ next_follow_time: 下次跟进时间
├─ create_time: 跟进时间
```

**跟进提醒规则**:
- 若设置了"下次跟进时间",系统在当天推送提醒
- 若客户"考虑中"超过7天未跟进,自动发送提醒
- 若到期前3天仍未成交,升级为紧急任务并推送给团队长

#### 4.4.3 续保成功标记

**成功判定条件**:
- 条件1: 业务员在系统内为该车辆发起新报价且状态=成功
- 条件2: 业务员手动标记"已续保"并填写新保单号
- 条件3: 对接保司API查询到该车辆新保单(自动标记)

**关联关系更新**:
```
续保成功后的数据更新:
1. 更新续保任务状态 → status=1(已续保)
2. 更新车辆档案 → last_insure_date=新投保日期
3. 生成新的报价记录 → 关联原任务ID
4. 统计业绩数据 → 续保保费计入业绩
5. 触发佣金计算 → 若有续保奖励政策
```

#### 4.5.2 服务订购与激活

**订购流程**:
```
[业务员] 在报价时勾选增值服务
    ↓
[系统] 计算服务费用,加入总保费
    ↓
[客户] 确认并支付
    ↓
[系统] 生成服务订单
    ├─ 订单号: VS20260214001
    ├─ 关联报价单号
    ├─ 服务类型
    ├─ 有效期: 跟随保险期间
    └─ 状态: 待激活
    ↓
[系统] 调用合作方API激活服务
    ├─ 传递客户手机号
    ├─ 传递服务类型和次数
    └─ 获取服务卡号/券码
    ↓
[系统] 发送激活短信给客户
    └─ "您的道路救援服务已激活,服务卡号:XXX,拨打400-XXX-XXXX使用"
```

**使用与核销**:
```
客户使用服务:
1. 拨打服务热线 → 报手机号和服务卡号
2. 服务商提供服务
3. 服务完成后,服务商通过API回调核销
   └─ 传递: 服务卡号、使用时间、服务类型、服务地点
4. 系统记录使用明细
   └─ 剩余次数 = 总次数 - 已使用次数
5. 若剩余次数=0,推送给业务员
   └─ "客户张三的道路救援服务已用完,可推荐续购"
```

#### 4.5.3 服务效果统计

**统计维度**:
- 增值服务销售额占比
- 各类服务销售量排行
- 服务使用率(已使用次数/总次数)
- 服务续购率

**优化建议生成**:
- 若某服务使用率<30%,提示"客户可能不需要,建议减少推荐"
- 若某服务使用率>80%,提示"客户依赖度高,可推荐升级版"

### 5.2 关键业务规则

#### 5.2.1 报价有效期规则

**有效期计算**:
```
报价单有效期 = 报价生成时间 + 3天

示例:
- 报价时间: 2026-02-14 15:30:00
- 有效期至: 2026-02-17 23:59:59
```

**过期处理**:
- 超过有效期的报价单,系统自动标记为"已过期"
- 客户打开过期报价H5页面时,显示"该报价已过期,请联系业务员重新报价"
- 业务员可基于过期报价"一键重新报价",车辆信息自动填充

#### 5.2.2 出险次数与NCD系数对应关系

**NCD系数表**(以人保为例):

| 上年度出险次数 | NCD系数 | 折扣说明 |
|--------------|---------|---------|
| 0次 | 0.85 | 8.5折 |
| 1次 | 1.00 | 无折扣 |
| 2次 | 1.25 | 上浮25% |
| 3次 | 1.50 | 上浮50% |
| 4次及以上 | 1.75 | 上浮75% |
| 发生伤亡事故 | 2.00 | 上浮100% |

**连续无赔优待**:
- 连续2年无出险: NCD系数 = 0.7
- 连续3年无出险: NCD系数 = 0.6

**获取方式**:
1. 对接保司API查询上年度出险记录(最准确)
2. 业务员询问客户并手动录入(需客户确认签字)
3. 默认按1次出险计算(保守估计)

#### 5.2.3 车龄与投保限制

| 车龄 | 交强险 | 商业险 | 特殊说明 |
|------|-------|-------|---------|
| 0-6年 | ✓ | ✓ | 无限制 |
| 7-10年 | ✓ | ✓ | 部分保司要求验车照片 |
| 11-15年 | ✓ | ✓ | 需人工核保,盗抢险受限 |
| 16-20年 | ✓ | 部分 | 仅基本险,多数保司拒保划痕险 |
| 20年以上 | ✓ | × | 商业险拒保,仅能购买交强险 |

**新能源车特殊规则**:
- 自燃险: 建议投保(电池风险)
- 涉水险: 不适用(纯电车无发动机)
- 车损险保额: 按电池折旧后价值计算

**降级策略**:
- 若3次调用均失败,熔断该保司10分钟

- 管理员可手动关闭/启用保司接口

**监控告警**:
- 单保司1小时内失败率>50%,发送告警短信

**故障恢复**:
- 数据库故障: 主从自动切换,RTO<5分钟

**数据恢复**:

### 7.3 安全性要求

#### 7.3.1 身份认证

**JWT令牌机制**:
- 登录成功后颁发JWT Token
- Token有效期: 7天
- 支持刷新Token,无需重复登录
- Token包含用户ID、角色、租户ID

**双因素认证(可选)**:
- 敏感操作(删除客户、修改佣金)需短信验证码
- 异地登录需二次验证

#### 7.3.2 数据加密

**传输加密**:
- 全站HTTPS(TLS 1.2+)
- API接口签名验证

**存储加密**:
- 身份证号: AES-256加密存储
- 手机号: 中间4位脱敏展示
- 密码: BCrypt哈希存储

#### 7.3.3 权限控制

**RBAC模型**:
- 角色: 业务员、团队长、内勤、管理员
- 权限: 增删改查细粒度控制
- 数据隔离: 业务员只能看自己的客户

**操作审计**:
- 记录所有修改操作(谁、何时、改了什么)
- 敏感操作留痕(删除、导出、修改权限)
- 审计日志保留1年

#### 7.4.1 移动端体验

### 附录C: 常见问题FAQ

**Q1: OCR识别失败率过高怎么办?**
A: 
1. 引导用户在光线充足环境拍摄
2. 提供拍摄示例图,标注正确角度
3. 支持多次重拍
4. 兜底方案:手动输入

**Q2: 保司接口全部超时怎么办?**
A:
1. 检查网络连通性
2. 查看保司是否系统维护
3. 启用离线报价模式(基于历史费率估算)
4. 人工线下报价兜底

**Q3: 并发报价时某个保司特别慢怎么办?**
A:
1. 设置单保司超时10秒,超时即返回失败
2. 不等待慢保司,优先展示成功的保司
3. 慢保司结果异步更新(可选)

**Q4: 续保提醒短信发送失败怎么办?**
A:
1. 短信发送失败自动重试3次
2. 仍失败则记录失败日志
3. 管理员每日查看失败清单,人工跟进

**Q5: 客户投诉报价不准怎么办?**
A:
1. 查看报价日志,核对请求参数
2. 联系保司核实费率
3. 若系统问题,及时修复并补偿
4. 建立报价纠错机制
