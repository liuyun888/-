## 1. 个险保单录入

### 1.1 入口与整体流程

菜单路径：寿险 → 保单管理 → 个险保单 → 保单录入

点击【保单录入】按钮，右侧或新页面弹出三步骤向导表单（Step 1 / Step 2 / Step 3），每步骤完成后点【下一步】，第三步点【提交】完成录入。

### 1.2 第一步：基本信息

| 字段名 | 是否必填 | 说明 |
|---|---|---|
| 险种 | ★必填 | 下拉枚举：寿险/重疾/医疗/意外/年金/万能险，从 `ins_life_product_type` 枚举加载 |
| 保险公司 | ★必填 | 下拉，从已配置的寿险合作保司中选取 |
| 产品名称 | ★必填 | 联动险种+保司，下拉选择已上架产品，`ins_life_product` 表中 `status=1` 的记录 |
| 保单号 | ★必填 | 手动输入；后端去重校验：同一保司+保单号不允许重复，重复则返回错误"保单号已存在" |
| 起保日期 | ★必填 | 日期选择器 |
| 到期日期 | ★必填 | 日期选择器；后端校验：到期日期须 > 起保日期 |
| 缴费方式 | ★必填 | 枚举：趸缴/年缴/半年缴/季缴/月缴 |
| 缴费期间 | ★必填 | 数字输入，单位年；趸缴时固定为1年，其他缴费方式手动输入 |
| 年度保费 | ★必填 | 金额，精度两位小数，单位元 |
| 缴费频率 | 选填 | 一般与缴费方式联动自动带出（年=1次/年，月=12次/年等） |
| 件数状态 | 选填 | 枚举：新单/续保/转保 |
| 录入业务员 | ★必填 | 默认当前操作人；管理员可修改为其他在职业务员 |
| 签单日期 | 选填 | 默认等于起保日期 |
| 备注 | 选填 | 文本，最多500字 |

**后端校验逻辑（第一步提交时）：**
- 保单号 + 保险公司组合唯一性校验（查 `ins_policy_life` 表）
- 起保日期/到期日期格式校验
- 年度保费 > 0

### 1.3 第二步：被保人 & 投保人信息

**投保人（必须有且仅有一个）：**

点击【选择客户】弹出客户搜索弹窗 → 输入姓名/证件号搜索 → 选中后自动填充如下字段；也可手动填写。

| 字段名 | 是否必填 | 说明 |
|---|---|---|
| 投保人姓名 | ★必填 | |
| 证件类型 | ★必填 | 枚举：身份证/护照/港澳通行证 |
| 证件号码 | ★必填 | 存储时 AES-256 加密（复用系统已有加密工具类）；展示时脱敏 |
| 出生日期 | ★必填 | 身份证号可自动解析带出 |
| 性别 | ★必填 | 单选，可从身份证解析 |
| 联系电话 | 选填 | |

**被保人（至少一个，可多个）：**

点击【添加被保人】按钮，出现一行被保人录入区。每个被保人字段与投保人类似，额外增加：

| 字段名 | 是否必填 | 说明 |
|---|---|---|
| 与投保人关系 | ★必填 | 枚举：本人/配偶/子女/父母/其他 |
| 受益人类型 | 选填 | 枚举：法定受益人/指定受益人 |
| 受益人信息 | 条件必填 | 若受益人类型=指定受益人，则受益人姓名、证件号为必填 |

每个被保人填写完毕后，需点击该被保人行的【保存】按钮，下方列表出现该被保人一行，方视为添加成功（前端校验此行为，避免用户填了不点保存就提交）。

**自动客户匹配逻辑（后端）：**
- 被保人/投保人提交时，按证件号（解密后精确匹配）查询 `ins_customer` 表
- 匹配到则更新 `ins_policy_life.customer_id`；未匹配则自动创建新客户记录（`ins_customer` 插入新行），再关联

### 1.4 第三步：附件上传

支持分类上传以下类型的附件，每类最多 5 个文件，文件大小限制单个 ≤ 20MB：
- 保单首页
- 缴费通知单
- 健康告知书
- 体检报告
- 其他材料

上传方式：点击对应分类的【上传】按钮 → 选择文件 → 前端直传 OSS，获取 OSS 路径后提交。

点击第三步的【提交】按钮后：
1. 后端完整校验所有三步数据
2. 事务内执行：
   - INSERT `ins_policy_life`（保单主表）
   - INSERT `ins_policy_life_insured`（被保人表，可多行）
   - INSERT `ins_policy_attachment`（附件表）
   - 触发 MQ 消息（`InsOrderProducer.publishPolicyCreated`），异步触发佣金预计算
3. 返回成功，前端弹出"录入成功"提示，跳转到保单列表

## 2. 个险保单查询与导出

### 2.1 搜索条件

列表页顶部展示筛选区，支持以下条件（均为选填，支持组合查询）：

| 筛选项 | 类型 | 说明 |
|---|---|---|
| 保单号 | 文本输入 | 模糊匹配 |
| 被保人姓名 | 文本输入 | 模糊匹配 |
| 投保人姓名 | 文本输入 | 模糊匹配 |
| 险种 | 下拉多选 | |
| 保险公司 | 下拉多选 | |
| 保单状态 | 下拉多选 | 有效/失效/减额缴清/退保/延期缴费/保单贷款 |
| 起保日期范围 | 日期区间 | |
| 到期日期范围 | 日期区间 | |
| 录入业务员 | 下拉搜索 | |

点击【查询】发起 POST 请求，分页返回（默认20条/页）。

### 2.2 列表字段与自定义列

默认显示列：保单号、被保人、险种、保险公司、年度保费、缴费方式、起保日期、到期日期、保单状态、录入业务员、操作。

点击列表右上角【列设置】按钮 → 弹出字段选择弹窗 → 用户勾选需要显示的列 → 点保存 → 前端将配置写入用户偏好（接口：`/admin-api/ins/life/policy/column-config`，按当前用户 ID 存储）。

### 2.3 操作列

每行操作列包含：详情、编辑、核对、验真（若保司支持）、导出单条。

- **详情**：点击打开只读弹窗，展示保单全量信息（含被保人/附件）
- **编辑**：重新打开三步骤表单，带入已有数据，提交后更新。非核心字段直接更新；核心字段（保单号/保险公司/年度保费/险种）修改则走审批流（见第9节）
- **核对**：手动触发与保司数据比对（见第4节）
- **验真**：调用保司验真接口（需配置保司工号），返回验真结果展示

### 2.4 批量导出

列表上方【导出】按钮：
- 导出当前查询条件下的所有记录（不受分页限制）
- 数量 ≤ 2000条：同步生成 Excel，直接下载
- 数量 > 2000条：异步任务，使用 EasyExcel 分批写入，任务完成后写入 `ins_export_task` 表，前端轮询或站内信通知下载链接
- 导出字段：所有保单关键字段，证件号**脱敏**（仅显示首2位和后4位，中间用***代替）

## 3. 批量导入保单

### 3.1 操作流程

1. 点击列表页【批量导入】按钮 → 弹出导入向导弹窗
2. 第一步选择险种（确定使用哪个导入模板，不同险种字段有差异）
3. 点击【下载模板】下载对应险种的 Excel 模板
4. 用户在本地填写 Excel，返回系统点击【选择文件】上传（限制：xlsx格式，≤5000行，文件大小≤20MB）
5. 上传后后端 **预解析**：使用 EasyExcel 读取前20行进行数据校验，返回预览结果展示给用户（包含列名映射、每列校验状态）
6. 用户确认预览无误后，点击【开始导入】正式提交
7. 若存在错误行：弹窗提示"有N行数据存在问题，是否跳过错误行继续导入？"，用户选择"跳过"或"取消"

### 3.2 后端导入逻辑

1. 读取 Excel（EasyExcel）按行解析，每行映射到 `InsLifePolicyImportDTO`
2. 逐行校验：必填字段为空、保单号格式、日期格式、金额格式
3. 对校验通过的行：
   - 查重：保单号+保险公司已存在则标记为"重复"，不插入
   - 自动匹配客户（同录入逻辑，按证件号精确匹配）
   - 批量 INSERT，每批500条（MyBatis Plus `saveBatch`），使用事务
4. 全部处理完毕后生成导入报告：
   - 成功N条、失败N条（含原因）、重复N条
   - 报告写入 `ins_import_log` 表（字段：batch_no, total, success, fail, repeat, file_url, operator_id, create_time）
5. 发送站内信通知操作人：导入任务完成，成功X条，失败Y条

### 3.3 导入模板字段（以寿险-重疾险为例）

模板表头：保单号*、险种*、保险公司*、产品名称*、起保日期*、到期日期*、年度保费*、缴费方式*、缴费期间*、投保人姓名*、投保人证件号*、被保人姓名*、被保人证件号*、被保人出生日期*、被保人性别*、录入业务员（工号）

## 4. 保单核对

### 4.1 功能说明

将保司下发的对账数据与系统保单数据进行比对，找出差异。

### 4.2 操作流程

1. 菜单：寿险 → 保单管理 → 个险保单 → 保单核对
2. 点击【导入保司数据】→ 选择保司 → 下载该保司的对账模板 → 填写后上传
3. 上传完成后，点击【开始核对】，后端异步执行比对（数据量大时可能耗时，完成后发站内信通知）
4. 核对完成后展示差异报告页面

### 4.3 差异分类与处理

后端比对逻辑：以保单号 + 被保人证件号（AES解密后比对）为匹配键，逐行比对：

| 差异类型 | 判断逻辑 | 展示颜色 |
|---|---|---|
| 系统有/保司无 | 系统中存在该保单，保司数据中没有 | 橙色 |
| 保司有/系统无 | 保司数据中存在，系统中不存在 | 红色 |
| 金额差异 | 两边都有该保单，但年度保费不一致 | 黄色 |
| 完全一致 | 两边数据一致 | 绿色 |

每条差异记录提供三个处理按钮：
- **接受保司**：以保司数据为准，更新系统保单数据（需记录变更日志）
- **保留系统**：标记该差异为"已处理-保留系统"，不修改数据
- **手动修正**：打开编辑弹窗，手动修改数据后保存

核对记录归档到 `ins_policy_reconcile` 表（字段：reconcile_no、insurer_id、operate_date、total_count、diff_count、status、operator_id）。核对报告可导出为 Excel。

## 6. 回执管理

### 6.1 功能概述

菜单：寿险 → 保单管理 → 回执管理

记录每张保单的回执签收情况，监控逾期未回执的保单。

### 6.2 回执状态更新

**手动更新（单条）：**
- 列表中点击某条回执记录的【编辑】按钮
- 修改字段：回执类型（签收回执/电子回执）、回执时间（日期）、处理人（当前操作人）
- 保存后更新 `ins_policy_receipt` 表对应记录的 `receipt_status = 1`（已回执）、`receipt_time`、`handler_id`

**批量导入更新：**
- 下载批量更新模板（字段：保单号、回执类型、回执时间）
- 上传 Excel → 后端按保单号匹配更新 `ins_policy_receipt` 表
- 导入结果报告同批量导入保单逻辑

### 6.3 逾期预警定时任务

XXL-Job Handler：`InsReceiptReminderJob`，每天 09:00 执行：
1. 查询：`ins_policy_receipt` 中 `receipt_status = 0`（未回执）且 `承保时间 + 30天 < now()`
2. 对每条记录：取对应保单的 `agent_id`，发送站内信：内容"保单[保单号]回执已逾期，请及时处理"
3. 更新 `is_overdue = 1`

### 6.4 回执统计看板

展示两个指标（聚合查询）：
- **逾期率** = `is_overdue=1` 数量 / 总保单数量
- **完成率** = `receipt_status=1` 数量 / 总保单数量

按时间维度（月/季/年）筛选，数据以卡片形式展示。

## 7. 回访管理

### 7.1 功能概述

菜单：寿险 → 保单管理 → 回访管理

监管要求：新单承保后30天内必须完成回访。系统需记录回访情况并自动预警。

### 7.2 新增回访记录

点击【新增回访】按钮，弹出新增弹窗：

| 字段名 | 是否必填 | 说明 |
|---|---|---|
| 保单号 | ★必填 | 输入保单号后，后端查询关联带出：被保人姓名、险种、承保日期、业务员名称 |
| 回访时间 | ★必填 | 日期时间选择器 |
| 回访方式 | ★必填 | 枚举：电话/上门/视频/微信 |
| 客户反馈 | ★必填 | 枚举：满意/一般/不满意/拒绝回访 |
| 回访结论 | ★必填 | 文本，最多1000字 |
| 是否在30天内 | 只读字段 | 后端根据承保日期自动计算并显示"合规"/"逾期"标签 |
| 附件 | 选填 | 支持上传回访录音/截图，OSS存储，最多5个文件 |

保存时 INSERT `ins_life_visit_record`（字段：policy_id, visit_time, visit_type, customer_feedback, visit_conclusion, attachment_urls, is_compliant, operator_id）

### 7.3 批量导入回访记录

场景：保司系统有批量回访数据需要导入。流程同保单批量导入，模板字段：保单号、回访时间、回访方式、客户反馈、回访结论。

### 7.4 回访提醒定时任务

XXL-Job Handler：`InsVisitReminderJob`，每天 09:00 执行：
1. 查询：`ins_policy_life` 中 `承保日期 + 25天 >= now()` 且 `承保日期 + 30天 >= now()` 且该保单 `ins_life_visit_record` 中无回访记录
2. 即承保25-30天内仍未回访的保单
3. 发站内信给业务员："保单[保单号]即将超出30天回访期限，请尽快完成回访"

### 7.5 回访统计

- **回访率** = 已回访保单数 / 应回访保单总数（承保后30天已过的保单）
- **逾期回访率** = 超30天才回访或未回访数 / 总数
- 导出回访报告（EasyExcel，含所有回访记录明细）

## 8. 保单状态变更

### 8.1 操作入口

菜单：寿险 → 保单管理 → 保单修改 → 状态变更

在保单列表操作列点击【状态变更】按钮，弹出状态变更弹窗。

### 8.2 状态流转规则

状态机（`ins_policy_life.policy_status`）：

```
有效（ACTIVE）
  ├── → 失效（LAPSE）          触发：欠费未续缴
  ├── → 减额缴清（REDUCED_PAID）  触发：客户申请停缴，保额等比例缩减
  ├── → 延期缴费（GRACE_PERIOD）  触发：宽限期内
  ├── → 保单贷款（LOAN）         触发：客户以保单现金价值贷款
  └── → 退保（SURRENDERED）     触发：客户申请退保
```

非有效状态不可再变更（除非特殊审批）。

### 8.3 变更表单字段

| 字段名 | 是否必填 | 说明 |
|---|---|---|
| 目标状态 | ★必填 | 下拉，只展示当前状态可流转到的目标状态 |
| 变更原因 | ★必填 | 文本，最多500字 |
| 生效日期 | ★必填 | 默认今天，可选未来日期（不可选过去日期） |
| 证明材料 | 条件必填 | 退保时必须上传，其他状态选填 |

### 8.4 后端处理逻辑

1. 校验状态流转合法性（不合法返回错误）
2. UPDATE `ins_policy_life.policy_status` = 目标状态，`status_change_date` = 生效日期
3. INSERT `ins_policy_status_log`（old_status, new_status, change_reason, effective_date, operator_id, change_time）
4. 更新续期管理队列（`ins_life_renewal_track`）：若保单失效/退保，将对应续期记录标记为"已结束"
5. **退保特殊处理**：发送 MQ 消息，触发佣金回收计算（佣金系统消费消息，计算应回收的已发放佣金）
6. 非退保变更不影响已计算佣金

## 9. 保单修改与批量影像件

### 9.1 保单字段修改

**非核心字段（直接修改）：** 联系电话、收件地址、备注、附件

**核心字段（需审批流）：** 保单号、保险公司、险种、产品名称、年度保费、起保日期、到期日期、被保人信息

**核心字段修改流程：**
1. 点击【编辑】，在编辑弹窗中修改核心字段后，提交时检测到有核心字段变更
2. 弹出确认框："检测到核心字段变更，需提交审批"
3. 用户确认 → 创建审批工单（Flowable），状态为"待审核"
4. 审批通过后：UPDATE `ins_policy_life` 对应字段；INSERT `ins_policy_change_log`（field_name, old_value, new_value, approve_no, approve_time, operator_id）
5. 审批拒绝后：保单数据不变，通知申请人

### 9.2 批量影像件上传

1. 在保单列表勾选多个保单（最多50个），点击【批量上传影像件】
2. 弹出上传弹窗，选择影像件分类（保单首页/条款页/健康告知书/其他）
3. 上传文件后，后端按如下命名规则归档到 OSS：
   - 路径：`/life-policy/{insurer_code}/{policy_no}/{category}/{filename}`
4. 批量关联写入 `ins_policy_attachment` 表（每个保单+分类+文件URL一行）
5. 支持同一分类多次上传（追加，不覆盖旧文件）

## 10. 保全维护

### 10.1 保全类型

枚举值（`ins_life_conservation_type`）：
- `BENEFICIARY_CHANGE`：变更受益人
- `POLICY_HOLDER_CHANGE`：变更投保人
- `SUM_REDUCE`：减保
- `PREMIUM_STOP`：停缴
- `REINSTATEMENT`：复效
- `CONVERT`：转换（险种转换）

### 10.2 新增保全

菜单：寿险 → 保单管理 → 保全维护 → 新增保全

点击【新增】按钮，弹出保全录入弹窗：

| 字段名 | 是否必填 | 说明 |
|---|---|---|
| 保全类型 | ★必填 | 下拉，从保全类型枚举选择 |
| 保单号 | ★必填 | 输入后带出保单基本信息（被保人/险种/保司） |
| 被保人 | 只读带出 | 从保单关联带出 |
| 申请内容 | ★必填 | 文本描述，根据保全类型提示填写内容（如"变更受益人"需填写新受益人信息） |
| 申请日期 | ★必填 | 默认今天 |
| 附件材料 | ★必填 | 上传保全相关证明文件，至少1个 |

**不同保全类型的扩展字段：**
- 变更受益人：新受益人姓名★、新受益人证件号★、与被保人关系★
- 减保：新保额★（需 > 0 且 < 原保额）
- 复效：欠缴保费金额★（系统自动计算）

提交后 INSERT `ins_life_conservation`，状态为 `PENDING`（待审核）。

### 10.3 审批流程

保全申请提交后自动进入 Flowable 审批流（`LIFE_CONSERVATION_APPROVE`）：

```
申请人提交 → 机构负责人审批 → 平台管理员审核 → 审批通过
```

- 审批通过：UPDATE `ins_policy_life` 对应字段（根据保全类型更新不同字段）；`ins_life_conservation.status = APPROVED`
- 审批拒绝：`ins_life_conservation.status = REJECTED`；站内信通知申请人，附拒绝原因

### 10.4 批量保全

点击【批量保全】→ 下载批量保全模板 → 填写（保全类型/保单号/申请内容） → 上传。

后端解析后为每行创建一条 `ins_life_conservation` 记录，批量进入审批队列。

导入结果报告展示：成功N条进入审批 / 失败N条（含原因）。

### 10.5 保全记录查询

筛选：保单号、保全类型、状态（待审核/已通过/已拒绝）、申请日期范围。

## 11. 孤儿单分配与轨迹

### 11.1 孤儿单定义

业务员离职后（`sys_user.status = RESIGNED`），其名下的寿险保单即成为孤儿单，需分配给在职业务员继续跟进。

**孤儿单识别逻辑（XXL-Job，每天执行）：**
1. 查询当天离职的业务员（`sys_user_change_log` 中今日离职记录）
2. 将其名下所有 `policy_status = ACTIVE` 的保单标记到孤儿单池：`ins_life_orphan.status = UNASSIGNED`

### 11.2 孤儿单池列表

菜单：寿险 → 孤儿单管理 → 孤儿单池

展示所有未分配（`UNASSIGNED`）或归属离职业务员的保单，列表字段：保单号、被保人、险种、保司、年度保费、原业务员（已离职）、入池时间、状态。

筛选：险种、保司、入池日期范围、状态（未分配/已分配）。

### 11.3 分配操作

**单条分配：**
1. 点击某行的【分配】按钮
2. 弹出分配弹窗，字段：
   - 接手业务员★：下拉搜索，只显示本机构在职业务员
   - 分配原因★：文本，最多200字
3. 点确定后：
   - UPDATE `ins_policy_life.agent_id` = 新业务员ID
   - UPDATE `ins_life_orphan.status = ASSIGNED`、`assigned_agent_id` = 新业务员ID、`assigned_time` = now()
   - INSERT `ins_life_orphan_log`（policy_id, from_agent_id, to_agent_id, reason, operator_id, assign_time）

**批量分配：**
1. 勾选多个孤儿单（同一机构下）
2. 点击【批量分配】按钮
3. 弹窗选择接手业务员 + 分配原因
4. 后端循环处理，每条保单单独记录分配轨迹
5. 全部处理在同一个事务中（任意失败则回滚）

### 11.4 分配轨迹查询

菜单：寿险 → 孤儿单管理 → 分配轨迹

筛选：保单号、被保人姓名、原业务员、接手业务员、分配时间范围。

列表展示 `ins_life_orphan_log` 所有记录，每条记录可查看完整的分配历史（一个保单可能被多次分配）。
