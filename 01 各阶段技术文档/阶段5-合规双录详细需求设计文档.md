## 一、概述

### 1.1 模块背景

根据《保险销售行为管理办法》、银保监会关于《保险销售行为可回溯管理暂行办法》等监管要求,保险中介机构在销售人身保险新型产品、60岁以上客户投保、趸交保费超5万元或期交保费超2万元等场景下,**必须实施双录(录音+录像)**,确保销售过程可回溯、关键环节有留痕。

本阶段目标是构建一套完整的**合规双录与回溯存证系统**,涵盖远程/现场双录、AI实时质检、行为轨迹存证、审计后台及第三方公证存证,满足监管要求的同时提升销售合规性。

### 1.2 业务价值

1. **监管合规**:满足银保监会双录要求,避免因合规问题导致的行政处罚
2. **风险防控**:通过AI质检实时识别销售误导、话术违规等风险行为
3. **纠纷取证**:客户投诉时可快速调取双录视频作为证据链
4. **效率提升**:可视化流程配置,支持断点重录,降低业务员操作难度
5. **数据安全**:对接区块链存证,确保证据不可篡改

## 三、功能模块详细设计

### 3.1 双录引擎

#### 3.1.1 双录触发判断

**业务规则**:

1. **年龄触发**: 投保人或被保险人年龄≥60岁
2. **保费触发**: 趸交保费≥5万元 或 期交保费≥2万元
3. **产品触发**: 分红险、万能险、投连险等新型产品
4. **风险触发**: 保险公司要求双录的其他情形

**实现逻辑**:

```
双录触发判断服务 (DoubleRecordTriggerService)

方法: checkNeedDoubleRecord(OrderDO order)
输入: 订单对象
输出: Boolean (是否需要双录) + String (触发原因)

判断流程:
1. 获取投保人、被保险人年龄
2. 计算保费金额(趸交/期交)
3. 查询产品类型配置
4. 匹配双录规则库
5. 返回判断结果及原因
```

**接口定义**:

```
POST /admin-api/insurance/double-record/check-trigger

Request Body:
{
  "orderId": 123456,
  "applicantAge": 65,
  "insuredAge": 65,
  "productType": 1,  // 1-分红险
  "paymentType": 1,  // 1-趸交 2-期交
  "premium": 60000,
  "paymentPeriod": 10
}

Response:
{
  "code": 0,
  "data": {
    "needDoubleRecord": true,
    "triggerReasons": [
      "投保人年龄65岁,超过60岁阈值",
      "趸交保费6万元,超过5万元阈值"
    ],
    "recordScene": 1  // 双录场景编码
  }
}
```

#### 3.1.2 话术模板配置

**管理后台功能**:

1. **模板列表**: 展示所有话术模板,支持按产品类型、版本筛选
2. **模板新增**: 可视化配置话术节点
3. **模板编辑**: 修改节点内容、关键词、禁用词
4. **模板版本管理**: 支持多版本并存,可指定生效版本

**话术节点配置器**:

```
节点配置表单:
- 节点序号 (自动生成)
- 节点名称 (如: "身份确认"、"产品告知"、"风险提示")
- 节点类型 (必读/选读)
- 话术内容 (支持富文本编辑)
- 预期回应 (客户应有的反馈)
- 检查点 (用于AI质检)
- 关键词 (必须包含的词汇)
- 禁用词 (不能出现的词汇)
- 最小时长 (该节点最短耗时,单位:秒)

操作按钮:
- 添加节点
- 删除节点
- 上移/下移节点
- 预览完整话术
```

**接口定义**:

```
GET /admin-api/insurance/double-record/template/list
POST /admin-api/insurance/double-record/template/create
PUT /admin-api/insurance/double-record/template/update
DELETE /admin-api/insurance/double-record/template/delete/{id}
GET /admin-api/insurance/double-record/template/get/{id}
```

#### 3.1.3 远程/现场双录流程

**远程双录 (基于RTC实时音视频)**:

**技术选型**: 
- 腾讯云TRTC / 声网Agora / 网易云信

**流程设计**:

```
业务员端 (APP/PC):
1. 发起双录邀请 -> 生成RTC房间ID
2. 将邀请链接发送给客户(短信/微信)
3. 进入RTC房间
4. 按照话术模板逐节点朗读
5. 客户实时视频交互
6. 完成所有节点后结束双录
7. 视频自动上传云端存储

1. 收到邀请链接,点击进入
```

#### 3.2.1 ASR语音转文字

**技术方案**:

```

- 返回时间戳对齐
- 支持语气词过滤
```

**业务流程**:

```
双录开始 -> 实时传输音频流给ASR服务
             ↓
          ASR返回文字片段 + 时间戳
             ↓
          保存到质检记录表 (asr_text字段)
             ↓
          实时关键词匹配
             ↓
       发现禁用词 -> 立即弹窗提醒业务员
```

**关键词检测逻辑**:

```
必读关键词检测:
1. 获取话术模板的keyWords数组
2. 对ASR文本进行分词
3. 计算关键词覆盖率 = 命中数 / 必读关键词总数
4. 覆盖率 < 100% -> 质检不通过

禁用词检测:
1. 获取话术模板的forbiddenWords数组
2. 对ASR文本进行全文匹配
3. 发现任意禁用词 -> 记录违规项 + 扣分
4. 常见禁用词:
   - "保本"、"无风险"、"固定收益"
   - "保证赚钱"、"稳赚不赔"
   - "银行存款"、"储蓄"
```

```
3. 后台记录: 标记为"疑似快速跳过"
4. 质检标记: 人工质检时重点关注
```

**实现方式**:

```
前端计时器:
1. 页面加载时启动计时器
2. 每秒检查当前停留时长
3. 未达到最低时长时,"下一步"按钮置灰
4. 达到最低时长后,按钮变为可点击状态
5. 页面离开时上报最终停留时长

示例代码(Vue):
data() {
  return {
    stayDuration: 0,
    minDuration: 30,  // 最低30秒
    canNext: false
  }
},
mounted() {
  this.timer = setInterval(() => {
    this.stayDuration++;
    if (this.stayDuration >= this.minDuration) {
      this.canNext = true;
    }
  }, 1000);
},
beforeDestroy() {
  clearInterval(this.timer);
  this.reportStayDuration();
}
```

### 3.5 存证加固

#### 3.5.1 区块链存证集成

**区块链平台选择**:

```
推荐平台:
1. 蚂蚁链 (阿里云)
   - 优势: 与支付宝实名体系打通,存证公信力强
   - 适用场景: 金融、保险等强监管行业
   
2. 腾讯至信链
   - 优势: 与微信生态集成,易于推广
```

```
1. 生成文件哈希

   - 存储到ins_blockchain_evidence表
```

#### 3.5.3 证据链不可篡改验证

**防篡改机制**:

```
1. 文件哈希校验
   - 存证时计算原始文件哈希
   - 验证时重新计算文件哈希
   - 比对两次哈希值是否一致

2. 区块链时间戳
   - 区块链上记录存证时间
   - 时间戳不可逆、不可篡改
   - 可证明文件在某时间点已存在

3. 公证书法律效力
   - 公证书具有法律证明力
   - 法院诉讼中可直接作为证据
   - 对方需举证推翻

4. 多重备份
   - 文件存储在多个云端节点
   - 区块链存证在多个节点
   - 公证书线上线下双保险
```

## 五、业务规则与逻辑

### 5.1 双录触发规则

```
规则1: 投保人或被保险人年龄≥60岁 -> 必须双录
规则2: 趸交保费≥5万元 -> 必须双录
规则3: 期交保费≥2万元 -> 必须双录
规则4: 产品类型为分红险/万能险/投连险 -> 必须双录
规则5: 保险公司特别要求双录的产品 -> 必须双录

优先级:
满足任意一条规则即触发双录

特殊处理:
- 线上投保默认不双录,仅做行为轨迹存证
- 线下投保或电话销售强制双录
- 可在产品配置中自定义双录规则
```

### 5.2 话术节点必读逻辑

```
必读节点定义:
在话术模板的script_nodes中,nodeType为"必读"的节点

必读检测:
1. 业务员必须按顺序完成所有必读节点
2. 不能跳过必读节点
3. 必读节点最小时长必须满足

未读处理:
1. 前端阻止进入下一节点
2. AI质检检测到未读 -> 质检不通过
3. 人工质检发现未读 -> 要求重录

关键词检测:
必读节点的keyWords数组中的词汇必须全部出现在ASR文本中
```

### 5.3 重录次数限制

```
默认重录次数: 3次

重录场景:
1. 网络中断导致录制失败
2. 业务员操作失误
3. AI质检发现违规
4. 客户要求重录

重录处理:
1. 重录次数≤3: 业务员可自行重录
2. 重录次数>3: 需主管审批
3. 重录次数>5: 需合规部门介入调查

重录记录:
每次重录都记录在ins_double_record表的retry_count字段
记录最后重录时间last_retry_time
```

### 5.4 质检评分规则

```
初始分: 100分

扣分项:
1. 使用禁用词: 每次扣20分
2. 未读关键条款: 每次扣10分
3. 跳过必读节点: 每次扣15分
4. 关键词遗漏: 每次扣5分
5. 身份证翻拍: 扣30分
6. 人脸识别不通过: 扣30分

评分等级:
- 90-100分: 优秀
- 80-89分: 良好
- 60-79分: 合格
- <60分: 不合格

不合格处理:
1. 质检结论为"不通过"
2. 业务员需重录
3. 保单暂缓承保
4. 记录到业务员绩效档案
```

### 5.5 存证时效要求

```
存证时限:
- 双录完成后24小时内必须完成区块链存证
- 双录完成后72小时内必须完成AI质检
- 双录完成后7个工作日内必须完成人工质检

超时处理:
- 超时未存证 -> 系统自动告警,通知合规部门
- 超时未质检 -> 系统自动分配给质检员
- 超时未完成质检 -> 上报主管

存储期限:
- 双录视频至少保留5年(监管要求)
- 行为轨迹数据至少保留3年
- 区块链存证永久保存
```

## 七、性能优化建议

### 7.1 视频上传优化

```
分片上传:
- 将大视频文件分成多个小片(如5MB/片)
- 并行上传多个分片
- 上传失败的分片可单独重传
- 所有分片上传完成后合并

断点续传:
- 记录已上传的分片
- 网络中断后从断点继续上传
- 不需要重新上传已完成的部分

压缩优化:
- 使用H.264编码
- 分辨率建议: 1280x720 (720P)
- 码率建议: 1500kbps
- 帧率建议: 25fps
```

### 7.2 AI质检异步处理

```
异步任务队列:
- 使用RabbitMQ/RocketMQ消息队列
- 双录完成后发送质检任务到队列
- 后台消费者异步处理质检

任务优先级:
- 高优先级: VIP客户、大额保单
- 普通优先级: 常规保单
- 低优先级: 测试数据

并发控制:
- 限制同时执行的质检任务数量
- 避免AI服务过载
- 合理分配计算资源
```

### 7.3 存储成本优化

```
生命周期管理:
- 0-3个月: 标准存储(热数据,快速访问)
- 3-12个月: 低频存储(访问频率降低)
- 12个月+: 归档存储(合规保留,几乎不访问)

压缩存储:
- 视频使用H.264编码压缩
- JSON数据使用gzip压缩
- 预计可节省50%以上存储空间

CDN加速:
- 视频文件使用CDN分发
- 减少源站带宽消耗
- 提升访问速度
```
