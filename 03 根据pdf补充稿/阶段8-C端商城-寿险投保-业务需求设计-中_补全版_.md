## 一、投保流程总览

用户从产品详情页点击「立即投保」后，进入以下四步线性流程（步骤条显示在页面顶部，已完成步骤为绿色勾，当前步骤高亮）：

```
第1步：健康告知  →  第2步：填写信息  →  第3步：缴费确认  →  第4步：支付&核保
```

每步完成后方可进入下一步，不可跳过。点击顶部步骤条的已完成步骤可返回修改。离开页面时弹出"确认放弃本次投保？"提示，确认后清除当前投保草稿数据。

**投保草稿机制**：用户每次开始投保时，后端创建一条 `ins_life_order_draft` 记录（`draft_id`），前端所有步骤的表单数据实时通过 `PUT /app-api/ins/life/order/draft/{draft_id}` 保存，防止意外退出导致数据丢失。草稿 24 小时未完成则自动过期（XXL-Job 定时清理）。

## 二、功能点详细业务逻辑

### 2.1 健康告知问卷（C端）

> **对应排期表功能点**：健康告知问卷(C端) | 前端 2天 + 后端 1.5天
>
> **技术难点**：动态问卷复用AI保障规划组件；初步智能核保规则引擎

#### 2.1.1 页面触发

点击产品详情页「立即投保」按钮后：

1. 若用户未登录，先跳转微信授权登录页，登录后回跳并携带 `product_id`。
2. 若用户已登录，前端调用 `POST /app-api/ins/life/order/start`，入参 `{product_id}`，后端创建草稿记录，返回 `draft_id`，前端跳转至健康告知问卷页并携带 `draft_id`。

**归因追踪**：若用户从业务员分享链接进入，URL 中携带 `agent_id` 参数，`start` 接口须将其写入草稿 `agent_id` 字段，后续保单归因使用。

#### 2.1.2 问卷加载

前端调用 `GET /app-api/ins/life/product/{product_id}/health-notice`，返回该产品的健康告知问题列表（来自 `ins_life_product.health_notice_json`）。

`health_notice_json` 数据结构：

```json
[
  {
    "question_id": "q1",
    "question_text": "您是否患有高血压（含临界高血压）？",
    "type": "YN",
    "required": true,
    "sub_questions": [
      {
        "question_id": "q1_1",
        "condition": "Y",
        "question_text": "请描述您的血压情况（收缩压/舒张压数值、是否服药）",
        "type": "TEXT",
        "required": true
      }
    ]
  },
  {
    "question_id": "q2",
    "question_text": "过去5年内，您是否因病住院超过7天？",
    "type": "YN",
    "required": true,
    "sub_questions": []
  }
]
```

#### 2.1.3 问卷渲染规则

- 每道题显示题目序号、题目文字，提供「是」「否」两个选项按钮（互斥单选）。
- **必须全部作答**，未作答题目高亮边框提示（红色），点击「下一步」时前端校验所有 `required = true` 的题目是否已作答，未完成则阻止提交并滚动到第一个未答题目。
- 当用户选择「是」时，该题红色高亮提示"注意：此选项可能影响承保结论"，且若该题有 `sub_questions`，则动态展开追问题目，追问题目同样必填。
- 当用户选择「否」时，追问题目折叠隐藏，清空追问题目已填内容。
- 不允许跳过任何题目，不提供"跳过此题"按钮。
- 问卷底部勾选框：**"本人已如实告知，并了解如未如实告知可能影响理赔"**，**必须勾选**才能点击「提交健康告知」按钮（按钮否则置灰）。

**页面布局细节**：
- 顶部展示产品名称和投保步骤条（第1步高亮）。
- 问卷题目列表垂直滚动，每道题之间有分隔线。
- 每道题答案按钮为左右排列的「是」「否」两个圆角矩形按钮，选中状态为填充色（主题色），未选中为描边白底。
- 页面底部固定「提交健康告知」按钮，不随列表滚动。

#### 2.1.4 提交健康告知与智能核保

点击「提交健康告知」，前端调用 `POST /app-api/ins/life/order/health-notice`：

入参：

```json
{
  "draft_id": "xxx",
  "product_id": 12345,
  "answers": [
    {"question_id": "q1", "answer": "N"},
    {"question_id": "q2", "answer": "Y", "sub_answers": [{"question_id": "q2_1", "answer": "2021年因阑尾炎住院10天"}]}
  ]
}
```

**后端处理步骤**：

1. 校验 `draft_id` 归属当前用户且状态为"健康告知"步骤。
2. 校验所有 `required = true` 的问题均已提交答案（防止前端绕过校验）。
3. 将答案序列化存入 `ins_life_order_draft.health_notice_answers`（JSON 字段）。
4. **执行初步智能核保**（同步，不超过 3 秒）：从 `ins_life_product.underwriting_rules` 加载规则，使用规则引擎（Drools 或 JSON 规则解析器）逐条评估答案。

核保规则示例：

```json
{
  "rules": [
    {
      "condition": "q1.answer == 'Y'",
      "result": "REJECT",
      "reason": "患有高血压，暂不支持线上投保，建议联系专属顾问"
    },
    {
      "condition": "q2.answer == 'Y'",
      "result": "MANUAL_REVIEW",
      "reason": "近5年有住院史，需人工核保，审核周期1-3个工作日"
    }
  ],
  "default_result": "PASS"
}
```

5. 核保结果处理：

| 核保结果 | 处理逻辑 |
|---|---|
| `PASS` | 更新草稿步骤为"填写信息"，返回 `{result: "PASS"}`，前端自动跳转第2步 |
| `REJECT` | 更新草稿状态为"已拒绝"，返回 `{result: "REJECT", reason: "xxx"}`，前端展示拒保原因页，提示"建议联系专属顾问"，显示业务员联系按钮 |
| `MANUAL_REVIEW` | 此场景简化处理：弹窗告知用户"您的情况需进一步了解，请联系专属顾问协助投保"，引导用户拨打客服或联系业务员，本次投保流程终止 |

**拒保/人工审核后的引导设计**：
- 拒保页：展示拒保原因（脱敏，不暴露医学细节）、「联系专属业务员」按钮（拨号或跳转聊天）、「查看其他产品」按钮（跳转产品列表并过滤掉该产品）。
- 人工审核弹窗：展示业务员头像+姓名（若有归因业务员），「立即联系」按钮（拨号或微信客服），「稍后再说」关闭弹窗。

### 2.2 投保人 / 被保人信息（C端）

> **对应排期表功能点**：投保人/被保人信息(C端) | 前端 1.5天 + 后端 1天
>
> **技术难点**：受益人多人比例配置；未成年人特殊处理
>
> **参考 PDF**：`110`（保单录入中关系人/被保人/受益人信息维护逻辑）

#### 2.2.1 页面加载逻辑

进入第2步页面后，调用 `GET /app-api/ins/life/order/draft/{draft_id}/policy-holders`，返回页面预填信息：

- 投保人信息：从 `member` 表读取当前登录用户的认证信息（姓名、身份证号、手机号），已实名认证则自动填入，未认证则提示完成实名认证。
- 被保人信息：默认不填，等待用户操作。
- 受益人信息：默认"法定受益人"。

**未完成实名认证处理**：若用户未实名认证，页面顶部展示橙色提醒横幅"您尚未完成实名认证，请先认证再投保"，并提供「去认证」按钮跳转实名认证页面，认证完成后自动返回本步骤。

#### 2.2.2 投保人信息（只读/部分可编辑）

| 字段 | 必填 | 说明 |
|---|---|---|
| 姓名 | ✅ | 已认证用户只读，不可修改 |
| 身份证号 | ✅ | 已认证用户只读，不可修改（脱敏显示，如：张*明，110***1990***0011） |
| 手机号 | ✅ | 已认证用户只读，不可修改 |
| 证件有效期 | ✅ | 可编辑，选择日期（身份证长期有效则勾选"长期"） |
| 常住地址 | ✅ | 省市区三级联动选择 + 详细地址文本输入 |

**投保人与被保人关系**：下拉选择，选项：本人/配偶/父母/子女/其他，默认"本人"（当被保人与投保人相同时自动设为"本人"）。

#### 2.2.3 被保人信息（核心填写区）

**被保人来源选择**（三选一，顶部 Tab 切换）：

- **与投保人相同**：点击后自动填入投保人信息，无需再填。
- **从家庭成员选取**：调用 `GET /app-api/member/family/list` 返回用户已维护的家庭成员列表，点击选取后自动填入，可编辑。家庭成员列表以卡片形式展示（头像+姓名+关系），无家庭成员时显示"暂无家庭成员，去添加"入口，跳转至「家庭成员管理」页。
- **手动填写**：所有字段手动输入。

被保人表单字段：

| 字段 | 必填 | 说明与校验 |
|---|---|---|
| 姓名 | ✅ | 2-20个汉字 |
| 性别 | ✅ | 单选：男/女 |
| 出生日期 | ✅ | 日期选择器，后端根据产品 `min_insure_age`/`max_insure_age` 校验是否在投保年龄范围内；**未成年人（< 18岁）特殊处理见下文** |
| 证件类型 | ✅ | 下拉：居民身份证/护照/港澳台通行证 |
| 证件号码 | ✅ | 身份证号自动校验18位格式+末位校验码；身份证号需与出生日期一致 |
| 证件有效期 | ✅ | 日期选择器，不得早于今日 |
| 手机号 | ✅ | 11位手机号格式校验 |
| 常住地址 | ✅ | 省市区+详细地址 |
| 职业类别 | ✅ | 三级联动选择（职业大类→小类→具体职业），影响健康告知分类和核保结论 |

**职业类别说明**：从 `ins_life_occupation` 字典表读取三级职业数据。高风险职业（如：矿工、消防员等 `risk_level >= 3`）选择后页面弹出提示"您选择的职业属于高风险职业，部分产品可能无法承保或需要加费"。职业信息存入草稿 `insured_info.occupation`（JSON 字段）。

**OCR辅助填写**：被保人填写区提供「扫描身份证」入口，点击后调起微信扫码/相册能力，上传身份证图片至 `POST /app-api/ins/life/order/ocr/id-card`，后端调用OCR服务识别姓名、性别、出生日期、身份证号、地址，识别结果自动填入表单对应字段（用户仍可手动修改），识别失败则提示"识别失败，请手动填写"。

OCR识别结果映射规则：
- 识别出的性别（男/女）→ 自动选中对应性别单选按钮
- 识别出的出生日期（格式：19900101）→ 转换为 `1990-01-01` 填入日期选择器
- 识别出的住址 → 尝试解析省市区部分并填入省市区选择器，其余填入详细地址

**未成年被保人特殊处理**：当被保人出生日期计算出年龄 < 18岁时：

1. 页面自动展开"监护人信息"区域（不可折叠）。
2. 监护人信息字段：姓名（必填）、与被保人关系（必填，下拉：父亲/母亲/其他监护人）、身份证号（必填）、手机号（必填）。
3. 页面底部强制勾选项：**"本人确认为被保人法定监护人，同意以未成年人为被保人投保"**，**必须勾选**才能进入下一步。
4. 监护人信息存入 `ins_life_order_draft.guardian_info`（JSON 字段）。

**特殊年龄限制补充**：
- 0-28天（新生儿）：大多数产品不支持，如产品 `min_insure_days` 有设置则按设置校验，否则提示"被保人年龄过小，不符合本产品投保条件"。
- 部分产品对成年后不支持续保（如少儿险），需在此步骤提前告知用户。

#### 2.2.4 受益人配置

受益人区域默认折叠，展示"法定受益人"（文字说明：按法定继承顺序）。用户可切换为"指定受益人"，展开后支持：

- 点击「添加受益人」按钮，弹出受益人信息表单（姓名、与被保人关系、证件类型、证件号、受益顺序 1/2/3、受益比例输入框）。
- **最多添加3名**受益人（超出3名时按钮置灰，提示"最多添加3名受益人"）。
- **受益比例总和必须等于100%**（每次添加/修改后实时计算剩余可分配比例，最后一名受益人比例自动填充剩余比例，但用户可手动修改；提交时后端校验总比例是否精确等于100，否则返回错误"受益人比例之和必须为100%"）。
- 删除某受益人时弹出确认框，确认后移除该条记录，剩余受益人比例不自动重新分配（由用户手动调整）。

受益人数据存入 `ins_life_order_draft.beneficiary_list`（JSON 字段）。

**受益人与被保人关系**：下拉选项：配偶/父母/子女/兄弟姐妹/其他。

**受益顺序说明**：受益顺序1为第一顺序受益人，在第一顺序受益人放弃或无法受益时，依次由第二、第三顺序受益人受益。同一顺序多名受益人按各自比例受益。

#### 2.2.5 保存与提交

用户点击「下一步」时，前端校验所有必填字段，通过后调用 `PUT /app-api/ins/life/order/draft/{draft_id}/policy-holders`。

后端校验：

1. 被保人年龄是否在产品投保年龄范围内（根据出生日期计算）。
2. 身份证号格式与出生日期是否一致（前6位为地区码，7-14位为出生日期，倒数第2位奇偶性为性别）。
3. 若指定受益人，受益比例总和是否等于100（整数精度，允许误差 ±0.01%）。
4. 未成年被保人时，监护人信息是否完整。

校验通过则更新草稿，将步骤推进到"缴费确认"，返回成功。校验失败返回对应字段的错误信息，前端定位到对应字段并显示红色提示文字。

**数据入库**：此步骤数据**暂存草稿表**，不写入正式保单表，等支付成功后统一写入。

### 2.3 缴费方式与保费确认（C端）

> **对应排期表功能点**：缴费方式与保费确认(C端) | 前端 1.5天 + 后端 1天
>
> **技术难点**：代扣授权签约对接银行/支付宝；与合规双录模块联动
>
> **参考 PDF**：`110`（缴费信息标星项必填逻辑）

#### 2.3.1 页面展示内容

进入第3步页面，调用 `GET /app-api/ins/life/order/draft/{draft_id}/payment-confirm`，后端根据产品配置和被保人年龄计算保费，返回：

| 展示项 | 说明 |
|---|---|
| 缴费期限 | 从产品 `payment_period_list` 中取，用户已在被保人信息步骤选择（若未选则在此步骤补选，单选 Tab） |
| 保额 | 用户在试算时选定的保额，只读展示 |
| 年缴保费 | 后端实时计算，按被保人年龄+性别+保额+缴费期从费率表精确匹配 |
| 月缴保费 | 年缴保费 / 12（部分产品支持月缴，从 `ins_life_product.support_monthly` 字段判断；不支持则不展示） |
| 首次缴费日 | 默认为今日，用户可选择未来7天内某日（超过7天不允许） |
| 保障起始日 | 等于首次缴费日（部分产品为次日 0:00 生效，由 `ins_life_product.effective_delay_days` 字段配置） |
| 保单到期日 | 根据保障期限计算 |
| 总缴保费 | 年缴保费 × 缴费年数，保至终身的产品显示"根据缴费期限计算" |

**缴费期限选项展示**：以 Tab 形式横向排列，如"10年缴""20年缴""缴至60岁"，点击后实时刷新保费金额（重新调用后端保费计算）。

**缴费方式选择**（下拉或卡片选择）：

- 微信支付（默认选中）
- 支付宝支付
- 银行卡（若用户已绑定银行卡，则显示卡末四位；未绑定则显示"绑定银行卡"入口）

**期缴代扣授权**（仅期缴产品显示，趸缴不显示）：

页面底部展示"开通自动续费"开关（默认关闭）。用户开启后，弹出代扣授权说明弹窗："授权后每期保费到期前3天，系统将自动从您选择的支付方式中扣款，如余额不足则推送提醒，您可手动补缴。"用户点击「同意授权」后，开关状态设为开启，记录 `auto_deduct_authorized = true` 至草稿。代扣签约在支付成功后调用支付宝/银行API完成实际签约，本步骤仅记录用户意向。

**代扣授权说明补充**：
- 支付宝代扣：通过支付宝签约授权 API（`alipay.user.agreement.page.sign`），签约后平台获得周期性扣款授权。
- 银行卡代扣：通过银行代扣协议接口签约，需用户提供银行卡号和预留手机号进行二次短信验证。
- 若用户未开通代扣，保单续期时系统会在缴费到期前30/15/7/1天推送提醒，由用户手动缴费。

**合规双录判断**（前端感知）：

调用接口时，后端返回字段 `dual_record_required`（取自产品配置 `ins_life_product.dual_record_required`）。若 `dual_record_required = true`，前端在页面底部展示提示框："根据监管要求，本产品投保需完成双录（音视频留存），请在支付前完成"。

**双录触发条件**（监管规定，后端判断逻辑）：
- 保险期间超过1年的人身险产品（含重疾险、寿险、年金险等）。
- 保费金额 ≥ 指定阈值（由 `ins_life_product.dual_record_premium_threshold` 配置，默认 1000 元）。
- 满足以上条件之一则 `dual_record_required = true`。

**必读协议勾选**：

页面底部勾选框：**"我已阅读并同意《投保须知》《电子保单条款》《健康告知书》《个人信息授权书》"**。**必须勾选**才能激活「去支付」按钮（否则置灰）。每个文件名为可点击链接，弹出对应PDF预览弹窗。

**协议文件来源**：从 `ins_life_product.agreement_files`（JSON 数组字段）读取，字段结构：
```json
[
  {"name": "投保须知", "url": "https://oss.xxx.com/agreements/xxx.pdf"},
  {"name": "电子保单条款", "url": "https://oss.xxx.com/agreements/xxx.pdf"},
  {"name": "健康告知书", "url": "https://oss.xxx.com/agreements/xxx.pdf"},
  {"name": "个人信息授权书", "url": "https://oss.xxx.com/agreements/xxx.pdf"}
]
```

#### 2.3.2 保费计算逻辑（后端）

接口 `GET /app-api/ins/life/order/draft/{draft_id}/payment-confirm`：

1. 从草稿读取 `product_id`、被保人出生日期/性别、选定保额。
2. 根据出生日期计算投保年龄（按"周岁"计算，即生日当天算整岁）。
3. 从 `ins_life_product_rate` 表查询费率（联合索引：`product_id + age + gender + payment_period + coverage_amount`），精确匹配返回年缴保费。
4. 若无精确匹配，按差值最近的相邻年龄插值（线性插值）。
5. 若产品支持月缴（`support_monthly = 1`），月缴保费 = 年缴 / 12，保留2位小数，向上取整到分。
6. 返回保费明细 VO。

**保费计算边界情况**：
- 若被保人今日正好是生日，以今日生效后的新年龄计算保费。
- 若费率表不存在该年龄段数据，返回错误"该年龄段暂无报价，请联系专属顾问"，前端展示对应提示。
- 保费金额超过单次支付限额（微信/支付宝均有限额，默认 ¥50,000/次），前端展示"本次保费超出单次支付限额，请选择银行卡支付"，并强制切换缴费方式。

#### 2.3.3 提交缴费确认

用户点击「去支付」时，前端校验：

- 缴费期限已选择
- 缴费方式已选择
- 协议勾选框已勾选

通过后调用 `PUT /app-api/ins/life/order/draft/{draft_id}/payment-info`，保存缴费期限、缴费方式、代扣授权意向、首次缴费日至草稿。

若 `dual_record_required = true` 且草稿中 `dual_record_done = false`，则后端返回错误码 `DUAL_RECORD_REQUIRED`，前端捕获此错误码后跳转至合规双录流程（调用 `intermediary-module-ins-compliance` 的双录发起接口），双录完成后回调更新草稿 `dual_record_done = true`，然后用户重新点击「去支付」进入第4步。

### 2.4 支付与核保等待（C端）

> **对应排期表功能点**：支付与核保等待(C端) | 前端 1.5天 + 后端 1天
>
> **技术难点**：核保状态轮询或WebSocket推送；附加条件用户交互设计；超时退款
>
> **参考 PDF**：`111`（保单状态说明：已承保/待回执/中止/终止/退保等状态）；`116`（回执管理，核保后生成保单后续的回执操作）

#### 2.4.1 支付发起

用户进入第4步后，前端调用 `POST /app-api/ins/life/order/submit`，入参 `{draft_id}`。

**后端处理步骤**：

1. 再次校验草稿完整性（健康告知已完成、投保人/被保人信息完整、缴费信息完整、双录条件满足）。
2. **创建订单**：向 `ins_order` 表写入一条主订单记录（`order_no` 使用雪花算法生成，`order_type = LIFE`，`status = PAYING`），同时向 `ins_life_policy_draft` 写入保单草稿（含投保人/被保人/受益人/健康告知全量信息）。
3. **发起支付**：调用 `intermediary-module-pay` 的支付 Feign 接口 `PayOrderApi.createOrder()`，入参包含 `order_no`、金额（首期保费 = 年缴保费，若首次缴费日非今日则今日不扣款仅创建订单）、支付方式、回调 URL（`/app-api/ins/life/order/pay-callback`）。支付模块返回 `pay_order_id` 和唤起支付所需参数（微信 `prepay_id` 或支付宝 `trade_no`）。
4. 返回给前端：`{order_no, pay_params}`，前端唤起对应支付 SDK。

**支付超时处理**：若用户唤起支付页面后超过 30 分钟未完成支付（微信/支付宝订单超时），前端检测到支付取消或超时后，展示"支付已超时，请重新发起"提示，并提供「重新支付」按钮（重新调用 `submit` 接口或直接重用已有订单号重发起支付，由 `pay_order` 状态判断）。

#### 2.4.2 支付回调处理

支付成功回调接口 `POST /app-api/ins/life/order/pay-callback`（由支付模块内部调用，非直接对外）：

1. 验证签名（复用支付模块签名验证）。
2. 根据 `order_no` 更新 `ins_order.status = PAID`。
3. 向 RocketMQ 发送 `ins_life_order_paid` 消息，消息体 `{order_no}`。

**支付成功消费者**（`InsLifeOrderPaidConsumer`）：

1. 从草稿表读取全量投保信息。
2. **正式写入保单数据**（数据库操作，需 Seata 分布式事务）：
   - `ins_life_policy`（保单主表）：写入 `policy_no`（由保司API返回或内部生成预保单号）、`product_id`、`insurer_id`、`holder_info`（JSON）、`status = UNDERWRITING`（核保中）。
   - `ins_life_policy_insured`（被保人/受益人表）：写入被保人信息、受益人列表。
   - `ins_order.status = UNDERWRITING`。
3. **调用保司核保API**（异步，向 RocketMQ 发送 `ins_life_underwriting_request` 消息，由保司适配器消费者处理）。
4. **处理代扣签约**：若 `auto_deduct_authorized = true`，调用支付宝/银行代扣签约 API，签约成功后更新 `ins_life_policy.auto_deduct_status = ACTIVE`，失败则记录日志并推送用户"代扣签约失败，请手动续费"通知。
5. **推送"支付成功，核保中"通知**给用户（极光推送 + 微信小程序订阅消息）。
6. **写入回执待处理记录**（参考 PDF `116`）：创建 `ins_life_policy_receipt` 回执记录，`status = PENDING`，等待核保通过后由业务员或后台完成回执确认。

#### 2.4.3 核保等待页面

支付成功后，前端展示核保等待页（不跳转走，停留在此页面）：

- 顶部大图标：转圈动画（Loading 状态）
- 文字：「核保审核中，预计 30 分钟内完成」
- 提示：「核保结果将通过消息通知您，您可先离开此页面」
- 「查看我的保单」按钮：点击跳转至「我的寿险保单」列表页，保单状态显示"核保中"

**核保等待页面详细布局**：
- 顶部：产品名称、保额、年缴保费（只读展示）
- 中部：Loading 动画 + 进度说明（已完成：支付成功 → 核保中：等待保司审核 → 待完成：保单生效）
- 底部：两个按钮「查看保单」「返回首页」

前端通过 **WebSocket** 订阅核保结果推送（连接至网关的 WebSocket 端点，频道：`/topic/underwriting/{order_no}`）。后端核保结果回来后，通过 WebSocket 推送结果给前端，前端根据结果更新页面状态（无需用户手动刷新）。若 WebSocket 不可用（降级方案），前端每隔 10 秒轮询一次 `GET /app-api/ins/life/order/{order_no}/underwriting-status`，最多轮询 30 次（5 分钟），超时后停止轮询并提示"核保审核中，结果将通过消息通知您"。

#### 2.4.4 核保结果处理

核保结果回调或保司主动推送后，后端处理（`InsLifeUnderwritingResultConsumer`）：

**情形一：标准承保（`result = STANDARD`）**

1. 更新 `ins_life_policy.status = ACTIVE`（保单生效）。
2. 更新 `ins_order.status = COMPLETED`。
3. 生成电子保单（调用保司 API 下载 PDF，存至 OSS，更新 `ins_life_policy.e_policy_url`）。
4. **创建回执记录**（参考 PDF `116`）：更新 `ins_life_policy_receipt.status = WAITING`（待回执），业务员端将在管理后台看到待回执提醒；若回执方式为线上电子签，则直接通过H5签署链接推送给用户。
5. **创建回访记录**（参考 PDF `118`）：创建 `ins_life_policy_visit` 回访记录，`status = PENDING`，归因业务员在管理后台/移动端可见待回访提醒，需在规定时间内完成回访。
6. 推送通知："恭喜！您的[产品名]保单已承保生效，保单号[xxx]，点击查看"（极光推送 + 微信订阅消息）。
7. 通过 WebSocket 推送 `{result: "STANDARD", policy_no: "xxx"}` 给等待页面。

**情形二：附加条件承保（`result = CONDITIONAL`）**

附加条件类型包括：加费（`type = EXTRA_PREMIUM`，增加额外保费）或除外（`type = EXCLUSION`，排除某些保障责任）。

1. 后端将条件信息写入 `ins_life_policy.underwriting_conditions`（JSON）。
2. 更新 `ins_life_policy.status = WAITING_CONDITION_CONFIRM`（等待用户确认）。
3. 通过 WebSocket 推送 `{result: "CONDITIONAL", conditions: [{type, description, extra_premium}]}` 给前端。
4. 前端展示条件说明弹窗，包含：条件类型与说明文字、若是加费则显示"需补缴 ¥xxx"、两个按钮「接受条件并继续」「拒绝承保并退款」。
5. 用户点击「接受条件并继续」：前端调用 `POST /app-api/ins/life/order/{order_no}/accept-condition`，若有加费则发起补缴支付流程，否则直接确认。后端更新 `ins_life_policy.status = ACTIVE`，生成电子保单，推送生效通知。
6. 用户点击「拒绝承保并退款」：前端调用 `POST /app-api/ins/life/order/{order_no}/reject-condition`，后端发起退款（调用支付模块 `PayOrderApi.refund()`，全额退款），更新 `ins_life_policy.status = CANCELLED`，`ins_order.status = REFUNDED`，推送退款通知"您的退款将在1-7个工作日退回原支付账户"。

**附加条件弹窗超时处理**：若用户未在 48 小时内对条件弹窗做出选择（可能关闭了APP），系统在「我的保单」页面持续显示"待确认"状态和条件说明，用户下次进入APP时弹窗重新展示。超过 7 天未确认则视为拒绝，自动发起退款。

**情形三：拒保（`result = REJECT`）**

1. 更新 `ins_life_policy.status = REJECTED`，`ins_order.status = REFUNDED`。
2. 自动发起全额退款（调用支付模块退款接口）。
3. 通过 WebSocket 推送拒保原因（脱敏处理，如"根据您的健康情况，本次投保暂无法承保"，不暴露具体医学判断）。
4. 前端展示拒保结果页，显示拒保原因（简化描述）、退款告知、「联系专属顾问」按钮。

**情形四：超时 24 小时未返回核保结果**

XXL-Job 定时任务（每小时执行一次）扫描 `ins_life_policy.status = UNDERWRITING` 且 `create_time < NOW() - 24小时` 的记录：

1. 发送站内信给用户："您的核保申请审核中，我们将尽快为您处理"。
2. 触发告警通知客服工作群（企业微信 Webhook），消息内容："超时核保提醒：保单[policy_no]，用户[姓名]，产品[产品名]，已超24小时未返回核保结果，请人工跟进"。

- **立即缴费/立即补缴**：调用支付模块重新发起缴费（复用投保支付流程的支付模块，`order_type = LIFE_RENEWAL`）。

#### 3.1.4 逾期缴费提醒弹窗

当用户进入保单列表或详情页时，若存在逾期缴费保单，页面弹出提醒弹窗：

- 标题：「缴费提醒」
- 内容：「您的[产品名]保单第[N]期保费已逾期[X天]，请尽快补缴，否则保单将于[日期]失效」
- 按钮：「立即补缴」「稍后再说」

**逾期状态计算规则**（XXL-Job 每天凌晨 2:00 执行）：

```sql
-- 扫描到期未缴费的保单
UPDATE ins_life_policy 
SET status = 'LAPSED', lapsed_date = NOW()
WHERE status = 'OVERDUE' 
  AND DATEDIFF(NOW(), next_payment_date) > grace_period_days  -- grace_period_days 默认60天
```

宽限期内（逾期60天以内）状态为 `OVERDUE`（已逾期但保单仍在保障期），宽限期满后状态变为 `LAPSED`（失效）。

#### 3.2.3 续期缴费流程

用户点击「立即缴费」入口触发续期缴费：

1. 前端调用 `POST /app-api/ins/life/policy/{policy_id}/renew`，入参：`{payment_method, renewal_period}（可复用上次缴费方式）`。
2. 后端创建续期订单 `ins_order`（`order_type = LIFE_RENEWAL`），调用支付模块发起支付。
3. 支付成功后，回调更新 `ins_life_policy.next_payment_date`（加一年/一月），更新 `ins_life_policy.status = ACTIVE`（若之前是 OVERDUE 则恢复生效），并在 `ins_life_renewal_record` 表写入续期记录（参考 PDF `130` 续期跟踪字段）。
4. 续期成功推送通知："[产品名]第[N]期保费缴纳成功，感谢您的续费"。

**续期缴费记录表** `ins_life_renewal_record`（参考 PDF `130` 续期查询字段）：

#### 3.2.4 代扣签约到期重签

代扣签约到期（`auto_deduct_status = EXPIRED`，通常1年有效期）时，提醒用户重新签约：

- XXL-Job 每天扫描代扣签约即将到期（提前7天）的保单，推送消息："您的[产品名]自动续费授权将于[日期]到期，请续签以保障保费自动缴纳"。
- 用户进入保单详情，显示"代扣签约已过期，点击重新签约"入口按钮，点击后重新调起支付宝/银行签约流程。

**代扣状态枚举**（`ins_life_policy.auto_deduct_status`）：

| 状态值 | 说明 |
|---|---|
| `NONE` | 未开通代扣 |
| `SIGNING` | 签约中（支付成功后异步签约期间） |
| `ACTIVE` | 代扣生效中 |
| `EXPIRED` | 签约已到期 |
| `CANCELLED` | 用户主动取消 |
| `FAILED` | 签约失败 |

#### 3.2.5 年金险/两全险到期给付提醒

对于到期给付型产品（`ins_life_product.product_type = ANNUITY` 或 `ENDOWMENT`），在满期日/年金给付日前推送提醒：

- 提前30天：推送"您的[产品名]将于[日期]到期给付¥[N]元，届时将自动打款至原支付账户"。
- 给付当天：推送"恭喜！您的[产品名]到期给付¥[N]元已发起，请注意查收"。

给付触发接口：`POST /app-api/ins/life/policy/{policy_id}/maturity-payment`，由 XXL-Job 在满期日调用，实际打款通过支付模块退款/转账接口实现。
