### 问题一：健康险核保逻辑分散（health-underwriting-service）

#### 原始问题

健康告知问卷和等待期逻辑被分散在非车险（阶段2）和寿险（阶段7/8）两个模块，后期极易出现：
- 两套问卷渲染前端组件，维护成本翻倍
- 核保规则配置分两个入口，运营困惑
- 健康数据存两张表，后期稽核困难

#### 已规划处理方案（来自 `架构风险工时评估` Sheet）

| 任务编号 | 任务 | 工时 | 建议阶段 | 优先级 |
|----------|------|------|----------|--------|
| #1 | health-underwriting-service 微服务骨架搭建 | 2天后端 | 阶段1末 | P0 |
| #2 | 健康告知问卷模板管理后台 | 3前+2后=5天 | 阶段2前完成 | P0 |
| #3 | 健康告知问卷 H5 填写页面（B端+C端复用） | 2前+1.5后=3.5天 | 阶段3同步 | P0 |
| #4 | 等待期规则配置后台 | 1前+1后=2天 | 阶段2 | P0 |
| #5 | 核保决策引擎（后端服务） | 2天后端 | 阶段2 | P0 |
| #6 | 非车险模块接入健康核保服务 | 0.5前+1.5后=2天 | 阶段5 | P1 |
| #7 | 寿险模块接入健康核保服务 | 1前+2后=3天 | 阶段7 | P1 |

**问题一小计：7.5天前端 + 12天后端 = 19.5天**

#### ✅ 评审结论：处理方案可行，以下补充三点注意事项

**补充1：与 AI 智能中台（阶段4）的问卷引擎存在重叠（见问题四）**  
`health-underwriting-service` 的问卷模板管理（任务#2）与阶段4 `AI智能中台` 的问卷模板管理在数据结构和功能上高度重合，**建议统一共用一套问卷引擎**，节省约 3 天工时（排期表备注已提及，但需落实到具体 Sheet 中）。

**补充2：等待期规则的作用范围需明确**  
等待期逻辑目前规划在健康核保服务内（任务#4），但等待期本质上是保单生效日计算逻辑，车险没有等待期，非车健康险有 30-180 天，寿险有 90-180 天。建议等待期规则服务由 `health-underwriting-service` 统一对外提供，所有险种的核保流程均调用同一接口查询当前产品的等待期天数。

**补充3：阶段8 C端健康告知已有独立实现描述，需整合**  
在 `阶段8-C端商城-寿险投保` Sheet 的 Row 5，描述了"健康告知问卷(C端)"功能，其中提到"动态问卷复用AI保障规划组件"，但并未明确调用 `health-underwriting-service`。务必在该 Sheet 的开发说明中明确：**C端健康告知页面复用任务#3的 H5 组件，后端调用 `health-underwriting-service` 的核保决策接口**，而不是独立实现。

### 问题四（新发现）：问卷引擎在 AI 中台与健康核保两处独立建设 🔴 高危

#### 问题描述

排期表中存在**两套问卷模板系统**，分别在不同模块各自建设：

| 位置 | Sheet | 内容描述 |
|------|-------|----------|
| AI 智能中台 | `阶段4-AI智能中台` Row 1 | "PC后台新建/编辑问卷模板，含题目列表管理、题目类型（单选/多选/数字/日期）、选项配置、跳题逻辑JSON配置" |
| 健康核保服务 | `架构风险工时评估` 任务#2 | "问卷模板CRUD：模板列表、题目列表管理、题目类型（是否题/单选/多选/文本输入）、跳题逻辑JSON可视化配置" |

这两套问卷系统的功能描述**高度重合**，但却被规划在两个不同模块里各自实现，会导致：
- 两套数据库表（`ai_questionnaire_template` + `health_questionnaire_template`）
- 两套前端问卷渲染组件
- 运营人员要在两个入口维护相似的模板
- 后期打通时需要重构合并

#### 处理方案

**将问卷引擎抽取为公共基础设施，供 AI 中台和健康核保两个业务域共同调用。**

##### 数据库设计

```sql
-- 统一问卷模板主表（通用，不区分业务场景）
CREATE TABLE ins_questionnaire_template (
    id          BIGINT PRIMARY KEY COMMENT '模板ID',
    template_code VARCHAR(64) NOT NULL UNIQUE COMMENT '模板编码',
    template_name VARCHAR(128) NOT NULL COMMENT '模板名称',
    scene_type  TINYINT NOT NULL COMMENT '使用场景：1-健康告知 2-保障规划 3-满意度调查',
    product_id  BIGINT COMMENT '关联产品ID（null=通用模板）',
    insurer_id  BIGINT COMMENT '关联保司ID（null=通用模板）',
    version     INT NOT NULL DEFAULT 1 COMMENT '版本号，修改自动递增',
    status      TINYINT NOT NULL DEFAULT 0 COMMENT '状态：0-草稿 1-启用 2-停用',
    skip_logic  JSON COMMENT '跳题逻辑配置JSON',
    tenant_id   BIGINT NOT NULL,
    creator     VARCHAR(64),
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updater     VARCHAR(64),
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    deleted     TINYINT NOT NULL DEFAULT 0,
    INDEX idx_scene_status (scene_type, status, deleted),
    INDEX idx_product_id (product_id)
) COMMENT '统一问卷模板表';

-- 题目表（复用，通过 template_id 区分归属）
CREATE TABLE ins_questionnaire_question (
    id              BIGINT PRIMARY KEY,
    template_id     BIGINT NOT NULL COMMENT '模板ID',
    question_code   VARCHAR(64) NOT NULL COMMENT '题目编码（用于跳题引用）',
    question_text   VARCHAR(512) NOT NULL COMMENT '题目文本',
    question_type   TINYINT NOT NULL COMMENT '类型：1-是否题 2-单选 3-多选 4-文本 5-数字 6-日期',
    is_required     TINYINT NOT NULL DEFAULT 1 COMMENT '是否必答',
    is_sensitive    TINYINT NOT NULL DEFAULT 0 COMMENT '是否敏感题（健康核保相关）',
    sort            INT NOT NULL DEFAULT 0 COMMENT '排序',
    options_json    JSON COMMENT '选项列表（选择题用）',
    skip_rule_json  JSON COMMENT '跳题规则（本题触发的跳转）',
    risk_weight     DECIMAL(5,2) COMMENT '风险权重（健康核保用）',
    INDEX idx_template_sort (template_id, sort, deleted)
) COMMENT '问卷题目表';
```

##### 服务拆分建议

```
ins-questionnaire-service（新增，公共问卷引擎）
├── QuestionnaireTemplateController    // 模板管理 CRUD
├── QuestionnaireAnswerController      // 作答记录管理
├── SkipLogicEngine                    // 跳题逻辑引擎（Aviator表达式）
└── QuestionnaireAnswerService         // 作答提交+暂存+断点续答

调用方：
├── health-underwriting-service（健康告知场景）
│   └── 调用 QuestionnaireTemplateController 筛选 scene_type=1 的模板
│   └── 核保决策在自身服务内处理
└── ai-planning-service（AI保障规划场景）
    └── 调用 QuestionnaireTemplateController 筛选 scene_type=2 的模板
    └── 保障缺口计算在自身服务内处理
```

### 问题五（新发现）：保费试算引擎三处重复实现 🔴 高危

#### 问题描述

保费试算（Groovy 脚本引擎 + 费率表查询）在三个模块中各自描述，但核心逻辑高度重合：

| 位置 | Sheet | 相关描述 |
|------|-------|----------|
| 非车险试算 | `阶段2-业务员App-非车险展业` Row 11 | "后端从产品表读取Groovy脚本(premium_formula)和费率表(rate_table JSON)；构建参数Map传入Groovy引擎执行（超时5秒）；缓存Key=quotation:{productId}:{age}...TTL 24小时" |
| 寿险试算 | `阶段8-业务员App-寿险展业` Row 3 | "**调用与非车险同一Groovy试算引擎**，险种字段扩展" |
| C端寿险试算 | `阶段8-C端商城-寿险投保` Row 14 | "费率表区间查询SQL；多方案同时展示；前端实时计算" |

值得注意的是，阶段8 B端寿险试算**已经明确说明"调用与非车险同一Groovy试算引擎"**，这是好的意识，但并没有在排期表中体现出独立的引擎服务，导致：
- 车险报价引擎（阶段1，`carins` 模块）和非车险试算引擎（阶段2，`noncar` 模块）是两套独立代码
- 寿险试算虽然声称复用，但没有明确的共享服务落地方案
- C端寿险试算描述"前端实时计算"，与后端 Groovy 引擎方案混用，存在计算结果不一致的风险

#### 处理方案

**抽取统一保费试算引擎 `premium-calculation-service`，所有险种共用。**

##### 核心设计思路

```java
// 统一试算请求VO（险种参数不同，通过 insuranceType + params Map 传入）
public class PremiumCalcRequest {
    private Long productId;
    private InsuranceTypeEnum insuranceType; // CAR / NON_CAR / LIFE
    private Integer age;
    private GenderEnum gender;
    private BigDecimal coverageAmount;
    private Integer payPeriod;       // 缴费期（寿险）
    private Integer coveragePeriod;  // 保障期
    private OccClassEnum occClass;   // 职业类别（非车险）
    private Boolean nonSmoker;       // 是否不吸烟（寿险重疾）
    private Map<String, Object> extParams; // 险种专属扩展参数
}

// 统一缓存Key（险种+产品+全部影响因子）
// quotation:{insuranceType}:{productId}:{age}:{gender}:{amount}:{payPeriod}:{covPeriod}:{...}
```

##### 服务架构

```
premium-calculation-service
├── PremiumCalcController
│   ├── POST /calc/single      // 单次试算
│   └── POST /calc/batch       // 批量试算（计划书对比用）
├── GroovyScriptEngine         // Groovy 脚本执行器（复用现有逻辑，统一管理）
├── RateTableService           // 费率表区间查询（替代各处独立实现）
├── PremiumCalcCache           // Redis 缓存管理（统一缓存策略）
└── PremiumCalcHistoryService  // 试算历史记录（各端共享）

调用方：
├── carins-service（车险报价 → 改造为调用此服务）
├── noncar-service（非车险试算）
├── life-service（寿险试算）
└── c-mall-service（C端前台试算）
```

各模块分别引入短信 SDK、微信 API、JPush/极光推送 SDK，存在以下问题：
- 相同的限频逻辑（手机号每日上限、时段控制）被重复实现

- 无全局消息发送记录，出现通知异常难以排查

#### 处理方案

├── 渠道路由（短信/App Push/微信服务号/站内信）

### 问题七（新发现）：文件上传 / OSS 管理各模块独立封装 🟡 中危

#### 问题描述

排期表中，文件上传至 OSS 的逻辑在以下场景各自描述：
- 双录视频上传（`阶段5`，私有 Bucket，命名规范独立）
- 产品封面图上传（`阶段1-基础建设`）
- 保单影像件上传（`阶段7`，`ins_life_policy_attachment` 表）
- 被保人档案影像（`阶段7` 客户影像管理）
- 计划书 PDF 上传（`阶段2` / `阶段8`，OSS 临时 URL）
- 财务对账文件上传（`阶段6`）

各处独立封装 OSS 上传，存在：
- Bucket 命名规范不一致，导致权限管理混乱
- 文件大小限制、格式校验规则各处不同
- OSS 临时 URL 有效期各处硬编码（有的2小时、有的24小时）

#### 处理方案

复用 yudao-cloud 框架已有的 `yudao-module-infra` 文件模块（`infra:file`），统一文件上传和访问，**约定以下目录规范**：

```
OSS Bucket 目录结构约定：
/{tenant_id}/
├── policy-attachment/    # 保单影像件（私有，长期有效）
├── insured-doc/          # 被保人证件文件（私有，长期有效）
├── double-record/        # 双录视频（私有，30天自动清理→OSS生命周期）
├── product-image/        # 产品封面图（公开读）
├── report/               # 各类报告PDF（私有，24h临时URL）
├── reconcile/            # 对账文件（私有，长期有效）
└── temp/                 # 临时文件（3天自动清理）
```

> 此项改造工时极少（约 0.5 天后端，制定规范文档），但需在阶段0数据库设计期同步约定，避免后期各模块各自设计目录。

### 问题八（新发现）：佣金计算引擎在非车险和寿险两个模块各自触发 🟡 中危

#### 问题描述

佣金触发入口在排期中出现在多处：

| 触发点 | 位置 |
|--------|------|
| 车险保单录入后触发佣金匹配 | `阶段1-PC管理后台-车险业务` Row 1 |
| 非车保单保存后触发佣金匹配 | `阶段2-PC管理后台-非车险业务` Row 1 |
| 寿险保单录入后触发佣金预计算 | `阶段7-PC管理后台-寿险体系` Row 1 |
| 非车险保单导入后佣金匹配 | `阶段2-PC管理后台-非车险业务` Row 3 |

`阶段2-PC管理后台-佣金系统` Sheet 中已规划了佣金计算引擎，但各险种的**触发入口**分散在各模块，且寿险的佣金规则（首年佣金/续年佣金/增值服务佣金）与非车险、车险规则存在差异，容易形成两套计算逻辑。

#### 处理方案

确认 `commission-service`（阶段2佣金系统）为统一的佣金计算服务，各险种保单完成录入/出单后，**统一通过 MQ 异步消息触发佣金计算**，而不是在各模块内部同步调用：

```
保单完成 → 发送 MQ 消息（topic: ins.policy.completed）
                    ↓
         commission-service 消费消息
                    ↓
  按 policy_type + product_id 匹配佣金规则
                    ↓
  写入 commission_record（统一表）
```

关键是在数据库设计期（阶段0/阶段1）就明确 `commission_record` 表结构支持所有险种，避免后期非车险和寿险各自建一张佣金表。

> 此项无需额外大工时，主要是在阶段1数据库设计时做好规范约定（1天架构评审）。

### 问题九（新发现）：续保提醒逻辑在三端各自实现 🟡 中危

#### 问题描述

续保提醒功能在三个位置各自描述了实现逻辑：

| 端 | 位置 | 描述 |
|----|------|------|
| C端 | `阶段3-C端商城` Row 76 | "到期前60/30/15/7/1天向用户发送续保提醒（App Push+微信服务号+短信）；若有归因业务员则同步提醒业务员跟进" |
| PC端 | `阶段0` Row 30 | "XXL-Job每日08:00执行：扫描ins_policy（到期日=今天+30天）→取关联ins_policy_member_bind（renew_remind=true）→调微信订阅消息API" |
| B端 | `阶段2-PC管理后台-客户CRM`（续期模块） | 客户续期跟踪功能，但提醒触发逻辑不明确 |

三处都在扫描保单到期日并发送通知，但扫描范围、通知渠道、提醒节点（1天/7天/30天/60天）各不统一，将导致用户收到重复通知或漏通知。

#### 处理方案

**统一由 `renewal-task-service`（或复用佣金系统的 XXL-Job 调度）集中管理续保提醒任务**，各端不再独立实现扫描逻辑：

```sql
-- 续保提醒配置表（可配置哪些节点发提醒）
CREATE TABLE ins_renewal_remind_config (
    id              BIGINT PRIMARY KEY,
    remind_days     INT NOT NULL COMMENT '提前N天提醒',
    insurance_type  TINYINT COMMENT '险种（null=全部）',
    channels        JSON COMMENT '通知渠道列表（APP_PUSH/SMS/WECHAT）',
    notify_agent    TINYINT DEFAULT 1 COMMENT '是否同时提醒业务员',
    status          TINYINT DEFAULT 1,
    tenant_id       BIGINT
) COMMENT '续保提醒规则配置';
```

统一的定时任务每天08:00扫描一次，匹配各配置节点的提单，调用 `message-center-service` 发送（与问题六联动处理）。

> 此项与问题六（消息中台）和问题八（MQ驱动）关联，需统一设计，工时约1.5天后端。
